## Architecture

Three layers, split by change frequency.

- **`core/`** — Types, constants, hook engine, state machine. Rarely changes.
- **`hooks/`** — Delegation guard, keyword detection, mode execution, verification, error recovery, memory. Changes most often.
- **`runtime/`** — Claude Code stdin/stdout bridge, settings registration, tmux parallel execution.

21 files, ~2,600 lines.

## Tool Interception

When Claude Code is about to call a tool, a PreToolUse event fires. `bridge.ts` reads JSON from stdin (`tool_name`, `file_path`, etc.) and dispatches it through the HookEngine.

`delegation.ts` intercepts at three layers:

1. **Bash guard** — Only allows read-only commands (`ls`, `git status`, `bun test`, etc.) directly. Modifying commands get a delegation reminder.
2. **Agent guard** — Blocks Agent calls without `subagent_type`. Prevents generic delegation.
3. **Write/Edit guard** — Paths on the allow list (`.harness/`, `CLAUDE.md`, etc.) pass through. Source files (`.ts`, `.py`, `.rs`) get stronger messages. `warn` mode injects a warning; `strict` mode blocks execution.

This is where it fundamentally differs from a system prompt. A prompt is a suggestion. A PreToolUse hook intervenes before execution.

## State Machine

In pipeline mode, when a phase finishes, Claude naturally tries to stop. The Stop event triggers the stop guard, which checks `isLastPhase()`. Not the last phase — Stop is blocked, and `transition()` advances to the next one.

When verification finds problems, it loops back to execute. `data.retries` increments; after three attempts, it exits. Only at the last phase does `endMode()` get called and state get cleared.

All state persists in a single `.harness/state.json` — mode, current phase, phase list, data, transition history. One reader, one writer, no lock needed.

## Parallel Execution

One call: `runTeam(tasks, { concurrency: 4, maxRetries: 2 })`. Internally, tasks are split into batches of four. A tmux session is created, workers attach as split panes. Each worker runs `claude --print` and writes results to `.harness/team/results/{taskId}`.

The leader polls every five seconds. A `[DONE]` marker means complete; error patterns (`error:`, `fatal:`, `FAILED`, etc.) classify failures. Failed tasks retry up to twice. When a batch finishes, the next one spawns. The final `TeamReport` contains success list, failure list, and total time.
