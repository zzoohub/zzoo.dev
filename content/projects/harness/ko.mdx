Claude Code는 혼자서도 코드를 읽고, 쓰고, 테스트한다. 근데 큰 작업을 시키면 한계가 온다 — 컨텍스트가 넘치고, 검증을 건너뛰고, 맥락을 잃는다. 오케스트레이터 하네스는 이걸 푸는 레이어다. Claude Code의 라이프사이클에 훅을 걸어서, 작업을 서브에이전트에 나누고, 진행을 추적하고, 완료를 검증한다.

oh-my-claudecode는 Claude 생태계에서 가장 널리 쓰이는 하네스 중 하나다. 19개 에이전트, 30개 스킬, 35개 MCP 도구 — 200개 파일, 수만 줄. 여러 프로젝트에 쓰다 보니 패턴이 보였다. 에이전트와 스킬은 프로젝트마다 새로 짠다. 근데 오케스트레이션 흐름 — 위임 강제, 진행 추적, 완료 검증 — 은 매번 같았다.

Claude Harness는 그 오케스트레이터만 떼어내서 만든 경량 하네스다. Claude Code의 기존 에이전트와 스킬을 그대로 활용하는 구조. 19개 파일, 2,371줄. 런타임 의존성 제로.

## 설치 & 사용

Claude Code 플러그인으로 동작한다. 설치하면 설정 없이 바로 쓸 수 있다:

```bash
/plugin marketplace add zzoohub/claude-plugins
/plugin install claude-harness@zzoohub-plugins
```

설치 끝. 이제 아무 프로젝트에서:

```
autopilot 인증 모듈 리팩토링해줘
```

이게 전부다. Claude Code에 이미 설치된 에이전트와 스킬을 자동으로 활용한다. 별도 설정 파일 필요 없음.

## 핵심 흐름

`autopilot`을 치면 4단계를 자동 순회한다:

1. **Understand** — 요구사항 분석, 제약 조건 파악
2. **Plan** — 단계별 작업 계획 수립
3. **Execute** — 서브에이전트에 위임하여 구현
4. **Verify** — 테스트, 빌드로 직접 검증

오케스트레이터는 직접 코드를 쓰지 못한다 — Write, Edit는 소스 파일에 대해 차단되고, 서브에이전트에 위임해야 한다. 검증 실패 시 Execute로 돌아간다. 마지막 phase까지 완료해야 멈출 수 있다.

`loop`을 치면 실행 → 검증을 될 때까지 반복한다 (최대 20회). 대규모 작업은 tmux로 4개씩 병렬 실행.

세션이 죽어도 상태가 사라지지 않는다. 모드, phase, 작업 메모리 — 전부 디스크에 영속되고 다음 세션에서 자동 복원된다.

## 커스텀 설정 (선택)

기본 동작을 바꾸고 싶다면 `.claude/harness.json`을 추가한다:

```json
{
  "agents": {
    "implementer": {
      "description": "코드 구현 전담",
      "prompt": "주어진 태스크를 구현하라.",
      "model": "sonnet"
    }
  },
  "systemPromptSuffix": "항상 한국어로 응답하라."
}
```

없으면 Claude Code의 기본 에이전트를 자동 활용한다.
