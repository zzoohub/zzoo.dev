Claude Code에 큰 작업을 시키면, 혼자서 다 한다. 탐색하고, 코드 쓰고, 테스트 돌리고, 에러 고치고 — 한 세션에서 전부 처리한다. 그리고 그게 문제다.

한 에이전트가 모든 걸 하면 컨텍스트가 넘친다. 탐색이 구현을 밀어내고, 구현이 검증을 건너뛰게 만들고, 에러 복구가 원래 목표를 잊게 한다. 결국 "다 했습니다"인데 테스트는 안 돌렸고 절반은 빠져 있다.

Harness는 오케스트레이터에게서 코드 작성 도구를 빼앗는다. Write, Edit, NotebookEdit — 소스 파일에 대해서는 전부 차단. 할 수 있는 건 딱 하나, 서브에이전트에게 위임하는 것뿐이다.

## 훅과 상태 머신

두 축이 있다.

**Hooks** — Claude Code의 라이프사이클 이벤트에 개입한다. 세션 시작, 프롬프트 입력, 도구 호출 전/후, 중단. 다섯 지점에서 컨텍스트를 주입하거나, 도구를 차단하거나, 검증을 강제한다.

**State machine** — 모드와 phase를 디스크에 기록한다. "pipeline의 구현 phase, retry 1회 소진" 같은 상태가 세션이 죽어도 사라지지 않는다. 다음 세션이 시작되면 자동으로 복원된다.

## "autopilot"이라고 치면

분석 → 설계 → 구현 → 검증을 자동으로 순회한다. 한 phase가 끝나면 Claude가 멈추려 해도 stop guard가 차단하고 다음 phase로 넘긴다. 검증에서 실패하면 구현으로 돌아가서 고치고 다시 검증한다. 마지막 phase까지 가야 비로소 멈출 수 있다.

"loop"이라고 치면 실행 → 검증을 될 때까지 반복한다. 대규모 작업은 tmux로 4개씩 병렬 실행.

oh-my-claudecode에서 핵심만 추출했다. 200개 파일 → 19개. 수만 줄 → 2,371줄. 런타임 의존성 제로.
