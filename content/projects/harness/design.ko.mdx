## 관찰

oh-my-claudecode를 여러 프로젝트에 쓰면서 알게 된 게 있다. 에이전트 프롬프트는 프로젝트마다 새로 쓰고, 스킬은 도메인에 맞게 바꾸고, MCP 도구는 필요에 따라 붙이거나 뗀다. 근데 오케스트레이션 레이어 — 위임을 강제하는 훅, 워크플로우를 추적하는 상태 머신, 완료를 검증하는 로직, 에러를 감지하고 복구하는 흐름 — 는 매번 똑같았다.

200개 파일 중에서 진짜 재사용 가능한 부분은 19개 파일의 오케스트레이션 코어였다. Harness는 그 관찰에서 시작했다.

그리고 그 코어에서 가장 중요한 역할은 하나다 — AI 에이전트가 혼자 다 하려는 습관을 막는 것. 에이전트는 기본적으로 위임하지 않는다. 도구가 있으면 쓴다. 할 수 있으면 한다. 그 자체는 효율적이지만, 작업이 커지면 단일 컨텍스트의 한계와 충돌한다.

## 누구를 위한 것인가

**재현, 30대 개발자.** Claude Code를 매일 쓴다. 작은 버그 수정은 완벽하게 처리되는데, 대규모 리팩토링을 시키면 중간에 맥락을 잃는다. 80%는 Claude가 해내고, 나머지 20% — 빠진 부분 찾기, 테스트 안 돌린 거 확인, 맥락 다시 주입 — 을 자기가 한다. "관리하는 시간이 코딩 시간보다 길어지는 지점이 있다."

**은지, 팀 리드.** 서브에이전트를 병렬로 돌리고 싶다. 근데 결과를 모으고, 검증하고, 실패한 걸 다시 돌리는 과정이 전부 수동이다. 에이전트가 "완료했습니다"라고 보고하면 진짜 됐는지 하나하나 확인해야 한다. "에이전트를 관리하는 에이전트가 필요하다."

## 제품 결정

**오케스트레이터만 남기고 나머지는 꽂아 쓰게 했다.** 에이전트 프롬프트, 스킬, MCP 도구, 모델 라우팅 — 이것들은 프로젝트마다 다르다. 식사 기록 앱의 에이전트와 인프라 마이그레이션 에이전트가 같을 수 없다. OMC는 이 모든 걸 빌트인으로 품고 있었지만, Harness는 오케스트레이션 흐름만 제공하고 나머지는 사용자가 `.claude/harness.json`에서 프로젝트에 맞게 정의한다. 프레임워크가 책임지는 건 흐름뿐이다 — 위임하고, 추적하고, 검증한다.

**위임은 가이드라인이 아니라 차단이다.** "서브에이전트에 위임하세요"라는 시스템 프롬프트는 무시된다. 에이전트는 할 수 있으면 한다 — 프롬프트는 제안이지 제약이 아니니까. 그래서 도구 자체를 빼앗았다. Write, Edit, NotebookEdit — 소스 파일에 대해서는 전부 차단. 설정 파일(`.harness/`, `CLAUDE.md`)만 허용하고, 코드는 서브에이전트만 건드릴 수 있다.

반대 선택은 경고만 주는 'warn' 모드다. "오늘부터 야식 줄이세요" 같은 거다 — 듣긴 하는데 행동은 안 바뀐다. 기본을 'strict'으로 잡았다.

**자연어로 모드를 켠다.** "autopilot 인증 모듈 리팩토링해줘" — 이 한 줄이면 pipeline 모드가 시작된다. 별도 명령어나 UI 없이, 프롬프트에서 키워드를 감지한다. 코드 블록은 건너뛰고, 대소문자 구분 없이 매칭한다. 한국어도 된다 — "계속", "파이프라인", "중단".

트레이드오프가 있다. 가끔 오탐이 난다. "loop 구조를 설명해줘"에서 키워드가 잡힐 수 있다. 수용했다 — 의도치 않은 모드 시작은 금방 알아차리지만, 매번 명시적 명령어를 치는 귀찮음은 누적된다.

**상태는 디스크에 산다.** `.harness/state.json`에 모드, phase, 데이터가 기록된다. 세션이 죽어도, 컨텍스트가 압축되어도, "pipeline의 구현 phase, retry 1회 소진"이 남아 있다. 다음 세션이 시작되면 자동으로 복원되고, 작업 메모리(`<remember>` 태그로 기록한 것)도 함께 돌아온다.

JSON 파일 하나로 충분하다. 상태 머신이니까. 읽고 쓰는 건 한 프로세스뿐이고, 동시성 문제가 없다.

**"서브에이전트는 거짓말한다."** 서브에이전트가 "완료했습니다"라고 보고하면, 오케스트레이터에게 메시지가 주입된다: "이 주장을 믿지 마라. 직접 테스트를 돌리고, 코드를 읽고, 빌드를 확인해라." git diff 요약도 같이 보여준다. 에이전트의 자기 보고를 신뢰하지 않는 것 — 이게 오케스트레이터가 존재하는 이유다.
