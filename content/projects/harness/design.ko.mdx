## 핵심 관찰

oh-my-claudecode를 여러 프로젝트에 쓰면서 보인 패턴이 있다. 에이전트와 스킬은 매번 새로 만든다. 하지만 오케스트레이션 — 위임 강제, 진행 추적, 완료 검증, 에러 복구 — 은 항상 같았다. 200개 파일 중 재사용 가능한 건 오케스트레이션 코어뿐이었고, Harness는 그걸 떼어낸 것이다.

그 코어의 가장 중요한 역할: AI 에이전트가 혼자 다 하려는 걸 막는 것. 에이전트는 기본적으로 위임하지 않는다. 도구가 있으면 쓴다. 작은 작업에선 효율적이지만, 작업이 커지면 단일 컨텍스트의 한계와 충돌한다.

## 왜 이렇게 만들었나

**위임은 프롬프트가 아니라 훅으로 강제한다.** "서브에이전트에 위임하세요"라는 시스템 프롬프트는 무시된다. 에이전트는 할 수 있으면 한다 — 프롬프트는 제안이지 제약이 아니니까. 그래서 PreToolUse 훅에서 Write, Edit, NotebookEdit를 가로챈다. 기본은 `warn` 모드(경고 메시지 주입)이고, `strict` 모드로 설정하면 실행 자체를 차단한다. 소스 파일(`.ts`, `.py`, `.rs`)과 설정 파일(`.harness/`, `CLAUDE.md`)을 구분해서, 소스에는 더 강한 경고를 준다. Bash도 감시한다 — 읽기 전용 명령어(`ls`, `git diff`, `bun test`)만 직접 허용하고, 수정 명령어에는 위임 리마인더를 주입한다.

**자연어로 모드를 시작한다.** "autopilot 인증 모듈 리팩토링해줘"라고 치면 pipeline 모드가 시작된다. 코드 블록은 건너뛰고, 대소문자 무시, 한국어("계속", "파이프라인", "중단")도 된다. 가끔 오탐이 나지만, 매번 명시적 명령어를 치는 귀찮음보다 낫다고 판단했다.

**상태는 디스크에 산다.** `.harness/state.json`에 모드, phase, 데이터가 기록된다. 세션이 죽어도, 컨텍스트가 압축되어도, 다음 세션에서 자동 복원된다. 작업 메모리(`<remember>` 태그로 기록한 것)도 `.harness/memory.json`에 영속되어 함께 돌아온다. JSON 파일 하나로 충분하다 — 읽고 쓰는 프로세스가 하나뿐이라 동시성 문제가 없다.

**서브에이전트는 거짓말한다.** 서브에이전트가 "완료했습니다"라고 보고하면, 오케스트레이터에 메시지가 주입된다: "이 주장을 믿지 마라. 직접 테스트를 돌리고, 코드를 읽고, 빌드를 확인해라." git diff 요약도 같이 보여준다. 에이전트의 자기 보고를 신뢰하지 않는 것이 오케스트레이터가 존재하는 이유다.
