## 시스템 아키텍처

Idea Fork는 하나의 핵심 관찰에 기반한 분리 경로 아키텍처를 사용합니다: 데이터 파이프라인과 사용자 대면 읽기 경로의 요구사항이 완전히 다릅니다. 파이프라인은 Python의 ML 생태계, 스케줄 실행, 대규모 LLM API 통합이 필요합니다. 읽기 경로는 저지연 페이지 렌더링이 필요하고 유휴 시 비용이 0이어야 합니다. 둘을 같은 서버에 밀어넣으면 돈을 낭비하고 지연만 추가됩니다.

**분리 구조:**

- **웹 앱 (Next.js on Vercel)** — `@neondatabase/serverless`를 통해 Neon PostgreSQL에 직접 쿼리. 모든 사용자 대면 읽기와 유일한 쓰기 액션(Server Actions를 통한 브리프 평가)을 처리. 프로덕션 읽기 경로에 API 서버 없음.
- **API + 파이프라인 (FastAPI on GCP Cloud Run)** — 일일 스케줄 파이프라인으로만 실행. 5개 이상의 소스에서 데이터를 가져오고, LLM으로 게시물을 태깅하고, 클러스터링한 후 브리프를 합성. 같은 Neon 데이터베이스에 쓰기. 사용자 요청 서빙을 위해 배포되지 않음.
- **데이터베이스 (Neon PostgreSQL)** — 단일 진실 소스. 웹 앱(읽기)과 파이프라인(쓰기) 모두 접근. Scale-to-zero 가격으로 유휴 시 비용이 거의 0.

데이터 액세스 레이어는 `DATA_SOURCE` 환경 변수로 전환 가능합니다. 프로덕션에서는 `neon`(Next.js가 DB에 직접 쿼리). 로컬 개발에서는 `api`(Next.js가 FastAPI 서버 호출). API 서버로 프로덕션 읽기를 되돌리려면 환경 변수만 바꾸면 됩니다 — 코드 변경 불필요.

```
사용자 → Next.js (Vercel) → Neon DB ← FastAPI 파이프라인 (Cloud Run)
           ↑ 직접 읽기                      ↑ 스케줄에 따라 쓰기
```

## 아키텍처 결정 기록

### ADR-1: Python (FastAPI) 백엔드 선택

파이프라인은 LLM 태깅(google-genai SDK), 임베딩 벡터화(Gemini Embedding API), 밀도 기반 클러스터링(scikit-learn HDBSCAN), 다중 소스 데이터 수집을 수행합니다. Python의 ML/데이터 생태계 덕분에 이 모든 게 수월했습니다. 대안은 TypeScript 엔드투엔드였지만, scikit-learn, numpy, google-genai SDK는 Python에서 상당히 더 성숙합니다.

병목은 LLM API 지연(배치 합성에 30초 이상)이지 서버 사이드 컴퓨팅이 아닙니다. Rust의 속도 이점은 여기서 보이지 않습니다. Python의 async 지원(FastAPI + asyncpg)이 I/O 바운드 파이프라인을 효율적으로 처리합니다.

트레이드오프: 더 높은 메모리 사용량과 Cloud Run에서 1-2초 콜드 스타트 vs Rust의 ~200ms. 파이프라인이 사용자 대면이 아니므로 수용 가능.

### ADR-2: 단일 서비스 + 스케줄 파이프라인

API 서버와 데이터 파이프라인은 같은 코드베이스, 도메인 모델, 데이터베이스를 공유합니다. 별도의 마이크로서비스가 아닙니다 — 파이프라인은 같은 Docker 이미지를 사용해 Cloud Run Job으로 cron 스케줄에 따라 실행됩니다.

대안은 두 서비스(API + 파이프라인)로 분리하는 것이었습니다. 기각 이유 — 혼자 개발하는 입장에서 두 개의 배포 파이프라인, 두 세트의 모니터링, 서비스 간 네트워킹을 관리하는 운영 오버헤드를 정당화할 수 없습니다. MVP 부하(~500 WAU, ~50 req/s 피크)에서 워크로드는 독립적 스케일링이 필요하지 않습니다.

파이프라인이 하루 수백만 게시물을 처리할 정도로 커지면, 별도 서비스로 추출하는 게 수월합니다 — 헥사고날 아키텍처가 이미 도메인 로직을 인프라에서 분리하고 있으니까요.

### ADR-3: 헥사고날 아키텍처 (포트 & 어댑터)

파이프라인은 Reddit API, Product Hunt API(GraphQL), App Store, Play Store, RSS 피드, Google Gemini(세 개의 다른 모델 엔드포인트), Google Trends와 통합됩니다. 이 중 어떤 것이든 변경되거나, 깨지거나, 교체가 필요할 수 있습니다.

헥사고날 아키텍처는 각 통합 지점에 대해 포트(Python의 Protocol ABC)를 정의합니다. 어댑터가 프로토콜을 구현합니다. 도메인 로직은 포트에 의존하고, 구체적인 어댑터에는 절대 의존하지 않습니다. Reddit에서 다른 포럼 소스로 교체하거나, 합성을 Gemini에서 Claude로 바꾸려면 비즈니스 로직을 건드리지 않고 새 어댑터만 작성하면 됩니다.

코드 구조는 도메인 모듈을 따릅니다: `post`, `brief`, `product`, `pipeline`. 각 모듈이 자체 도메인 로직, 포트 정의, 어댑터 구현을 소유합니다. 읽기 API는 독립적 — 읽기 경로에서 교차 모듈 쿼리 없음.

### ADR-4: Neon을 유일한 데이터 저장소로

PostgreSQL이 모든 것을 처리합니다: 관계형 데이터, 전문 검색(tsvector + pg_trgm), JSON 저장(브리프 비정규화 데이터를 위한 JSONB), 커서 기반 페이지네이션. Redis 캐시도, 검색 서비스도, 문서 저장소도 없음.

10K DAU 미만에서 PostgreSQL은 적절한 인덱스로 읽기 부하를 직접 처리합니다. Redis를 추가하면 하루에 한 번 쓰는 파이프라인에 캐시 무효화 복잡성이 생깁니다. Meilisearch를 추가하면 pg_trgm이 이미 제공하는 검색 품질을 위해 운영 표면이 2배가 됩니다.

Neon의 scale-to-zero 가격은 매출 전 MVP에 결정적 — 유휴 시 데이터베이스 비용이 $0. Neon의 브랜칭 모델은 별도 인스턴스 없이 개발을 위한 격리된 데이터베이스 환경을 제공합니다.

### ADR-7: LLM 클러스터링 대신 임베딩 클러스터링 (HDBSCAN)

원래 접근법은 모든 게시물을 LLM에 보내고 그룹화를 요청하는 것이었습니다. 세 가지 벽에 부딪혔습니다: O(n) 토큰 비용, 컨텍스트 윈도우 제한(호출당 ~50개 게시물), 볼륨 증가에 따른 정확도 저하.

대체: Gemini Embedding API로 게시물을 벡터화한 후 HDBSCAN(밀도 기반, scikit-learn)으로 클러스터링. HDBSCAN은 자동으로 클러스터 수를 결정하고, 노이즈 포인트(클러스터에 안 맞는 아웃라이어)를 처리하며, 수천 개 게시물로 스케일됩니다. LLM은 각 클러스터에 라벨을 붙이는 데만 사용 — 모든 게시물이 담긴 거대 프롬프트 대신 클러스터당 짧고 저렴한 호출.

K-means는 클러스터 수를 미리 지정해야 해서 기각. 로컬 sentence-transformers는 Gemini 임베딩이 품질이 더 높고 API 비용이 무시할 수준이라 기각.

### ADR-8: Next.js에서 Neon 직접 접근

프로덕션에서 Next.js는 `@neondatabase/serverless`를 사용해 Neon에 직접 쿼리합니다. API 서버가 프로덕션 읽기 경로에 없습니다. 네트워크 홉 하나를 제거하고(Next.js → Cloud Run → Neon이 Next.js → Neon으로) 크리티컬 패스에서 인프라 컴포넌트를 하나 없앱니다.

각 엔티티의 데이터 액세스 파일이 `DATA_SOURCE`를 확인하고 Neon 쿼리 모듈이나 `apiFetch` 클라이언트를 동적으로 임포트합니다. `server-only` 패키지가 데이터베이스 코드의 우발적 클라이언트 사이드 임포트를 방지합니다. 평가 쓰기는 Next.js Server Actions가 Neon에 직접 씁니다.

트레이드오프: SQL 쿼리가 TypeScript에 중복됨(Python 원본에서 포팅). 스키마 변경 시 두 코드베이스 모두 업데이트 필요. 쿼리 표면이 작고 안정적이므로 수용 가능.

## 데이터 파이프라인 상세

파이프라인은 GCP Cloud Scheduler에 의해 매일 트리거됩니다. Cloud Run(1 vCPU, 1Gi RAM, 0-3 인스턴스)에서 실행되며 실행당 모든 소스에서 ~4K 게시물을 처리합니다. Advisory lock(`pg_try_advisory_lock`)이 중복 실행을 방지합니다.

### 1단계: Fetch

`asyncio.gather()`를 통한 동시 데이터 수집, 세마포어 제어 리뷰 페칭(동시 3개):

- **Reddit** — ~44개 서브레딧, 로컬 전용 파이프라인 트리거(API 정책으로 프로덕션에서 미사용). 데이터는 프로덕션 DB로 전달.
- **RSS** — Hacker News(50+ 포인트 임계값), TechCrunch. feedparser로 파싱.
- **App Store** — iTunes Search API, 20개 키워드 카테고리로 리뷰 발견.
- **Play Store** — google-play-scraper(비공식), 20개 키워드 카테고리.
- **Product Hunt** — GraphQL API, 토큰 인증, 실행당 30개 제품.

게시물은 `INSERT ... ON CONFLICT (source, external_id)`로 upsert — 재페치된 게시물은 중복 생성 없이 점수와 댓글 수를 업데이트.

### 2단계: Tag

20개 게시물 배치를 Gemini Flash Lite로 전송. 각 게시물에 할당:

- `post_type` — 10개 카테고리 중 하나: 7개 실행 가능(need, complaint, feature_request, alternative_seeking, comparison, question, review) + 3개 비실행 가능(showcase, discussion, other)
- `sentiment` — positive, negative, neutral, 또는 mixed
- 토픽 태그 — 2-3개 자유 형식 카테고리 태그, 기존 태그 재사용을 위한 few-shot 프롬프팅

실패한 배치는 `tagging_status = 'failed'`로 표시되고 다음 실행에서 재시도. 실행 가능한 포스트 타입만 클러스터링 단계에 투입됩니다.

### 3단계: Score

매칭된 게시물 태그를 기반으로 제품의 `trending_score`와 `complaint_count` 업데이트. 제품은 `product_tag` → `post_tag` join 경로를 통해 불만 게시물과 연결 — 공유 태그가 연결 메커니즘.

### 4단계: Cluster

200개 태그된 게시물 배치를 Gemini Embedding API로 벡터화, HDBSCAN(`min_cluster_size=3`, `min_samples=2`)으로 클러스터링. 노이즈 포인트(어떤 클러스터에도 안 맞는 게시물)는 "기타"로 그룹화. 각 클러스터는 짧은 Gemini Flash Lite 호출로 라벨을 받습니다.

임베딩 + HDBSCAN 접근법이 원래의 LLM 기반 클러스터링을 대체하여, 비용을 ~10배 절감하고 컨텍스트 윈도우 병목을 제거했습니다.

### 5단계: Synthesize

각 클러스터에 대해 파이프라인이:

1. 관련 키워드의 Google Trends 데이터를 가져옴(병렬)
2. 데이터베이스에서 관련 제품을 가져옴(병렬)
3. 클러스터의 게시물, 트렌드 데이터, 경쟁 맥락을 Gemini Flash(temperature 0.9)에 전송
4. Gemini가 완전한 브리프를 작성: 제목, 요약, 문제 정의, 기회 분석, 솔루션 방향, 수요 시그널, 경쟁 환경, 소스 스냅샷

각 브리프는 소스 참조를 두 곳에 저장합니다: 정규화된 `brief_source` 테이블(참조 무결성)과 `source_snapshots` JSONB 컬럼(join 없는 빠른 읽기). 이 이중 비정규화는 의도적 — 브리프 상세 페이지의 읽기 경로는 테이블 간 join이 필요하지 않습니다.

## 데이터베이스 설계

### 스키마 개요

4개 도메인 영역에 걸친 10개 테이블:

**게시물 (수요 시그널):**
- `post` — 소스에 구애받지 않는 컬럼을 가진 원시 사용자 불만(`source`, `external_id`, non-Reddit의 경우 `subreddit` nullable). 전문 검색을 위한 `tsvector` 컬럼. `deleted_at`을 통한 소프트 삭제.
- `tag` — 자유 형식 카테고리 태그(SaaS, Mobile App, Developer Tools). `post_tag`를 통한 다대다.
- `post_tag` — 게시물과 태그를 연결하는 junction 테이블.

**클러스터 & 브리프 (합성):**
- `cluster` — 관련 게시물 그룹. LLM이 생성한 라벨과 요약.
- `cluster_post` — Junction 테이블. 게시물은 하나의 클러스터에 속함.
- `brief` — AI 생성 기회 브리프. `demand_signals`, `solution_directions`, `source_snapshots`를 위한 비정규화 JSONB 컬럼.
- `brief_source` — 브리프와 소스 게시물 간 정규화 FK 관계. 설계상 `source_snapshots`와 중복.

**제품 (공급 시그널):**
- `product` — Product Hunt, App Store, Play Store의 제품. `source` + `external_id` 패턴(게시물과 동일). 쿼리 시 `DISTINCT ON (lower(name))`으로 중복 제거.
- `product_tag` — 제품과 태그를 연결하는 junction 테이블.
- `product_post` — 공유 태그를 통해 제품을 불만 게시물에 연결.

**평가:**
- `rating` — 익명 브리프 평가. `UNIQUE(brief_id, session_id)`로 중복 방지. `brief.upvote_count` / `downvote_count`는 인라인 업데이트되는 비정규화 카운터.

### 핵심 설계 결정

**ENUM 대신 TEXT + CHECK** — `post_type`과 `sentiment`는 PostgreSQL ENUM 대신 CHECK 제약이 있는 TEXT 컬럼 사용. 새 타입 추가 시 `ALTER TABLE`(빠름)만 필요하고, `ALTER TYPE`(구버전 PostgreSQL에서 테이블 재작성이 필요할 수 있음)은 불필요.

**브리프의 이중 비정규화** — `brief_source`는 참조 무결성을 위해 존재. `brief.source_snapshots`는 브리프 상세 페이지가 게시물 테이블과 join 없이 단일 쿼리로 로드되게 하기 위해 존재. JSONB는 생성 시점의 스냅샷 — 소스 게시물이 나중에 삭제되어도 브리프는 인용을 유지.

**제네릭 소스 컬럼** — 게시물이 Reddit 전용 컬럼 이름 대신 `source` + `external_id` 사용. 새 플랫폼 추가 시 스키마 변경 없이 멀티소스 전략을 지원.

**키셋 페이지네이션** — 모든 목록 엔드포인트가 커서 기반 페이지네이션 사용(`{sort_field, id}`를 인코딩한 base64 JSON 커서). 어떤 페이지 깊이에서도 상수 시간 성능, 선형적으로 저하되는 OFFSET과 대비.

### 인덱스 전략

피드 성능이 크리티컬 패스(랜딩 페이지):

- **`idx_post_feed`** — `(external_created_at DESC, id DESC) WHERE deleted_at IS NULL` 부분 B-tree 인덱스. 기본 피드 정렬 커버.
- **`idx_post_search_vector`** — 전문 `@@` 쿼리를 위한 GIN 인덱스.
- **`idx_post_title_trgm`** — `LIKE '%keyword%'` 퍼지 검색을 위한 GIN/pg_trgm 인덱스.

모든 게시물 인덱스는 부분 인덱스(`WHERE deleted_at IS NULL`)로 소프트 삭제된 행 제외 — 아카이브된 게시물이 인덱스 공간을 소비하지 않음.

제품 검색은 `ILIKE` 쿼리를 위한 `product.name`의 pg_trgm GIN 인덱스 사용. 브리프 목록은 발행된 브리프의 키셋 페이지네이션을 위한 `published_at` 부분 B-tree 인덱스 사용.

## API 설계

일관된 컨벤션의 REST API:

- **Envelope**: 모든 응답에 `{ data, meta }`. `meta`에 페이지네이션 커서 포함.
- **에러**: RFC 9457 Problem Details(`application/problem+json`) — `type`, `title`, `status`, `detail`.
- **페이지네이션**: 커서 기반(키셋). MVP에서 전진 전용. `cursor` + `limit`(최대 100) 파라미터.
- **필터링**: 쿼리 스트링 파라미터(`?tag=saas&post_type=complaint`). 다중 값은 쉼표 구분.
- **캐싱**: 컬렉션 `Cache-Control: public, max-age=60`, 상세 엔드포인트 `max-age=300`. 파이프라인이 하루 한 번 쓰기 — 60초 지연 수용 가능.

주요 엔드포인트:

| 엔드포인트 | 목적 | 캐시 |
|-----------|------|------|
| `GET /v1/posts` | 피드 목록 — 필터/정렬/검색/페이지네이션 | 60초 |
| `GET /v1/briefs` | 브리프 목록, 발행된 것만 | 60초 |
| `GET /v1/briefs/{slug}` | 브리프 상세 — `source_snapshots` 포함(join 없음) | 300초 |
| `GET /v1/products` | 제품 목록 — 트렌딩/불만 수 정렬 | 60초 |
| `GET /v1/products/{slug}` | 제품 상세 + 연결된 불만 + 관련 브리프 | 300초 |
| `POST /v1/briefs/{id}/ratings` | 익명 평가(세션 쿠키, 분당 10개 제한) | no-store |

유일한 쓰기 엔드포인트(평가)는 `slowapi`로 세션당 분당 10개 요청 속도 제한. 중복 평가는 UNIQUE 제약에 걸려 409 반환.

## 인프라 & 배포

| 컴포넌트 | 플랫폼 | 주요 설정 |
|---------|--------|----------|
| 웹 앱 | Vercel | main 푸시 시 자동 배포, PR에 프리뷰 배포 |
| API + 파이프라인 | GCP Cloud Run | 1 vCPU, 1Gi RAM, 0-3 인스턴스, scale-to-zero |
| 파이프라인 트리거 | GCP Cloud Scheduler | 일일 cron |
| 데이터베이스 | Neon PostgreSQL | us-east-1, scale-to-zero, 개발 환경용 브랜칭 |
| IaC | Pulumi (TypeScript) | GCP + Neon + Vercel의 선언적 관리 |

**CI/CD:** GitHub Actions가 모든 배포를 처리. 웹은 Vercel의 GitHub 통합으로 배포(제로 설정). API는 Docker 빌드 → GCP Artifact Registry → Cloud Run 롤링 업데이트. 롤백: Vercel 즉시 롤백; Cloud Run 이전 리비전으로 트래픽 분할.

**환경 토폴로지:** 개발(localhost + Neon 브랜치), 프리뷰(Vercel 프리뷰 URL + Neon 브랜치), 프로덕션(Vercel + Cloud Run + Neon 메인 브랜치). 스테이징 환경 없음 — Neon의 브랜칭 모델이 충분한 격리를 제공.

## 성능 고려사항

**예상 부하 (MVP):** 500 WAU → ~100 DAU → ~10명 동시 사용자 피크 → ~50 req/s. 시스템은 이 수준에 맞게 사이징, 스케일을 위한 게 아님 — 100K 사용자를 위한 조기 최적화는 개발 시간 낭비.

**피드 지연 목표:** < 200ms TTFB(p95). 부분 인덱스가 있는 직접 Neon 쿼리로 달성. 이 부하에서 캐시 레이어 불필요.

**파이프라인 예산:** 일일 LLM 비용 < $5. 태깅에 Gemini Flash Lite(저렴, 빠름), 합성에 Gemini Flash(높은 품질). 일일 비용이 임계값을 초과하면 파이프라인을 중단하는 예산 서킷 브레이커.

**스케일링 트리거 (포스트 MVP):**
- p95 > 300ms → Neon 읽기 레플리카 추가
- DAU > 10K → Redis 캐싱 레이어 추가
- 파이프라인 > 4시간/일 → 더 많은 CPU를 가진 전용 Cloud Run Job으로 추출

## 관측성

- **에러 추적**: Next.js와 FastAPI 모두 Sentry. 웹 소스맵 포함.
- **분석**: PostHog으로 페이지 뷰, 브리프 읽기, 소스 링크 클릭(클라이언트 사이드) 및 파이프라인 메트릭(서버 사이드).
- **로깅**: Cloud Run 빌트인(Cloud Logging). FastAPI에서 구조화된 JSON.
- **알림**: Cloud Monitoring — API 에러율 > 5%와 파이프라인 실패에 크리티컬 알림; LLM 비용 초과와 지연 스파이크에 경고.

MVP에서 분산 트레이싱 없음. 단일 서비스, 추적할 서비스 간 호출 없음.
