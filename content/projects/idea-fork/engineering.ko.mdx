## 시스템 구조

매일 파이프라인이 Reddit, RSS, 앱스토어, Product Hunt에서 게시물을 수집하고, LLM으로 태깅하고, 불만을 테마별로 클러스터링해서 기회 브리프를 합성한다. Next.js 프론트엔드가 Neon PostgreSQL을 직접 읽는다 — 읽기 경로에 API 서버 없음. 파이프라인은 스케줄에 따라 쓰고, 사용자는 방문할 때 읽는다.

## 수천 개 게시물을 LLM 없이 클러스터링

처음에는 모든 게시물을 LLM에 보내 불만을 그룹화하려 했다. 세 벽에 막혔다: O(n) 토큰 비용, 호출당 ~50개의 컨텍스트 윈도우 한계, 볼륨 증가에 따른 정확도 저하.

대체안은 각 도구가 잘하는 일을 나눈다. Gemini Embedding API가 게시물을 벡터화한다. HDBSCAN(밀도 기반 클러스터링, scikit-learn)이 벡터를 그룹화한다 — 클러스터 수를 자동 결정하고, 노이즈를 처리하고, 수천 개까지 확장된다. LLM은 그룹화가 끝난 뒤 클러스터 라벨링에만 쓴다 — 모든 게시물이 담긴 거대 프롬프트 대신 클러스터당 짧은 호출. 클러스터링 비용이 ~10배 줄었다.

K-means는 클러스터 수를 미리 지정해야 해서 배제했다 — 오늘 배치에 불만 테마가 3개인지 30개인지 모르는 상황에서 불가능하다. HDBSCAN은 데이터에서 구조를 발견한다.

## 읽기 경로에서 서버 제거

파이프라인은 Python ML 생태계, 스케줄 실행, 대량 LLM 통합이 필요하다. 읽기 경로는 서브초 렌더링과 유휴 비용 제로가 필요하다. 서로 반대 방향으로 끌리니 분리한다: 파이프라인(FastAPI on Cloud Run)이 Neon에 스케줄 기록, Next.js on Vercel이 `@neondatabase/serverless`로 Neon을 직접 읽기.

SQL이 두 언어에 존재한다 — Python에서 쓰기, TypeScript에서 읽기. 쿼리 표면이 ~10개로 안정적이라 중복은 감당할 만하다. 대안 — 읽기 서빙용 API를 24시간 띄우기 — 은 하루 한 번 쓰는 파이프라인에 비용이 맞지 않는다.

`server-only`가 클라이언트 사이드 DB 임포트를 차단한다. PostgreSQL이 추가 인프라 없이 전부 처리한다: 관계형 데이터, 전문 검색(tsvector + pg_trgm), JSON 저장(JSONB), 커서 기반 페이지네이션.

## 원시 게시물에서 브리프까지 — 파이프라인

5단계, 실행당 ~4K 게시물, Cloud Scheduler로 매일 트리거. Advisory lock이 중복 실행을 방지한다.

Fetch가 5개 소스에서 동시 수집하고 `ON CONFLICT`로 upsert. Tag가 20개 배치로 Gemini Flash Lite에 보내 유형 분류(실행 가능 7 + 비실행 3), 감성, 토픽 태깅. Score가 공유 태그 기반으로 제품 트렌딩 지표를 갱신. Cluster가 태그된 게시물을 200개씩 벡터화하고 HDBSCAN으로 클러스터링한 뒤 LLM이 라벨링. Synthesize가 클러스터별 Google Trends 데이터와 관련 제품을 가져온 뒤 Gemini Flash가 풀 브리프를 생성한다 — 문제 정의, 분석, 방향, 경쟁 환경.

각 브리프의 소스 참조를 두 곳에 저장한다: 정규화된 `brief_source` 테이블(참조 무결성)과 `source_snapshots` JSONB 컬럼(join 없는 읽기). 읽기 경로는 빠른 비정규화 접근이, 쓰기 경로는 참조 안전성이 필요하다. 의도적 중복, 각 복사본이 자기 접근 패턴에 최적화된다.
