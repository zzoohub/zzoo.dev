## System Overview

A daily pipeline collects posts from Reddit, RSS, app stores, and Product Hunt, tags them via LLM, clusters complaints into themes, and synthesizes opportunity briefs. A Next.js frontend reads from Neon PostgreSQL directly — no API server in the read path. The pipeline writes on schedule; users read whenever they visit.

## Clustering Thousands of Posts Without an LLM

The first approach sent all posts to an LLM and asked it to group complaints into themes. Three walls: O(n) token cost, context window limit of ~50 posts per call, and accuracy degrading as volume grew.

The replacement splits the work by what each tool does best. Gemini Embedding API vectorizes posts. HDBSCAN (density-based clustering, scikit-learn) groups the vectors — it determines cluster count automatically, handles noise, and scales to thousands. The LLM only labels each cluster after grouping — one short call per cluster instead of one massive prompt. Clustering cost dropped ~10x.

K-means was ruled out because it requires specifying cluster count upfront — impossible when today's batch could contain 3 complaint themes or 30. HDBSCAN discovers the structure from the data.

## Eliminating the Read-Path Server

The pipeline needs Python's ML ecosystem, scheduled execution, and heavy LLM integration. The read path needs sub-second rendering and zero cost at idle. These pull in opposite directions, so the system splits: the pipeline (FastAPI on Cloud Run) writes to Neon on schedule, and Next.js on Vercel queries Neon directly via `@neondatabase/serverless`.

This means SQL lives in two languages — Python for writes, TypeScript for reads. The query surface is ~10 stable queries, so the duplication is manageable. The alternative — keeping an API server running 24/7 to serve reads — costs money for a pipeline that writes once daily.

`server-only` prevents accidental client-side database imports. PostgreSQL handles everything without extra infrastructure: relational data, full-text search (tsvector + pg_trgm), JSON storage (JSONB), and cursor-based pagination.

## From Raw Posts to Briefs — The Pipeline

Five stages, ~4K posts per run, triggered daily by Cloud Scheduler. An advisory lock prevents overlapping executions.

Fetch pulls concurrently from five source types, upserting via `ON CONFLICT`. Tag sends batches of 20 to Gemini Flash Lite for type classification (7 actionable + 3 non-actionable), sentiment, and topic tagging. Score updates trending metrics on products based on shared tags. Cluster vectorizes tagged posts in batches of 200, runs HDBSCAN, and labels each cluster via LLM. Synthesize fetches Google Trends data and related products per cluster, then Gemini Flash generates a full brief — problem statement, analysis, potential directions, competitive landscape.

Each brief stores source references in two places: a normalized `brief_source` table for referential integrity and a `source_snapshots` JSONB column for join-free reads. The read path needs fast denormalized access; the write path needs referential safety. Intentional duplication, each copy optimized for its access pattern.
