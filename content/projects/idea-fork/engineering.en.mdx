## System Architecture

Idea Fork uses a split-path architecture built around one key observation: the data pipeline and the user-facing read path have completely different requirements. The pipeline needs Python's ML ecosystem, scheduled execution, and heavy LLM API integration. The read path needs low-latency page rendering and costs nothing when idle. Forcing both through the same server wastes money and adds latency.

**The split:**

- **Web App (Next.js on Vercel)** — Queries Neon PostgreSQL directly via `@neondatabase/serverless`. No API server in the production read path.
- **API + Pipeline (FastAPI on GCP Cloud Run)** — Runs only as a daily scheduled pipeline. Fetches data from 5+ sources, tags posts via LLM, clusters them, and synthesizes briefs. Not deployed for serving user requests.
- **Database (Neon PostgreSQL)** — Single source of truth. Scale-to-zero pricing means near-zero cost when idle.

The data access layer is switchable via the `DATA_SOURCE` env var (`neon` for production, `api` for local dev). Switching back to the API server for production reads requires only changing an environment variable.

```
User → Next.js (Vercel) → Neon DB ← FastAPI Pipeline (Cloud Run)
         ↑ reads directly                  ↑ writes on schedule
```

## Key Architecture Decisions

### ADR-1: Python for the Pipeline

The pipeline does LLM tagging, embedding vectorization, density-based clustering (scikit-learn), and multi-source data ingestion. Python's ML/data ecosystem made this straightforward. TypeScript end-to-end was considered, but scikit-learn, numpy, and the google-genai SDK are significantly more mature in Python.

The bottleneck is LLM API latency (30s+ for batch synthesis), not compute. Rust's speed advantage would be invisible here. Trade-off: higher memory and 1-2s cold starts on Cloud Run, acceptable because the pipeline is not user-facing.

### ADR-7: HDBSCAN over LLM Clustering

The original approach sent all posts to the LLM and asked it to group them. This hit three walls: O(n) token cost, context window limits (around 50 posts per call), and degrading accuracy with volume.

The replacement: vectorize posts via Gemini Embedding API, then cluster via HDBSCAN (density-based, from scikit-learn). HDBSCAN automatically determines cluster count, handles noise points, and scales to thousands of posts. The LLM is used only for labeling each cluster — a short, cheap call per cluster instead of a massive prompt containing all posts. This cut clustering costs by roughly 10x.

K-means was rejected because it requires specifying cluster count upfront. Local sentence-transformers were rejected because Gemini embeddings are higher quality at negligible API cost.

### ADR-8: Direct Neon Access from Next.js

In production, Next.js queries Neon directly using `@neondatabase/serverless`, eliminating a network hop (Next.js to Cloud Run to Neon becomes Next.js to Neon). The `server-only` package prevents accidental client-side imports of database code. Rating writes use Next.js Server Actions that write to Neon directly.

Trade-off: SQL queries are duplicated in TypeScript (ported from the Python originals). Acceptable because the query surface is small and stable.

PostgreSQL handles everything with no additional infrastructure: relational data, full-text search (tsvector + pg_trgm), JSON storage (JSONB for denormalized brief data), and cursor-based pagination. At MVP scale, adding Redis or a dedicated search service would only increase ops complexity for a pipeline that writes once daily.

## Data Pipeline

The pipeline runs daily via GCP Cloud Scheduler, processing around 4K posts per run. An advisory lock prevents overlapping runs.

**Five stages:**

1. **Fetch** — Concurrent collection from Reddit, RSS (Hacker News, TechCrunch), App Store, Play Store, and Product Hunt. Posts are upserted via `ON CONFLICT` to avoid duplicates.
2. **Tag** — Batches of 20 posts sent to Gemini Flash Lite. Each post receives a type classification (7 actionable + 3 non-actionable categories), sentiment, and 2-3 topic tags.
3. **Score** — Updates trending scores and complaint counts on products based on shared tags with posts.
4. **Cluster** — Batches of 200 tagged posts vectorized via Gemini Embedding API, then clustered via HDBSCAN. Each cluster gets a label from a short LLM call.
5. **Synthesize** — For each cluster, fetches Google Trends data and related products, then sends everything to Gemini Flash to generate a complete opportunity brief (problem statement, opportunity analysis, solution directions, competitive landscape).

Each brief stores source references in two places: a normalized `brief_source` table (referential integrity) and a `source_snapshots` JSONB column (fast reads without joins). This dual storage is intentional — the read path never needs cross-table joins for a brief detail page.

## Infrastructure

- **Web App**: Vercel (auto-deploy on push to main, preview deploys on PRs)
- **Pipeline**: GCP Cloud Run (1 vCPU, 1Gi RAM, scale-to-zero)
- **Database**: Neon PostgreSQL (scale-to-zero, branching for dev environments)
- **IaC**: Pulumi (TypeScript) for declarative management of GCP + Neon + Vercel
- **CI/CD**: GitHub Actions — web via Vercel integration, API via Docker to Artifact Registry to Cloud Run

No staging environment. Neon's branching model provides sufficient isolation across development, preview, and production.
