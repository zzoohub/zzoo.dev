## 핵심 인사이트

"스타트업 아이디어 찾기" 도구들은 전부 중간에서 멈춥니다. Gummysearch는 Reddit 스레드를 보여주고, SparkToro는 오디언스를 보여주고, Exploding Topics는 트렌드 그래프를 보여줍니다. 전부 재료만 던져주고 요리는 알아서 하라는 거죠.

진짜 병목은 불만을 찾는 게 아닙니다 — 그 불만을 이미 존재하는 제품과 연결하고 빈틈을 발견하는 겁니다. 아이디어가 실제로 살아있는 곳은 합성 단계인데, 어떤 도구도 여기를 건드리지 않습니다.

Idea Fork의 가설: **수요를 공급에 대조하고, AI가 그 갭을 출처가 명시된 실행 가능한 브리프로 합성하게 하자.**

## 타겟 사용자

**사이드 프로젝트 탐색가** — 풀타임 직장에 주 10시간 정도 뭔가 만들 여유가 있는 개발자. 이전에 출시 경험이 있고, "시작하기에 충분한 수준"이 뭔지 압니다. 다만 Reddit에서 시그널 찾으려고 5시간씩 쓸 시간이 없습니다. 큐레이션된, 사전 소화된 기회를 통해 "이거 만들어볼까?"로 바로 건너뛰고 싶은 사람입니다.

**검증 우선 창업자** — 검증 전에 만들다가 데인 적 있는 사람. "사람들이 이걸 원할 것 같아" 류의 추론에 알레르기가 있습니다. 증거가 필요합니다: 실제 불만, 실제 수치, 실제 출처 링크. 아이디어에 3개월을 투자하기 전에, 5개 커뮤니티의 47명이 같은 문제에 독립적으로 좌절하고 있다는 걸 확인하고 싶은 사람입니다.

**내부 혁신가** — 중견 기업의 PM으로, 리더십에 새로운 기능이나 제품을 제안해야 하는 사람. 내부 피드백 채널은 에코 챔버입니다. 외부 수요 시그널이 필요합니다 — 고객 지원팀이 아니라 낯선 사람들에게 사용자가 뭘 불평하는지요.

## 핵심 루프

제품의 사용자 여정은 의도적인 좁히기 패턴을 따릅니다 — 넓은 탐색이 구체적이고 실행 가능한 인사이트로 수렴합니다:

1. **피드 탐색** — Reddit, 앱스토어, RSS에서 온 태그된 사용자 불만 스트림. 유형(니즈, 불만, 기능 요청)과 카테고리로 필터링. 원시 수요 시그널 — 하루 수백 개의 게시물이 검색이 아니라 스캔할 수 있게 정리됩니다.

2. **제품 탐색** — Product Hunt, App Store, Play Store의 트렌딩 제품, 각각 불만 수와 함께 표시. 뭐가 존재하고 어디서 실패하는지 보여줍니다. 방정식의 공급 측면.

3. **AI 브리프 읽기** — 수요와 공급이 만나는 곳. 시스템이 관련 불만을 클러스터링하고, 기존 제품과 교차 참조한 후, 브리프를 합성합니다: 문제 정의, 수요 시그널(게시물 수, 커뮤니티 수, 기간), 제안된 솔루션 방향. 모든 주장에 소스 게시물로 연결되는 위첨자가 있습니다.

4. **검증** — 아무 출처 링크나 탭하면 원본 게시물을 읽을 수 있습니다. 브리프는 하나의 논증이지만, 증거는 항상 한 클릭 거리에 있습니다.

이 진행은 의도적입니다: 피드가 직관을 키우고, 제품이 맥락을 더하고, 브리프가 결론을 전달합니다. 대부분의 사용자는 브리프를 통해 들어오지만(SEO 랜딩 페이지), 파워 유저는 아직 합성되지 않은 시그널을 찾기 위해 피드를 탐색하는 습관을 만들어갑니다.

## 프로덕트 디자인 결정

### 제로 마찰 익명 접근

가입 없음. 페이월 없음. 온보딩 플로우 없음. 사이트를 열면 피드가 보입니다.

이건 단순한 UX 선호가 아니라 전략적 판단이었습니다. 이 제품의 가치는 사용해봐야만 증명됩니다. 핵심 가치를 경험하기 전 로그인 벽은 전환을 죽입니다. 먼저 브리프 10개를 탐색하게 하세요. 일단 빠져들면, 프리미엄 기능(알림, 내보내기, 커스텀 파이프라인)이 계정을 정당화할 수 있습니다.

MVP 전체에서 유일한 "쓰기" 액션은 브리프 평가(좋아요/싫어요)이고, 그마저도 익명 세션 쿠키로 작동합니다. PII 수집 없음, GDPR 두통 없음, 만들고 유지할 인증 시스템 없음.

### 3부 구조: 피드 / 제품 / 브리프

왜 브리프만 보여주지 않을까요? 신뢰에는 투명성이 필요하기 때문입니다.

브리프는 AI 생성 요약입니다. 그것만 보여주면 블랙박스를 신뢰하라는 겁니다. 피드가 존재하는 이유는 사용자가 원재료를 볼 수 있게 하기 위해서입니다. 제품이 존재하는 이유는 공급 환경을 직접 평가할 수 있게 하기 위해서입니다. 브리프는 합성이지만, 나머지 두 탭이 영수증입니다.

이 구조는 또한 서로 다른 사용자 의도에 대응합니다: 어떤 사용자는 원시 시그널을 탐색하고 싶고(피드), 어떤 사용자는 경쟁 정보를 원하고(제품), 어떤 사용자는 사전 소화된 기회를 원합니다(브리프). 세 탭 IA는 사용자가 시스템의 파이프라인을 이해할 필요 없이 이런 의도에 깔끔하게 매핑됩니다.

### 증거 우선 AI

브리프의 모든 주장은 실제 소스 게시물에 연결됩니다. 있으면 좋은 기능이 아니라 핵심 신뢰 메커니즘입니다.

대안은 "AI가 47개 게시물을 분석해서 X라고 결론 내렸습니다, 우리를 믿으세요"였습니다. ChatGPT에서는 통합니다(사용자가 출처 없는 의견을 기대하니까요). 하지만 데이터 기반 발견이 가치 제안인 제품에서는 실패합니다. 브리프가 "5개 커뮤니티의 사용자들이 모바일 노트 앱 성능에 좌절하고 있다"고 말하면, 클릭해서 실제로 뭐라고 썼는지 읽을 수 있어야 합니다.

이 제약이 전체 데이터 아키텍처를 형성했습니다: 모든 브리프가 `source_snapshots`를 저장해서 인용이 join 없이 렌더링되고, 정규화된 `brief_source` 테이블이 그 아래에서 참조 무결성을 유지합니다.

### 모바일 퍼스트, 엄지 퍼스트

주요 액션이 엄지 영역에 위치합니다. 모바일 하단 탭 바는 세 탭(피드, 브리프, 제품) — 세 핵심 경험과 매칭됩니다. 데스크톱은 인라인 검색이 있는 수평 상단 바를 사용합니다.

프로그레시브 깊이는 모바일 카드가 제목, 소스 배지, 감정 지표만 보여준다는 뜻입니다. 탭하면 전체 게시물. 긴 텍스트는 2줄에서 line-clamp되고 확장/축소됩니다. 첫눈에 정보 과부하 없음.

## UX 원칙

모든 화면을 형성한 여섯 가지 규칙:

1. **콘텐츠가 인터페이스다** — 피드가 홈페이지. 스플래시 스크린 없음, 히어로 섹션 없음, "Idea Fork에 오신 것을 환영합니다" 모달 없음. 처음 보이는 것은 최신 불만 스트림.

2. **화면당 하나의 액션** — 각 페이지에 하나의 주요 액션. 브리프 상세의 주요 액션은 "출처 검증". 제품 상세의 주요 액션은 "불만 탐색". 보조 액션(평가, 공유)은 시각적으로 억제.

3. **프로그레시브 깊이** — 목록 페이지는 카드 수준 요약. 상세 페이지는 전체 콘텐츠. 소스 링크는 원본 게시물로. 각 레이어가 강요하지 않고 깊이를 더함.

4. **증거가 보인다** — AI 주장에 인라인 인용 위첨자. 수요 시그널은 정확한 수치를 보여줌(47개 게시물, 5개 커뮤니티, 평균 점수 142). 모호한 "많은 사용자가 보고합니다..." 없음.

5. **제로 마찰** — 로그인 없음, 온보딩 없음, 팝업 없음. 쿠키 동의 배너조차 없음(추적 쿠키를 설정하지 않으니까 — 분석은 자사, 프라이버시 존중 도구 사용).

6. **모바일 퍼스트, 엄지 퍼스트** — 모든 터치 타겟 최소 44x44pt. 필터 칩은 수평 스크롤. Intersection observer로 무한 스크롤. 폰 화면에서 탭할 페이지네이션 컨트롤 없음.

## 정보 구조

```
idea-fork.com
├── / (피드)               ← 랜딩 페이지. 불만 스트림.
├── /briefs               ← 브리프 목록. 기회 카드 그리드.
│   └── /briefs/[slug]    ← 브리프 상세. 전체 합성 + 소스.
├── /products             ← 제품 목록. 트렌딩 제품 + 불만.
│   └── /products/[slug]  ← 제품 상세. 불만, 테마, 관련 브리프.
└── /search?q=...         ← 교차 엔티티 검색. 탭: 전체, 브리프, 제품, 게시물.
```

네비게이션에 세 개의 최상위 탭. 검색은 유틸리티이지 탭이 아닙니다 — 데스크톱에서는 헤더에서 접근, 모바일에서는 풀스크린 오버레이. 모든 엔티티가 관련 엔티티로 연결됩니다: 게시물은 포함된 브리프로, 브리프는 소스 게시물로, 제품은 불만과 관련 브리프로. 막다른 길 없음.

## 경쟁 포지셔닝

| 도구 | 잘하는 것 | 멈추는 곳 | Idea Fork의 각도 |
|------|----------|----------|-----------------|
| Gummysearch | 깊은 Reddit 커뮤니티 탐색 | 수동 쿼리 작성 필요, 합성 없음 | 제로 셋업 — 파이프라인이 매일 실행, 브리프 전달 |
| SparkToro | 오디언스 리서치와 인구통계 | 마케팅 중심, 제품 기회 경로 없음 | 오디언스가 아닌 수요-공급 갭을 매핑 |
| Exploding Topics | 트렌드 데이터와 떠오르는 토픽 | 뭐가 성장하는지 보여줌, 왜/뭘 만들지는 아님 | 피드 우선 발견 + AI 생성 브리프 |
| 수동 브라우징 + ChatGPT | 유연하고 무료 | 집계 데이터 없음, 종단 시그널 없음 | 실제 귀속이 있는 멀티소스 파이프라인 |

핵심 차별화는 엔드투엔드 커버리지입니다. 다른 도구들은 방정식의 한쪽만 줍니다(수요 또는 공급, 원시 데이터 또는 합성). Idea Fork는 전체 파이프라인을 커버합니다: 원시 시그널 → 집계 → 합성 → 소스 링크가 있는 브리프.

## 비즈니스 모델

**1단계 (현재): 무료 탐색, 수요 검증.** 로그인 없음, 페이월 없음. 참여도 측정: 사용자가 브리프를 읽나? 소스 링크를 클릭하나? 매주 돌아오나? 프리미엄 대기자 명단이 의향 시그널을 수집.

**2단계 (계획): 프리미엄.** 무료 = 공개 브리프와 피드. 프리미엄 = 맞춤 알림(관심사에 매칭되는 새 브리프 알림), 브리프 내보내기(PDF/Notion), 히스토리컬 트렌드 데이터, 워크플로우 구축을 위한 API 접근.

**왜 처음부터 유료가 아닌가?** 제품은 매출 전에 리텐션 증명이 필요합니다. 검증되지 않은 제품의 페이월은 제품이 작동하는지 알려줄 바로 그 사용자들을 걸러냅니다.

## 로드맵

**1단계 — MVP (출시 완료):** 전체 파이프라인, 세 경험 모두(피드, 제품, 브리프), 검색, 익명 평가. 8주 빌드.

**2단계 — 검증 및 반복:** 사용자 인터뷰, 브리프 품질 스코어링, 평가 피드백 기반 프롬프트 튜닝. X/Twitter를 수요 소스로 추가. +4주.

**3단계 — 수익화:** 프리미엄 기능, 사용자 계정, 커스텀 파이프라인. 2단계 시그널 기반으로 타임라인 결정.
