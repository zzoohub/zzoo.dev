## 문제

인디 해커와 초기 단계 창업자들은 Reddit, Hacker News, Product Hunt, 앱스토어에서 제품 기회를 수동으로 조사하는 데 주당 5시간 이상을 소비합니다. 프로세스는 파편화되어 있습니다: 수요 시그널(사용자 불만)은 한 곳에, 공급 데이터(기존 제품)는 다른 곳에 있으며, 그 갭을 합성하는 데 상당한 수동 노력이 필요합니다. 이 리서치의 대부분은 지저분한 스프레드시트와 모호한 직감을 낳을 뿐 — 실행 가능한 인사이트가 아닙니다.

## 솔루션

Idea Fork는 수요-공급 리서치 파이프라인 전체를 자동화합니다. 일일 5단계 데이터 파이프라인이 44개 서브레딧, RSS 피드, 앱스토어 리뷰에서 사용자 불만을 수집하고, Product Hunt와 앱스토어에서 트렌딩 제품을 추적하며, Gemini LLM 파이프라인을 사용해 그 갭을 출처가 명시된 제품 기회 브리프로 합성합니다.

### 분리 경로 아키텍처

웹 앱은 `@neondatabase/serverless`를 통해 Neon PostgreSQL에서 직접 읽어, 읽기 작업에서 API 서버를 완전히 우회합니다. FastAPI 백엔드는 예약된 파이프라인으로만 실행됩니다 — 프로덕션 읽기 경로에 있지 않습니다. 이를 통해 사용자의 지연을 제거하고 파이프라인이 실행되지 않을 때 인프라 비용을 거의 0에 가깝게 유지합니다.

### 헥사고날 백엔드

FastAPI 서비스는 Protocol ABC를 도메인 포트로 사용하는 헥사고날 아키텍처를 사용합니다. 이를 통해 비즈니스 로직을 건드리지 않고 데이터 소스(Reddit, App Store, Product Hunt)를 교체할 수 있었습니다. 각 외부 어댑터는 독립적으로 테스트 가능합니다.

### 증거가 있는 AI

모든 AI 생성 브리프에는 `source_snapshots`가 포함됩니다 — 각 주장을 원본 게시물에 연결하는 비정규화된 JSONB입니다. 이 이중 접근법(무결성을 위한 정규화된 `brief_source` 테이블 + 빠른 읽기를 위한 JSONB)을 통해 사용자가 추가 데이터베이스 조인 없이 주장을 검증할 수 있습니다.

## 핵심 설계 결정

- **Node.js 대신 Python 백엔드** — 풍부한 ML/LLM 생태계(scikit-learn HDBSCAN, google-genai SDK)가 결정적 요인; 파이프라인 병목은 LLM 지연이지 컴퓨팅이 아님
- **LLM 클러스터링 대신 HDBSCAN** — LLM 기반 클러스터링의 O(n) 비용 문제; HDBSCAN은 클러스터 수를 자동 감지하고 노이즈를 우아하게 처리
- **Next.js에서 Neon 직접 읽기** — 읽기 경로 병목으로서의 API 서버 제거; 개발 유연성을 위해 `DATA_SOURCE` 환경 변수로 전환 가능
- **평가를 위한 익명 세션** — 제로 마찰; httpOnly 쿠키 기반 속도 제한이 MVP 규모에서 방향성 시그널에 충분

## 결과

- **5단계 자동화 파이프라인** — 수동 개입 없이 5개 이상의 데이터 소스에서 매일 실행
- **350+ 브리프 생성** — 각각 원본 사용자 게시물로 연결되는 출처 귀속 포함
- **6개 언어 i18n** — en, es, pt-BR, id, ja, ko의 완전한 국제화
- **제로-투-쓰리 오토스케일링** — 파이프라인이 실행되지 않을 때 Cloud Run이 0으로 스케일 다운하여 비용 최소화
- **10개 테이블 PostgreSQL 스키마** — 게시물, 제품, 브리프, 클러스터, 태그, 평가를 지원하는 정규화 설계
