## System Overview

A React Native/Expo mobile client talks to a Rust/Axum API on Cloud Run. Photos upload directly from the client to Cloudflare R2 via presigned URLs — the API never touches photo bytes. AI nutrition analysis runs in background Tokio tasks, decoupled from the request-response cycle. Data lives in Neon PostgreSQL.

## Two Data Backends, One Interface

The app runs in two modes — guest (local MMKV storage) and authenticated (API + TanStack Query) — but the UI never knows which. A `useEntryData()` hook abstracts the difference: same interface, same return types, different backends.

Guest mode writes to device storage and triggers re-renders via a Zustand version bump. Auth mode enqueues captures into a background upload processor — photos go to R2 in parallel, then the server entry is created, AI triggered, and query caches invalidated.

The tricky part is login migration. All guest entries transfer to the server in one batch, then local storage clears. Every feature — capture, feed, search, statistics — must work against two data sources with different ID formats: UUID for guest, integer for auth. Without the `useEntryData` abstraction, this distinction would leak into 30+ UI components.

## Presigned Upload — Keeping Bytes Off the Server

Photo bytes never touch the API server. The mobile client requests a presigned URL, the API generates a 15-minute PUT URL for R2, the client uploads directly, then links the photo to the entry via API. This keeps API memory low, bandwidth costs down, and Cloud Run scaling simple.

Photos upload in parallel per entry, but entries process sequentially to prevent race conditions in the upload queue. One platform detail that cost real debugging time: on Android, `fetch()` can't read `file://` URIs. The upload code uses `XMLHttpRequest` instead. Doesn't show up in architecture diagrams, but it's the kind of thing that blocks a release for two days.

## Trigger-and-Poll AI on Scale-to-Zero

AI analysis can't hold a connection open for 30 seconds on a serverless platform. `POST /diary/{id}/analyze` validates ownership, inserts a pending record, spawns a Tokio background task, and returns 202 immediately. The endpoint is idempotent — returns existing results if pending or complete, retries only on failure.

The background task downloads entry photos (validating each URL against the R2 prefix to prevent SSRF), sends them to Gemini Vision with a structured JSON schema, and writes parsed nutrition data back to the database.

The client polls every 3 seconds via TanStack Query. On Cloud Run, this polling does double duty: it keeps the instance warm while the Tokio task runs (5-30 seconds), preventing scale-to-zero mid-analysis. A recovery query on startup resets analyses stuck in processing for over 5 minutes — handling the case where Cloud Run killed the instance before the task finished.
