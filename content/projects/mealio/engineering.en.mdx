## Architecture

Mealio is a three-package monorepo: `mobile` (React Native/Expo), `api` (Rust/Axum), and `infra` (Pulumi). Each deploys independently through path-filtered CI/CD.

The data flow is simple: **Mobile --&gt; API --&gt; PostgreSQL + R2**. One critical rule: photo bytes never touch the API server. They upload directly from the client to Cloudflare R2 via presigned URLs. AI analysis runs in a background Tokio task, fully decoupled from the request-response cycle.

```
Mobile Client (React Native / Expo)
  |
  |-- HTTPS --> API Server (Rust / Axum on Cloud Run)
  |                |-- SQLx --> Neon PostgreSQL
  |                |-- AWS SDK --> Cloudflare R2
  |                |-- reqwest --> Gemini Vision API
  |
  |-- Presigned PUT --> Cloudflare R2 (direct upload)
```

## Dual-Mode Data Architecture

The app runs in two modes — guest (MMKV local storage) and authenticated (API + TanStack Query) — but the UI layer never knows which is active. A `useEntryData()` hook abstracts the difference completely.

Guest mode writes entries to device storage and triggers re-renders via a Zustand version bump. Auth mode enqueues captures into a background upload processor that uploads photos to R2 in parallel, creates the server entry, triggers AI analysis, and invalidates cached queries.

The interesting part is login migration: all guest entries transfer to the server in a single batch, then local storage clears. This sounds simple, but it means every feature — capture, feed, search, statistics — needs to work against two different data sources with different ID formats (UUID for guest, integer for auth). The `useEntryData` abstraction keeps this from leaking into 30+ UI components.

## Presigned Upload Pipeline

Photos bypass the API entirely. This was deliberate — keeping photo bytes off the server means lower memory usage, lower bandwidth costs, and simpler horizontal scaling on Cloud Run.

The flow: mobile requests a presigned URL, API generates a 15-minute PUT URL for R2, mobile uploads directly, then links the photo to the entry via API. Photos upload in parallel per entry, but entries process sequentially to prevent race conditions in the upload queue.

One platform quirk worth noting: on Android, `XMLHttpRequest` is used instead of `fetch()` because `fetch()` cannot read `file://` URIs. This kind of detail doesn't show up in architecture diagrams but costs real debugging time.

## Trigger-and-Poll AI Analysis

AI analysis avoids long-lived HTTP connections entirely. The client calls `POST /diary/{id}/analyze`, the API validates ownership, inserts a pending analysis record, spawns a background Tokio task, and returns 202 immediately. The endpoint is idempotent — it returns existing results if already pending or completed, only retries on failure.

The background task downloads entry photos (validating each URL against the R2 prefix to prevent SSRF), sends them to Gemini Vision API with a structured JSON schema, and writes parsed nutrition data back to the database.

The client polls with TanStack Query every 3 seconds. This polling serves double duty on Cloud Run: it keeps the instance alive while the background Tokio task runs (typically 5-30 seconds), preventing the instance from scaling to zero mid-analysis. On server startup, a recovery query resets analyses stuck in processing for over 5 minutes.

## Key Decisions

**Rust over Node.js for the API** — The API is mostly HTTP handlers and SQL queries — nothing that requires Rust's systems programming strengths. But on Cloud Run, Rust gives roughly 100ms cold starts, 512MB memory is plenty, and SQLx provides compile-time checked SQL. No query typos, no column mismatches, no type coercion bugs at runtime. The trade-off is slower development velocity, which matters less for a solo project with a stable API surface.

**Cloudflare R2 over S3** — Zero egress fees are essential for a photo-heavy app where every diary view loads multiple images. The S3-compatible API means standard AWS SDK tooling works unchanged.

**Scale-to-zero everything** — Cloud Run, Neon (serverless Postgres), and R2 all scale to zero. The entire stack costs $0 at idle. For a solo project without predictable traffic, paying only for actual usage is the only sustainable model.
