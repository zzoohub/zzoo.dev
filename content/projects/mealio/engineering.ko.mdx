## 아키텍처

Mealio는 3개 패키지 모노레포다: `mobile` (React Native/Expo), `api` (Rust/Axum), `infra` (Pulumi). 각 패키지는 경로 필터링된 CI/CD로 독립 배포된다.

데이터 흐름은 단순하다: **Mobile --&gt; API --&gt; PostgreSQL + R2**. 핵심 규칙 하나: 사진 바이트는 API 서버를 절대 거치지 않는다. presigned URL을 통해 클라이언트에서 Cloudflare R2로 직접 업로드된다. AI 분석은 백그라운드 Tokio 태스크에서 실행되며 요청-응답 사이클과 완전히 분리되어 있다.

```
Mobile Client (React Native / Expo)
  |
  |-- HTTPS --> API Server (Rust / Axum on Cloud Run)
  |                |-- SQLx --> Neon PostgreSQL
  |                |-- AWS SDK --> Cloudflare R2
  |                |-- reqwest --> Gemini Vision API
  |
  |-- Presigned PUT --> Cloudflare R2 (직접 업로드)
```

## 듀얼 모드 데이터 아키텍처

앱은 게스트(MMKV 로컬 저장)와 인증(API + TanStack Query) 두 모드로 동작하지만, UI 레이어는 어떤 모드인지 모른다. `useEntryData()` 훅이 차이를 완전히 추상화한다.

게스트 모드는 디바이스 저장소에 항목을 쓰고 Zustand 버전 범프로 리렌더를 트리거한다. 인증 모드는 캡처를 백그라운드 업로드 프로세서에 넣어서 R2에 사진을 병렬 업로드하고, 서버 항목을 생성하고, AI 분석을 트리거하고, 캐시를 무효화한다.

재미있는 부분은 로그인 마이그레이션이다. 모든 게스트 항목이 단일 배치로 서버에 전송되고 로컬 저장소가 클리어된다. 단순해 보이지만, 이건 모든 기능 — 캡처, 피드, 검색, 통계 — 이 서로 다른 ID 형식(게스트는 UUID, 인증은 integer)의 두 데이터 소스에 대해 동작해야 한다는 뜻이다. `useEntryData` 추상화가 이걸 30개 넘는 UI 컴포넌트에 새어나가지 않게 막는다.

## Presigned 업로드 파이프라인

사진은 API를 완전히 우회한다. 의도된 설계다 — 사진 바이트를 서버에서 처리하지 않으면 메모리 사용이 줄고, 대역폭 비용이 낮아지고, Cloud Run 수평 스케일링이 단순해진다.

흐름: 모바일이 presigned URL을 요청하고, API가 15분짜리 PUT URL을 R2용으로 생성하고, 모바일이 직접 업로드한 뒤 API를 통해 사진을 항목에 연결한다. 항목당 사진은 병렬 업로드되지만, 항목 자체는 업로드 큐의 레이스 컨디션 방지를 위해 순차 처리된다.

플랫폼 특이사항 하나: Android에서는 `fetch()`가 `file://` URI를 읽지 못해서 `XMLHttpRequest`를 사용한다. 아키텍처 다이어그램에는 안 나오지만 실제 디버깅 시간을 잡아먹는 종류의 디테일이다.

## Trigger-and-Poll AI 분석

AI 분석은 장시간 HTTP 연결을 완전히 피한다. 클라이언트가 `POST /diary/{id}/analyze`를 호출하면, API가 소유권을 확인하고, pending 분석 레코드를 삽입하고, 백그라운드 Tokio 태스크를 생성한 뒤 즉시 202를 반환한다. 엔드포인트는 멱등성을 보장한다 — pending이나 completed면 기존 결과를 반환하고 failed일 때만 재시도한다.

백그라운드 태스크는 항목 사진을 다운로드하고(각 URL을 R2 접두사와 대조해 SSRF 방어), Gemini Vision API에 구조화된 JSON 스키마와 함께 전송한 뒤, 파싱된 영양 데이터를 DB에 기록한다.

클라이언트는 TanStack Query로 3초마다 폴링한다. 이 폴링은 Cloud Run에서 이중 역할을 한다: 백그라운드 Tokio 태스크가 실행되는 동안(보통 5-30초) 인스턴스를 활성 상태로 유지해서 분석 중에 인스턴스가 제로로 스케일 다운되는 걸 방지한다. 서버 시작 시에는 복구 쿼리가 5분 이상 processing 상태인 분석을 리셋한다.

## 핵심 결정

**API에 Node.js 대신 Rust** — API는 대부분 HTTP 핸들러와 SQL 쿼리다. Rust의 시스템 프로그래밍 강점이 필요한 건 아니다. 하지만 Cloud Run 위에서 Rust는 약 100ms 콜드 스타트, 512MB면 충분한 메모리, 그리고 SQLx의 컴파일 타임 SQL 검증을 제공한다. 쿼리 오타, 컬럼 불일치, 런타임 타입 변환 버그가 없다. 트레이드오프는 개발 속도가 느리다는 건데, API 표면이 안정된 1인 프로젝트에서는 덜 중요하다.

**S3 대신 Cloudflare R2** — 사진 중심 앱에서 이그레스 비용 제로는 필수다. 다이어리를 볼 때마다 여러 이미지를 로드하니까. S3 호환 API 덕분에 표준 AWS SDK 도구가 그대로 동작한다.

**전부 제로 스케일** — Cloud Run, Neon(서버리스 Postgres), R2 모두 제로로 스케일 다운된다. 유휴 시 전체 스택 비용이 $0이다. 트래픽을 예측할 수 없는 1인 프로젝트에서는 실제 사용량에만 과금되는 것이 유일하게 지속 가능한 모델이다.
