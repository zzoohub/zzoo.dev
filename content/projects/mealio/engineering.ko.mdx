## 시스템 구조

React Native/Expo 모바일 클라이언트가 Cloud Run의 Rust/Axum API와 통신한다. 사진은 presigned URL로 클라이언트에서 Cloudflare R2에 직접 업로드된다 — API는 사진 바이트를 건드리지 않는다. AI 영양 분석은 백그라운드 Tokio 태스크에서 실행되며 요청-응답 사이클과 분리된다. 데이터는 Neon PostgreSQL에 저장된다.

## 두 데이터 백엔드, 하나의 인터페이스

앱이 게스트(MMKV 로컬 저장)와 인증(API + TanStack Query) 두 모드로 동작하지만, UI는 어느 쪽인지 모른다. `useEntryData()` 훅이 차이를 추상화한다 — 같은 인터페이스, 같은 리턴 타입, 다른 백엔드.

게스트 모드는 디바이스 저장소에 쓰고 Zustand 버전 범프로 리렌더를 트리거한다. 인증 모드는 캡처를 백그라운드 업로드 프로세서에 넣어서 R2에 사진 병렬 업로드, 서버 항목 생성, AI 트리거, 캐시 무효화를 수행한다.

까다로운 부분은 로그인 마이그레이션이다. 게스트 항목 전체가 한 배치로 서버에 전송되고 로컬이 클리어된다. 모든 기능(캡처, 피드, 검색, 통계)이 두 데이터 소스에서 다른 ID 형식(게스트 UUID, 인증 integer)으로 동작해야 한다. `useEntryData` 추상화가 없으면 이 구분이 30개 넘는 UI 컴포넌트로 새어나간다.

## Presigned 업로드 — 바이트를 서버에서 빼기

사진 바이트는 API 서버를 안 거친다. 모바일이 presigned URL을 요청하면 API가 15분짜리 R2 PUT URL을 생성하고, 모바일이 직접 업로드한 뒤 API로 사진을 항목에 연결한다. API 메모리가 낮아지고, 대역폭 비용이 줄고, Cloud Run 스케일링이 단순해진다.

항목당 사진은 병렬 업로드, 항목 자체는 업로드 큐 레이스 컨디션 방지를 위해 순차 처리. 실제 디버깅 시간을 잡아먹은 플랫폼 디테일: Android에서 `fetch()`가 `file://` URI를 읽지 못한다. 업로드 코드에 `XMLHttpRequest`를 쓴다. 아키텍처 다이어그램에는 안 나오지만 릴리즈를 이틀 막은 문제다.

## Trigger-and-Poll AI — 제로 스케일 환경의 백그라운드 처리

서버리스 플랫폼에서 30초짜리 커넥션을 붙잡고 있을 수 없다. `POST /diary/{id}/analyze`가 소유권을 확인하고, pending 레코드를 삽입하고, Tokio 백그라운드 태스크를 생성한 뒤 즉시 202를 반환한다. 멱등성 보장 — pending이나 complete면 기존 결과를 반환하고 실패 시에만 재시도.

백그라운드 태스크가 항목 사진을 다운로드하고(SSRF 방지를 위해 URL을 R2 접두사와 대조), Gemini Vision에 구조화된 JSON 스키마와 함께 전송한 뒤, 파싱된 영양 데이터를 DB에 기록한다.

클라이언트가 TanStack Query로 3초마다 폴링한다. Cloud Run에서 이 폴링이 이중 역할을 한다: Tokio 태스크가 돌아가는 동안(5-30초) 인스턴스를 활성 상태로 유지해서 분석 중 제로 스케일 다운을 방지한다. 서버 시작 시 복구 쿼리가 5분 이상 processing 상태인 분석을 리셋한다 — Cloud Run이 태스크 완료 전에 인스턴스를 죽인 경우를 처리한다.
