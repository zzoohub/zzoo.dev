## 독자 문제

개발자 포트폴리오는 독자가 한 명인 것처럼 만든다. 아니다. 채용 담당자, 동료 개발자, 검색 크롤러가 같은 URL에 와서 전혀 다른 걸 원한다.

채용 담당자는 5초 안에 신호가 필요하다: 이 사람이 내 문제를 해결할 수 있는가? 개발자는 아키텍처 결정과 코드로 바로 건너뛴다. 크롤러는 파싱 가능한 구조화 데이터가 필요하다 — JSON-LD, 깔끔한 헤딩, 시맨틱 HTML.

독자 하나를 위한 설계는 레이아웃 문제다. 셋을 동시에 설계하는 건 정보 아키텍처 문제다. 이 차이가 사이트의 모든 결정을 이끌었다.

## 누가 방문하는가

**평가자 — 5초.**

스타트업 대표나 엔지니어링 매니저. 훑으면서 신호를 찾는다. 프로젝트 페이지에 도착해서 5초 안에 계속 읽을지 떠날지 결정한다. 성과가 중요하고 구현은 관심 없다. Engineering 탭은 클릭하지 않는다.

Above the Fold에 필요한 것: 이 사람이 뭘 만드는지, 배포할 능력이 있다는 증거, 연락할 경로 하나.

**동료 — Engineering 탭.**

블로그 포스트나 프로젝트 링크로 사이트를 발견한 동료 개발자. Overview를 통째로 건너뛴다. 트레이드오프, 아키텍처 다이어그램, "왜 X를 안 썼는가"를 보러 왔다.

필요한 것: 결정에 대한 솔직한 이유. 세일즈 피치가 아니라.

**기계 — 구조화 데이터.**

검색 결과를 만드는 구글 크롤러. 답변을 작성하는 Perplexity. "X 전문 개발자"에 대한 응답을 만드는 ChatGPT. 이들은 산문을 읽지 않는다 — 스키마, 헤딩, 메타데이터를 파싱한다.

필요한 것: JSON-LD 엔티티, 시맨틱 HTML, hreflang 대체 링크, 추측 없이 추출할 수 있는 깔끔한 콘텐츠.

## 긴 페이지 하나가 아니라 탭 셋

당연한 설계는 프로젝트당 스크롤 한 페이지 — 문제, 해결, 기술 스택, 아키텍처를 순서대로 나열하는 것이었다. 대부분의 포트폴리오가 이렇게 한다.

문제: 모든 독자를 모든 섹션으로 끌고 간다. 엔지니어가 아키텍처에 도달하려면 비즈니스 맥락을 거쳐야 한다. 비기술적 평가자가 코드 블록을 만나면 이탈한다.

세 탭 시스템이 각 독자에게 자기 선택을 허용한다:

- **Overview** — 이것이 뭔지, 왜 존재하는지, 무슨 일이 있었는지. 짧고, 성과 중심, 스캔 가능.
- **Design** — 프로덕트 결정, 사용자 사고, 트레이드오프. 뭘 거부했고 왜.
- **Engineering** — 아키텍처, 구현, 기술 선택.

트레이드오프는 발견성이다. 방문자가 Design이나 Engineering 탭의 존재를 모를 수 있다. 완화책: 탭 바가 항상 보이고, Overview가 다른 탭의 내용을 자연스럽게 언급한다.

## 5초 테스트

모든 페이지가 Above the Fold에서 하나의 질문에 답한다: 이 개발자가 나를 도울 수 있는가?

이 원칙이 흔한 포트폴리오 패턴들을 배제한다. 전체 화면 Hero 이미지 없음. 콘텐츠가 나타나기 전 애니메이션 없음. 프로젝트 리스트 위의 "About Me" 없음. 방문자의 첫 5초는 기다리는 데가 아니라 읽는 데 쓰인다.

홈 페이지가 바로 value proposition, availability indicator, 프로젝트 리스트로 시작한다. 스플래시 스크린 없음. 방문자와 콘텐츠 사이의 장식적 간격 없음.

## 디자인 제약으로서의 검색 가시성

SEO는 보통 사이트를 다 만든 뒤에 붙인다. 여기서는 처음부터 HTML 구조를 결정한 구조적 제약이다.

세 발견 채널을 동시에:

- **전통 검색** — 시맨틱 HTML, hreflang이 포함된 XML 사이트맵, 로케일별 canonical URL.
- **답변 엔진** — 모든 엔티티 타입의 JSON-LD 스키마: Person, WebSite, BlogPosting, SoftwareApplication, BreadcrumbList.
- **생성형 AI** — 사이트 루트의 `llms.txt`와 LLM이 추출하고 인용할 수 있는 깔끔한 MDX.

디자인에 미치는 영향: 모든 헤딩은 의미 있어야 한다(기발할 필요 없다). 모든 페이지는 주제 하나가 명확해야 한다. 모든 메타데이터는 기계가 읽을 수 있어야 한다. 창의적이지만 모호한 섹션 제목, 주제가 여러 개인 페이지, 크롤러가 파싱 못 하는 JavaScript 렌더링 — 전부 배제된다.

## 이중 언어 UX

언어 두 개, 버튼 하나. 영어에서 "KO", 한국어에서 "EN". 드롭다운은 클릭만 추가할 뿐이다.

더 깊은 결정: 영어와 한국어는 서로의 번역이 아니다. 각 언어로 자연스럽게 쓴 병렬 콘텐츠다. 구조는 공유하되 표현은 각 언어에 자연스럽다. 번역체의 어색함을 피하기 위해서다.

한국어가 없으면 404가 아니라 영어로 폴백한다. 새 콘텐츠를 한쪽 언어로 먼저 배포 가능. 제약은 "항상 이중 언어"가 아니라 "항상 접근 가능"이다.

## 의도적으로 빠뜨린 것들

**연락 폼 없음.** 폼은 백엔드가 필요하다. 백엔드는 서버가 필요하다. 서버는 공격 면적, 유지보수, 스팸 필터링이다. `mailto` 링크면 된다. 모바일에는 클립보드 복사 폴백이 있다.

**애니메이션 라이브러리 없음.** 모든 트랜지션이 CSS다 — Hover 150ms, 모바일 내비 200ms, 플로팅 CTA 300ms. 성능 예산(초기 JS 100KB gzip 이하, FCP 1.0초 미만)이 Framer Motion을 허용하지 않는다.

**이미지 캐러셀 없음.** 프로젝트 스크린샷은 정적이다. 캐러셀은 JavaScript, 인터랙션 복잡성, 접근성 부담을 추가하고 정보 이득은 미미하다. 잘 고른 이미지 하나가 자동 회전하는 여섯 장보다 더 많은 걸 전달한다.

**CMS 없음.** 콘텐츠는 git 저장소의 MDX다. CMS는 의존성, 계정, 콘텐츠와 코드 사이의 동기화 레이어를 추가한다. 1인 빌더에게 git의 파일이 더 빠르고, 버전 관리가 쉽고, 잃어버릴 수가 없다.

빠진 기능이 아니다. 하나의 제약이 만든 결과물이다: 백엔드 제로, 런타임 제로, 최소 면적. 제약 자체가 제품이다.

## 왜 템플릿이 아닌가

포트폴리오 템플릿 — Next.js, Astro, Gatsby — 은 레이아웃과 스타일을 준다. 콘텐츠 아키텍처는 안 준다. 세 탭 프로젝트 문서, Zod 검증 프론트매터, 로케일 폴백이 되는 이중 언어 콘텐츠, AI 발견용 구조화 데이터를 포함하는 템플릿은 없다.

Notion과 WordPress는 편하지만 구조화 출력의 통제권이 없다. JSON-LD 스키마, hreflang 대체 링크, `llms.txt` 모두 지원 안 한다. 쓰기 편의에 최적화되어 있지, 기계가 콘텐츠를 소비하는 방식에는 아니다.

이 사이트는 어려운 길을 택했다: 아키텍처를 직접 만들고, 출력을 소유하고, 같은 정적 HTML로 모든 독자를 서빙한다.
