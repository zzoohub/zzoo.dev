이 사이트의 모든 디자인 결정은 하나의 질문에서 출발합니다: 방문자가 5초 안에 이 개발자가 자신의 문제를 해결할 수 있는지 판단하고, 가장 짧은 경로로 연락할 수 있는가?

이 질문과, 여기서 도출한 다섯 가지 원칙이 정보 구조, 콘텐츠 전략, 비주얼 디자인, 검색 가시성 접근 방식 전체를 결정했습니다.

## 디자인 철학: 5초 안에 신뢰 확보

사이트는 다섯 가지 UX 원칙을 따릅니다. 원칙 간 충돌이 있을 때는 우선순위 순서대로 적용합니다:

1. **5초 안에 신뢰 확보** — Hero 섹션이 첫인상의 전체 무게를 감당합니다. Value proposition, CTA 하나, 증거 하나(availability badge나 성과 지표)가 모든 뷰포트의 Above the Fold에 보여야 합니다. 장식적 요소 없음. 모호함 없음.

2. **기능이 아닌 성과** — 콘텐츠는 개발자가 아는 것이 아니라 클라이언트가 얻는 것 중심으로 구성합니다. 케이스 스터디는 기술 스택이 아니라 결과로 시작합니다. 블로그는 명확한 takeaway가 있습니다. 기술은 보조 근거이지, 헤드라인이 아닙니다.

3. **점진적 공개** — 현재 결정에 필요한 것만 보여줍니다. 홈페이지는 요약, 각 페이지는 한 단계 깊이. 깊이가 필요한 사용자는 찾을 수 있고, 필요 없는 사용자는 압도당하지 않습니다.

4. **마찰 없는 연락** — 모든 페이지에 명확하고 쉬운 연락 경로가 있습니다. 연락은 이메일 전용(폼 없음, 백엔드 없음)이므로 패턴은 `mailto` 링크를 Primary 버튼으로 스타일링하고, 모바일 사용자를 위한 클립보드 복사 폴백을 제공하는 것입니다.

5. **성능이 곧 UX** — 느린 사이트는 부주의의 신호입니다. 모든 디자인 결정은 성능 예산의 제약을 받습니다: 초기 JS 100KB gzip 이하, FCP 1.0초 미만, Lighthouse 95+. 이 때문에 무거운 애니메이션 라이브러리, 클라이언트 사이드 라우팅 트랜지션, 대형 이미지 캐러셀은 배제됩니다. 모션은 CSS 전용.

## 정보 구조

사이트맵은 의도적으로 간결합니다. Primary 내비게이션은 네 항목 — Home, Projects, Blog, About — 으로 인지 부하를 낮게 유지합니다. 보조 페이지(Now, Contact)는 푸터와 문맥적 CTA에서 접근 가능하지만 헤더에서 주의를 빼앗지 않습니다.

모든 페이지는 전환 퍼널을 염두에 두고 설계했습니다:

- **인지** — 검색, AI 답변, 추천으로 방문자 유입
- **관심** — Hero가 5초 안에 신뢰 확립
- **평가** — 케이스 스터디가 성과 증명, 블로그가 전문성 시연
- **신뢰** — 추천사, 기술적 깊이, About 페이지가 확신 구축
- **행동** — 사이트 곳곳의 CTA가 이메일 연락으로 유도

Breadcrumb은 사용자가 두 단계 깊이에 있는 상세 페이지(프로젝트, 블로그 포스트)에서만 표시하며, `Projects / 프로젝트명` 형식을 사용합니다.

## 콘텐츠 전략: 3-Layer 프로젝트 문서

프로젝트마다 세 개의 탭으로 문서화하며, 각각 다른 독자와 깊이를 겨냥합니다:

- **Overview** — 누구나 읽는 프로덕트 브리프: 이것이 무엇인지, 왜 존재하는지, 결과는 무엇인지. 스캔 가능하고, 성과 중심이며, 짧습니다.
- **Design** — 디자인 마인드의 방문자를 위한 프로덕트 디자인 결정: UX 철학, 정보 구조, 콘텐츠 전략, 검색 가시성 접근 방식.
- **Engineering** — 엔지니어를 위한 기술 심화: 아키텍처, 프레임워크 선택, 빌드 파이프라인, 배포.

이 구조는 점진적 공개 원칙을 따릅니다. 채용 담당자는 Overview를 읽고, 프로덕트 디자이너는 Design을 클릭하고, 동료 엔지니어는 Engineering으로 갑니다. 각 독자가 다른 사람을 위한 콘텐츠를 헤치지 않고 정확히 필요한 것을 얻습니다.

모든 프론트매터는 Zod 스키마로 검증됩니다. 콘텐츠는 로케일별 MDX 파일로 관리되며, 번역이 없으면 영어 콘텐츠로 자동 폴백합니다.

## 검색 가시성 — 디자인 결정으로서

검색 가시성은 사후 고려가 아닙니다. HTML 구조, 콘텐츠 포맷, 메타데이터 아키텍처를 결정하는 디자인 제약입니다. 사이트는 세 가지 발견 채널을 동시에 겨냥합니다.

### SEO — 전통 검색

올바른 헤딩 구조, 시맨틱 랜드마크, 접근성 있는 마크업으로 구조화된 HTML. 모든 페이지가 canonical URL과 양 로케일의 hreflang 대체 링크를 선언합니다. XML 사이트맵이 전체 라우트를 커버합니다. 정적 생성이 검색 엔진이 보상하는 1초 미만 로딩을 보장합니다.

### AEO — 답변 엔진 최적화

Google AI Overview나 Perplexity 같은 AI 시스템이 답변을 추출할 때 구조화된 데이터에 의존합니다. 사이트에 `WebSite`, `Person`, `BlogPosting`, `SoftwareApplication`, `BreadcrumbList` JSON-LD 스키마를 임베딩하여 답변 엔진에 기계 판독 가능한 맥락을 제공합니다. RSS 피드가 인덱싱을 위한 콘텐츠 파이프라인을 제공하고, 모든 페이지에 Open Graph와 Twitter Card 메타데이터가 적용됩니다.

### GEO — 생성형 엔진 최적화

ChatGPT나 Claude를 구동하는 LLM을 위해 사이트 루트에 `llms.txt`를 제공합니다 — 사이트의 목적, 구조, 콘텐츠를 LLM이 직접 소비할 수 있는 포맷으로 기술한 일반 텍스트 파일입니다. 명확한 헤딩 구조의 깔끔한 MDX 콘텐츠는 생성형 모델이 정확하게 추출하고 인용하기 쉽게 합니다. 한영 이중 언어 지원이 언어 경계를 넘어 발견 가능성을 보장합니다.

## 이중 언어 UX

언어 전환기는 대체 로케일을 표시하는 텍스트 버튼입니다(영어에서 "KO", 한국어에서 "EN"). 드롭다운이 아닙니다 — 로케일이 두 개뿐이라 탭 한 번으로 전환하는 것이 더 빠르고 클릭을 줄입니다.

전환 시 현재 페이지 경로가 유지됩니다. 번역 버전이 없으면 404 대신 영어 콘텐츠로 폴백합니다.

타이포그래피는 두 문자 체계를 처리합니다: 라틴 텍스트에 Geist Sans, 한국어는 시스템 폰트를 폴백으로 사용합니다. 한국어 콘텐츠에는 `word-break: keep-all`을 적용하여 단어 중간 줄바꿈을 방지합니다. 한국어 텍스트가 일반적으로 더 간결하므로 버튼 패딩(`px-6`)이 레이아웃 시프트 없이 양쪽 언어를 수용합니다.

## 다크/라이트 모드

테마 전환은 기본적으로 시스템 설정을 따르며, 수동 오버라이드는 `localStorage`로 유지합니다. `next-themes` 블로킹 스크립트가 로딩 시 잘못된 테마의 깜빡임을 방지합니다.

토글 아이콘은 전환될 테마를 보여줍니다(다크 모드에서 태양, 라이트 모드에서 달) — 상태가 아닌 액션을 표시하는 컨벤션입니다. 모든 컴포넌트, D2 다이어그램, 코드 블록이 양쪽 테마에 맞게 스타일링됩니다. 양쪽 모드 모두 WCAG AA를 통과하는 색상 대비를 확보합니다 — 텍스트 대 배경 약 18:1, muted 텍스트 4.6:1 이상.

`prefers-reduced-motion`을 존중하여 사용자가 모션 감소를 요청하면 모든 애니메이션 지속 시간을 거의 0으로 줄입니다.

## CTA 위계와 배치

CTA는 세 단계 위계를 따릅니다:

| 단계 | 스타일 | 용도 |
|------|--------|------|
| Primary | 단색 배경, 고대비 | 섹션당 1개, 페이지당 최대 2-3개 |
| Secondary | 테두리 아웃라인 | 보조 대안 |
| Inline | 화살표가 있는 텍스트 링크 | 콘텐츠, 카드, 리스트 내 |

CTA 카피는 맥락에 맞게 바뀝니다: 홈페이지에서 "나의 작업 보기", 프로젝트 상세에서 "비슷한 과제를 겪고 계신가요?", 블로그에서 "함께 일하고 싶으신가요?". Availability badge는 색상 코딩된 필 형태에 맥동하는 점이 있으며, 홈 Hero, Contact, 푸터에 세 가지 크기로 표시됩니다.

긴 페이지에서는 Hero를 지나 스크롤하면 플로팅 CTA가 나타납니다: 데스크톱에서는 우하단 고정 필, 모바일에서는 닫기 옵션이 있는 전체 너비 하단 바.

## 모바일 퍼스트 반응형 전략

디자인은 세 개의 breakpoint를 기준으로 모바일 퍼스트로 구축했습니다: 모바일(기본), 태블릿(`md:` 768px), 데스크톱(`lg:` 1024px). 콘텐츠는 `max-w-5xl`(1024px)로 최대 너비를 제한합니다 — 더 넓은 뷰포트는 더 넓은 콘텐츠가 아니라 더 넓은 마진을 보여줍니다.

주요 반응형 동작:

- **내비게이션** — 모바일에서 햄버거 메뉴, 전체 너비 드롭다운으로 콘텐츠를 아래로 밀어냅니다(오버레이 아님). 데스크톱에서는 네 개의 수평 링크.
- **Hero** — 모바일에서 중앙 정렬과 세로 배치 버튼, 데스크톱에서 좌측 정렬과 인라인 버튼. 헤딩은 36px에서 48px로 커집니다.
- **프로젝트 카드** — 모바일에서 단일 컬럼, 데스크톱에서 2컬럼 그리드.
- **푸터** — 모바일에서 단일 컬럼, 데스크톱에서 3컬럼 그리드.

모든 인터랙티브 요소는 44x44px 최소 터치 타겟을 충족합니다. 버튼은 모바일에서 전체 너비입니다. 아이콘 버튼은 패딩으로 최소 타겟 영역에 도달합니다.

## 접근성

사이트는 모든 인터랙션에서 WCAG 2.1 AA 준수를 목표합니다:

- **문서 구조** — 페이지당 `h1` 하나, 엄격한 헤딩 위계, 시맨틱 랜드마크(`header`, `nav`, `main`, `footer`)에 각각 고유한 `aria-label`.
- **Skip link** — 포커스 시에만 표시, 모든 페이지의 첫 번째 포커스 가능 요소로 좌상단에 나타남.
- **포커스 관리** — `:focus-visible` 링(`:focus` 아님), 탭 순서가 시각적 순서를 따름, 모바일 내비는 열린 상태에서 포커스 트랩 적용.
- **스크린 리더** — 아이콘 전용 버튼에 `aria-label`, 장식 요소는 숨김, 읽기 시간에 전체 텍스트를 위한 `aria-label` 사용, D2 다이어그램에 의미 있는 alt 텍스트.
- **색상 독립성** — 색상만으로 정보를 전달하지 않음. Availability badge는 색상과 텍스트 모두 사용. 상태 표시기는 색상 외에 굵기나 채움 변화로 보완.
- **키보드 내비게이션** — 전체 흐름: skip link, 로고, 내비 링크, 언어 전환기, 테마 토글, 메인 콘텐츠, 푸터 링크. `Escape`로 모바일 메뉴를 닫고 포커스를 복귀.

## 컴포넌트 패턴

컴포넌트 시스템은 일관된 spacing과 스타일링 토큰을 사용합니다:

- **카드** — `border border-border rounded-xl`에 `p-6`(기본) 또는 `p-8`(featured). Hover 시 150ms에 걸쳐 테두리 색상 변화와 미묘한 그림자 추가.
- **섹션** — `py-16 md:py-20` 수직 패딩, 반응형 수평 패딩의 `max-w-5xl` 컨테이너 안에 콘텐츠 배치.
- **타이포그래피** — Hero h1 36/48px, 섹션 h2 28/32px, 카드 h3 20/24px, 본문 산문 16px(행간 1.75), 코드 14px(Geist Mono).
- **태그** — 콘텐츠 태그에 작은 필(`bg-muted px-2 py-0.5 text-xs rounded-md`), 필터에는 테두리가 있는 인터랙티브 필.
- **모션** — Hover에 150ms, 모바일 내비에 200ms, 플로팅 CTA에 300ms의 CSS 전용 트랜지션. 애니메이션 라이브러리 없음.
