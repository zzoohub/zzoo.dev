이 사이트를 만들면서 깨달은 것이 있다. 개발자 포트폴리오에는 이제 독자가 두 부류다. 사람은 역량을 평가하러 오고, 기계는 구조화된 사실을 추출하러 온다. 사람만 위한 포트폴리오는 절반이 보이지 않는 셈이다.

이 깨달음이 정보 구조부터 콘텐츠 포맷, 의도적으로 빠뜨린 것까지 모든 디자인 결정을 이끌었다.

## 이 사이트를 읽는 사람들

**평가자.** 스타트업 대표나 채용 담당자. 신호를 찾으며 훑는다. 프로젝트 페이지에 도착해서 5초 안에 이 개발자가 자기 문제를 해결할 수 있는지 판단하고, 연락할 경로를 찾으려 한다. 엔지니어링 탭은 읽지 않는다. 스크롤을 내리지 않을 수도 있다.

**동료 개발자.** 어떻게 만들었는지 궁금한 사람. Overview를 건너뛰고 바로 Engineering이나 Design 탭으로 간다. 성과나 CTA가 아니라 트레이드오프와 결정이 궁금하다.

**기계.** 검색 크롤러, AI 답변 엔진, "X 전문 개발자"에 대한 응답을 만드는 LLM. 시각적 완성도가 아니라 구조화된 데이터 — JSON-LD, 깔끔한 헤딩, 시맨틱 HTML — 가 필요하다.

세 독자가 같은 URL에서 완전히 다른 것을 원한다. 세 탭 시스템이 존재하는 이유는 이들 모두를 다른 사람을 위한 콘텐츠를 헤치지 않고 서빙하기 위해서다.

## 5초 테스트

모든 페이지가 Above the Fold에서 하나의 질문에 답한다: 이 개발자가 내 문제를 해결할 수 있는가? Hero 섹션이 첫인상의 전체 무게를 감당한다 — value proposition, CTA 하나, 증거 하나(availability badge나 구체적 성과). 장식적 요소는 없다.

당연해 보이지만, 이 원칙이 많은 흔한 포트폴리오 패턴을 배제한다. 전체 화면 Hero 이미지 없음. 콘텐츠가 나타나기 전 애니메이션 트랜지션 없음. 프로젝트 리스트 위에 "About Me" 없음. 방문자의 첫 5초는 기다리는 게 아니라 읽는 데 쓰인다.

## 구조로서의 점진적 공개

사이트맵은 의도적으로 간결하다. Primary 내비게이션은 네 항목(Home, Projects, Blog, About). 보조 페이지(Now, Contact)는 푸터와 문맥적 CTA에서만 접근 가능하고 헤더에서 주의를 빼앗지 않는다.

각 프로젝트 안에서 세 탭 구조가 점진적 공개를 구체화한다:

- **Overview** — 이것이 무엇인지, 왜 존재하는지, 결과는. 스캔 가능하고, 짧고, 성과 중심.
- **Design** — 프로덕트 씽킹: UX 결정, 트레이드오프, 뭘 거부했고 왜.
- **Engineering** — 아키텍처, 기술 선택, 흥미로운 구현 디테일.

채용 담당자는 탭 하나를 읽고, 프로덕트 디자이너는 둘을 읽고, 동료 엔지니어는 셋 다 읽는다. 아무도 다른 사람을 위한 콘텐츠를 읽지 않는다.

## 왜 긴 페이지 하나가 아니라 탭 셋인가

대안은 프로젝트당 스크롤 한 페이지 — 문제, 해결, 기술 스택, 아키텍처를 순서대로 나열하는 것이었다. 대부분의 포트폴리오가 이렇게 한다.

문제: 모든 독자를 모든 섹션으로 끌고 간다. 엔지니어가 아키텍처 다이어그램에 도달하려면 비즈니스 컨텍스트를 거쳐야 한다. 비기술적 평가자가 코드 블록을 만나면 이탈한다.

탭은 각 독자가 스스로 선택하게 한다. 트레이드오프는 발견성이다 — 방문자가 Design이나 Engineering 탭의 존재를 모를 수 있다. 완화책: 탭 바가 항상 보이고, Overview가 다른 탭의 내용을 자연스럽게 언급한다.

## 디자인 제약으로서의 검색 가시성

SEO는 보통 사이트를 다 만든 뒤에 덧붙이는 것이다. 여기서는 처음부터 HTML 구조, 콘텐츠 포맷, 메타데이터 아키텍처를 결정한 구조적 제약이다.

전통 검색(시맨틱 HTML, 사이트맵, hreflang), 답변 엔진(모든 엔티티 타입의 JSON-LD 스키마), 생성형 AI(사이트 루트의 `llms.txt`와 LLM이 추출하고 인용할 수 있는 깔끔한 MDX) — 세 채널을 동시에 겨냥한다.

디자인에 미치는 영향: 모든 헤딩은 의미 있어야 하고(기발할 필요 없다), 모든 페이지는 하나의 명확한 주제를 가져야 하고, 모든 메타데이터는 기계가 읽을 수 있어야 한다. 창의적이지만 모호한 섹션 제목, 여러 주제를 다루는 페이지, 크롤러가 파싱할 수 없는 JavaScript 렌더링 콘텐츠는 여기서 허용되지 않는다.

## 이중 언어 UX: 번역이 아니라 병렬 콘텐츠

로케일이 두 개뿐이니 언어 전환은 버튼 하나다 — 영어에서 "KO", 한국어에서 "EN". 드롭다운은 클릭만 하나 추가할 뿐이다.

더 깊은 디자인 결정: 영어와 한국어는 서로의 번역이 아니다. 각 언어로 자연스럽게 쓴 병렬 콘텐츠다. 구조는 같지만 표현은 각 언어에 자연스럽다. 번역체의 어색함을 피하기 위해서다.

한국어 번역이 없으면 404가 아니라 영어 콘텐츠로 폴백한다. 새 콘텐츠를 한 언어로 먼저 배포할 수 있다는 뜻이다 — 제약은 "항상 이중 언어"가 아니라 "항상 접근 가능"이다.

## 의도적으로 빠뜨린 것들

**연락 폼 없음.** 폼은 백엔드가 필요하고, 백엔드는 서버가 필요하고, 서버는 공격 면적과 유지보수와 스팸을 의미한다. `mailto` 링크면 충분하다. 모바일 사용자에게는 클립보드 복사 폴백이 있다.

**애니메이션 라이브러리 없음.** 모든 트랜지션이 CSS 전용이다 — Hover 150ms, 모바일 내비 200ms, 플로팅 CTA 300ms. 성능 예산(초기 JS 100KB gzip 이하, FCP 1.0초 미만)이 Framer Motion 같은 라이브러리를 허용하지 않는다.

**이미지 캐러셀 없음.** 프로젝트 이미지는 의도적으로 정적이다. 캐러셀은 JavaScript, 인터랙션 복잡성, 접근성 부담을 추가하는데 정보 이득은 미미하다. 잘 고른 Hero 이미지 하나가 자동 회전하는 여섯 장보다 더 많은 걸 전달한다.

이것들은 한계가 아니다. 이 사이트의 핵심 제약이 만들어낸 결과물이다: 백엔드 제로, 런타임 제로, 최소 공격 면적 — 그러면서도 풍부하고, 이중 언어이고, 콘텐츠 중심인 사이트.
