서버 없음. 데이터베이스 없음. API 라우트 없음. 런타임 없음. 모든 페이지가 빌드 타임에 사전 렌더링되어 Vercel 엣지 네트워크에서 정적 HTML로 서빙된다. 이건 한계가 아니라 핵심 아키텍처 결정이다. 백엔드가 없다는 건 공격 면적 최소화, 거의 즉각적인 로딩, 그리고 `main`에 push하면 끝나는 배포 모델을 의미한다.

## 왜 SSR이 아니라 SSG인가

콘텐츠가 바뀌는 시점은 방문자가 올 때가 아니라 개발자가 커밋을 push할 때다. 사용자별 데이터도, 인증도, 폼 제출도 없다. SSR은 콜드 스타트, 서버 비용, 캐싱 전략이라는 복잡성만 추가하고 이 유스케이스에서는 이점이 전무하다.

트레이드오프: 콘텐츠를 바꿀 때마다 리빌드가 필요하다. 하지만 Vercel의 GitHub 연동으로 `main` push부터 배포까지 2분이면 된다. 실시간 요구사항이 없는 콘텐츠 사이트에는 충분히 빠르다.

빌드 출력이 이 제약을 확인해 준다. 모든 라우트가 Next.js 빌드 로그에서 `●`(static)을 보여야 한다. `ƒ`(dynamic)가 하나라도 나타나면 뭔가 잘못된 것이다.

## next-intl의 SSG 함정

다국어는 `next-intl`로 두 로케일을 지원한다. 구현은 간단하지만, 정적 생성을 조용히 깨뜨리는 함정이 하나 있다.

서버 컴포넌트는 페이지의 `params`에서 `setRequestLocale(locale)`을 호출해야 한다. 직관적인 대안인 `getLocale()`은 정적 생성 중에 존재하지 않는 요청 시점 컨텍스트에 의존한다. 빌드는 성공하지만 런타임에 로케일이 틀린다.

두 번째 제약: async 서버 컴포넌트에서 `useTranslations()`를 직접 호출할 수 없다. 우회 방법은 각 페이지를 async 외부 컴포넌트(params 처리, 데이터 fetch)와 sync 내부 컴포넌트(훅 호출)로 분리하는 것이다. 번역 설정도 동적 `import()` 대신 명시적 정적 import를 사용한다 — SSG 호환성을 위한 또 하나의 요구사항이다.

## 콘텐츠 시스템: 안전망으로서의 Zod

콘텐츠는 저장소에 파일로 저장되는 MDX다. CMS도 데이터베이스도 없다. 블로그 포스트, 프로젝트 페이지, About 페이지 각각이 로케일별 MDX 파일을 가진다. 프로젝트는 세 탭을 지원한다: Overview(필수), Design(선택), Engineering(선택).

Zod 관련 결정 두 가지가 중요하다:

**선택 필드의 우아한 저하.** 필수 필드가 없으면 빌드를 의도적으로 깨뜨린다 — title 없는 프로젝트는 에러다. 선택 필드는 `.catch(undefined)`로 유효하지 않은 데이터를 조용히 드롭한다. 프론트매터의 잘못된 `video` URL 하나가 사이트 전체를 멈추지 않는다.

**로케일 기반 읽기 시간.** 영어는 단어 수(200 WPM), 한국어는 글자 수(CJK 분당 300자)를 쓴다. 콘텐츠가 아니라 로케일 기반 감지다 — 단순하고 예측 가능하며, 하나의 포스트가 항상 한 언어인 이중 언어 콘텐츠에 정확하다.

## D2 다이어그램: 빌드 타임, 듀얼 테마

아키텍처 다이어그램은 선언적 다이어그래밍 언어인 D2를 사용한다. 빌드 시 각 `.d2` 파일을 두 번 컴파일해서 — 라이트 한 번, 다크 한 번 — `/public/diagrams/`에 쌍을 이루는 SVG를 생성한다.

렌더링 컴포넌트는 hydration 안전을 위해 `useSyncExternalStore`를 사용한다. 서버에서는 항상 라이트 테마를 렌더링하고, 클라이언트 마운트 시 사용자 테마로 전환한다. 서버가 테마를 추측하려 할 때 발생하는 hydration 불일치를 방지한다.

## 세 채널 SEO

SEO 구현은 플러그인이 아니라 모든 페이지가 `generateMetadata()`를 통해 호출하는 모듈(`lib/seo.ts`)이다. 각 페이지가 로케일별 title, description, canonical URL, hreflang 대체 링크, Open Graph 태그를 생성한다.

JSON-LD 스키마는 다섯 엔티티 타입을 커버한다: `WebSite`, `Person`, `BlogPosting`, `SoftwareApplication`, `BreadcrumbList`. 출력은 XSS 방지를 위해 JSON-LD script 태그에서 꺾쇠괄호와 앰퍼샌드를 이스케이프한다.

사이트 루트의 `llms.txt`는 LLM에게 사이트 구조와 콘텐츠의 일반 텍스트 요약을 제공한다 — 아직 형성 중인 컨벤션이지만, 지원 비용은 제로다.

## 부재를 통한 보안

보안 헤더(CSP, HSTS, X-Frame-Options)는 `next.config.ts`에서 설정한다. CSP `frame-src`는 YouTube(프로젝트 영상)와 Giscus(블로그 댓글)만 허용한다. 나머지는 전부 차단.

하지만 진짜 보안 이야기는 존재하지 않는 것들이다. API 엔드포인트 없음. 폼 처리 없음. 사용자 입력 처리 없음. 데이터베이스 연결 없음. 정적 사이트의 공격 면적은 CDN 자체다 — 그건 Vercel의 문제이지, 내 문제가 아니다.
