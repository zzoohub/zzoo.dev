## 핵심 제약: SSG

모든 페이지가 빌드 타임에 사전 렌더링되어 Vercel 엣지에서 정적 HTML로 서빙된다. 서버 없음. 데이터베이스 없음. API 라우트 없음. 예산 결정이 아니라 아키텍처의 토대다.

콘텐츠가 바뀌는 시점은 방문자가 올 때가 아니라 개발자가 커밋을 push할 때다. SSR은 콜드 스타트, 서버 비용, 캐싱 복잡성만 추가하고 이점이 없다. 빌드 출력이 제약을 증명한다: 모든 라우트가 Next.js 로그에서 `●`(static)을 보여야 한다. `ƒ`(dynamic)가 나타나면 뭔가 깨진 거다.

트레이드오프: 콘텐츠를 바꿀 때마다 리빌드가 필요하다. Vercel의 GitHub 연동으로 2분이면 배포된다. 실시간 요구사항이 없는 콘텐츠 사이트에는 충분히 빠르다.

## next-intl의 SSG 함정

두 로케일(EN, KO)을 `next-intl`로 지원한다. 간단해 보이지만, 정적 생성을 조용히 깨뜨리는 함정이 하나 있다.

서버 컴포넌트는 페이지 params에서 `setRequestLocale(locale)`을 호출해야 한다. 직관적인 대안 `getLocale()`은 정적 생성 중 존재하지 않는 요청 시점 컨텍스트에 의존한다. 빌드는 성공하지만 런타임에 로케일이 틀린다. 하루짜리 디버깅에 한 줄짜리 수정.

두 번째 제약: async 서버 컴포넌트에서 `useTranslations()`를 직접 호출할 수 없다. 해결 방법은 각 페이지를 async 외부 컴포넌트(params 처리, 데이터 fetch)와 sync 내부 컴포넌트(훅 호출)로 분리하는 것이다. i18n 설정도 동적 `import()` 대신 명시적 정적 import를 쓴다 — 놓치기 쉬운 SSG 호환성 요구사항이다.

## 콘텐츠 시스템

저장소의 파일 기반 MDX. CMS 없음. 블로그 포스트, 프로젝트 페이지, About 페이지 각각이 로케일별 파일을 가진다. 프로젝트는 선택적 탭 세 개를 지원한다: Overview(필수), Design, Engineering.

Zod 관련 결정 두 가지가 시스템을 형성했다:

**필수 필드는 빌드를 깨뜨린다.** title이 없는 프로젝트는 의도적으로 검증 실패한다. 에러가 `bun run build` 시점에 뜨지, 배포 이후에 뜨지 않는다.

**선택 필드는 우아하게 저하된다.** 유효하지 않은 선택 데이터 — 잘못된 video URL, 빠진 thumbnail — 는 `.catch(undefined)`로 조용히 드롭된다. 나쁜 필드 하나가 사이트 전체를 멈추지 않는다.

읽기 시간은 로케일 기반이다. 영어: 단어 수(200 WPM). 한국어: 글자 수(CJK 분당 300자). 콘텐츠가 아니라 로케일 기반으로 감지한다 — 한 포스트가 항상 한 언어이니 정확하고 단순하다.

## 빌드 파이프라인

Next.js 전에 세 개의 prebuild 스크립트가 실행된다:

1. **copy-images** — 프로젝트 이미지를 `content/`에서 `public/`로 복사. slug 형식 검증. 이미지 확장자 화이트리스트.
2. **build-diagrams** — D2 파일마다 두 번 컴파일: 라이트 테마와 다크 테마, 쌍을 이루는 SVG 생성. D2 CLI가 없으면 기존 SVG로 우아하게 폴백.
3. **generate-rss** — 영어 블로그 포스트로 RSS 피드 생성. 모든 콘텐츠 XML 이스케이프.

D2 렌더링 컴포넌트는 hydration 안전을 위해 `useSyncExternalStore`를 쓴다. 서버에서는 항상 라이트 테마를 렌더링하고, 클라이언트 마운트 시 사용자 테마로 전환한다. 서버가 테마를 추측할 때 생기는 hydration 불일치를 방지한다.

## 세 채널 SEO

SEO 레이어는 플러그인이 아니라 모듈(`lib/seo.ts`)이다. 모든 페이지가 `generateMetadata()`를 통해 호출해서 로케일별 title, description, canonical URL, hreflang 대체 링크, Open Graph 태그를 생성한다.

JSON-LD가 다섯 엔티티 타입을 커버한다: `WebSite`, `Person`, `BlogPosting`, `SoftwareApplication`, `BreadcrumbList`. XSS 방지를 위해 script 태그에서 꺾쇠괄호와 앰퍼샌드를 이스케이프한다.

사이트 루트의 `llms.txt`가 LLM에 사이트 구조의 일반 텍스트 지도를 제공한다. 아직 형성 중인 컨벤션이지만, 비용은 파일 하나다.

## 부재를 통한 보안

보안 헤더(CSP, HSTS, X-Frame-Options)는 `next.config.ts`에 있다. CSP `frame-src`는 YouTube와 Giscus만 허용한다. 나머지는 전부 차단.

진짜 보안은 존재하지 않는 것들이다. API 엔드포인트 없음. 폼 처리 없음. 사용자 입력 없음. 데이터베이스 연결 없음. 정적 사이트의 공격 면적은 CDN이다 — 그건 Vercel의 문제이지, 이 코드베이스의 문제가 아니다.
