## 시스템 구조

Vercel 엣지에 배포되는 정적 생성 Next.js 사이트. 서버, 데이터베이스, API 라우트 없음. 콘텐츠는 저장소의 MDX 파일 — 블로그 포스트, 프로젝트 페이지, About 페이지 각각이 로케일별 파일(EN, KO)을 가진다. 세 개의 prebuild 스크립트(이미지 복사, D2 다이어그램 컴파일, RSS 생성)가 Next.js 전에 실행되어 모든 페이지를 정적 HTML로 렌더링한다.

## SSG라는 제약 — 그리고 그것이 만드는 함정

모든 페이지가 빌드 타임에 사전 렌더링된다. 콘텐츠가 바뀌는 시점은 방문자가 올 때가 아니라 개발자가 커밋을 push할 때다. 빌드 출력이 제약을 증명한다: 모든 라우트가 Next.js 로그에서 `●`(static)을 보여야 한다. `ƒ`(dynamic)가 나타나면 뭔가 깨진 거다.

`next-intl`로 두 로케일을 추가하면 이 제약을 조용히 깨뜨리는 함정이 생긴다. 서버 컴포넌트가 현재 로케일을 알아야 하는데, 직관적인 호출인 `getLocale()`은 정적 생성 중 존재하지 않는 요청 시점 컨텍스트에 의존한다. 빌드는 성공하지만 런타임에 로케일이 틀린다. 수정은 페이지 params에서 `setRequestLocale(locale)` 호출. 한 줄, 하루짜리 디버깅.

두 번째 제약이 뒤따른다: async 서버 컴포넌트에서 `useTranslations()`를 직접 호출할 수 없다. 각 페이지를 async 외부 컴포넌트(params 처리, 데이터 fetch)와 sync 내부 컴포넌트(훅 호출)로 분리한다. i18n 설정도 동적 `import()` 대신 명시적 정적 import — 놓치기 쉬운 SSG 호환성 요구사항이다.

## 이중 콘텐츠 검증

Zod 스키마가 빌드 타임에 모든 프런트매터를 검증한다. 두 가지 규칙이 시스템을 형성한다:

**필수 필드는 빌드를 깨뜨린다.** title이 없는 프로젝트, date가 없는 블로그 포스트는 의도적으로 검증 실패한다. 에러가 `bun run build` 시점에 뜨지, 배포 이후에 뜨지 않는다. 구조적 문제를 빌드 타임에 잡으면 배포된 사이트가 항상 구조적으로 유효하다.

**선택 필드는 우아하게 저하된다.** 잘못된 video URL, 빠진 thumbnail, 유효하지 않은 이미지 경로는 `.catch(undefined)`로 조용히 드롭된다. 나쁜 선택 필드 하나가 페이지를 멈추지 않는다. 해당 요소 없이 렌더링된다.

이 이중 접근으로 콘텐츠 추가가 두렵지 않다. 필수 필드가 진짜 실수를 조기에 잡고, 선택 필드가 불완전함을 벌 없이 허용한다.

## 빌드 파이프라인과 하이드레이션 안전한 테마

세 개의 prebuild 스크립트가 Next.js 전에 실행된다:

**copy-images**가 프로젝트 이미지를 `content/`에서 `public/`로 복사하며 slug 형식과 이미지 확장자를 검증한다. **build-diagrams**가 D2 파일마다 두 번 컴파일 — 라이트 테마와 다크 테마 — 해서 쌍을 이루는 SVG를 생성한다. D2 CLI가 없으면 기존 SVG로 폴백. **generate-rss**가 영어 블로그 포스트로 RSS 피드를 만들고 모든 콘텐츠를 XML 이스케이프한다.

D2 다이어그램 컴포넌트가 하이드레이션 문제를 해결한다. 서버가 사용자의 테마 선호를 모르니 항상 라이트 변형을 렌더링한다. 클라이언트가 마운트 시 `useSyncExternalStore`로 올바른 테마로 전환한다. 서버가 테마를 추측할 때 생기는 하이드레이션 불일치를 방지한다 — 잠깐의 잘못된 콘텐츠가 React 하이드레이션 에러보다 낫다.
