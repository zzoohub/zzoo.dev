개발자 포트폴리오 대부분에는 조용한 문제가 있다. 사람은 읽을 수 있지만, 기계는 못 읽는다. 구글이 인덱싱은 해주겠지만, AI 답변 엔진이나 LLM이 이 사람이 누구인지, 뭘 만들었는지, 왜 중요한지를 구조적으로 이해할 방법이 없다.

이 사이트는 하나의 실험이다. 서버 없이, 데이터베이스 없이, 런타임 없이 — 순수 정적 HTML만으로 사람에게도 충분히 풍부하고, AI에게도 구조적이고, 자연스럽게 이중 언어인 사이트를 만들 수 있을까?

## 세 가지 발견 채널, 하나의 아키텍처

콘텐츠가 발견되는 경로를 세 가지로 설계했다:

1. **검색 엔진** — 정적 생성, 시맨틱 HTML, hreflang이 포함된 XML 사이트맵, canonical URL. 결과: Lighthouse 95+, 1초 미만 로딩.
2. **답변 엔진** — JSON-LD 스키마(`Person`, `WebSite`, `BlogPosting`, `SoftwareApplication`, `BreadcrumbList`)가 Perplexity와 Google AI Overview에 모든 페이지의 맥락을 기계가 읽을 수 있는 형태로 제공한다.
3. **생성형 AI** — 사이트 루트의 `llms.txt`와 깔끔한 MDX 콘텐츠가 LLM의 정확한 추출과 인용을 돕는다.

대부분의 사이트는 이 중 하나만 최적화한다. 이 사이트는 세 가지 모두를 일급 제약 조건으로 취급한다.

## 결과물이 아니라 결정을 담는 콘텐츠

프로젝트마다 세 개의 탭이 있다 — Overview, Design, Engineering — 각각 다른 독자를 위한 것이다. 채용 담당자는 Overview를 훑고, 프로덕트 씽커는 Design 탭에서 트레이드오프를 읽고, 동료 엔지니어는 Engineering에서 아키텍처 결정을 확인한다.

모든 프론트매터는 Zod로 검증한다. 콘텐츠는 로케일별 MDX 파일로 관리하며 한국어 번역이 없으면 영어로 자동 폴백한다. CMS도 데이터베이스도 없다 — git 저장소의 파일이 전부다.

## 처음부터 이중 언어

영어와 한국어는 나중에 붙인 게 아니다. 모든 페이지, 모든 프로젝트 탭, 모든 UI 문자열이 두 언어로 존재한다. 언어 전환은 버튼 하나 — 로케일이 두 개뿐인데 드롭다운을 쓸 이유가 없다. 번역이 없으면 영어로 자연스럽게 폴백하므로, 새 콘텐츠를 한 언어로 먼저 배포할 수 있다.

## 전부 오픈소스다

[코드베이스 전체](https://github.com/zzoohub/zzoo.dev)가 공개되어 있다. Next.js 16, React 19, TypeScript, Tailwind CSS v4로 만들었다. 포크해서 콘텐츠를 바꾸고 Vercel에 배포하면 된다. 가져갈 만한 건 콘텐츠 아키텍처다 — 세 탭 프로젝트 시스템, Zod 스키마, 세 채널 동시 SEO.
