This is a purely static site. No server-side rendering, no API routes, no database, no backend of any kind. Every page is pre-rendered at build time and served as static HTML from Vercel's global edge network. The architectural constraint is intentional: a minimal attack surface, near-instant load times, and a deployment model that is essentially "push to main and forget."

## Architecture: SSG-Only

The entire site is statically generated via Next.js 16's App Router. At build time, `generateStaticParams()` pre-renders every combination of locale and slug. The build output shows `●` (static) for every route — never `ƒ` (dynamic).

Why SSG over SSR? The content changes when the developer pushes a commit, not when a user visits. There is no user-specific data, no authentication, no form submissions. SSR would add complexity (cold starts, server costs, caching strategies) with zero benefit for this use case.

The tradeoff: any content change requires a rebuild. But with Vercel's GitHub integration, pushing to `main` triggers an automatic build and deploy in under two minutes.

## Server Components First

React 19 Server Components are the default. Most pages ship zero client-side JavaScript beyond the framework runtime. Client components are used sparingly — blog tag filter, theme toggle, mobile nav, clipboard copy, D2 diagram switcher — each a thin interactive wrapper around a server-rendered parent.

## Content System

Content is file-based MDX stored alongside the code. Blog posts, project pages, and the about page each have per-locale MDX files. Projects support a three-tab structure: Overview (required), Design (optional), and Engineering (optional).

Two design decisions worth noting:

**Locale fallback.** When a requested locale is missing, the system falls back to `en.mdx`. English is always the baseline, and Korean translations can be added incrementally without breaking any routes.

**Zod-validated frontmatter.** All frontmatter is parsed through Zod schemas. Required fields throw on missing values. Optional fields use `.catch(undefined)` — invalid optional data is silently dropped rather than breaking the build. This means a single malformed optional field never takes down the entire site.

**Reading time by script.** English uses word count (200 WPM), Korean uses character count (300 CPM for CJK). Detection is locale-based, not content-based — simple and predictable.

## i18n and the SSG Constraint

Internationalization uses `next-intl` with two locales: `en` and `ko`. UI strings live in JSON translation files; content translations are separate MDX files.

The critical SSG gotcha: server components must call `setRequestLocale(locale)` from the page's `params`, not `getLocale()`. The latter relies on request-time context that does not exist during static generation — it silently breaks the build.

For async server components that need translations, the workaround is splitting into an async outer component (resolves params, fetches data) and a sync inner component (calls `useTranslations()`). The translation config also uses explicit static imports rather than dynamic `import()` for SSG compatibility.

## D2 Diagram Pipeline

Architecture diagrams are authored in D2, a declarative diagramming language. At build time, each `.d2` file is compiled twice — once for light theme, once for dark — producing paired SVGs.

The `D2Diagram` component uses `useSyncExternalStore` for hydration safety: it renders the light theme on the server, then switches to the user's theme on client mount. This avoids hydration mismatches while keeping diagrams theme-aware.

```tsx
<img src="/diagrams/arch-light.svg" className="dark:hidden" />
<img src="/diagrams/arch-dark.svg" className="hidden dark:block" />
```

## SEO

Every page uses `generateMetadata()` to produce per-locale titles, descriptions, canonical URLs, hreflang alternates, and Open Graph tags. A centralized `lib/seo.ts` module provides builders for page metadata, canonical URLs, alternate links, and structured data (JSON-LD for website, person, article, project, breadcrumb, and FAQ types).

JSON-LD output is sanitized by escaping angle brackets and ampersands to prevent XSS in script tags. Additional SEO infrastructure includes an auto-generated XML sitemap with hreflang, an RSS feed built from English blog posts, robots.txt, and an llms.txt file for LLM discovery.

## Security

Security headers (CSP, HSTS, X-Frame-Options) are configured in `next.config.ts`. The CSP `frame-src` allows only YouTube and Giscus. The "no backend" constraint means the attack surface is minimal by design — no API endpoints, no form handling, no user input processing, no database connections.
