No server. No database. No API routes. No runtime. Every page is pre-rendered at build time and served as static HTML from Vercel's edge network. This is not a limitation — it is the central architectural decision. Zero backend means a minimal attack surface, near-instant load times, and a deployment model where pushing to `main` is the entire workflow.

## Why SSG, not SSR

The content changes when the developer pushes a commit, not when a visitor arrives. There is no user-specific data, no authentication, no form submissions. SSR would add cold starts, server costs, and caching strategies — complexity with zero benefit for this use case.

The tradeoff: every content change requires a rebuild. But Vercel's GitHub integration turns a push to `main` into a deployed site in under two minutes. For a content site with no real-time requirements, that is fast enough.

The build output confirms the constraint: every route shows `●` (static) in the Next.js build log. If `ƒ` (dynamic) ever appears, something is wrong.

## The SSG gotcha with next-intl

Internationalization uses `next-intl` with two locales. The implementation is straightforward — except for one trap that silently breaks static generation.

Server components must call `setRequestLocale(locale)` from the page's `params`. The intuitive alternative, `getLocale()`, relies on request-time context that does not exist during static generation. The build succeeds, but the locale is wrong at runtime.

A second constraint: async server components cannot call `useTranslations()` directly. The workaround splits each page into an async outer component (resolves params, fetches data) and a sync inner component (calls hooks). The translation config uses explicit static imports instead of dynamic `import()` — another SSG compatibility requirement.

## Content system: Zod as the safety net

Content is file-based MDX stored in the repository. No CMS, no database. Blog posts, project pages, and the about page each have per-locale MDX files. Projects support three tabs: Overview (always present), Design (optional), Engineering (optional).

Two Zod decisions matter:

**Graceful degradation for optional fields.** Required fields throw on missing values — a project without a title breaks the build intentionally. Optional fields use `.catch(undefined)`, silently dropping invalid data. A malformed `video` URL in frontmatter does not take down the entire site.

**Reading time by locale.** English uses word count at 200 WPM. Korean uses character count at 300 CPM for CJK text. The detection is locale-based, not content-based — simple, predictable, and correct for bilingual content where a single post is always in one language.

## D2 diagrams: build-time, dual-theme

Architecture diagrams use D2, a declarative diagramming language. Each `.d2` file is compiled twice at build time — once for light theme, once for dark — producing paired SVGs in `/public/diagrams/`.

The rendering component uses `useSyncExternalStore` for hydration safety. On the server, it always renders the light variant. On client mount, it switches to match the user's theme. This avoids the hydration mismatch that would occur if the server tried to guess the theme.

## Triple-channel SEO

The SEO implementation is not a plugin — it is a module (`lib/seo.ts`) that every page calls through `generateMetadata()`. Each page produces locale-specific titles, descriptions, canonical URLs, hreflang alternates, and Open Graph tags.

JSON-LD schemas cover five entity types: `WebSite`, `Person`, `BlogPosting`, `SoftwareApplication`, and `BreadcrumbList`. The output is sanitized — angle brackets and ampersands are escaped in the JSON-LD script tags to prevent XSS injection.

The `llms.txt` file at the site root gives LLMs a plain-text summary of the site's structure and content — a convention that is still emerging, but costs nothing to support.

## Security through absence

Security headers — CSP, HSTS, X-Frame-Options — are configured in `next.config.ts`. The CSP `frame-src` allows only YouTube (for project videos) and Giscus (for blog comments). Everything else is blocked.

But the real security story is what does not exist. No API endpoints. No form handling. No user input processing. No database connections. The attack surface of a static site is the CDN itself — and that is Vercel's problem, not mine.
