## SSG as the core constraint

Every page is pre-rendered at build time and served as static HTML from Vercel's edge. No server. No database. No API routes. This is not a budget decision — it is the architectural foundation.

Content changes when the developer pushes a commit, not when a visitor arrives. SSR would add cold starts, server costs, and caching complexity for zero benefit. The build output proves the constraint: every route shows `●` (static) in the Next.js log. If `ƒ` (dynamic) appears, something broke.

The trade-off: every content change triggers a rebuild. Vercel's GitHub integration deploys in under two minutes. For a content site with no real-time requirements, that is fast enough.

## The next-intl SSG Trap

Two locales (EN, KO) via `next-intl`. Straightforward — except for one trap that silently breaks static generation.

Server components must call `setRequestLocale(locale)` from page params. The intuitive alternative — `getLocale()` — depends on request-time context that does not exist during static generation. The build succeeds, but the locale is wrong at runtime. A one-line fix for a day-long debug.

A second constraint: async server components cannot call `useTranslations()` directly. The fix splits each page into an async outer component (resolves params, fetches data) and a sync inner component (calls hooks). The i18n config uses explicit static imports instead of dynamic `import()` — another SSG compatibility requirement that is easy to miss.

## Content System

File-based MDX in the repository. No CMS. Blog posts, project pages, and the about page each have per-locale files. Projects support three optional tabs: Overview (always present), Design, Engineering.

Two Zod decisions shaped the system:

**Required fields break the build.** A project without a title fails validation intentionally. The error surfaces at `bun run build`, not after deployment.

**Optional fields degrade gracefully.** Invalid optional data — a malformed video URL, a missing thumbnail — is silently dropped via `.catch(undefined)`. One bad field does not take down the site.

Reading time is locale-aware. English: word count at 200 WPM. Korean: character count at 300 CPM. Detection is locale-based, not content-based — each post is always in one language, so this is correct and simple.

## Build Pipeline

Three prebuild scripts run before Next.js:

1. **copy-images** — moves project images from `content/` to `public/`. Validates slug format. Whitelists image extensions.
2. **build-diagrams** — compiles each D2 file twice: light theme and dark theme, producing paired SVGs. Falls back gracefully if the D2 CLI is missing.
3. **generate-rss** — produces an RSS feed from English blog posts. XML-escapes all content.

The D2 rendering component uses `useSyncExternalStore` for hydration safety. Server always renders the light variant. Client switches to the user's theme on mount. This avoids the hydration mismatch that would occur if the server tried to guess the theme.

## Triple-Channel SEO

The SEO layer is a module (`lib/seo.ts`), not a plugin. Every page calls it through `generateMetadata()` to produce locale-specific titles, descriptions, canonical URLs, hreflang alternates, and Open Graph tags.

JSON-LD covers five entity types: `WebSite`, `Person`, `BlogPosting`, `SoftwareApplication`, `BreadcrumbList`. The output is sanitized — angle brackets and ampersands are escaped in script tags to prevent XSS.

An `llms.txt` file at the site root gives LLMs a plain-text map of the site. The convention is still emerging, but the cost is one file.

## Security Through Absence

Security headers — CSP, HSTS, X-Frame-Options — live in `next.config.ts`. The CSP `frame-src` allows only YouTube and Giscus. Everything else is blocked.

The real security is what does not exist. No API endpoints. No form processing. No user input. No database connections. The attack surface of a static site is the CDN — and that is Vercel's problem, not this codebase's.
