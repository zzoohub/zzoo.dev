## System Overview

A statically-generated Next.js site deployed on Vercel's edge. No server, no database, no API routes. Content lives as MDX files in the repository — blog posts, project pages, and the about page, each with per-locale variants (EN, KO). Three prebuild scripts (image copy, D2 diagram compilation, RSS generation) run before Next.js renders every page to static HTML.

## SSG as Constraint — And the Trap It Sets

Every page is pre-rendered at build time. Content changes when the developer pushes a commit, not when a visitor arrives. The build output proves the constraint: every route shows `●` (static) in the Next.js log. If `ƒ` (dynamic) appears, something broke.

Adding two locales via `next-intl` introduces a trap that silently breaks this constraint. Server components need the current locale, and the intuitive call — `getLocale()` — depends on request-time context that doesn't exist during static generation. The build succeeds, but the locale is wrong at runtime. The fix is `setRequestLocale(locale)` from page params. One line, one day of debugging.

A second constraint follows: async server components can't call `useTranslations()` directly. Each page splits into an async outer component (resolves params, fetches data) and a sync inner component (calls hooks). The i18n config uses explicit static imports instead of dynamic `import()` — another SSG compatibility requirement that's easy to miss.

## Two-Tier Content Validation

Zod schemas validate all frontmatter at build time. Two rules shape the system:

**Required fields break the build.** A project without a title, a blog post without a date — these fail validation intentionally. The error surfaces at `bun run build`, not after deployment. Catching structural problems at build time means the deployed site is always structurally valid.

**Optional fields degrade gracefully.** A malformed video URL, a missing thumbnail, an invalid image path — silently dropped via `.catch(undefined)`. One bad optional field doesn't take down the page. The page renders without that element.

This two-tier approach means content can be added without fear. Required fields catch real mistakes early. Optional fields allow imperfection without penalty.

## Build Pipeline and Hydration-Safe Theming

Three prebuild scripts run before Next.js:

**copy-images** moves project images from `content/` to `public/`, validating slug format and whitelisting image extensions. **build-diagrams** compiles each D2 file twice — light and dark theme — producing paired SVGs. Falls back gracefully if the D2 CLI is missing. **generate-rss** builds an RSS feed from English blog posts, XML-escaping all content.

The D2 diagram component solves a hydration problem. The server doesn't know the user's theme preference, so it always renders the light variant. The client switches to the correct theme on mount using `useSyncExternalStore`. This avoids the hydration mismatch that would occur if the server tried to guess — a flash of wrong content is better than a React hydration error.
