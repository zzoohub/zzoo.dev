## 문제

써봤던 모든 사이트 차단기에는 같은 치명적 결함이 있었습니다: 차단 해제 버튼. 시간 예약 차단, 비밀번호 보호, 쿨다운 타이머 — 이것들은 모두 유혹의 순간에 의지력이 작동한다고 가정합니다. 하지만 의지력이 무너지는 게 바로 그 순간입니다. 오전 9시에 트위터를 차단한 사람이 오후 2시에 "5분만 해제"를 누르는 사람과 같은 사람이니까요.

하나의 진실을 존중하는 차단기가 필요했습니다: **되돌릴 수 있으면, 되돌리게 된다**.

## 솔루션

Permanent Site Blocker는 단 하나의 급진적 설계 결정을 중심으로 만든 Chrome 확장 프로그램입니다 — **차단 해제가 없습니다**. 한번 차단된 사이트를 제거하는 유일한 방법은 확장 프로그램을 완전히 재설치하는 것뿐입니다. 비밀번호 우회도, 타이머도, 예외도 없습니다.

### 이중 레이어 차단 아키텍처

단일 차단 메커니즘에는 빈틈이 있습니다. 네트워크 수준 규칙은 캐시된 페이지를 잡지 못하고, 탭 모니터링은 미리 로드된 콘텐츠를 놓칩니다. 그래서 두 개의 독립적인 레이어를 사용합니다:

- **레이어 1: 네트워크 수준 차단** — Chrome의 `declarativeNetRequest` API로 요청이 서버에 도달하기 전에 차단합니다. 지연 시간 제로, 서비스 워커가 비활성 상태여도 작동합니다.
- **레이어 2: 탭 모니터링** — `tabs.onUpdated` 리스너가 빠져나가는 것들을 잡습니다: 캐시된 페이지, 미리 렌더링된 콘텐츠, 주소창에서의 직접 이동.

각 레이어는 독립적으로 실패합니다. 하나가 고장나도 다른 하나가 여전히 차단합니다.

### 유연한 패턴 매칭

단순한 도메인 차단이 대부분의 경우를 커버하지만, 실제 방해 요소는 더 미묘합니다. YouTube 자체가 문제가 아니라 YouTube Shorts가 문제입니다. 그래서 세 가지 차단 모드를 지원합니다:

- **도메인 차단** — `twitter.com`으로 전체 사이트 차단 (`www.` 변형 포함)
- **경로별 차단** — `youtube.com/shorts`로 Shorts만 차단하고 나머지 YouTube는 유지
- **정규식 패턴** — `youtube\.com.*[가-힣]`로 RE2 구문을 사용해 한국어 YouTube 콘텐츠 차단

### 프라이버시 우선, 의존성 제로

빌드 시스템 없음. npm 패키지 없음. 분석 도구 없음. 외부 서버 없음. 전체 확장 프로그램은 바닐라 JavaScript로 — 약 300줄의 코드가 정확히 한 가지를 잘 합니다. 차단된 사이트는 Chrome의 내장 스토리지 동기화를 통해 기기 간 동기화되며, 모든 데이터가 Chrome 자체 인프라 안에 머무릅니다.

## 핵심 설계 결정

- **차단 해제 없음은 의도적** — 되돌릴 수 없음이 핵심 가치 제안이지, 누락된 기능이 아닙니다. 모든 우회 메커니즘(비밀번호, 타이머, 쿨다운)은 의지력이 결국 열어버릴 문입니다.
- **바닐라 JS, 빌드 과정 없음** — ~300줄 확장 프로그램에 번들러는 가치 없이 복잡성만 더합니다. 코드베이스는 감사 가능하고 즉시 배포 가능합니다.
- **도메인당 2개 규칙** — 각 표준 항목은 `example.com`과 `www.example.com` 모두에 대한 규칙을 생성하여, `www`를 추가하거나 제거하는 흔한 우회를 차단합니다.
- **정규식 자동 무시** — 잘못된 정규식 패턴은 서비스 워커를 크래시시키지 않고 조용히 건너뜁니다. 하나의 잘못된 패턴이 나머지 모든 차단을 비활성화해서는 안 됩니다.

## 결과

- **이중 레이어 차단** — 알려진 우회 경로 제로
- **밀리초 이하 차단** — 네트워크 수준 규칙 인터셉션
- **기기 간 동기화** — Chrome 네이티브 스토리지 인프라 활용
- **외부 의존성 제로** — 빌드 과정 없음, npm 없음, 분석 도구 없음
- **~300줄**의 감사 가능한 바닐라 JavaScript
