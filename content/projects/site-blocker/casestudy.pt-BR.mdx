## Problema

Todo bloqueador de sites que eu tentei tinha a mesma falha fatal: um botão de desbloquear. Bloqueio programado, proteção por senha, timers de cooldown — todos assumem força de vontade no momento da tentação. Mas é exatamente quando a força de vontade falha. O procrastinador que bloqueou o Twitter às 9h é a mesma pessoa que clica em "desativar por 5 minutos" às 14h.

Eu precisava de um bloqueador que respeitasse uma verdade: **se você pode desfazer, você vai desfazer**.

## Solução

Permanent Site Blocker é uma extensão Chrome construída em torno de uma única decisão radical de design — **não existe desbloquear**. Uma vez que um site é bloqueado, a única forma de removê-lo é reinstalar a extensão inteira. Sem bypass de senha, sem timer, sem exceções.

### Arquitetura de Bloqueio em Duas Camadas

Um único mecanismo de bloqueio deixa brechas. Regras de nível de rede não pegam páginas em cache; monitoramento de abas perde conteúdo pré-carregado. Então a extensão usa duas camadas independentes:

- **Camada 1: Bloqueio em nível de rede** — Usa a API `declarativeNetRequest` do Chrome para matar requisições antes de chegarem ao servidor. Zero latência, funciona mesmo se o service worker estiver inativo.
- **Camada 2: Monitoramento de abas** — Um listener `tabs.onUpdated` captura qualquer coisa que passe: páginas em cache, conteúdo pré-renderizado, navegação direta pela barra de endereços.

Cada camada falha independentemente. Se uma quebra, a outra ainda bloqueia.

### Matching de Padrões Flexível

Bloqueio simples de domínio cobre a maioria dos casos, mas distração real é mais nuançada. YouTube em si não é o problema — YouTube Shorts é. Então a extensão suporta três modos de bloqueio:

- **Bloqueio de domínio** — `twitter.com` bloqueia o site inteiro (incluindo variantes `www.`)
- **Bloqueio por caminho** — `youtube.com/shorts` bloqueia apenas Shorts mantendo o resto do YouTube acessível
- **Padrões regex** — `youtube\.com.*[가-힣]` bloqueia conteúdo coreano do YouTube usando sintaxe RE2

### Privacy-First, Zero Dependências

Sem sistema de build. Sem pacotes npm. Sem analytics. Sem servidores externos. A extensão inteira é JavaScript puro — cerca de 300 linhas de código que fazem exatamente uma coisa bem. Sites bloqueados sincronizam entre dispositivos pelo sync de armazenamento built-in do Chrome, mantendo todos os dados dentro da infraestrutura do próprio Chrome.

## Decisões de Design Chave

- **Não desbloquear é intencional** — Irreversibilidade é a proposta de valor central, não uma feature faltando. Todo mecanismo de bypass (senhas, timers, cooldowns) é uma porta que a força de vontade eventualmente vai abrir.
- **JS puro, sem build step** — Para uma extensão de ~300 linhas, um bundler adiciona complexidade sem valor. A codebase permanece auditável e deployável instantaneamente.
- **2 regras por domínio** — Cada entrada padrão gera regras tanto para `example.com` quanto `www.example.com`, eliminando um bypass comum onde usuários simplesmente adicionam ou removem `www.`.
- **Falha silenciosa de regex** — Padrões regex inválidos são ignorados silenciosamente em vez de crashar o service worker. Um padrão ruim não deveria desativar o bloqueio de todo o resto.

## Resultados

- **Bloqueio em duas camadas** sem caminhos de bypass conhecidos
- **Bloqueio sub-milissegundo** via interceptação de regras em nível de rede
- **Sync cross-device** pela infraestrutura nativa de armazenamento do Chrome
- **Zero dependências externas** — sem build step, sem npm, sem analytics
- **~300 linhas** de JavaScript puro auditável
