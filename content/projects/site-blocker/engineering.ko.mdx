## 시스템 구조

세 개의 격리된 런타임 컨텍스트 — 팝업, 서비스 워커, 차단 페이지 — 가 `chrome.storage.sync`만으로 통신한다. Chrome이 격리를 강제한다. 컨텍스트 간 직접 함수 호출은 불가능하다. 팝업이 스토리지에 쓰면 서비스 워커가 변경을 감지해 차단 규칙을 관리하고, 차단 페이지가 리다이렉트 시 표시된다. 전체 코드는 바닐라 JavaScript ~860줄, 파일 4개.

## 이중 레이어 차단

차단 레이어 하나로는 부족하다. Chrome의 `declarativeNetRequest`가 네트워크 수준에서 요청을 죽인다 — 빠르고, 안정적이고, 서비스 워커 생명주기와 독립적이다. 하지만 캐시 페이지, 프리렌더링된 콘텐츠, 특정 주소창 이동은 네트워크를 안 타서 빠져나간다.

두 번째 레이어인 `tabs.onUpdated` 리스너가 모든 URL 변경을 차단 목록과 대조하고, 일치하면 `blocked.html`로 리다이렉트한다. 두 독립 시스템이 서로의 사각지대를 커버한다.

트레이드오프: 네트워크 차단은 Chrome 기본 오류 페이지를, 탭 차단은 커스텀 동기 부여 페이지를 보여준다. 사소한 UX 불일치지만, 빈틈 없는 커버리지를 위해 수용한다. 같은 URL에 두 레이어가 모두 작동해도 무해하다 — 네트워크가 로딩을 막고, 탭 리다이렉트는 어쨌든 로드된 경우에만 작동한다.

모든 규칙이 dynamic rule(재시작 후 유지)과 session rule(ID 오프셋 +1000) 양쪽으로 등록된다. 한 메커니즘이 조용히 실패해도 나머지가 차단한다.

## 증분이 아닌 전체 재생성

차단 목록이 바뀌면 서비스 워커가 모든 기존 규칙을 삭제하고 전체 세트를 처음부터 재생성한다. 표준 항목은 규칙 2개(도메인 + `www.` 변형), 정규식 항목은 `regexFilter` 규칙 1개.

증분 업데이트는 규칙 ID와 사이트의 매핑 추적, 삭제 시 ID 재사용 처리, 스토리지와 규칙 엔진 간 상태 diff가 필요하다. 코드가 많아지고 엣지 케이스가 늘어나지만 성능 이득은 미미하다. 일반적인 사용자가 50개 미만의 사이트를 차단하고 — 전체 재생성은 100ms 이하. 영리함보다 단순함.

## 인코딩 문제

사용자가 `youtube\.com.*[가-힣]`로 한국어 YouTube 콘텐츠를 차단한다. 브라우저가 URL에서 한글을 퍼센트 인코딩한다: `%ED%95%9C%EA%B8%80`. 디코딩 없이는 정규식이 매칭되지 않는다.

탭 모니터링 레이어가 `decodeURIComponent`로 URL을 디코딩한 뒤 매칭한다. 네트워크 레이어는 Chrome RE2 엔진으로 원본 URL에서 작동한다 — RE2가 인코딩을 다르게 처리한다. 양쪽 경로 모두 필요하다. 각각이 상대가 처리 못하는 부분을 맡는다.

이 디테일 하나 — 매칭 전 디코딩 — 가 나머지 매칭 로직 전체보다 디버깅 시간이 더 걸렸다. 수정은 한 줄. 찾는 데 하루.
