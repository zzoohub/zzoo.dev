## 아키텍처

세 개의 격리된 런타임 컨텍스트가 `chrome.storage.sync`만으로 통신한다. Chrome이 격리를 강제한다 — 컨텍스트 간 직접 함수 호출은 불가능하다.

```
Popup (popup.js)           → 사이트 추가, 입력 검증
Service Worker (background.js) → 차단 규칙 관리, 탭 모니터링
Blocked Page (blocked.js)  → 차단 시 동기 부여 메시지 표시
```

팝업이 스토리지에 쓰면 서비스 워커가 `storage.onChanged` 이벤트로 반응한다. 메시지 패싱도 없고 공유 상태도 없다. 전체 코드는 4개 파일 ~860줄 — 프레임워크 없음, 빌드 과정 없음, TypeScript 없음.

## 이중 레이어 차단

레이어 하나론 부족하다. Chrome의 `declarativeNetRequest`는 서버에 도달하기 전에 요청을 차단한다 — 빠르고 안정적이다. 하지만 캐시 페이지, 프리렌더링된 콘텐츠, 특정 주소창 이동은 네트워크를 타지 않아서 빠져나간다.

두 번째 레이어인 `tabs.onUpdated`가 모든 URL 변경을 차단 목록과 대조하고, 일치하면 `blocked.html`로 리다이렉트한다. 두 독립 시스템이 서로의 빈틈을 메운다.

트레이드오프: 네트워크 차단은 Chrome 기본 오류 페이지를, 탭 차단은 커스텀 동기 부여 페이지를 보여준다. 사소한 UX 불일치지만 빈틈 제로를 위해 수용한다.

같은 URL에 두 레이어가 모두 작동해도 무해하다. 네트워크가 로딩을 막고, 탭 리다이렉트는 어쨌든 로드된 경우에만 작동한다.

## 증분이 아닌 전체 재생성

차단 목록이 바뀌면 서비스 워커가 모든 규칙을 삭제하고 처음부터 전체 세트를 재생성한다. 표준 항목은 규칙 2개씩(도메인 + `www.` 변형), 정규식 항목은 `regexFilter` 규칙 1개. 모든 규칙은 dynamic rule(재시작 후에도 유지)과 session rule(ID 오프셋 +1000) 양쪽으로 등록해 심층 방어를 제공한다.

증분 업데이트는 사이트별 규칙 ID 추적, 재사용 처리, 스토리지와 규칙 엔진 간 상태 diff가 필요하다. 코드가 많아지고 엣지 케이스도 늘어나지만 이득은 미미하다. 일반적인 사용자는 50개 미만의 항목을 가지고 있고 — 전체 재생성은 100ms 이하다.

## 인코딩 문제

사용자가 `youtube\.com.*[가-힣]`로 한국어 YouTube 콘텐츠를 차단하면, 브라우저가 URL에서 한글을 퍼센트 인코딩한다. 디코딩 없이는 정규식이 `%ED%95%9C%EA%B8%80`에 매칭되지 않는다.

탭 모니터링 레이어는 `decodeURIComponent`로 URL을 디코딩한 뒤 매칭한다. 네트워크 레이어는 Chrome의 RE2 엔진으로 원본 URL에서 작동한다. 양쪽 경로 모두 필요하다 — 각각이 상대가 처리 못하는 부분을 맡는다. 이 디테일 하나 — 매칭 전 디코딩 — 가 나머지 매칭 로직 전체보다 디버깅 시간이 더 걸렸다.

## 핵심 결정

**빌드 시스템 없음.** 3개 파일, ~860줄, 바닐라 JavaScript. 소스가 작성된 그대로 배포된다. `tabs` 권한을 가진 보안 민감한 확장 프로그램에서 감사 가능성은 개발자 편의보다 중요하다.

**`storage.local` 대신 `storage.sync`.** 공짜로 기기 간 동기화. 100KB 쿼터로 ~2,000개 항목 지원 — 10~50개 사이트를 차단하는 도구에서 절대 한계에 닿지 않는다.

**Manifest V3.** 새 Chrome Web Store 제출에 필수다. 서비스 워커 모델이 stateless 설계를 강제한다 — Chrome이 언제든 워커를 죽일 수 있다. 재시작 시 스토리지를 읽고 규칙을 다시 만든다. 결과적으로 맞는 제약이었다: 잃을 인메모리 상태가 없다.
