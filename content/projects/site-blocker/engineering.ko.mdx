## 아키텍처

Permanent Site Blocker는 이벤트 기반 내부 통신을 사용하는 단일 프로세스 Chrome 확장 프로그램입니다. 서버 없음, 데이터베이스 없음, 네트워크 인프라 없음. 전체 시스템이 Chrome의 확장 프로그램 샌드박스 안에서 실행됩니다 -- 직접 함수 호출 대신 Chrome의 이벤트 시스템을 통해 통신하는 세 개의 격리된 런타임 컨텍스트로 구성됩니다.

```
Popup (popup.html/js)          → 사용자 입력, 유효성 검사, 목록 렌더링
Service Worker (background.js) → 규칙 관리, 탭 모니터링, 마이그레이션
Blocked Page (blocked.html/js) → URL 표시, 파비콘, 동기 부여 메시지
```

세 컨텍스트는 서로의 함수를 직접 호출할 수 없습니다. 팝업은 `chrome.storage.sync`에 쓰고, 서비스 워커는 `storage.onChanged` 이벤트를 수신하여 반응합니다. 이 격리는 아키텍처적 선택이 아니라 Chrome의 보안 모델입니다 -- 하지만 결과적으로 깔끔한 관심사 분리를 만들어냅니다.

전체 코드베이스: 3개 파일에 걸친 ~300줄의 바닐라 JavaScript. 프레임워크 없음, 빌드 과정 없음, 모듈 시스템 없음, TypeScript 없음. 전체 시스템을 10분이면 읽을 수 있습니다.

## 이중 레이어 차단

단일 차단 메커니즘에는 빈틈이 있습니다. 확장 프로그램은 서로가 놓치는 것을 잡아주는 두 개의 독립적인 레이어를 사용합니다.

### 레이어 1: declarativeNetRequest (네트워크 수준)

Chrome의 `declarativeNetRequest` API는 브라우저의 네트워크 스택에 직접 차단 규칙을 등록합니다. 요청이 서버에 도달하기 전에 차단됩니다 -- 지연 시간 제로, 페이지 로드 없음, 콘텐츠 깜빡임 없음. 이 레이어가 대부분의 차단을 처리합니다.

규칙은 `main_frame` 리소스 타입만 대상으로 합니다. 차단된 도메인의 이미지, 스크립트, iframe이 다른 페이지에 나타나는 경우에는 차단하지 않습니다. 사용자가 차단하지 않은 사이트를 고장내지 않기 위해서입니다.

핵심 장점: declarativeNetRequest 규칙은 서비스 워커와 독립적으로 유지됩니다. Chrome이 리소스 절약을 위해 서비스 워커를 종료해도 차단은 중단 없이 계속됩니다.

### 레이어 2: tabs.onUpdated (탭 수준 폴백)

`tabs.onUpdated` 리스너가 모든 탭 URL 변경을 차단 목록과 대조합니다. 일치하면 탭을 `blocked.html`로 리다이렉트합니다.

이 레이어가 존재하는 이유는 declarativeNetRequest에 알려진 빈틈이 있기 때문입니다:
- **캐시된 페이지** -- 브라우저가 네트워크 요청 없이 캐시에서 페이지를 제공할 수 있음
- **프리렌더링된 페이지** -- Chrome의 투기적 프리렌더링이 네트워크 규칙을 우회할 수 있음
- **브라우저 내부 네비게이션** -- 특정 주소창 이동이 네트워크 수준 규칙을 트리거하지 않음

### 왜 두 레이어 모두 필요한가

중복은 의도적입니다. 같은 URL에 두 레이어가 모두 실행되는 것은 무해합니다 -- 네트워크 차단이 페이지 로드를 방지하고, 탭 리다이렉트는 페이지가 어떻게든 로드된 경우에만 트리거됩니다. 각 레이어는 독립적으로 실패합니다: declarativeNetRequest 장애가 탭 모니터링에 영향을 주지 않으며, 그 반대도 마찬가지입니다.

사소한 UX 트레이드오프: 네트워크 수준 차단은 Chrome의 기본 오류 페이지를 보여주고, 탭 수준 차단은 동기 부여 메시지가 포함된 커스텀 `blocked.html`을 보여줍니다. 이 불일치는 커버리지 완성도를 위해 수용합니다.

## 규칙 생성

사이트가 추가되면 팝업이 `chrome.storage.sync`에 쓰고, `storage.onChanged` 이벤트가 발생합니다. 서비스 워커가 이 이벤트를 받아 `updateBlockRules()`를 실행합니다 -- 모든 차단 규칙의 전체 재생성입니다.

**표준 항목**은 각각 2개 규칙을 생성합니다: 정확한 도메인용 하나와 `www.` 변형용 하나. `twitter.com`을 차단하면 `www.twitter.com`도 자동으로 차단됩니다. 사용자 조작 불필요.

**정규식 항목**은 RE2 구문의 `regexFilter`를 사용하여 각각 1개 규칙을 생성합니다. Chrome이 적용 전에 RE2 패턴을 서버 측에서 검증합니다.

**이중 등록**: 모든 규칙은 dynamic rule(브라우저 재시작 간 유지)과 session rule(브라우저 종료 시 삭제, ID 오프셋 +1000)로 모두 등록됩니다. 이것이 심층 방어를 제공합니다 -- 한 규칙 타입이 적용에 실패해도 다른 하나가 여전히 차단합니다.

### 증분 업데이트 대신 전체 재생성

모든 스토리지 변경 시 `updateBlockRules()`는 기존 규칙을 모두 삭제하고 전체 규칙 세트를 처음부터 재생성합니다. 이것은 의도적입니다.

대안인 증분 규칙 업데이트는 사이트에 매핑된 규칙 ID 추적, ID 재사용 처리, 이전 상태와 새 상태의 diff가 필요합니다. 미미한 성능 이득을 위한 더 복잡한 코드와 더 많은 버그 표면. 전체 재생성 접근법은 O(n)으로, n은 차단 사이트 수입니다. 5,000개 규칙 상한 기준으로 최악의 경우 ~10,000개 규칙 연산. Chrome의 처리 용량 내에 충분합니다.

## 데이터 스토리지

**`storage.local` 대신 `chrome.storage.sync`를 선택한 이유**: 제로 인프라의 기기 간 동기화. Chrome이 동기화 프로토콜을 처리합니다. 사용자가 Chrome에 로그인하면 모든 브라우저에 차단 목록이 자동으로 나타납니다. 계정 없음, 서버 없음, 설정 없음.

트레이드오프는 용량입니다: `storage.sync`는 100KB 제한 vs `storage.local`의 10MB. 각 차단 사이트 항목은 ~50바이트로, 쿼터 내에서 약 2,000개 항목을 지원합니다. 사용자가 일반적으로 10-50개 사이트를 차단하는 도구에서 이는 충분합니다.

**데이터 스키마:**

```json
[
  { "url": "twitter.com", "isRegex": false },
  { "url": "youtube.com/shorts", "isRegex": false },
  { "url": "youtube\\.com.*[가-힣]", "isRegex": true }
]
```

서비스 워커는 설계상 stateless입니다. Chrome이 리소스 절약을 위해 언제든 종료할 수 있습니다. 이벤트가 발생하면(스토리지 변경, 탭 업데이트) Chrome이 워커를 재시작하고, 워커는 스토리지에서 읽어 모든 상태를 재구성합니다. 인메모리 영속성이 필요 없습니다.

## URL 매칭

`isUrlBlocked()` 함수는 모든 `tabs.onUpdated` 이벤트에서 실행되어, 각 URL을 전체 차단 목록과 대조합니다.

**표준 모드:**
1. URL 호스트네임에서 `www.` 접두사 제거
2. 저장된 호스트네임과 매칭 (서브도메인 지원)
3. 경로가 지정된 경우, URL 경로가 해당 경로로 시작하는지 확인

**정규식 모드:**
1. 전체 URL을 먼저 디코딩 (인코딩된 문자 처리, 예: `%ED%95%9C%EA%B8%80` → `한글`)
2. 디코딩된 URL에 대해 대소문자 무시 RegExp 매칭 적용
3. 잘못된 정규식 패턴은 try-catch로 조용히 건너뜀 -- 하나의 잘못된 패턴이 나머지 모든 차단을 비활성화하지 않음

URL 디코딩 단계가 중요합니다. 사용자가 한국어 콘텐츠를 대상으로 `youtube\.com.*[가-힣]`를 차단하면, 브라우저가 URL 바에서 해당 문자를 인코딩할 수 있습니다. 디코딩 없이는 정규식이 절대 매칭되지 않습니다.

## 레거시 마이그레이션

1.3 이전 버전은 차단 사이트를 일반 문자열 배열로 저장했습니다: `["twitter.com", "youtube.com"]`. 현재 형식은 정규식 플래그가 있는 객체를 사용합니다: `[{ url: "twitter.com", isRegex: false }]`.

마이그레이션은 `runtime.onInstalled`에서 자동으로 실행됩니다 -- 새 설치와 업데이트 모두 커버:

1. 스토리지에서 `blockedSites` 읽기
2. 요소 중 일반 문자열이 있는지 확인 (레거시 형식)
3. 있다면, 각 문자열을 `{ url: string, isRegex: false }`로 매핑
4. 마이그레이션된 배열을 다시 쓰기
5. 모든 규칙 재생성

새 형식은 이전 형식의 엄격한 상위 집합입니다. 데이터 손실 없음, 사용자 조작 불필요, 롤백 불필요.

## 보안

**Content Security Policy**는 모든 스크립트 소스를 `'self'`만으로 제한합니다. 이미지 소스는 `self`, `https://*.google.com`, `https://*.gstatic.com`, `data:` -- 파비콘에 필요한 최소한만 허용합니다. 인라인 스크립트 없음, 동적 코드 실행 없음, 원격 코드 로딩 없음.

**주입 표면 없음**: 정규식 패턴은 `new RegExp()` (안전한 생성자)와 declarativeNetRequest의 `regexFilter` (Chrome이 RE2 구문 검증)로 전달됩니다. 사용자 제공 문자열은 절대 코드로 실행되지 않습니다.

**입력 유효성 검사**: 팝업은 비어있지 않은 입력을 검증하고 스토리지 전에 `new RegExp()` try-catch로 정규식 패턴을 확인합니다. 서비스 워커는 자체 try-catch로 정규식 생성을 래핑하여 손상된 패턴이 워커를 크래시시키는 것을 방지합니다.

**전송 중 데이터**: 사용자 데이터는 외부 서버로 전송되지 않습니다. 유일한 아웃바운드 요청은 도메인 이름으로 Google 파비콘 서비스에 보내는 HTTPS GET입니다 -- 사용자 식별 불가.

## 아키텍처 결정

**빌드 시스템이나 프레임워크 없음** -- 입력 하나와 목록 하나인 320px 팝업에 React는 필요 없습니다. 총 ~300줄의 JavaScript 3개 파일에 TypeScript, 번들러, 모듈 시스템은 필요 없습니다. 전체 확장 프로그램 소스가 프로덕션에서 사람이 읽을 수 있습니다. 프라이버시 중심 확장 프로그램에서 이것은 기능입니다: 사용자가 코드가 정확히 무엇을 하는지 검사할 수 있습니다.

**MV2 대신 Manifest V3** -- MV3은 새로운 Chrome Web Store 제출에 필수입니다. MV2는 더 이상 사용되지 않습니다. 서비스 워커 모델(영구 백그라운드 페이지 대체)은 stateless 설계를 요구하는 대가로 더 효율적인 리소스 사용을 제공합니다 -- 결과적으로 올바른 제약이 되었습니다.

**네이티브 앱이나 VPN 대신 Chrome 확장 프로그램** -- Chrome 확장 프로그램은 VPN 기반 차단의 배터리 소모, 네트워크 오버헤드, 프라이버시 문제를 피합니다. 트레이드오프: Chrome/Chromium 생태계에 고정되고, 사용자가 확장 프로그램을 제거할 수 있습니다. 둘 다 수용된 제한입니다.

## 에러 처리

| 시나리오 | 대응 |
|----------|------|
| 잘못된 정규식 입력 | 팝업이 `new RegExp()` try-catch로 검증; 오류 표시; 제출 차단 |
| 스토리지의 손상된 정규식 | `isUrlBlocked()`가 해당 패턴을 조용히 건너뜀 |
| 스토리지 읽기 실패 | 스토리지 복구 전까지 차단 없음 |
| declarativeNetRequest 실패 | 탭 모니터링 레이어가 리다이렉트로 여전히 차단 |
| 작업 중 서비스 워커 종료 | 다음 이벤트가 전체 규칙 재생성 트리거 |
| Google Favicon 서비스 다운 | 깨진 이미지 아이콘; 기능적 영향 없음 |

이중 레이어 아키텍처 자체가 복원력 전략입니다. 두 차단 레이어는 독립적입니다 -- 서로에게 의존하지 않습니다.

## 성능

**규칙 적용**: `updateBlockRules()`는 스토리지 변경당 O(n). 모든 규칙 삭제 후 처음부터 재생성. 일반적인 사용량(100개 미만 항목)에서 밀리초 내 완료.

**탭 모니터링**: `isUrlBlocked()`는 모든 `tabs.onUpdated` 이벤트에서 전체 차단 목록을 순회. 네비게이션당 O(n). 작은 n과 빠른 문자열 연산으로 체감 불가.

**메모리**: 서비스 워커는 영구적 인메모리 상태를 유지하지 않음. Chrome이 워커의 라이프사이클과 독립적으로 declarativeNetRequest 규칙을 관리.

**팝업 렌더링**: 열릴 때 차단 목록을 동기적으로 렌더링. 수백 개 항목에서 지연이 있을 수 있지만, 대부분의 사용자는 100개 미만. 사용 사례에 수용 가능.

## 배포

CI/CD 파이프라인 없음. 빌드 과정 없음. 확장 프로그램은 HTML, CSS, JavaScript, 아이콘 파일의 정적 번들입니다.

```
개발자가 .zip 빌드
  → Chrome Web Store Developer Dashboard에 업로드
  → Chrome Web Store 심사
  → 게시
  → Chrome이 설치된 확장 프로그램 자동 업데이트 (수 시간 내)
```

개발 환경에서는 Chrome의 `chrome://extensions` 개발자 모드를 사용해 파일 시스템에서 직접 언패킹된 확장 프로그램을 로드합니다. 프로덕션 아티팩트는 .zip 파일입니다. 롤백은 이전 버전을 새 버전으로 업로드하는 것입니다.

단순함이 핵심입니다. ~300줄의 코드, 10개 파일, 제로 의존성, 제로 빌드 도구. 전체 확장 프로그램이 하나의 코드 리뷰에 들어갑니다.
