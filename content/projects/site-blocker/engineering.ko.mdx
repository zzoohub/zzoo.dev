## 아키텍처

확장 프로그램은 Chrome의 샌드박스 안에서만 실행된다 -- 서버 없음, 데이터베이스 없음, 외부 네트워크 없음. 세 개의 격리된 런타임 컨텍스트가 직접 함수 호출이 아닌 Chrome 이벤트 시스템으로 통신한다.

```
Popup (popup.html/js)          → 사용자 입력, 유효성 검사, 목록 렌더링
Service Worker (background.js) → 규칙 관리, 탭 모니터링
Blocked Page (blocked.html/js) → URL 표시, 파비콘, 동기 부여 메시지
```

팝업이 `chrome.storage.sync`에 쓰면, 서비스 워커가 `storage.onChanged` 이벤트를 수신해서 반응한다. 이 격리는 아키텍처적 선택이 아니라 Chrome의 보안 모델인데, 결과적으로 깔끔한 관심사 분리를 만들어낸다.

전체 코드: 3개 파일, ~300줄 바닐라 JavaScript. 프레임워크 없음, 빌드 과정 없음, TypeScript 없음. 10분이면 전부 읽을 수 있다.

## 이중 레이어 차단

단일 차단 메커니즘에는 빈틈이 있다. 두 개의 독립 레이어가 서로의 빈틈을 메운다.

**레이어 1: declarativeNetRequest**는 Chrome 네트워크 스택에 직접 차단 규칙을 등록한다. 요청이 서버에 도달하기 전에 차단된다 -- 지연 제로, 페이지 로드 없음, 콘텐츠 깜빡임 없음. `main_frame`만 대상이므로, 다른 페이지에 포함된 차단 도메인의 이미지나 스크립트는 정상 로드된다. 핵심 속성: 서비스 워커와 독립적으로 유지된다. Chrome이 워커를 종료해도 차단은 계속된다.

**레이어 2: tabs.onUpdated**는 모든 탭 URL 변경을 차단 목록과 대조한다. 일치하면 `blocked.html`로 리다이렉트한다. declarativeNetRequest에 알려진 빈틈이 있어서 필요하다: 네트워크 요청 없이 제공되는 캐시 페이지, Chrome의 투기적 프리렌더링, 네트워크 규칙을 건너뛰는 주소창 이동 등.

같은 URL에 두 레이어가 모두 실행되는 건 무해하다 -- 네트워크 차단이 로드를 막고, 탭 리다이렉트는 어떻게든 로드된 경우에만 작동한다. 사소한 UX 불일치가 있다: 네트워크 차단은 Chrome 기본 오류 페이지를, 탭 차단은 커스텀 동기 부여 페이지를 보여준다. 커버리지 완성도를 위해 수용한다.

## 규칙 생성: 증분이 아닌 전체 재생성

사이트가 추가되면 팝업이 `chrome.storage.sync`에 쓰고, `storage.onChanged`가 발생한다. 서비스 워커가 `updateBlockRules()`로 모든 차단 규칙을 처음부터 재생성한다.

표준 항목은 각각 2개 규칙 (정확한 도메인 + `www.` 변형), 정규식 항목은 RE2 구문의 `regexFilter`로 1개 규칙을 생성한다. 모든 규칙은 dynamic rule(재시작 간 유지)과 session rule(ID 오프셋 +1000) 양쪽으로 등록해 심층 방어를 제공한다.

증분 업데이트 대신 전체 재생성을 택한 이유가 있다. 증분 업데이트는 사이트에 매핑된 규칙 ID 추적, ID 재사용 처리, 상태 diff가 필요하다. 미미한 성능 이득을 위한 복잡성과 버그 표면 증가다. 전체 재생성은 O(n)이고, Chrome의 5,000개 규칙 상한 기준으로 충분히 처리 가능하다.

## URL 매칭과 인코딩 문제

`isUrlBlocked()`는 모든 `tabs.onUpdated` 이벤트에서 실행된다. 표준 모드는 호스트네임에서 `www.`를 제거하고, 서브도메인 지원으로 매칭하고, 경로 접두사를 확인한다. 정규식 모드는 대소문자 무시 RegExp 매칭을 적용한다. 잘못된 패턴은 조용히 건너뛴다 -- 하나의 잘못된 정규식이 나머지 차단을 비활성화하지 않는다.

흥미로운 엣지 케이스가 URL 디코딩이다. 사용자가 한국어 콘텐츠를 대상으로 `youtube\.com.*[가-힣]`를 차단하면, 브라우저가 URL에서 해당 문자를 인코딩한다. 디코딩 없이는 `%ED%95%9C%EA%B8%80` 같은 인코딩 형태에 정규식이 매칭되지 않는다. 탭 레이어는 URL을 디코딩한 후 매칭하고, 네트워크 레이어는 RE2로 원본 URL에서 작동한다. 각각이 상대방이 처리 못하는 부분을 담당하기 때문에 양쪽 경로 모두 필요하다.

## 핵심 결정

**빌드 시스템 없음** -- 입력 하나, 목록 하나인 320px 팝업에 React는 필요 없다. ~300줄 JavaScript 3개 파일에 TypeScript나 번들러도 필요 없다. 프로덕션 소스가 그대로 사람이 읽을 수 있다. 프라이버시 중심 확장 프로그램에서 이것 자체가 기능이다.

**storage.local 대신 chrome.storage.sync** -- 제로 인프라로 기기 간 동기화. 트레이드오프는 용량: 100KB 제한 vs 10MB. 항목당 ~50바이트로 약 2,000개를 지원한다. 10-50개 사이트를 차단하는 도구에 충분하다.

**Manifest V3** -- 새로운 Chrome Web Store 제출에 필수다. 서비스 워커 모델(영구 백그라운드 페이지 대체)이 stateless 설계를 요구하는데, 결과적으로 올바른 제약이 되었다. Chrome이 워커를 종료해도, 재시작 시 스토리지에서 읽어 상태를 재구성한다. 인메모리 영속성이 필요 없다.

**네이티브 앱이나 VPN 대신 Chrome 확장 프로그램** -- VPN 차단의 배터리 소모, 네트워크 오버헤드, 프라이버시 문제를 피한다. 트레이드오프: Chromium 생태계에 고정되고, 사용자가 확장 프로그램을 제거할 수 있다. 단순하고 투명한 도구를 지향하는 한, 수용 가능한 제한이다.
