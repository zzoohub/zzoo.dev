## 아키텍처

Permanent Site Blocker는 Chrome의 확장 프로그램 샌드박스 안에서만 실행됩니다 -- 서버 없음, 데이터베이스 없음, 네트워크 없음. 세 개의 격리된 런타임 컨텍스트가 직접 함수 호출 대신 Chrome의 이벤트 시스템을 통해 통신합니다.

```
Popup (popup.html/js)          → 사용자 입력, 유효성 검사, 목록 렌더링
Service Worker (background.js) → 규칙 관리, 탭 모니터링
Blocked Page (blocked.html/js) → URL 표시, 파비콘, 동기 부여 메시지
```

팝업은 `chrome.storage.sync`에 쓰고, 서비스 워커는 `storage.onChanged` 이벤트를 수신하여 반응합니다. 이 격리는 아키텍처적 선택이 아니라 Chrome의 보안 모델입니다 -- 하지만 결과적으로 깔끔한 관심사 분리를 만들어냅니다.

전체 코드베이스: 3개 파일에 걸친 ~300줄의 바닐라 JavaScript. 프레임워크 없음, 빌드 과정 없음, TypeScript 없음. 전체 시스템을 10분이면 읽을 수 있습니다.

## 이중 레이어 차단

단일 차단 메커니즘에는 빈틈이 있습니다. 확장 프로그램은 서로가 놓치는 것을 잡아주는 두 개의 독립적인 레이어를 사용합니다.

### 레이어 1: declarativeNetRequest (네트워크 수준)

Chrome의 `declarativeNetRequest` API는 브라우저의 네트워크 스택에 직접 차단 규칙을 등록합니다. 요청이 서버에 도달하기 전에 차단됩니다 -- 지연 시간 제로, 페이지 로드 없음, 콘텐츠 깜빡임 없음.

규칙은 `main_frame`만 대상으로 하므로, 차단된 도메인의 이미지나 스크립트가 다른 페이지에 나타나는 경우에는 영향을 주지 않습니다. 핵심 장점: 이 규칙들은 서비스 워커와 독립적으로 유지됩니다. Chrome이 워커를 언제든 종료해도 차단은 중단 없이 계속됩니다.

### 레이어 2: tabs.onUpdated (탭 수준 폴백)

`tabs.onUpdated` 리스너가 모든 탭 URL 변경을 차단 목록과 대조합니다. 일치하면 탭을 `blocked.html`로 리다이렉트합니다.

이 레이어가 존재하는 이유는 declarativeNetRequest에 알려진 빈틈이 있기 때문입니다:
- **캐시된 페이지** -- 네트워크 요청 없이 제공됨
- **프리렌더링된 페이지** -- Chrome의 투기적 프리렌더링이 네트워크 규칙을 우회할 수 있음
- **브라우저 내부 네비게이션** -- 특정 주소창 이동이 네트워크 수준 규칙을 건너뜀

### 왜 두 레이어 모두 필요한가

중복은 의도적입니다. 같은 URL에 두 레이어가 모두 실행되는 것은 무해합니다 -- 네트워크 차단이 로드를 방지하고, 탭 리다이렉트는 페이지가 어떻게든 로드된 경우에만 트리거됩니다. 각 레이어는 독립적으로 실패합니다.

사소한 UX 트레이드오프: 네트워크 수준 차단은 Chrome의 기본 오류 페이지를 보여주고, 탭 수준 차단은 동기 부여 메시지가 포함된 커스텀 `blocked.html`을 보여줍니다. 이 불일치는 커버리지 완성도를 위해 수용합니다.

## 규칙 생성

사이트가 추가되면 팝업이 `chrome.storage.sync`에 쓰고, `storage.onChanged` 이벤트가 발생합니다. 서비스 워커가 `updateBlockRules()`를 실행합니다 -- 모든 차단 규칙의 전체 재생성입니다.

**표준 항목**은 각각 2개 규칙을 생성합니다: 정확한 도메인용 하나와 `www.` 변형용 하나. **정규식 항목**은 RE2 구문의 `regexFilter`를 사용하여 1개 규칙을 생성합니다. **이중 등록**: 모든 규칙은 dynamic rule(재시작 간 유지)과 session rule(ID 오프셋 +1000)로 모두 등록되어 심층 방어를 제공합니다.

### 증분 업데이트 대신 전체 재생성

모든 변경 시 서비스 워커는 기존 규칙을 모두 삭제하고 전체 세트를 재생성합니다. 대안인 증분 업데이트는 사이트에 매핑된 규칙 ID 추적, ID 재사용 처리, 상태 diff가 필요합니다. 미미한 성능 이득을 위한 더 복잡한 코드와 더 많은 버그 표면. 전체 재생성은 O(n)으로, n은 차단 사이트 수입니다. Chrome의 5,000개 규칙 상한 기준으로 최악의 경우 ~10,000개 규칙 연산 -- 충분히 처리 가능합니다.

## 데이터 스토리지

**`storage.local` 대신 `chrome.storage.sync`를 선택한 이유**: 제로 인프라의 기기 간 동기화. 사용자가 Chrome에 로그인하면 모든 기기에 차단 목록이 자동으로 나타납니다. 계정 없음, 서버 없음.

트레이드오프는 용량입니다: `storage.sync`는 100KB 제한 vs `storage.local`의 10MB. 각 항목은 ~50바이트로, 약 2,000개 항목을 지원합니다. 사용자가 일반적으로 10-50개 사이트를 차단하는 도구에서 이는 충분합니다.

```json
[
  { "url": "twitter.com", "isRegex": false },
  { "url": "youtube.com/shorts", "isRegex": false },
  { "url": "youtube\\.com.*[가-힣]", "isRegex": true }
]
```

서비스 워커는 설계상 stateless입니다. Chrome이 언제든 종료할 수 있습니다. 이벤트가 발생하면 Chrome이 워커를 재시작하고, 워커는 스토리지에서 읽어 모든 상태를 재구성합니다. 인메모리 영속성이 필요 없습니다.

## URL 매칭

`isUrlBlocked()` 함수는 모든 `tabs.onUpdated` 이벤트에서 실행되어, 각 URL을 전체 차단 목록과 대조합니다.

**표준 모드**: 호스트네임에서 `www.`를 제거하고, 저장된 호스트네임과 서브도메인 지원으로 매칭하고, 경로가 지정된 경우 접두사를 확인합니다.

**정규식 모드**: 전체 URL을 먼저 디코딩한 후 대소문자 무시 RegExp 매칭을 적용합니다. 잘못된 패턴은 조용히 건너뜁니다 -- 하나의 잘못된 패턴이 나머지 모든 차단을 비활성화하지 않습니다.

URL 디코딩 단계가 핵심입니다. 사용자가 한국어 콘텐츠를 대상으로 `youtube\.com.*[가-힣]`를 차단하면, 브라우저가 URL에서 해당 문자를 인코딩합니다 (예: `%ED%95%9C%EA%B8%80`이 `한글`이 됨). 디코딩 없이는 정규식이 절대 매칭되지 않습니다.

## 아키텍처 결정

**빌드 시스템이나 프레임워크 없음** -- 입력 하나와 목록 하나인 320px 팝업에 React는 필요 없습니다. 총 ~300줄의 JavaScript 3개 파일에 TypeScript나 번들러는 필요 없습니다. 전체 소스가 프로덕션에서 사람이 읽을 수 있습니다. 프라이버시 중심 확장 프로그램에서 이것은 기능입니다: 사용자가 코드가 정확히 무엇을 하는지 검사할 수 있습니다.

**MV2 대신 Manifest V3** -- MV3은 새로운 Chrome Web Store 제출에 필수입니다. 서비스 워커 모델(영구 백그라운드 페이지 대체)은 stateless 설계를 요구합니다 -- 결과적으로 올바른 제약이 되었습니다.

**네이티브 앱이나 VPN 대신 Chrome 확장 프로그램** -- VPN 기반 차단의 배터리 소모, 네트워크 오버헤드, 프라이버시 문제를 피합니다. 트레이드오프: Chromium 생태계에 고정되고, 사용자가 확장 프로그램을 제거할 수 있습니다. 둘 다 수용된 제한입니다.

**보안 설계** -- CSP가 모든 스크립트를 `'self'`로 제한합니다. 정규식 패턴은 `new RegExp()` (안전한 생성자)와 Chrome의 RE2 검증기를 거칩니다. 사용자 입력은 절대 코드로 실행되지 않으며, 파비콘 요청 외에는 외부 서버로 데이터가 전송되지 않습니다.
