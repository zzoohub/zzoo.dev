## Architecture

Rust/Axum modular monolith with hexagonal architecture. One developer, one deployable unit. Projects, scenes, characters, and AI generation all share data heavily — microservices would mean every generation request fans out across 4 services. A monolith with domain-separated modules (Rust traits as port boundaries) avoids that overhead while staying extractable later.

The frontend is TanStack Start with SolidJS, chosen over React specifically for the timeline. Fine-grained reactivity means each scene node updates independently during drag-and-drop — no virtual DOM diffing across hundreds of elements. Deployed to Cloudflare Workers via Nitro adapter.

```
apps/api/
├── crates/llm/              # LLM provider abstraction (standalone crate)
├── src/domain/               # Business logic (zero infra dependencies)
│   ├── project/              # Project CRUD, story config
│   ├── timeline/             # Tracks, scenes, NLE positioning
│   ├── character/            # Characters, relationships
│   └── ai/                   # Prompt assembly, context compression
├── src/inbound/http/         # Axum handlers
└── src/outbound/             # SQLx, R2, JWT adapters
```

## What's Interesting

### Scenes as Clips, Not List Items

Scenes use `start_position` and `duration` as `DOUBLE PRECISION` — the NLE (non-linear editor) model. This is not a sorted list. Scenes have extent, which enables two things: midpoint insertion without reindexing (initial spacing of 1024.0 allows around 50 insertions before rebalancing), and automatic simultaneity detection across tracks via overlapping ranges. When generating prose for a scene, the system queries for scenes on other tracks with overlapping time ranges and includes them as "simultaneous events" in the prompt. The timeline structure literally shapes the AI context.

### The LLM Crate Boundary

The `llm` crate lives in a separate Cargo workspace. The compiler physically prevents it from importing API internals — it knows how to talk to LLM providers, nothing more. The API's `ai` module handles what to ask: prompt construction, context assembly, orchestration. This separation is enforced at compile time, not by convention.

Two provider implementations sit behind a single `LlmProvider` trait: Gemini Flash-Lite (primary, $0.10/1M tokens) and CF Workers AI (fallback, free tier). `LlmGateway` handles failover — if Gemini returns a timeout, 5xx, or 429, the request automatically retries on Workers AI. No sidecar, no gateway service. Just in-process Rust with zero latency overhead.

### Context Assembly

Each scene generation assembles a prompt from six structured sources: story config, scene details, character cards with relationships, compressed summaries of preceding scenes, simultaneous events from parallel tracks, and a preview of the next scene. Total: 2,500 to 5,000 tokens, deterministic given the same data.

The critical piece is context compression. After each draft completes, the system generates a 200-300 token summary preserving plot events, character states, and foreshadowing. These summaries replace full scene text in subsequent generation prompts. At 15 scenes with 250-token summaries, that's roughly 3,750 tokens — well within context limits. Without compression, 15 scenes of full text would blow past any reasonable context window.

### Draft Versioning Separated from Scene Metadata

Drafts live in a separate table with `(scene_id, version)` — not inline in the scene row. Each draft records its source (`ai_generation`, `ai_edit`, `manual`), the model used, and token counts. This keeps the frequently-read scene metadata table lean while preserving full generation history. Generation cost tracking uses `ON DELETE SET NULL` on project/scene foreign keys, so cost data survives entity deletion for billing analytics.

## Key Decisions

**Rust over Python** — No Python-only libraries needed. LLM calls are HTTP via `reqwest` + `serde`. Rust gives sub-ms latency, 10-30MB memory footprint, and compile-time safety. At launch traffic, the entire API runs on Cloud Run's free tier.

**Neon over Supabase** — Scale-to-zero pricing (free at launch), branch-based dev workflow for migrations, pgvector available for future embeddings. Co-located with Cloud Run in Virginia. Supabase Seoul was considered but rejected — the target is a global audience, and the 1-3ms cross-cloud latency between Neon and Cloud Run matters more than geographic proximity to Korea.

**SSE over WebSocket** — Token delivery only needs server-to-client streaming. SSE is stateless and HTTP-native. Cloud Run handles it without connection management overhead. WebSocket adds complexity for zero benefit. The frontend parses SSE via custom `fetch` + `ReadableStream` in SolidJS — no Vercel AI SDK since no SolidJS adapter exists.
