## System Overview

A SolidJS frontend talks to a Rust/Axum API. The API is a modular monolith — projects, timeline, characters, and AI generation live as domain modules inside one service. LLM calls go through a separate workspace crate (`crates/llm`) that fails over from Gemini Flash-Lite to CF Workers AI. Generation output streams to the client via SSE. Data lives in Neon PostgreSQL with hot paths cached in Upstash Redis.

## Scenes Are Clips, Not List Items

In web novels, scenes don't happen one after another. Character A acts in one place while Character B does something else somewhere else. Time overlaps. Tracks branch and merge. A sequential list can only say "scene 3 comes after scene 4."

Scenes are modeled as NLE (non-linear editor) clips. Each scene has a `start_position` and `duration`, placed on a track, draggable. This lets you answer "what's happening on other tracks while this scene plays out?" with a single range query.

The model carries through to AI generation. When generating a scene, the system includes simultaneous events — scenes on other tracks with overlapping time ranges — in the prompt. The act of placing a scene on the timeline is the act of giving the AI context.

## Context Compression — Fitting 20 Scenes into a Prompt

Generating a scene requires context from preceding scenes. Raw text from 15 scenes runs to tens of thousands of tokens. It won't fit the context window, and even if it did, cost scales linearly.

When a scene draft is finished, a compressed summary is generated immediately — ~200–300 tokens — and stored in the `scene_summary` table and cached in Redis. When generating the next scene, these summaries replace the full text of prior scenes. Summaries for 15 scenes total ~3,750 tokens — an ~80% reduction from raw text.

Prompt assembly is deterministic. Same data in, same prompt out. Global config, scene details, involved character cards, preceding scene summaries, simultaneous events, next scene preview — these pieces are assembled into a ~4–8K token prompt. No randomness.

## The LLM Crate — Compiler-Enforced Boundaries

Putting the LLM abstraction as a trait inside an API module keeps the boundary as a convention. Conventions erode over time. The `crates/llm` workspace crate makes it physical.

```rust
#[async_trait]
pub trait LlmProvider: Send + Sync {
    async fn generate(&self, req: GenerateRequest)
        -> Result<GenerateResponse>;
    async fn generate_stream(&self, req: GenerateRequest)
        -> Result<Pin<Box<dyn Stream<Item = Result<String>> + Send>>>;
}
```

This crate cannot import API internals — Cargo's dependency direction is one-way, and the compiler enforces it. The `llm` crate knows *how* to talk to LLM providers. The API's `ai` module knows *what* to ask. `LlmGateway` tries Gemini Flash-Lite first and fails over to CF Workers AI on timeout, 5xx, or 429.

The practical payoff of physical separation: the crate is independently testable. Provider mocks verify failover logic without spinning up Axum. If a CLI tool needs LLM access later, it just depends on the crate.
