## Architecture

Rust/Axum modular monolith with hexagonal architecture. One developer, one deployable unit. Projects, scenes, characters, and AI generation all share data heavily — microservices would fan out every generation request across 4 services. A monolith with domain modules separated by Rust traits avoids that overhead while staying extractable later.

The frontend is TanStack Start + SolidJS, chosen over React specifically for the timeline. Fine-grained reactivity means each scene node updates independently during drag-and-drop — no virtual DOM diffing across hundreds of elements. Deployed to Cloudflare Workers via Nitro adapter.

```
apps/api/
├── crates/llm/              # LLM provider abstraction (standalone crate)
├── src/domain/               # Business logic (zero infra deps)
│   ├── project/              # Project CRUD, story config
│   ├── timeline/             # Tracks, scenes, NLE positioning
│   ├── character/            # Characters, relationships
│   └── ai/                   # Prompt assembly, context compression
├── src/inbound/http/         # Axum handlers
└── src/outbound/             # SQLx, R2, JWT adapters
```

## Scenes as Clips, Not List Items

Scene `start_position` and `duration` are `DOUBLE PRECISION` — the NLE model. Not a sorted list. Scenes have extent, which enables two things: midpoint insertion without reindexing (initial spacing of 1024.0 allows ~50 insertions before rebalancing), and automatic simultaneity detection via overlapping ranges across tracks.

When generating prose for a scene, the system queries other tracks for overlapping time ranges and includes those as "simultaneous events" in the prompt. The timeline structure literally shapes the AI context.

## The LLM Crate Boundary

The `llm` crate lives in a separate Cargo workspace. The compiler physically prevents it from importing API internals — it only knows how to talk to LLM providers. The API's `ai` module handles what to ask: prompt construction, context assembly, orchestration. This separation is enforced at compile time, not by convention.

Two providers behind one `LlmProvider` trait: Gemini Flash-Lite (primary, $0.10/1M tokens) and CF Workers AI (fallback, free tier). `LlmGateway` handles failover — timeouts, 5xx, or 429 from Gemini automatically retry on Workers AI. No sidecar, no gateway service. In-process Rust, zero latency overhead.

## Context Assembly

Each scene generation assembles a prompt from six sources: story config, scene details, character cards with relationships, compressed summaries of preceding scenes, simultaneous events from parallel tracks, and a preview of the next scene. Total: 2,500–5,000 tokens, deterministic given the same data.

The critical piece is context compression. After each draft, the system generates a 200–300 token summary preserving plot events, character states, and foreshadowing. These summaries replace full scene text in subsequent prompts. 15 scenes at 250 tokens each = ~3,750 tokens, well within context limits. Without compression, full text from 15 scenes would blow past any reasonable context window.

## Key Decisions

**Rust over Python.** No Python-only libraries needed — LLM calls are HTTP via `reqwest` + `serde`. Rust gives sub-ms latency, 10–30MB memory, compile-time safety. At launch traffic, the entire API runs on Cloud Run's free tier.

**Neon over Supabase.** Scale-to-zero pricing (free at launch), branch-based dev workflow, pgvector for future embeddings. Co-located with Cloud Run in Virginia. Supabase Seoul was considered but rejected — the 1–3ms cross-cloud latency between Neon and Cloud Run matters more than geographic proximity to Korea for a global audience.

**SSE over WebSocket.** Token delivery only needs server-to-client streaming. SSE is stateless and HTTP-native. Cloud Run handles it without connection management overhead. The frontend parses SSE via custom `fetch` + `ReadableStream` in SolidJS — no Vercel AI SDK since no SolidJS adapter exists.
