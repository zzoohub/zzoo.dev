## Architecture

**Modular Monolith + Hexagonal (Rust / Axum)**

Solo developer building tightly coupled domains — projects, scenes, characters, and generation all share data extensively. A monolith avoids deployment overhead. Modules are separated by domain with clean interfaces (Rust traits as ports), extractable to services later if needed. Sub-ms responses, 10–30MB memory, compile-time safety.

The `llm` crate is a separate workspace crate — the compiler physically prevents it from importing API internals. This enforces the boundary between "how to talk to LLMs" and "what to ask them."

```
apps/api/
├── crates/llm/              # LLM provider abstraction (standalone)
├── src/domain/               # Business logic (no infra deps)
│   ├── project/              # Project CRUD, story config
│   ├── timeline/             # Tracks, scenes, branch/merge
│   ├── character/            # Characters, relationships
│   └── ai/                   # Prompt assembly, generation
├── src/inbound/http/         # HTTP adapters (Axum handlers)
└── src/outbound/             # Infra adapters (SQLx, R2, JWT)
```

**Frontend: TanStack Start + SolidJS**

SolidJS was chosen over React specifically for the timeline UI. Fine-grained reactivity means each scene node updates independently during drag-and-drop — no virtual DOM diffing across hundreds of elements. Deployed to Cloudflare Workers via Nitro adapter.

## Database Design

**11 tables in Phase 1.** Core design choices:

- **NLE positioning** — `scene.start_position` and `scene.duration` as `DOUBLE PRECISION`. Scenes are clips with extent, not list items. Initial spacing of 1024.0 allows ~50 midpoint insertions before rebalancing. Simultaneity across tracks via overlapping ranges.

- **Draft versioning** — Separate `draft` table with `(scene_id, version)`. Preserves full edit history, separates large text blobs from frequently-read scene metadata. Each draft records source (`ai_generation`, `ai_edit`, `manual`), model, and token counts.

- **Context compression** — `scene_summary` table (1:1 with scene) stores AI-compressed summaries (~200–300 tokens). Preceding summaries replace full scene text in generation prompts, keeping context manageable at 15+ scenes.

- **Generation cost tracking** — `generation_log` with `ON DELETE SET NULL` for project/scene FKs. Cost data survives entity deletion for billing analytics.

## Data Flows

Four primary flows, all streaming via SSE through `LlmGateway`:

1. **Idea to project** — User pastes a story idea. LLM returns structured JSON (project config, tracks, scenes, characters). Written to Postgres in a single transaction. Vague idea becomes a visual timeline in seconds.

2. **Scene draft generation** — System assembles context from six sources: story config, scene details, character cards, compressed preceding summaries, simultaneous events from parallel tracks, and next scene preview (~2,500–5,000 tokens total). Draft is versioned on completion; auto-applies if no existing content, otherwise user confirms.

3. **Direction-based edit** — User selects a passage and types a direction like "more tension." Same context assembly as generation, plus the selected text. New draft version recorded as `ai_edit`.

4. **Editor auto-save** — Edits debounce for 1–2 seconds, then PATCH directly. Scene status transitions from `ai_draft` to `edited` on first manual modification.

## AI / LLM Architecture

**Provider abstraction via `crates/llm`:**

```rust
#[async_trait]
pub trait LlmProvider: Send + Sync {
    async fn generate(&self, req: GenerateRequest) -> Result<GenerateResponse>;
    async fn generate_stream(&self, req: GenerateRequest)
        -> Result<Pin<Box<dyn Stream<Item = Result<String>> + Send>>>;
}
```

`LlmGateway` handles failover: Gemini Flash-Lite first, CF Workers AI on failure (timeout, 5xx, 429).

**Context assembly per scene (~2,500–5,000 tokens):**

| Source | Data | Tokens |
|--------|------|--------|
| System instruction | Generation rules, output format | ~500 |
| Global config | Genre, theme, era, POV, mood | ~100 |
| Scene details | Title, summary, location, mood | ~200 |
| Characters + relationships | Cards for involved characters | ~350–700 |
| Preceding summaries | Compressed prior scene summaries | ~1,000–3,000 |
| Simultaneous events | Overlapping scenes on other tracks | ~200–400 |
| Next scene preview | Title + summary | ~100 |

Assembly is deterministic given the same data — no randomness in prompt construction.

**Context compression:** After each draft completes, the system generates a ~200–300 token summary preserving key plot events, character states, and foreshadowing. Stored in Postgres (durable) and cached in Redis (fast reads). At 15 scenes with 250-token summaries, that is roughly 3,750 tokens — well within context limits.

**Streaming:** SSE via Axum's native `Sse` type + `async-stream`. Three endpoints: scene generation, direction-based edit, and project auto-structuring (3-phase: world, characters, timeline). Frontend consumes via custom `fetch` + `ReadableStream` parser in SolidJS — no Vercel AI SDK dependency since no SolidJS adapter exists.

## Key Decisions

**Rust/Axum over FastAPI** — No Python-only libraries needed. LLM calls are HTTP via `reqwest`. Rust gives sub-ms latency, 10–30MB memory, and compile-time safety at the lowest cloud cost.

**Monolith over microservices** — One developer, one deployable unit. Generation reads from every other module. One deployment = one CI pipeline, one container, one log stream. Revisit if the team grows or generation needs GPU compute.

**Neon over Supabase** — Scale-to-zero pricing (free at launch), branch-based dev workflow, pgvector for future embeddings. Co-located with Cloud Run in Virginia for low cross-cloud latency.

**`llm` as a separate workspace crate** — Compiler-enforced boundary. Independently testable with mock providers. Reusable across CLI tools or workers. Zero runtime cost — it is a library, not a service.

**SSE over WebSocket** — Token delivery only needs server-to-client streaming. SSE is stateless, HTTP-native, trivially handled by Cloud Run. WebSocket adds connection management for no benefit. Revisit when real-time collaboration is added.

**Google OAuth only (Phase 1)** — Simplest implementation. Adding Kakao is a two-way door — just another OAuth provider, addable in weeks. Revisit if acquisition data shows bounce at login.
