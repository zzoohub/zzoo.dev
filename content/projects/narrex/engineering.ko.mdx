## 아키텍처

Rust/Axum 모듈러 모놀리스, 헥사고날 아키텍처. 개발자 1명, 배포 단위 1개. 프로젝트, 장면, 캐릭터, AI 생성이 모두 데이터를 공유한다 — 마이크로서비스로 쪼개면 생성 요청 하나에 4개 서비스를 호출해야 한다. 도메인별 모듈을 Rust trait으로 분리한 모놀리스가 그 오버헤드를 피하면서도 나중에 추출 가능한 구조를 유지한다.

프론트엔드는 TanStack Start + SolidJS. 타임라인 때문에 React 대신 선택했다. 파인그레인드 반응성으로 드래그 앤 드롭 시 각 장면 노드가 독립적으로 업데이트된다 — 수백 개 엘리먼트에 대한 가상 DOM 디핑이 필요 없다. Nitro 어댑터로 Cloudflare Workers에 배포.

```
apps/api/
├── crates/llm/              # LLM 프로바이더 추상화 (독립 크레이트)
├── src/domain/               # 비즈니스 로직 (인프라 의존성 제로)
│   ├── project/              # 프로젝트 CRUD, 스토리 설정
│   ├── timeline/             # 트랙, 장면, NLE 포지셔닝
│   ├── character/            # 캐릭터, 관계
│   └── ai/                   # 프롬프트 조립, 컨텍스트 압축
├── src/inbound/http/         # Axum 핸들러
└── src/outbound/             # SQLx, R2, JWT 어댑터
```

## 흥미로운 부분

### 장면은 리스트 항목이 아니라 클립이다

장면의 `start_position`과 `duration`은 `DOUBLE PRECISION`이다 — NLE(비선형 편집기) 모델. 정렬된 리스트가 아니다. 장면이 범위를 가지기 때문에 두 가지가 가능하다: 리인덱싱 없는 중간 삽입(초기 간격 1024.0으로 리밸런싱 없이 약 50회 삽입 가능), 그리고 범위 겹침을 통한 트랙 간 동시성 자동 감지. 장면의 산문을 생성할 때, 시스템이 다른 트랙에서 시간 범위가 겹치는 장면을 조회해 "동시 사건"으로 프롬프트에 포함한다. 타임라인 구조가 그대로 AI 컨텍스트를 결정한다.

### LLM 크레이트 경계

`llm` 크레이트는 별도 Cargo 워크스페이스에 있다. 컴파일러가 API 내부를 물리적으로 임포트하지 못하게 막는다 — LLM 프로바이더와 어떻게 대화하는지만 안다. API의 `ai` 모듈이 무엇을 물어볼지를 담당한다: 프롬프트 구성, 컨텍스트 조립, 오케스트레이션. 이 분리는 컨벤션이 아니라 컴파일 타임에 강제된다.

하나의 `LlmProvider` trait 뒤에 두 개의 프로바이더 구현이 있다: Gemini Flash-Lite(주력, $0.10/1M 토큰)와 CF Workers AI(폴백, 무료 티어). `LlmGateway`가 페일오버를 처리한다 — Gemini가 타임아웃, 5xx, 429를 반환하면 자동으로 Workers AI에 재시도한다. 사이드카도, 게이트웨이 서비스도 없다. 레이턴시 오버헤드 제로의 인프로세스 Rust.

### 컨텍스트 어셈블리

장면 생성마다 6개의 구조화된 소스에서 프롬프트를 조립한다: 스토리 설정, 장면 상세, 관계가 포함된 캐릭터 카드, 선행 장면의 압축 요약, 병렬 트랙의 동시 사건, 다음 장면 프리뷰. 총 2,500~5,000 토큰, 동일 데이터면 결정적 결과.

핵심은 컨텍스트 압축이다. 각 초안 완료 후 시스템이 200~300 토큰 요약을 생성한다 — 플롯 이벤트, 캐릭터 상태, 복선을 보존한다. 이후 생성 프롬프트에서 전체 장면 텍스트 대신 이 요약이 들어간다. 15개 장면에 250토큰 요약이면 약 3,750 토큰 — 컨텍스트 한도 내에서 충분하다. 압축 없이 15개 장면의 전체 텍스트를 넣으면 어떤 합리적인 컨텍스트 윈도우도 초과한다.

### 초안 버전 관리와 장면 메타데이터의 분리

초안은 `(scene_id, version)`을 가진 별도 테이블에 저장한다 — 장면 행에 인라인이 아니다. 각 초안은 소스(`ai_generation`, `ai_edit`, `manual`), 사용 모델, 토큰 수를 기록한다. 자주 읽는 장면 메타데이터 테이블을 가볍게 유지하면서 전체 생성 히스토리를 보존한다. 생성 비용 추적은 프로젝트/장면 FK에 `ON DELETE SET NULL`을 적용해, 엔티티 삭제 후에도 과금 분석용 비용 데이터가 보존된다.

## 핵심 결정

**Rust over Python** — Python 전용 라이브러리가 필요 없다. LLM 호출은 `reqwest` + `serde`로 HTTP 통신. Rust는 서브밀리초 레이턴시, 10~30MB 메모리, 컴파일 타임 안전성을 준다. 출시 트래픽에서 전체 API가 Cloud Run 무료 티어로 돌아간다.

**Neon over Supabase** — 제로 스케일 과금(출시 시 무료), 마이그레이션용 브랜치 기반 개발 워크플로우, 향후 임베딩용 pgvector. Cloud Run과 버지니아에 공동 배치. Supabase 서울도 검토했지만 기각 — 타겟은 글로벌 사용자이고, Neon과 Cloud Run 간 1~3ms 크로스 클라우드 레이턴시가 한국과의 지리적 근접성보다 중요하다.

**SSE over WebSocket** — 토큰 전달에 필요한 건 서버에서 클라이언트로의 단방향 스트리밍뿐. SSE는 상태 비저장이고 HTTP 네이티브다. Cloud Run이 연결 관리 오버헤드 없이 처리한다. WebSocket은 이점 없이 복잡성만 추가한다. 프론트엔드는 SolidJS에서 커스텀 `fetch` + `ReadableStream`으로 SSE를 파싱한다 — SolidJS 어댑터가 없어 Vercel AI SDK 의존성 없음.
