## 시스템 구조

SolidJS 프론트엔드가 Rust/Axum API와 통신한다. API는 모듈러 모놀리스 — 프로젝트, 타임라인, 캐릭터, AI 생성이 하나의 서비스 안에서 도메인별 모듈로 나뉜다. LLM 호출은 별도 workspace crate(`crates/llm`)가 담당하고, Gemini Flash-Lite에서 CF Workers AI 순서로 failover한다. 생성 결과는 SSE로 스트리밍된다. 데이터는 Neon PostgreSQL에 저장되고, 핫 데이터는 Upstash Redis에 캐싱된다.

## 장면을 리스트가 아니라 클립으로 모델링

웹소설에서 장면은 순서대로 나열되지 않는다. A 캐릭터가 어딘가에서 행동하는 동안 B 캐릭터는 다른 곳에서 다른 일을 한다. 시간이 겹치고, 트랙이 갈라졌다 합쳐진다. 이걸 순차 리스트로 표현하면 "3번 장면 다음에 4번" 밖에 못 말한다.

장면을 NLE(비선형 편집기) 클립으로 모델링했다. 각 장면에 `start_position`(시작 지점)과 `duration`(길이)이 있다. 트랙 위에 배치되고, 드래그로 옮길 수 있다. 이렇게 하면 "이 장면이 진행되는 동안 다른 트랙에서 무슨 일이 벌어지고 있는가"를 range query 하나로 찾을 수 있다.

이 모델이 AI 생성까지 관통한다. 장면을 생성할 때 같은 시간대에 다른 트랙에서 진행 중인 장면들(simultaneous events)을 프롬프트에 포함시킨다. 작가가 타임라인에 장면을 배치하는 행위 자체가 AI에게 맥락을 건네는 행위다.

## 컨텍스트 압축 — 20개 장면을 프롬프트에 넣는 법

장면 하나를 생성하려면 앞선 장면들의 맥락이 필요하다. 장면 15개의 원문을 그대로 넣으면 토큰이 수만 개가 된다. 컨텍스트 윈도우에 안 들어가고, 들어가도 비용이 선형으로 늘어난다.

장면 초안이 완성되면 즉시 요약을 만든다. ~200–300 토큰짜리 압축 요약을 `scene_summary` 테이블에 저장하고, Redis에도 캐싱한다. 다음 장면을 생성할 때는 앞선 장면의 원문 대신 이 요약들을 프롬프트에 넣는다. 장면 15개의 요약이 ~3,750 토큰 — 원문 대비 ~80% 절감이다.

프롬프트 조립은 결정적(deterministic)이다. 같은 데이터를 넣으면 같은 프롬프트가 나온다. 전역 설정, 장면 정보, 관련 캐릭터 카드, 선행 장면 요약, 동시 진행 이벤트, 다음 장면 미리보기 — 이 조각들을 조립해서 ~4–8K 토큰짜리 프롬프트를 만든다. 랜덤 요소는 없다.

## LLM 크레이트 — 컴파일러가 경계를 강제한다

LLM 호출을 API 모듈 안에 trait로 두면 경계가 관습에 의존한다. 시간이 지나면 커플링이 스며든다. `crates/llm`을 별도 workspace crate로 분리했다.

```rust
#[async_trait]
pub trait LlmProvider: Send + Sync {
    async fn generate(&self, req: GenerateRequest)
        -> Result<GenerateResponse>;
    async fn generate_stream(&self, req: GenerateRequest)
        -> Result<Pin<Box<dyn Stream<Item = Result<String>> + Send>>>;
}
```

이 crate는 API 내부를 import할 수 없다 — Cargo 의존성 방향이 한쪽이라 컴파일러가 막는다. `llm` crate는 "어떻게 LLM과 대화할지"만 안다. "무엇을 물어볼지"는 API의 `ai` 모듈이 담당한다. `LlmGateway`가 Gemini Flash-Lite를 먼저 시도하고, 실패 시(타임아웃, 5xx, 429) CF Workers AI로 failover한다.

물리적 분리의 실질적 이점 — crate를 독립적으로 테스트할 수 있다. Axum을 안 띄우고도 provider mock으로 failover 로직을 검증한다. 나중에 CLI 도구에서 LLM이 필요하면 crate만 가져다 쓰면 된다.
