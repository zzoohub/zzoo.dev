## 문제

한국 웹소설 지망생들은 머릿속에 이야기가 있지만 원고로 만들지 못합니다. 부족한 건 창의력이 아닙니다. 수십 개의 사건을 구조화하고, 40화 이상에서 일관성을 유지하며, 화당 3,000~5,000자의 산문을 생산하는 노동이 병목입니다.

기존 AI 글쓰기 도구는 작업 단위가 잘못됐습니다. 문장 단위 자동완성(Sudowrite)은 구조 작업을 전부 작가에게 떠넘깁니다. 완전 자동 생성(ChatGPT, 독립형 생성기)은 창작 주도권을 빼앗고 몇 번의 대화 후 일관성을 잃습니다. 비주얼 스토리 구조와 AI 초안을 장면 단위로 결합한 도구는 한국에도, 해외에도 없었습니다.

작가는 장면과 플롯 흐름 단위로 사고합니다. 개별 문장이 아니라. 그 간극이 기회였습니다.

## 솔루션

Narrex는 스토리 구조 자체가 AI 프롬프트가 되는 비주얼 타임라인 에디터입니다. 장면, 캐릭터, 관계를 시각적으로 배치하면 — AI가 설정, 캐릭터, 이전 요약, 동시 사건 등 전체 서사 컨텍스트를 반영해 장면별 산문을 생성합니다. 프롬프트 엔지니어링이 필요 없습니다.

### 핵심 설계 결정

- **장면을 생성 단위로** — 문장도 아니고, 챕터도 아닙니다. 구조화된 타임라인 위의 장면이 소설가가 실제로 사고하는 자연스러운 단위입니다. AI가 설정, 캐릭터, 이전 요약, 병렬 트랙의 동시 사건까지 컨텍스트로 반영해 장면별 산문을 생성합니다.

- **NLE 타임라인 모델** — 영상 편집에서 차용했습니다. 장면은 단순 리스트 항목이 아니라 `start_position`과 `duration`을 가진 멀티트랙 클립입니다. 병렬 스토리라인, 중간점 삽입 방식의 드래그 앤 드롭(리넘버링 불필요), 범위 겹침을 통한 동시성 감지가 가능합니다.

- **사건-에피소드 분리** — 장면(무슨 일이 일어나는지)과 에피소드(독자에게 어떻게 전달되는지)는 독립적인 레이어입니다. 여러 작은 사건이 하나의 에피소드로 합쳐지거나, 하나의 큰 사건이 여러 에피소드에 걸칠 수 있습니다. 타임라인의 에피소드 구분선을 드래그하면 챕터가 재분할되고, AI 프롬프트가 자동으로 재구성됩니다.

- **Rust 모듈러 모놀리스** — 솔로 개발자가 긴밀하게 결합된 도메인을 구축합니다. 헥사고날 아키텍처(도메인 로직에 인프라 의존성 없음)를 적용한 단일 Axum 서비스로 배포 오버헤드를 없애면서도 모듈 분리 가능성을 유지합니다. 서브밀리초 응답, 10~30MB 메모리, 컴파일 타임 안전성.

- **LLM 크레이트를 워크스페이스 경계로** — 프로바이더 추상화가 별도 Rust 워크스페이스 크레이트(`crates/llm`)에 존재합니다. 컴파일러가 API 내부를 물리적으로 임포트하지 못하게 강제합니다. Gemini 2.5 Flash-Lite를 주력(장면당 ~$0.002), Cloudflare Workers AI를 무료 폴백으로 사용합니다. 페일오버는 자동입니다.

- **SolidJS로 타임라인 반응성 확보** — 파인그레인드 반응성이 타임라인 UI에 이상적입니다. 드래그 앤 드롭 시 많은 DOM 노드가 독립적으로 업데이트되는 상호작용 패턴에서 서지컬 DOM 업데이트가 가상 DOM 디핑보다 우수합니다.

## 현재 상태

Narrex는 Phase 1 개발 중입니다 — 핵심 루프 검증 단계: 아이디어 입력, 비주얼 구조 생성, 장면별 AI 초안 작성, 작가 편집.

- **장면당 ~$0.002 생성 비용** — Gemini 2.5 Flash-Lite, 장면당 ~4K 입력 + ~2K 출력 토큰
- **총 인프라 비용 ~$2~5/월** (출시 시점, 100 DAU) — Cloud Run, Neon, Upstash 모두 제로 스케일
- **컨텍스트 어셈블리 파이프라인** — 설정 + 캐릭터 + 압축 요약 + 동시 사건을 장면당 ~2,500~5,000 토큰 프롬프트로 자동 조립
- **Phase 1 PostgreSQL 테이블 11개** — NLE 위치 기반 장면, 버전 관리 초안(추가 전용), 생성 비용 로그, 캐릭터 관계 그래프
- **SSE 스트리밍** — Axum 네이티브 SSE + async-stream으로 첫 토큰 2초 이내
