学習曲線が最も急なプログラミング言語が、AIに任せるには最も安全だった。これは本来、意味が通らないはずである。

vibe coding -- 自然言語で欲しいものを記述し、AIに実装を書かせるやり方 -- が登場したとき、誰もが「簡単な言語」で使うものだと考えた。Pythonはsyntaxが少ない。JavaScriptはどこでも動く。関数ひとつに3つのlifetimeアノテーションが必要になり得る言語をわざわざ選ぶ理由があるだろうか。

ある。「難しい」と「危険」は同じ軸ではないからだ。そしてRustを人間にとって難しくしている性質が、まさにAIにとって信頼性を高めている。

私はしばらくの間、AIの助けを借りてRustでbackendサービスを書いている。同じやり方をPythonやJavaScriptで試した場合と比べて、Rustで生成されたcodeのほうが明らかに信頼できる。少しだけではない。大幅にだ。codeを拒絶することで有名な言語が、逆説的に、AI生成codeが最も信用に値する言語だった。

なぜそうなるのか。AIが質の悪いPythonを書いたとき何が起こるか考えてみるとよい。codeは動いてしまう。間違った結果を出すか、あるいは深夜3時のproductionで`TypeError`を吐いて落ちる。そのerror messageは実際の問題についてほとんど何も教えてくれない。feedbackのloopが劣悪なのだ。codeを生成し、実行し、不可解な挙動を観察し、曖昧なerrorとstack traceから何が壊れたかを推理しようとする。AIは暗闇の中で当て推量をしている。

ではAIが質の悪いRustを書いたとき何が起こるか。動かない。compilerが即座に拒絶し、こういうものを出力する。

```
error[E0502]: cannot borrow `v` as mutable because it is also borrowed as immutable
 --> src/main.rs:4:5
  |
3 |     let first = &v[0];
  |                  - immutable borrow occurs here
4 |     v.push(4);
  |     ^^^^^^^^^ mutable borrow occurs here
5 |     println!("{first}");
  |               ------- immutable borrow later used here
```

これはerror messageではない。構造化された授業である。正確なfile、正確な行、正確な競合を指し示し、多くの場合は修正方法まで提案する。これをAIに戻せば、何を変えるべきか正確にわかる。compile-fix-compileのloopは1回か2回の反復で収束する。曖昧さがないからだ。`undefined is not a function`と比較するとよい。あれはAIに作業材料を一切与えない。

これが誰も予想しなかった逆転である。Rustのcompiler -- 初心者を挫折させて辞めさせるあのもの -- が、AIにとって最良のpair programmerだった。

だが、より深い洞察はcorrectionのloopについてではない。Rustのcodeがcompileに通ったとき何が起こるか、という点にある。

Rustコミュニティにはこういう言い回しがある。compileが通れば、動く。虚勢に聞こえるかもしれないが、他のどのmainstream言語よりもこれは文字通りの真実に近い。type systemとborrow checkerがnull pointer dereference、data race、use-after-free、buffer overflowを排除する。testで検出するのではない。構造的に不可能にするのだ。compilerの承認印は、大半の言語のtest suite全体よりも強い正当性の保証である。

vibe codingにとって、これは決定的に重要である。AIが生成するすべての行を読んでいないとき -- syntaxを精査するのではなくvibeで操縦しているとき -- 別種のsafety netが必要になる。Pythonでは、そのsafety netは自分で書かなければならないtestであり、それが十分に網羅的であることを祈るしかない。Rustでは、safety netはcompilerであり、いかなるtest suiteにも検査できないことを検査する。

shared mutable stateを考えてみるとよい。softwareにおける最も陰湿なbugの源泉である。programの2つの部分が同じdataを変更する。その相互作用はtimingに依存し、実行順序に依存し、網羅的にtestすることがほぼ不可能な要因に依存する。Pythonを生成しているAIは、負荷がかかったときだけ顕在化する微妙なrace conditionを含むcodeを書くかもしれない。developmentでは絶対に捕まらない。Rustでは、ownershipモデルがこのカテゴリのbugを文字通り表現不可能にする。2つのものが同じdataをmutateしようとすると、programはcompileに通らない。「runtimeでcrashするかもしれない」ではない。compileに通らない。AIはconcurrency理論を理解する必要がない。compilerがそれを強制する。

強いtype systemは同様のことをより広い範囲で行う。関数signatureが`&[u8]`を受け取り`Result<Response, AppError>`を返すと宣言しているとき、AIは誤ってstringを渡したりerror処理を忘れたりできない。type systemがsolution spaceを極めて狭く制約するため、compileは通るが動作しないcodeの書き方がほとんど残らない。JavaScriptでは、AIはもっともらしく見えるが静かに間違ったことをするcodeを自由に生成できる。Rustでは、その自由の大部分が消えている。そしてそれは制限ではなく、featureである。

performanceの議論もあるが、この時点ではほとんどおまけのようなものだ。Pythonでvibe codingすると、AIは素朴な実装を書く。たいてい遅い。Rustでvibe codingすると、AIもやはり素朴な実装を書く。だがRustのzero-cost abstractionのおかげで、素朴な実装がすでに速い。iteratorは手書きのloopと同じmachine codeにcompileされる。latency spikeを引き起こすgarbage collectorがない。自分もAIもperformanceについて一切考えずに、C言語レベルのperformanceを手に入れる。

私はかつて、Rustに対する反論は説得力があると思っていた。学習曲線はほとんどのapplicationには割に合わない -- これが定説であり、合理的に見えた。だがこの議論は隠れた前提に依存していた。人間がその曲線を登らなければならない、という前提だ。すべてのdeveloperがlifetimeとtrait boundとborrow checkerの複雑さを内面化してから初めて生産的になれる、という前提だ。

AIが前提を変えた。developerはlifetimeを理解する必要がない。AIもlifetimeを本当に理解する必要がない。compilerがlifetimeを理解している。compilerはずっとlifetimeを理解してきた。問いは、規則が厳しすぎるかどうかではなかった。人間がそれらの規則をすべて同時に頭の中に保持できるかどうかだった。その問いはもう意味を持たない。

学習曲線を方程式から取り除いたあとに残るのは、memory safety、fearless concurrency、garbage collectorなしのnative performance -- これらを実質的に無料で与えてくれる言語である。かつてRustを大半のprojectにとって法外にしていたcostは、toolingが吸収した。

最も難しい言語が、vibe codingで最も安全な言語だった。難しさにもかかわらずではない。難しさゆえにだ。
