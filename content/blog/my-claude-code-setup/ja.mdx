一人の人間がチームを持つことはできるのか。

この問いに対する答えは自明に思える。チームとは複数の人間の集まりだ。セキュリティの人がいて、データベースの人がいて、テストの人がいる。それぞれが自分にはない角度からコードを見る。セキュリティエンジニアはフォームを見てSQLインジェクションを考える。DBAはfeatureの完成を見てインデックスの欠落を考える。一人の開発者がどんなに優秀でも、同時に五人分の視点を持つことはできない。

少なくとも、以前はそう思っていた。だが「チーム」という言葉が本当に指しているものを考え直してみると、少し違う風景が見えてくる。チームに本当に求めているものは、複数の人間ではない。複数の視点が一貫して適用されることだ。セキュリティエンジニアが貴重なのは、椅子をもう一つ埋めているからではない。自分にはないレンズでコードの全ての変更を見てくれるからだ。それも毎回。気分が乗った時だけではなく。

この区別は重要だ。なぜなら、ある問いが開かれるからである。一貫した視点が必要なのだとしたら、それを得るために本当に人間が必要なのか。

私はソロ開発者として複数のプロダクトを運営している。一年前、最も難しかったのはコードを書くことではなかった。思考モードの切り替えだった。featureの実装に没頭して、ReactのstateとUIのトランジションのことを考えている最中に、突然DBAのように考えるべき瞬間が来る。「このテーブルのforeign keyは正しいか。このschemaは10万レコードに耐えるか。」そういうスキルは身につけられる。だが同時に全てを適用することはできない。構築しているときは、構築のことを考えている。誰の脳も自発的にセキュリティ監査を行わない。だからチームが存在する。

Claude Codeの設定がこれを変えた。コードを速く書けるようになったのではない。特定の専門性を持つ同僚がいるのと本質的に同じ感覚を生み出した。驚きは、AIがコーディングを手伝えるということではない。適切な設定が「速くなった自分」とは質的に異なるものを生み出すということだ。持続的で専門化された視点が、頼まなくても全ての作業に関与する。

個々の部品は単純だ。組み合わせが予想外のことをする。

## 憲法

全てのプロジェクトはルートに`CLAUDE.md`というファイルを置くところから始まる。私はこれをそのcodebaseにおけるAIの行動を支配する憲法だと考えている。tech stack、アーキテクチャパターン、制約条件、そして最も重要なこととして、全ての変更に対してどのskillとagentを使わなければならないかを定義する。

全てのプロジェクトに共通するコアパターンはこうだ。

```
1. 全ての実装は指定されたskillを使用しなければならない
2. 実装後、2つのagentを並列で実行:
   - security-reviewer -> 監査 -> 修正
   - tester -> 変更コードのテスト -> 修正
```

これは提案ではない。ハードルールだ。Claude Codeは毎セッション開始時にこのファイルを読む。リマインドは不要だ。全てのfeature、全てのfix、全てのrefactorが同じpipelineを通る。憲法がそれを強制する。

プロジェクトごとに憲法は異なる。Next.jsサイトにはVercel公式のReact best practices skillを義務づけている。Rust APIには`axum`と`postgresql`のskillを義務づけている。React Nativeアプリにはmobile固有のskillを義務づけている。だがセキュリティとテストの並列パスは普遍的だ。例外なし。「テストは後で」もなし。

この効果は微妙だが深い。以前の私は、気が向いた時にテストを書き、覚えている時にセキュリティレビューをする類の開発者だった。今はシステムがその選択肢を与えない。一貫性は規律の問題ではなくなった。構造の問題になった。

## 制約としてのskill

私は7つのドメインにまたがる21のskillを持っている。アーキテクチャ、webフレームワーク、モバイル、バックエンド、データベース、AI/ML、品質。一部は自分で書いた。一部はフレームワークの作者から来ている。ここが面白い。

VercelはClaude Code用のReact best practices公式skillを配布している。8カテゴリ45ルール。ウォーターフォール防止、bundle最適化、server-sideパターン、re-render防止。フレームワークを作った人々が、自分たちのフレームワークをどう使ってほしいかを正確にエンコードした。

これが何を意味するか考えてほしい。何年もの間、best practiceは開発者が読むかもしれないし読まないかもしれないドキュメントの中にあった。古くなるブログ記事。半分しか覚えていないカンファレンスの発表。今やフレームワークの作者が、AIが従わなければならない機械可読な制約を配布している。読むかもしれないガイドではない。無視できないガードレールだ。

全ての主要ライブラリがskillを配布するようになれば、AI支援コードのベースライン品質は全体的に上がるだろう。今は私のカスタムskillがギャップを埋めている。公式のものがいずれほとんどをカバーするようになるはずだ。

skillについての核心的な洞察は、それがドキュメントではないということだ。ドキュメントは記述する。skillは制約する。Claude Codeがセキュリティチェックリストのskillを読み込むと、OWASP top 10について知っているだけではない。全項目を確認することを強制される。データベースskillを読み込むと、N+1クエリについて知っているだけではない。それを探すことを義務づけられる。知っていることと強制されることの違いは、教科書と、自分の基準を満たすまで承認しない同僚との違いだ。

## 仮想チーム

ここで「チーム」の感覚が本当に立ち上がる。私は7つのagentを動かしている。特定のタスクを自律的に処理する専門化されたサブプロセスだ。

セキュリティレビュアーは全ての実装後に実行される。インジェクション脆弱性、secret漏洩、race condition、IDORをスキャンする。発見を重要度で分類する。支払いロジックや認証に触れるものは直接私にエスカレーションされる。自動化すべきでない判断があるからだ。このagentは、今まで一緒に働いた中で最もパラノイアなセキュリティエンジニアの性格を持っている。ただし、パラノイアであることに疲れない。

テスターは行、ブランチ、関数、statementの全てで完全なカバレッジを目指す。何が変わったかを分析し、テスト戦略を立て、テストを書き、実行し、カバレッジが完成するまで反復する。厳格なルールが一つある。アプリケーションコードは絶対に変更しない。テストファイルだけだ。この制約が重要なのは、テスターが実装が提供するものをそのまま扱わなければならないということだ。「コードをテストしやすく直す」ことはできない。本物のQAエンジニアと同じだ。

データベースレビュアーはシニアDBAの視点を取る。核心的な問いは「このschemaは現実を生き延びるか」だ。欠落したforeign key、混在するID戦略、ドキュメント化されていない非正規化、全てnullableのアンチパターン。マルチテーブルのマイグレーションはエスカレーションされる。primary key戦略の変更もエスカレーションされる。featureに集中している時に私が忘れる質問を、このagentが問いかける。

他にもUIエンジニア、web vitals監査、UXデザイナー、コピーライターがいる。各agentはそれぞれのレンズを持つ。印象的なのは、各agentが私が理論的には持っているが同時には維持できない視点を体現していることだ。セキュリティが重要なことは知っている。テストカバレッジが重要なことも知っている。schema設計が重要なことも知っている。だがfeatureの構築に没頭しているとき、考えているのはfeatureのことだ。agentたちにはその問題がない。自分の一つのことだけを考え、毎回それを考える。

## 結合組織

skillとagentだけでも役立つ。それらをチームのように感じさせるのは結合組織だ。Claude Codeを私が実際に使っているシステムに接続するMCPサーバーがそれにあたる。

Context7は任意のライブラリの最新ドキュメントを提供するので、古い知識で作業することがない。Playwrightはテストと検証のためにブラウザを操作できるようにする。Sentryはプロダクションエラーをデバッグセッションに直接流し込むので、午前2時に何かが壊れても、朝座った時にはすでにコンテキストがある。Neonはマイグレーション実行、schema検査、クエリチューニングをターミナル間のコピペなしで可能にする。GitHubはPR、issue、コード検索への完全なアクセスを与える。CloudflareはWorkers、KV store、D1データベースの管理を可能にする。

各接続は個別には単純だ。だが合わせると、Claude Codeが人間のチームメンバーと同じコンテキストで作業することを意味する。本番データベースや最新のエラー率について推測しながら孤立して作業するのではない。生きたシステムに接続されている。

## 複利効果

予想しなかったことがある。個々の部品は控えめだ。セキュリティチェックリスト。Reactルールのセット。DBAレビュープロセス。エラートラッキング連携。どれか一つだけを見れば、「いいけど、ただのチェックリストだな」と思うだろう。

複利効果が驚きを生む。セキュリティレビュアーがテスターと並列で実行され、両方がデータベースレビュアーのschema分析にアクセスでき、全てがフレームワーク固有のskillの制約下で動作し、その全てがMCPサーバー経由でリアルタイムの本番データに接続されている時、結果は部品の合計ではない。質的に異なるものだ。全ての変更、全てのプロジェクトで一貫して維持される品質基準。私が活力に満ちていようが疲弊していようが、集中していようが散漫であろうが関係ない。

これは実のところ、私が経験してきたほとんどの人間のチームより一貫している。人間のチームには悪い日がある。締め切りのプレッシャーの下で手を抜く。普段なら全てをキャッチするシニアエンジニアが、転職活動中で気が散っている。品質基準は変動する。私のagentチームの品質基準は変動しない。構造的であって、動機に依存していないからだ。

投資は実質的なものだった。21のskillを構築し、7つのagentを設定し、各プロジェクトの詳細な憲法を書く。数週間かかった。だが投資はシステムに対してであって、個別のプロジェクトに対してではない。今や全ての新しいプロジェクト、全ての新しいfeatureが、蓄積された専門知識の恩恵を自動的に受ける。新しいプロジェクトでチームレベルの品質を維持するための限界費用はほぼゼロだ。

元の問いに戻る。一人の人間がチームを持つことはできるか。私が持っているものは人間的な意味でのチームではない。まだ適切な名前のない、何か新しいものだ。全ての成果物に関与する、持続的で専門化された一貫した視点の集合体。人間の同僚が持つ全く異なる人生経験から来る洞察は持てない。プロダクトの方向性そのものに異議を唱えることはできない。「そもそも解くべき問題が間違っている」と言うこともできない。

だが、一人の人間が創造的・戦略的な仕事をしている間に、全ての変更に対して例外なく複数の次元で同時に品質を維持するという特定の仕事に関しては、これまで経験したどんなものも及ばない。

セットアップが投資だ。リターンはそこから複利で増えていく。
