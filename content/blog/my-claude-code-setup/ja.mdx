一人の人間がチームを持つことはできるのか。

数ヶ月この問いに向き合っていた。答えは自明に思えた。チームとは複数の人間の集まりだ。採用する。それぞれが異なる専門性を持ち込む。セキュリティ担当がプロダクト担当の死角を突き、DBAがフロントエンド開発者が考えもしないschemaの問題を指摘する。一人がどんなに優秀でも、一人がチームにはなれない。

だが今は違う考えを持っている。「チーム」という言葉は、我々が思っていたより具体的な何かを指していた。チームに本当に求めているのは人数ではない。異なる視点が一貫して適用されることだ。セキュリティエンジニアが貴重なのは、椅子をもう一つ埋めるからではない。自分にはないレンズでコードを見て、それを毎回やるからだ。気分次第ではなく。

この区別が重要だ。扉が一つ開く。複数の人間なしに、複数の一貫した視点を得ることはできるのか。

私はソロ開発者として複数のプロダクトを運営している。最も難しかったのはコンテキストスイッチだった。セキュリティ、データベース、UX間の切り替えだけではない。プロダクトの方向性。分析。グロース。マーケティング。全て学ぶことはできる。同時に適用することはできない。機能実装に没頭している時にchurn rateのことは考えない。コードを書いている時にローンチ戦略を自発的に立てることはない。だから企業には部門が存在する。

Claude Codeのセットアップがこれを変えた。今やっていることを速くしたのではない。部門と呼ぶしかない何かを生み出した。要請するかどうかに関わらず、私の作業に関与する持続的で専門化された視点。AIがコーディングを手伝えることは驚きではない。驚きは、適切な設定がプロダクトマネージャー、セキュリティエンジニア、マーケター、データアナリストを同時に、一貫して生み出すことだ。

アーキテクチャを説明する。個々のピースは単純だ。組み合わせが予想外のことをする。

## 憲法

全てのプロジェクトはルートに`CLAUDE.md`というファイルを置くところから始まる。そのcodebaseにおけるAIの振る舞いを規定する憲法だ。tech stack、アーキテクチャパターン、制約条件、そして全ての変更に対してどのskillとagentを使わなければならないかを定義する。全てのプロジェクトに共通するコアパターン：

```
1. 全ての実装は指定されたskillを使用しなければならない
2. 実装後、2つのagentを並列で実行:
   - security-reviewer → 監査 → 修正
   - verifier → テスト実行、E2E、ブラウザ検証 → 報告
```

これは提案ではない。プロジェクトのDNAに刻まれたハードルールだ。Claude Codeは毎セッション開始時にこのファイルを読む。リマインドは不要だ。

だが憲法の役割は品質ゲートの強制だけではない。作業の種類ごとのskillパイプライン — 強制された順序を定義する。APIを構築する？まずデータベース設計、次にAPI仕様、次にフレームワーク固有のクエリパターンで実装。Webの機能？デザインシステム、次にコンポジションパターン、次にパフォーマンス最適化。蓄積されたベストプラクティスを正しい順序で、毎回強制するワークフローだ。

以前の私は、気が向いた時にテストを書き、覚えている時にセキュリティレビューをする類の開発者だった。今はシステムがその選択肢を与えない。一貫性は規律の問題ではなくなった。構造の問題になった。

## 制約としてのskill

29のskillを運用している。興味深いのは、ほぼ半数がコードと無関係だということだ。

技術的なものは当然ある。アーキテクチャ設計。データベースモデリング。API仕様。フロントエンドパターン。モバイル開発。国際化。セキュリティチェックリスト。3D Web。VercelはReact best practicesの公式Claude Code skillを配布している。8カテゴリ45ルール。フレームワークの作者が、自分たちのツールの使い方を正確にエンコードした。読むかもしれないガイドではない。無視できないガードレールだ。

だがその隣に、コピーライティング、メールマーケティング、SNSコンテンツ、広告クリエイティブ、価格戦略、競合分析、コンバージョン最適化、churn防止、検索可視性、プロダクトブリーフ、PRDのためのskillが並んでいる。

計画したわけではない。コード品質のためのskillを作り始めて、止まらなかった。SQLインジェクションを防ぐアーキテクチャが、悪いマーケティングコピーも同様に防ぐことが分かった。skillはドキュメントではない。ドキュメントは記述する。skillは制約する。Claude Codeがセキュリティチェックリストを読み込むと、OWASP top 10を知っているだけではない。全項目を確認することを強制される。コピーライティングskillを読み込むと、説得原則とブランドボイスのガイドラインに従うことを強制される。知っていることと強制されることの違いは、教科書と、自分の基準を満たすまで承認しない同僚の違いだ。

## 仮想チーム

ここで「チーム」の感覚が本当に立ち上がる。7つのagentを3つの部門に分けて運用している。

**Dev**にagentが2つ。セキュリティレビュアーは全ての実装後に実行される。インジェクション脆弱性、secret漏洩、race condition、アクセス制御の欠陥をスキャンする。重要度で分類し、支払いや認証に触れるものは直接私にエスカレーションする。自動化すべきでない判断があるからだ。ベリファイアはテストスイート全体をカバレッジ付きで実行し、E2Eテストを走らせ、実際のブラウザを開いてUIが正しく動作するか確認する。テストは書かない。存在するものを検証し、何が壊れたかを正確に報告する。この2つは全ての変更で並列実行される。

**Plan**にagentが2つ。プロダクトマネージャーがプロダクトブリーフとPRDを書く。問題のフレーミング、成功指標、詳細な要件定義。UXデザイナーが情報アーキテクチャ、ユーザーフロー、インタラクションデザインを認知原則とアクセシビリティ基準に基づいて行う。新しい機能を始める時、コードを1行書く前にこの2つが何を作るかを定義する。

**Biz**にagentが3つ。これは予想しなかった部分だ。データアナリストがPostHogに接続し、週次分析レポートを作成し、リテンション分析を実行し、機能ごとのKill/Keep/Scale推奨を出す。グロースオプティマイザーがリファラルプログラムを設計し、コンバージョンファネルを最適化し、churn防止を担当する。マーケターがローンチ戦略を立て、コンテンツカレンダーを書き、競合ポジショニングを管理する。

コード品質のためのシステムを作り始めて、結局ビジネス全体をカバーすることになった。各agentが、理論的には重要だと知っている視点を体現している。セキュリティが重要なのは知っている。リテンションが重要なのも知っている。だがコードを書いているとき、頭にあるのはコードだけだ。agentにはその問題がない。それぞれが自分の一つの関心を、毎回見ている。

## 結合組織

これらのagentがスクリプトではなく同僚のように感じられるのは、実際のシステムに接続されているからだ。21のpluginがインフラを提供する。3つのLSPサーバーがTypeScript、Python、Rustの型レベルインテリジェンスを与える。Expo、Pulumi、Vercelのフレームワークpluginが各自のコンベンションを持ち込む。ユーティリティpluginがダイアグラム生成、コード簡素化、新しいskillの作成を担当する。

その上に、MCPサーバーがagentを必要な生きたシステムに接続する。Context7が任意のライブラリのリアルタイムドキュメントを提供する。Playwrightがテストのためにブラウザを駆動する。Sentryがプロダクションエラーをデバッグセッションに流し込む。PostHogがデータアナリストにライブの分析データを供給する。Neonがデータベースマイグレーションを処理する。GitHubがPRとissueへの完全なアクセスを与える。CloudflareがWorkersとエッジストレージを管理する。Supabase、Slack、Gmail、Google Calendarが残りを埋める。

各接続は個別には単純だ。合わせると、データアナリストがメトリクスを推測するのではなく実データをクエリしているということだ。ベリファイアがUIの動作を推測するのではなくブラウザを開いて確認しているということだ。

## 複利効果

個々のピースは控えめだ。セキュリティチェックリスト。Reactルール。分析ダッシュボード。マーケティングプレイブック。どれか一つだけ見れば、ただのチェックリストに見える。

驚きは、それらが複利的に作用する時に生まれる。セキュリティレビュアーがベリファイアと並列で実行され、両方がフレームワーク固有のskill制約下で動作し、全てがライブのプロダクションシステムに接続されている時。結果は部品の合計ではない。自分のコンディションに関係なく一貫する品質だ。

投資は実質的だった。29のskill、3つの部門にまたがる7つのagent、21のplugin、プロジェクトごとの詳細な憲法。数週間のセットアップ。だが投資はシステムに対してであって、個別のプロジェクトではない。全ての新しいプロジェクトが自動的に恩恵を受ける。この運営レベルを新しいプロジェクトで維持する限界費用はほぼゼロだ。

一人の人間がチームを持つことはできるか。私が持っているものは人間的な意味でのチームではない。まだ適切な名前のない何かだ。専門化された視点の3つの部門が、私が生み出す全ての成果物に関与する。解くべき問題が間違っていると教えてくれることはできない。人間の同僚だけが持ち込める視点もない。

だが複数の次元で品質を例外なく維持しながら、一人の人間が戦略的思考を行うということに関しては。これまで経験したどんなものも及ばない。

セットアップが投資だ。リターンはそこから複利で増えていく。
