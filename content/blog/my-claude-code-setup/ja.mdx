最初にClaude Codeを使ったとき、ただの賢い自動補完だった。今は違う。プロジェクトの規約に従い、セキュリティレビューを実行し、100%カバレッジのテストを書き、シニアDBAのようにデータベーススキーマをレビューする。すべてのプロジェクトに同じ品質基準が自動的に適用される。

ツールが特別なわけじゃない。セットアップが違うだけだ。僕の完全なClaude Codeの設定を紹介する。

## CLAUDE.mdシステム

すべてのプロジェクトのルートに`CLAUDE.md`ファイルがある。そのコードベースにおけるAIの行動の憲法みたいなものだ。テックスタック、アーキテクチャパターン、主要な制約、そして最も重要なことに — **どのスキルとエージェントを使うべきか**を定義する。

すべてのプロジェクトに共通するクリティカルなパターン：

```
1. すべての実装は指定されたスキルを使用すること
2. 実装後、2つのエージェントを並列実行：
   - security-reviewer → 監査 → 修正
   - tester → 変更コードのテスト → 修正
```

これは提案ではない。ハードルールだ。すべての機能、すべてのバグ修正、すべてのリファクタリングがこのパイプラインを通る。`CLAUDE.md`ファイルがこれを自動化する — Claude Codeはすべてのセッション開始時にこのファイルを読み、言われなくてもワークフローに従う。

Next.jsのウェブサイトでは、CLAUDE.mdが`vercel-react-best-practices`と`vercel-composition-patterns`スキルを義務付けている。Rust APIでは`axum`と`postgresql`。React Nativeアプリでは`react-native`と`vercel-react-native-skills`。コードの1行目が書かれる前に、適切な専門知識がロードされる。

## 21のスキル：エンコードされた専門知識

スキルはClaude Codeがオンデマンドでロードするナレッジファイルだ。自分で書いたものもあれば、公式ソースのものもある。7つのドメインにわたって21のスキルを使っている：

**アーキテクチャ＆デザイン** — `data-modeling`、`api-design`、`ux-design`、`design-system`。システム設計に対する僕の考え方をエンコード：エンティティ抽出、RESTの規約、認知エルゴノミクス、デザイントークン。

**Webフレームワーク** — `vercel-react-best-practices`（8カテゴリにわたる45のルール）、`vercel-composition-patterns`、`nextjs`。`vercel-*`の接頭辞がついたスキルは**Vercelの公式提供** — 僕が書いたものではない。Reactベストプラクティスのスキルだけでも、ウォーターフォール防止、バンドル最適化、サーバーサイドパターン、再レンダリング回避、レンダリング戦略をカバーする。フレームワークベンダーがスキルを提供するのは画期的だ：フレームワークを作った人たちが、こう使ってほしいという使い方をエンコードしている。`nextjs`は公式スキルがカバーしていないギャップを埋めるために僕が書いたカスタムスキルだ。

**モバイル** — `react-native`、`vercel-react-native-skills`（こちらもVercel公式）。リストパフォーマンス、アニメーション、React Native固有の状態管理パターン。

**バックエンド** — `axum`、`axum-hexagonal`、`fastapi`、`fastapi-hexagonal`。RustとPythonの両スタックにヘキサゴナルアーキテクチャパターンを組み込み。

**データベース** — `postgresql`、`sqlite`、`neon-postgres`。ページネーション、全文検索、N+1クエリ、ゼロダウンタイムマイグレーション、パーティショニングの問題解決パターン。

**AI/ML** — `langgraph`。ReActエージェント、マルチエージェントシステム、ヒューマンインザループパターン。

**品質** — `security-checklists`、`web-vitals-checklist`、`i18n-patterns`、`web-design-guidelines`。

キーとなるインサイト：スキルはドキュメントではない。**制約**だ。Claude Codeが`vercel-react-best-practices`をロードすると、Reactパターンを知るだけでなく — 45の具体的なルールに従うことを強制される。ウォーターフォール防止はCRITICAL優先度。バンドルサイズ最適化はCRITICAL。これらはガイドラインではない。ガードレールだ。

これは大きなトレンドになると思う。より多くのフレームワーク作者やツールベンダーが公式スキルを公開するにつれて、AI生成コードのベースライン品質は全体的に向上する。すべての主要ライブラリが、読むかもしれないドキュメントとしてではなく、AIが従わなければならない制約としてベストプラクティスをエンコードしたスキルを提供する世界を想像してみてほしい。僕が書いたカスタムスキルは今日のギャップを埋めているが、公式スキルはカバーするドメインですでにより優れている。

## 7つのエージェント：バーチャルチーム

エージェントは特定のタスクを自律的に処理する専門サブプロセスだ。開発と計画に分かれて7つある：

### 開発エージェント

**security-reviewer** — すべての実装後にセキュリティ監査を実行。OWASP Top 10のスキャン、秘密情報の露出、SQLインジェクション、コマンドインジェクション、XSS、IDOR、レースコンディションをチェック。重要度で分類。決済ロジック、認証変更、暗号関連、コンプライアンス関連のコードは僕にエスカレーション。

**tester** — 行、ブランチ、関数、ステートメントの全メトリクスで100%テストカバレッジを目標にする。変更されたコードを分析し、テスト戦略を計画し、テストを書き、実行し、カバレッジが完了するまで反復する。重要なルール：アプリケーションコードは決して変更しない。テストファイルのみ。

**database-reviewer** — シニアDBAの視点でレビュー。重要な質問を投げかける：「このスキーマは現実に耐えるか？」FK制約の欠落、IDストラテジーの混在、文書化されていない非正規化、何でもnullableのアンチパターンをチェック。複数テーブルのマイグレーションとPKストラテジーの変更はエスカレーション。

**ui-engineer** — ビジネスロジックゼロの純粋なUIコンポーネントを構築。Webではshadcn/uiまたはデザインシステムスキルに従ったカスタムコンポーネントを使用。モバイルではデザインシステムスキルとプラットフォームネイティブパターンを使用。厳密な境界：状態管理なし、データフェッチなし、ルーティングなし。

**web-vitals-auditor** — 自動パフォーマンス監査。アプリをビルドしてサーブし、デスクトップとモバイルの両方でLighthouseを実行し、ターゲットと照合：LCP < 2.5s、INP < 200ms、CLS < 0.1、TTFB < 800ms、FCP < 1.8s。スコア、バンドル分析、具体的な推奨事項を含むレポートを出力。

### 計画エージェント

**ux-designer** — ファーストプリンシプルからデザインする。5ステッププロセス：問題定義→ユーザージャーニーマッピング→インタラクション設計→摩擦の除去→アクセシビリティ検証。プライマリアクションと情報階層付きのユーザーフロー、画面仕様を出力。

**copywriter** — プロダクトコピーとUXマイクロコピー。6つの説得原則：一つのメッセージのみ、確実性は最大報酬に勝る、新規性だけで十分、具体的なアクションを指定、集約の力、一つのメッセージとして書く。文字数とA/Bテスト提案付きの2-3バリエーションを提供。

UXデザイナーとコピーライターはパイプラインとして機能する：デザイナーがフローを作成し、その後コピーライターに引き渡して説得力のあるテキスト、CTA、オンボーディングコピー、エラーメッセージを作成する。

## ワークフローパイプライン

本当のパワーは個々のツールではなく — 再現可能なワークフローとしてどう組み合わさるかにある：

**標準開発：**
```
実装（スキル付き）→ [security-reviewer + tester]（並列）→ 修正 → 完了
```

**データモデリング：**
```
data-modelingスキル → database-reviewerエージェント → api-designスキル → 実装
```

**フロントエンド：**
```
nextjsスキル → 実装 → vercel-react-best-practices（レビュー）
```

**UXデザイン：**
```
ux-designerエージェント → copywriterエージェント（説得力のあるテキストのみ）
```

すべてのプロジェクトがCLAUDE.mdを通じて同じ開発ワークフローを強制する。データベース変更が関わる場合、スキーマレビューパイプラインが自動的に走る。セキュリティ＆テストの並列パスはすべての変更で発生する。例外なし。

## MCPサーバー：外部統合

Claude CodeはMCP（Model Context Protocol）サーバーを通じて外部サービスに接続する。僕のセットアップ：

- **Context7** — あらゆるライブラリのリアルタイムドキュメント検索
- **Playwright** — テストとスクリーンショットのためのブラウザ自動化
- **Sentry** — 本番問題のデバッグのためのエラートラッキング統合
- **Neon** — データベース操作とスキーマ管理の直接接続
- **Notion** — プロジェクト管理のためのワークスペース統合
- **GitHub** — 完全なGitHubワークフロー（PR、Issue、コード検索）
- **D2** — アーキテクチャ図の生成（テーマ付きSVGにコンパイル）
- **Cloudflare** — Workers、KV、R2、D1の管理

これらはあると嬉しい程度のものではない。Neon MCPがあれば、Claude Codeは僕がコンテキストスイッチングすることなく、マイグレーションの実行、スキーマの検査、クエリのチューニングができる。Sentry MCPがあれば、本番エラーがデバッグセッションに直接流れ込む。D2 MCPはプロジェクトドキュメントの一部として出荷されるアーキテクチャ図を生成する。

## 実際にどう見えるか

典型的なセッション：自然言語で機能を説明する。Claude CodeがCLAUDE.mdを読み、必要なスキルをロードし、実装を開始する。完了したら、2つのエージェントが並列で立ち上がる — セキュリティレビュワーが脆弱性をスキャンし、テスターが100%カバレッジのテストを書く。どちらかで問題が見つかれば修正される。そして僕がレビューしてリリース。

リンターを手動で実行しない。ボイラープレートテストを書かない。記憶を頼りにセキュリティチェックリストをこなさない。システムが処理してくれる。

投資は、21のスキルを構築し、7つのエージェントを設定し、各プロジェクトに詳細なCLAUDE.mdファイルを書くことだった。時間はかかった。でも今では、すべてのプロジェクトのすべての新機能が、蓄積されたナレッジの恩恵を自動的に受ける。

## 複利効果

各スキル、エージェント、CLAUDE.mdファイルは個別にはシンプルだ。セキュリティチェックリスト。一連のReactルール。DBAレビュープロセス。でも組み合わさると、チームがいるような感覚を生む：僕のパターンを知るアーキテクト、眠らないセキュリティエンジニア、100%カバレッジを譲らないQAリード、「これ現実に耐えるか？」と聞いてくるDBA。

複数のプロダクトを一人で出荷するソロ開発者にとって、それが燃え尽きるかスケールアップするかの分かれ目だ。

バイブは良い感じだ。エージェントたちが後ろを固めてくれる。
