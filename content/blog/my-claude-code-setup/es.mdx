Cuando usé Claude Code por primera vez, era solo un autocompletado inteligente. Ya no. Sigue las convenciones del proyecto, ejecuta revisiones de seguridad, escribe tests al 100% de cobertura y revisa schemas de bases de datos como un DBA senior. Cada proyecto recibe el mismo estándar de calidad, aplicado automáticamente.

No es que la herramienta sea excepcional. El setup es diferente. Aquí está mi configuración completa de Claude Code.

## El Sistema CLAUDE.md

Cada proyecto tiene un archivo `CLAUDE.md` en su raíz. Piénsalo como una constitución para el comportamiento de la IA en ese codebase. Define el tech stack, patrones de arquitectura, constraints clave y — lo más importante — **qué skills y agentes deben usarse**.

El patrón crítico en todos mis proyectos:

```
1. Toda implementación DEBE usar los skills especificados
2. Después de la implementación, ejecutar DOS agentes EN PARALELO:
   - security-reviewer → auditoría → corrección
   - tester → testear código modificado → corrección
```

Esto no es una sugerencia. Es una regla dura. Cada feature, cada bug fix, cada refactor pasa por este pipeline. El archivo `CLAUDE.md` lo hace automático — Claude Code lo lee al inicio de cada sesión y sigue el flujo sin que se lo pidan.

Para mi sitio Next.js, el CLAUDE.md exige los skills `vercel-react-best-practices` y `vercel-composition-patterns`. Para mi API en Rust, es `axum` y `postgresql`. Para la app React Native, `react-native` y `vercel-react-native-skills`. La expertise correcta se carga antes de que se escriba la primera línea de código.

## 21 Skills: Expertise Codificada

Los skills son archivos de conocimiento que Claude Code carga bajo demanda. Algunos los escribí yo, y otros vienen de fuentes oficiales. Uso 21 en siete dominios:

**Arquitectura y Diseño** — `data-modeling`, `api-design`, `ux-design`, `design-system`. Estos codifican cómo pienso sobre diseño de sistemas: extracción de entidades, convenciones REST, ergonomía cognitiva, design tokens.

**Frameworks Web** — `vercel-react-best-practices` (45 reglas en 8 categorías), `vercel-composition-patterns`, `nextjs`. Los skills con prefijo `vercel-*` son **proporcionados oficialmente por Vercel** — no algo que yo escribí. Solo el skill de mejores prácticas de React cubre prevención de waterfall, optimización de bundle, patrones server-side, prevención de re-renders y estrategias de renderizado. Que los vendors de frameworks publiquen skills como estos es un game changer: la gente que construyó el framework codifica exactamente cómo quieren que lo uses. `nextjs` es un skill custom que escribí para llenar los vacíos que los oficiales no cubren.

**Móvil** — `react-native`, `vercel-react-native-skills` (también oficial de Vercel). Performance de listas, animaciones, patrones de gestión de estado específicos para React Native.

**Backend** — `axum`, `axum-hexagonal`, `fastapi`, `fastapi-hexagonal`. Tanto stacks de Rust como Python con patrones de arquitectura hexagonal integrados.

**Base de Datos** — `postgresql`, `sqlite`, `neon-postgres`. Patrones de resolución de problemas para paginación, búsqueda full-text, queries N+1, migraciones sin downtime, particionamiento.

**AI/ML** — `langgraph`. Agentes ReAct, sistemas multi-agente, patrones human-in-the-loop.

**Calidad** — `security-checklists`, `web-vitals-checklist`, `i18n-patterns`, `web-design-guidelines`.

La idea clave: los skills no son documentación. Son **constraints**. Cuando Claude Code carga `vercel-react-best-practices`, no solo conoce patrones de React — está obligado a seguir 45 reglas específicas. La prevención de waterfall tiene prioridad CRITICAL. La optimización del tamaño del bundle es CRITICAL. Estas no son guías; son barandillas.

Espero que esto se convierta en una tendencia mayor. A medida que más autores de frameworks y vendors de herramientas publiquen skills oficiales, la calidad base del código generado por IA subirá en todos los ámbitos. Imagina que cada librería importante lance un skill que codifique sus mejores prácticas — no como docs que podrías leer, sino como constraints que la IA debe seguir. Los skills custom que escribí llenan los vacíos hoy, pero los oficiales ya son mejores para los dominios que cubren.

## 7 Agentes: Un Equipo Virtual

Los agentes son sub-procesos especializados que manejan tareas específicas de forma autónoma. Tengo siete, divididos entre desarrollo y planificación:

### Agentes de Desarrollo

**security-reviewer** — Ejecuta una auditoría de seguridad después de cada implementación. Escanea OWASP top 10, verifica exposición de secrets, SQL injection, command injection, XSS, IDOR, race conditions. Clasifica hallazgos por severidad. Me escala a mí para lógica de pagos, cambios de auth, criptografía o código relacionado con cumplimiento.

**tester** — Apunta a 100% de cobertura de tests en métricas de línea, branch, función y statement. Analiza el código modificado, planifica la estrategia de testing, escribe tests, los ejecuta e itera hasta que la cobertura esté completa. Regla importante: nunca modifica código de aplicación, solo archivos de test.

**database-reviewer** — Toma la perspectiva de un DBA senior. Hace las preguntas que importan: "¿Este schema sobrevivirá la realidad?" Verifica foreign key constraints faltantes, estrategias de ID mixtas, desnormalización sin documentar, anti-patrones de nullable-everything. Escala migraciones multi-tabla y cambios de estrategia de PK.

**ui-engineer** — Construye componentes de UI puros con cero lógica de negocio. En web, usa shadcn/ui o componentes custom siguiendo el skill de design system. En móvil, usa el skill `design-system` con patrones nativos de la plataforma. Límite estricto: sin gestión de estado, sin data fetching, sin routing.

**web-vitals-auditor** — Auditoría automatizada de performance. Construye y sirve la app, ejecuta Lighthouse en desktop y móvil, verifica contra objetivos: LCP < 2.5s, INP < 200ms, CLS < 0.1, TTFB < 800ms, FCP < 1.8s. Genera un reporte con scores, análisis de bundle y recomendaciones accionables.

### Agentes de Planificación

**ux-designer** — Diseña desde primeros principios. Proceso de cinco pasos: Definir el problema → Mapear user journeys → Diseñar interacciones → Eliminar fricción → Validar accesibilidad. Genera flujos de usuario, specs de pantalla con acciones primarias y jerarquía de información.

**copywriter** — Copy de producto y microcopy de UX con seis principios de persuasión: un solo mensaje, la certeza vence a la recompensa máxima, la novedad es suficiente, especificar acción concreta, poder de la agregación, escribir como un solo mensaje. Entrega 2–3 variaciones con conteo de caracteres y sugerencias de A/B test.

El diseñador de UX y el copywriter trabajan como pipeline: el diseñador crea el flujo, luego pasa al copywriter para texto persuasivo, CTAs, copy de onboarding y mensajes de error.

## Pipelines de Trabajo

El poder real no está en las herramientas individuales — está en cómo se componen en flujos repetibles:

**Desarrollo Estándar:**
```
Implementación (con skills) → [security-reviewer + tester] (paralelo) → Corrección → Listo
```

**Modelado de Datos:**
```
skill data-modeling → agente database-reviewer → skill api-design → implementación
```

**Frontend:**
```
skill nextjs → implementación → vercel-react-best-practices (review)
```

**Diseño UX:**
```
agente ux-designer → agente copywriter (solo texto persuasivo)
```

Cada proyecto aplica el mismo flujo de desarrollo a través de CLAUDE.md. El pipeline de revisión de schema se ejecuta automáticamente cuando hay cambios en la base de datos. El pase paralelo de seguridad-y-testing ocurre en cada cambio. Sin excepciones.

## Servidores MCP: Integraciones Externas

Claude Code se conecta a servicios externos a través de servidores MCP (Model Context Protocol). Mi setup incluye:

- **Context7** — Consulta de documentación en tiempo real para cualquier librería
- **Playwright** — Automatización de navegador para testing y capturas de pantalla
- **Sentry** — Integración de error tracking para debugging de issues en producción
- **Neon** — Operaciones directas de base de datos y gestión de schemas
- **Notion** — Integración de workspace para gestión de proyectos
- **GitHub** — Flujo completo de GitHub (PRs, issues, búsqueda de código)
- **D2** — Generación de diagramas de arquitectura (compilados a SVGs con temas)
- **Cloudflare** — Gestión de Workers, KV, R2, D1

Estos no son solo nice-to-haves. El MCP de Neon significa que Claude Code puede correr migraciones, inspeccionar schemas y optimizar queries sin que yo cambie de contexto. El MCP de Sentry significa que los errores de producción alimentan directamente las sesiones de debugging. El MCP de D2 genera diagramas de arquitectura que se envían como parte de la documentación del proyecto.

## Cómo Se Ve Esto en la Práctica

Una sesión típica: describo un feature en lenguaje natural. Claude Code lee el CLAUDE.md, carga los skills requeridos y empieza a implementar. Cuando termina, dos agentes arrancan en paralelo — el security reviewer escaneando vulnerabilidades, el tester escribiendo tests al 100% de cobertura. Si alguno encuentra problemas, se corrigen. Luego yo reviso y lanzo.

No ejecuto linters manualmente. No escribo tests de boilerplate. No hago checklists de seguridad de memoria. El sistema lo maneja.

La inversión fue construir los 21 skills, configurar los 7 agentes y escribir archivos CLAUDE.md detallados para cada proyecto. Eso tomó tiempo. Pero ahora cada nueva feature, en cada proyecto, se beneficia de ese conocimiento acumulado — automáticamente.

## El Efecto Compuesto

Cada skill, agente y archivo CLAUDE.md es individualmente simple. Un checklist de seguridad. Un set de reglas de React. Un proceso de revisión de DBA. Pero compuestos juntos, crean algo que se siente como tener un equipo: un arquitecto que conoce mis patrones, un ingeniero de seguridad que nunca duerme, un líder de QA que insiste en 100% de cobertura, y un DBA que pregunta "¿esto sobrevivirá la realidad?"

Para un desarrollador solo lanzando múltiples productos, esa es la diferencia entre quemarse y escalar.

Las vibes son buenas. Los agentes me cubren las espaldas.
