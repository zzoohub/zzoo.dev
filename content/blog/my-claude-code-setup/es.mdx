¿Puede una sola persona tener un equipo?

Pasé meses dándole vueltas a esta pregunta porque la respuesta parecía obvia. Un equipo es un grupo de personas. Contratas a alguien de seguridad, de bases de datos, de testing. Cada uno mira tu código desde un ángulo que tú no puedes mantener. Una persona sola no puede ser un equipo.

Excepto que la palabra "equipo" apuntaba a algo más preciso de lo que creíamos. Lo que realmente quieres de un equipo no son más personas. Son más perspectivas aplicadas de manera consistente. La ingeniera de seguridad no es valiosa porque ocupa una silla extra. Es valiosa porque mira tu código a través de un lente que tú no tienes, y lo hace siempre, no solo cuando está de humor.

Esa distinción abre una puerta. ¿Puedes obtener múltiples perspectivas consistentes sin múltiples personas?

Desarrollo varios productos como desarrollador solo. Lo más difícil solía ser el cambio de contexto, y no solo entre pensamiento de seguridad, bases de datos y UX. También dirección de producto. Analítica. Crecimiento. Marketing. Puedes aprender todo eso. No puedes aplicarlo simultáneamente. Cuando estás profundo implementando una feature, no te detienes a pensar en tu tasa de abandono. Cuando escribes código, no planeas espontáneamente una estrategia de lanzamiento. Por eso las empresas tienen departamentos.

Mi configuración de Claude Code cambió esto. No haciéndome más rápido en lo que ya hago. Sino creando algo que genuinamente se siente como departamentos: perspectivas persistentes y especializadas que se involucran con mi trabajo, lo pida yo o no. La sorpresa no es que la IA ayuda a programar. La sorpresa es que la configuración correcta te da un product manager, un ingeniero de seguridad, un marketer y un analista de datos. Simultáneamente. Consistentemente.

Déjame explicar la arquitectura, porque cada pieza es simple. Es la composición la que produce algo inesperado.

## La constitución

Todo proyecto empieza con un archivo llamado `CLAUDE.md` en la raíz — una constitución para cómo la IA se comporta en ese código. Define el stack, patrones de arquitectura, restricciones, y qué skills y agentes deben usarse en cada cambio. El patrón central que aparece en todos mis proyectos:

```
1. Toda implementación DEBE usar los skills especificados
2. Después de implementar, ejecutar dos agentes en paralelo:
   - security-reviewer → auditoría → corrección
   - verifier → ejecutar tests, E2E, verificación en navegador → reporte
```

No es una sugerencia. Es una regla dura grabada en el ADN del proyecto. Claude Code lee este archivo al inicio de cada sesión. No necesita que se lo recuerden.

Pero la constitución hace más que imponer gates de calidad. Define pipelines de skills — secuencias forzadas específicas para cada tipo de trabajo. ¿Construyendo un API? Diseño de base de datos primero, luego especificación del API, luego implementación con patrones de consulta del framework. ¿Feature web? Sistema de diseño, luego patrones de composición, luego optimización de rendimiento. Cada workflow es una marcha forzada a través de mejores prácticas acumuladas, en el orden correcto, cada vez.

Yo era de esos que escribían tests cuando les daba la gana y hacían revisiones de seguridad cuando se acordaban. Ahora el sistema no me da esa opción. La consistencia dejó de ser un problema de disciplina y se volvió estructural.

## Skills como restricciones

Tengo 29 skills, y aquí es donde se pone interesante: casi la mitad no tiene nada que ver con código.

Los obvios son técnicos. Diseño de arquitectura. Modelado de bases de datos. Especificaciones de API. Patrones de frontend. Desarrollo móvil. Internacionalización. Checklists de seguridad. Web 3D. Vercel distribuye skills oficiales de Claude Code para mejores prácticas de React — 45 reglas en ocho categorías. Los autores del framework codificaron cómo quieren que uses sus herramientas. No guías que podrías leer. Barreras que no puedes ignorar.

Pero junto a ellos hay skills de copywriting, email marketing, contenido para redes sociales, creativos de anuncios, estrategia de precios, análisis de competidores, optimización de conversión, prevención de abandono, visibilidad en buscadores, product briefs y PRDs.

No lo planifiqué. Empecé construyendo skills para calidad de código y no paré. La misma arquitectura que previene inyección SQL resultó prevenir igualmente bien el mal copy de marketing. Un skill no es documentación — la documentación describe, los skills restringen. Cuando Claude Code carga el checklist de seguridad, está obligado a verificar cada punto OWASP. Cuando carga el skill de copywriting, está obligado a seguir principios de persuasión y guías de voz de marca. La diferencia entre saber y estar obligado es la diferencia entre un libro de texto y un colega que no firma hasta que el trabajo cumple su estándar.

## El equipo virtual

Aquí es donde realmente emerge la sensación de "equipo." Ejecuto siete agentes organizados en tres departamentos.

**Dev** tiene dos agentes. El security reviewer se ejecuta después de cada implementación, escaneando inyecciones, exposición de secretos, race conditions y fallas de control de acceso. Clasifica por severidad y escala todo lo que toca pagos o autenticación — hay decisiones que no deberían automatizarse. El verifier ejecuta toda la suite de tests con cobertura, corre tests E2E, y luego abre un navegador real para verificar que la UI funciona correctamente. No escribe tests. Valida lo que existe y reporta exactamente qué se rompió. Estos dos corren en paralelo en cada cambio.

**Plan** tiene dos agentes. El product manager escribe briefs de producto y PRDs — definición de problemas, métricas de éxito, requisitos detallados. La UX designer maneja arquitectura de información, flujos de usuario y diseño de interacción, basándose en principios cognitivos y estándares de accesibilidad. Cuando empiezo una feature nueva, estos dos definen qué se construye antes de que se escriba una línea de código.

**Biz** tiene tres agentes. Esta es la parte que no esperaba. El data analyst se conecta a PostHog y produce reportes analíticos semanales, ejecuta análisis de retención y hace recomendaciones Kill/Keep/Scale sobre features. El growth optimizer diseña programas de referidos, optimiza funnels de conversión y maneja prevención de abandono. El marketer construye estrategias de lanzamiento, escribe calendarios de contenido y gestiona posicionamiento competitivo.

Piensa en lo que eso significa. Cuando construyo una feature, el security reviewer y el verifier revisan el código. Pero antes de eso, el product manager y la UX designer definieron qué construir. Y después del deploy, el data analyst mide el impacto, el growth optimizer experimenta con conversión, y el marketer resuelve la distribución. Planificar, construir, verificar, medir, crecer. Agentes especializados en cada etapa.

## El tejido conectivo

Lo que hace que estos agentes se sientan como colegas en vez de scripts es su conexión con sistemas reales. Veintiún plugins proveen la infraestructura — tres servidores LSP dan inteligencia a nivel de tipos para TypeScript, Python y Rust. Plugins de frameworks de Expo, Pulumi y Vercel traen sus propias convenciones. Plugins de utilidad manejan generación de diagramas, simplificación de código y creación de nuevos skills.

Sobre eso, servidores MCP conectan a los agentes con los sistemas vivos que necesitan. Context7 provee documentación de librerías en tiempo real. Playwright controla un navegador para testing. Sentry canaliza errores de producción a sesiones de debugging. PostHog alimenta al data analyst con analítica en vivo. Neon maneja migraciones de base de datos. GitHub da acceso completo a PRs e issues. Cloudflare gestiona Workers y almacenamiento edge. Supabase, Slack, Gmail y Google Calendar cubren el resto.

Cada conexión es simple. Juntas significan que el data analyst no adivina métricas — consulta datos reales. El verifier no adivina si la UI funciona — abre un navegador y verifica.

## El efecto compuesto

Cada pieza individualmente es modesta. Un checklist de seguridad. Reglas de React. Un dashboard analítico. Un playbook de marketing. Cualquiera de ellas parece solo un checklist.

La sorpresa es lo que pasa cuando se componen. Cuando el security reviewer corre en paralelo con el verifier, y ambos operan bajo restricciones de skills específicos del framework, y el product manager definió los requisitos, y el data analyst medirá los resultados, y el marketer lo distribuirá, y todo eso está conectado a sistemas de producción en vivo — el resultado no es la suma de las partes. Es una operación que se mantiene consistente en cada cambio, en cada proyecto, esté yo energizado o agotado.

La inversión fue real. Veintinueve skills, siete agentes en tres departamentos, veintiún plugins, constituciones detalladas para cada proyecto. Semanas de setup. Pero la inversión fue en el sistema, no en ningún proyecto individual. Cada proyecto nuevo se beneficia automáticamente. El costo marginal de mantener esta operación en un proyecto nuevo es cercano a cero.

¿Puede una sola persona tener un equipo? Lo que tengo no es un equipo en el sentido humano. Es algo que todavía no tiene un buen nombre — tres departamentos de perspectivas especializadas que se involucran con cada pieza de trabajo que produzco. No solo el código, sino las decisiones de producto, la analítica, el marketing, los experimentos de crecimiento. No puede cuestionar si estoy resolviendo el problema correcto. No puede traer la perspectiva de una experiencia de vida completamente diferente.

Pero para mantener calidad y consistencia operativa a lo largo del ciclo de vida completo de un producto, mientras una sola persona hace el pensamiento creativo y estratégico, nada de lo que he experimentado se le acerca.

La configuración es la inversión. Los retornos se componen a partir de ahí.
