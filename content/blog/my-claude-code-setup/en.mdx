Can one person have a team?

I spent months on this question because the answer seemed obvious. A team is a group of people. You hire them. Each brings different expertise. The security person catches what the product person misses. The DBA questions schemas the frontend developer never considers. One person can't be a team.

Except the word "team" was pointing at something more specific than we realized. What you want from a team isn't people. It's perspectives applied consistently. A security engineer is valuable not because she's another body in a chair, but because she looks at your code through a lens you don't have, and she does it every time — not when she feels like it.

That distinction opens a door. Can you get multiple consistent perspectives without multiple people?

I run several products as a solo developer. The hardest part used to be context-switching — not just between security thinking, database thinking, and UX thinking, but further out. Product direction. Analytics. Growth. Marketing. You can learn all of these. You can't apply them simultaneously. When you're deep in implementing a feature, you don't pause to think about your churn rate. When you're writing code, you don't spontaneously plan a launch strategy. That's why companies have departments.

My Claude Code setup changed this. Not by making me faster at what I already do. By creating what genuinely feels like departments — persistent, specialized perspectives that engage with my work whether I ask or not. The surprise isn't that AI helps you code. The surprise is that the right configuration gives you a product manager, a security engineer, a marketer, and a data analyst. Simultaneously. Consistently.

Let me explain the architecture, because individually the pieces are simple. It's the composition that does something unexpected.

## The Constitution

Every project starts with `CLAUDE.md` at the root — a constitution for how the AI behaves in that codebase. It defines the tech stack, architecture patterns, constraints, and which skills and agents must be used for every change. The core pattern that appears in all of mine:

```
1. All implementation MUST use specified skills
2. After implementation, run two agents in parallel:
   - security-reviewer → audit → fix
   - verifier → run tests, E2E, browser verify → report
```

This isn't a suggestion. It's a hard rule baked into the project's DNA. Claude Code reads this file at the start of every session. It doesn't need reminding.

But the constitution does more than enforce quality gates. It defines skill pipelines — forced sequences specific to each type of work. Building an API? Database design first, then API specification, then implementation with framework-specific query patterns. Building a web feature? Design system, then composition patterns, then performance optimization. Each workflow is a forced march through accumulated best practices, in the right order, every time.

I used to write tests when I felt like it and do security reviews when I remembered. Now the system doesn't give me that option. Consistency stopped being a discipline problem and became a structural one.

## Skills as Constraints

I run 29 skills, and here's where it gets interesting: nearly half of them have nothing to do with code.

The obvious ones are technical. Architecture design. Database modeling. API specifications. Frontend patterns. Mobile development. Internationalization. Security checklists. 3D web. Vercel ships official Claude Code skills for React best practices — 45 rules across eight categories. The framework authors encoded how they want you to use their tools. Not guidelines you might read. Guardrails you can't ignore.

But sitting alongside those are skills for copywriting, email marketing, social media content, ad creative, pricing strategy, competitor analysis, conversion optimization, churn prevention, search visibility, product briefs, and PRDs.

I didn't plan this. I started building skills for code quality and kept going. The same architecture that prevents SQL injection turned out to prevent bad marketing copy just as well. A skill isn't documentation — documentation describes, skills constrain. When Claude Code loads the security checklist, it's compelled to check every OWASP item. When it loads the copywriting skill, it's compelled to follow persuasion principles and voice guidelines. The difference between knowing and being compelled is the difference between a textbook and a colleague who won't sign off until the work meets their standard.

## The Virtual Team

This is where the "team" feeling really emerges. I run seven agents organized into three departments.

**Dev** has two agents. The security reviewer runs after every implementation, scanning for injection vulnerabilities, secrets exposure, race conditions, and access control flaws. It classifies by severity and escalates anything touching payment or authentication — some decisions shouldn't be automated. The verifier runs the full test suite with coverage, executes E2E tests, then opens a real browser to check the UI works correctly. It doesn't write tests. It validates what exists and reports exactly what broke. These two run in parallel on every change.

**Plan** has two agents. The product manager writes product briefs and PRDs — problem framing, success metrics, detailed requirements. The UX designer handles information architecture, user flows, and interaction design, grounded in cognitive principles and accessibility standards. When I start a new feature, these two define what gets built before a line of code is written.

**Biz** has three agents. This is the part I didn't expect. The data analyst connects to PostHog and produces weekly analytics reports, runs retention analysis, and makes Kill/Keep/Scale recommendations on features. The growth optimizer designs referral programs, optimizes conversion funnels, and handles churn prevention. The marketer builds launch strategies, writes content calendars, and manages competitive positioning.

Think about what that means. When I build a feature, the security reviewer and verifier check the code. But before that, the product manager and UX designer defined what to build. And after deployment, the data analyst measures impact, the growth optimizer experiments with conversion, and the marketer figures out distribution. Plan, build, verify, measure, grow. Specialized agents at every stage.

## The Connective Tissue

What makes these agents feel like colleagues rather than scripts is their connection to real systems. Twenty-one plugins provide the infrastructure — three LSP servers give type-level intelligence for TypeScript, Python, and Rust. Framework plugins from Expo, Pulumi, and Vercel bring their own conventions. Utility plugins handle diagram generation, code simplification, and building new skills.

On top of that, MCP servers connect agents to the living systems they need. Context7 provides real-time library documentation. Playwright drives a browser for testing. Sentry pipes production errors into debugging sessions. PostHog feeds live analytics to the data analyst. Neon handles database migrations. GitHub gives full access to PRs and issues. Cloudflare manages Workers and edge storage. Supabase, Slack, Gmail, and Google Calendar fill in the rest.

Each connection is simple. Together they mean the data analyst isn't guessing about metrics — it's querying real data. The verifier isn't guessing if the UI works — it's opening a browser and checking.

## The Compound Effect

Each piece individually is modest. A security checklist. React rules. An analytics dashboard. A marketing playbook. Any one of them looks like just a checklist.

The surprise is what happens when they compound. When the security reviewer runs in parallel with the verifier, and both operate under framework-specific skill constraints, and the product manager defined the requirements, and the data analyst will measure the results, and the marketer will distribute it, and all of that connects to live production systems — the result isn't the sum of the parts. It's an operation that stays consistent across every change, every project, whether I'm energized or exhausted.

The investment was real. Twenty-nine skills, seven agents across three departments, twenty-one plugins, detailed constitutions for each project. Weeks of setup. But the investment was in the system, not in any individual project. Every new project automatically benefits. The marginal cost of maintaining this operation on a new project is close to zero.

Can one person have a team? What I have isn't a team in the human sense. It's something that doesn't have a good name yet — three departments of specialized perspectives that engage with every piece of work I produce. Not just the code, but the product decisions, the analytics, the marketing, the growth experiments. It can't push back on whether I'm solving the right problem. It can't bring insight from a completely different life experience.

But for maintaining quality and operational consistency across the full lifecycle of a product, while one person does the creative and strategic thinking? Nothing I've experienced comes close.

The setup is the investment. The returns compound from there.
