혼자서 팀을 가질 수 있을까?

몇 달간 이 질문을 붙잡고 있었다. 답은 당연해 보였다. 팀은 사람들이다. 뽑아야 한다. 보안 담당이 내 맹점을 잡고, DBA가 내가 신경도 안 쓴 스키마를 짚는다. 혼자서는 아무리 잘해도 팀이 될 수 없다.

근데 지금은 좀 다르게 본다. "팀"이 가리키는 게 사실 머릿수가 아니었다. 진짜 원하는 건 다른 관점이 빠짐없이 적용되는 거다. 보안 엔지니어가 귀한 건 사람이 하나 더 있어서가 아니다. 내가 못 보는 걸 보고, 그걸 매번 해주니까다. 기분 탈 때만이 아니라.

이 구분이 문을 하나 연다. 사람 없이도 그런 관점들을 유지할 수 있을까?

나는 솔로 개발자로 여러 프로덕트를 돌린다. 제일 힘든 건 컨텍스트 스위칭이었다. 기능 만들다가 갑자기 보안 생각, 스키마 생각, UX 생각. 거기에 프로덕트 방향이나 마케팅까지. 하나하나는 배울 수 있다. 동시에 돌리는 건 불가능하다. 코드 짜면서 이탈률 걱정하는 사람은 없다. 그래서 회사에 부서가 있는 거다.

내 Claude Code 셋업이 이걸 바꿨다. 코딩이 빨라진 게 아니다. 진짜 부서 같은 게 생겼다. 안 시켜도 알아서 개입하는, 각자 전문성을 가진 관점들. AI가 코딩 도와주는 건 별로 안 놀랍다. 놀라운 건 설정을 제대로 하면 PM, 보안 엔지니어, 마케터, 데이터 분석가가 동시에 붙는다는 거다.

아키텍처를 설명하겠다. 하나하나는 단순하다. 합치면 다른 게 나온다.

## 헌법

모든 프로젝트 루트에 `CLAUDE.md`가 있다. AI가 이 코드에서 어떻게 굴지를 정하는 헌법이다. 기술 스택, 아키텍처, 제약사항, 그리고 어떤 skill과 agent를 써야 하는지. 핵심 패턴은 이렇다:

```
1. 모든 구현은 지정된 skill을 사용
2. 구현 후, 두 agent를 병렬 실행:
   - security-reviewer → 감사 → 수정
   - verifier → 테스트, E2E, 브라우저 검증 → 보고
```

제안이 아니다. 하드 룰이다. Claude Code가 세션 시작할 때 이걸 읽는다. 상기시킬 필요 없다.

헌법은 품질 게이트만 거는 게 아니다. 작업별 skill 순서도 정한다. API 만든다? DB 설계부터, API 스펙, 그다음 프레임워크별 구현. 웹 기능이면? 디자인 시스템, 합성 패턴, 성능 최적화 순서. 맞는 순서로, 매번 강제된다.

원래 나는 테스트 기분 날 때 쓰고, 보안 리뷰 생각날 때 하는 타입이었다. 지금은 시스템이 그 선택지를 안 준다. 일관성이 의지 문제에서 구조 문제가 됐다.

## 제약으로서의 Skill

29개 skill을 쓴다. 재밌는 건 거의 절반이 코드랑 상관없다는 거다.

기술 쪽은 당연히 있다. 아키텍처, DB 모델링, API 스펙, 프론트엔드 패턴, 모바일, 국제화, 보안 체크리스트, 3D 웹. Vercel이 React best practices 공식 skill을 만들어놨다. 8개 카테고리에 45개 규칙. 프레임워크 만든 사람들이 "이렇게 써라"를 직접 심어둔 거다. 읽을 수도 있는 가이드가 아니라, 무시할 수 없는 가드레일.

근데 그 옆에 카피라이팅, 이메일 마케팅, 소셜 콘텐츠, 광고, 가격 전략, 경쟁사 분석, 전환율 최적화, 이탈 방지, 검색 노출, 프로덕트 브리프, PRD용 skill이 줄줄이 있다.

계획한 게 아니다. 코드 품질 skill 만들기 시작했는데 멈출 수가 없었다. SQL 인젝션 막는 구조가 엉터리 마케팅 카피도 똑같이 막더라.

Skill은 문서가 아니다. 문서는 설명한다. Skill은 강제한다. 보안 skill을 로드하면 OWASP 항목을 "아는" 데서 안 끝난다. 하나하나 확인하게 강제된다. 카피라이팅 skill을 로드하면 설득 원칙을 따르게 강제된다. 아는 것과 강제되는 건 다르다. 교과서와, 기준 안 되면 절대 OK 안 하는 동료의 차이다.

## 가상 팀

여기서 "팀" 느낌이 진짜 나온다. 7개 agent를 3개 부서로 돌린다.

**Dev** — agent 2개. Security reviewer가 매 구현 후 돈다. 인젝션, 시크릿 노출, 레이스 컨디션, 접근 제어. 심각도별로 분류하고, 결제나 인증 건드리는 건 나한테 직접 올린다. 자동화하면 안 되는 결정이니까. Verifier는 전체 테스트를 커버리지랑 같이 돌리고, E2E 치고, 진짜 브라우저 열어서 UI 확인한다. 테스트를 짜지는 않는다. 있는 걸 검증하고 뭐가 깨졌는지 보고한다. 둘은 매번 병렬로 돈다.

**Plan** — agent 2개. Product manager가 브리프와 PRD를 쓴다. 문제 정의, 성공 지표, 상세 요구사항. UX designer가 정보 구조, 사용자 흐름, 인터랙션을 잡는다. 인지 원칙이랑 접근성 기준 맞춰서. 새 기능 시작할 때, 코드 한 줄 전에 이 둘이 뭘 만들지를 정한다.

**Biz** — agent 3개. 이건 예상 못 했다. Data analyst가 PostHog에 붙어서 주간 리포트 만들고, 리텐션 분석 돌리고, Kill/Keep/Scale 판정을 내린다. Growth optimizer가 리퍼럴 설계하고, 전환 퍼널 만지고, 이탈 방지를 맡는다. Marketer가 런칭 전략 세우고, 콘텐츠 캘린더 쓰고, 포지셔닝을 잡는다.

코드 품질만 챙기는 시스템을 만들려고 했는데, 결국 사업 전체를 커버하게 됐다. 각 agent가 내가 이론적으로는 아는 관점을 체화하고 있다. 보안이 중요한 건 안다. 리텐션이 중요한 것도 안다. 근데 코드 짜고 있으면 코드만 생각한다. agent들은 그 문제가 없다. 자기 한 가지만 보고, 매번 그렇게 한다.

## 연결 조직

agent들이 스크립트가 아니라 동료처럼 느껴지는 건, 진짜 시스템에 물려 있어서다. 21개 plugin이 인프라를 깔아준다. TypeScript, Python, Rust LSP가 타입 수준 지능을 주고, Expo, Pulumi, Vercel plugin이 프레임워크 컨벤션을 가져오고, 유틸리티 plugin이 다이어그램, 코드 정리, skill 제작을 맡는다.

그 위에 MCP 서버가 agent를 살아 있는 시스템에 연결한다. Context7이 라이브러리 최신 문서를 주고, Playwright가 브라우저를 돌리고, Sentry가 프로덕션 에러를 흘려보내고, PostHog가 분석 데이터를 먹이고, Neon이 마이그레이션을 처리하고, GitHub이 PR과 이슈를 열어주고, Cloudflare가 Workers를 관리한다. Supabase, Slack, Gmail, Google Calendar까지.

하나하나는 단순한 연결이다. 합치면 다른 얘기가 된다. Data analyst가 지표를 "아마 이 정도겠지" 하는 게 아니라 진짜 데이터를 쿼리한다. Verifier가 UI를 "되겠지" 하는 게 아니라 브라우저 열어서 직접 본다.

## 복리 효과

하나하나는 소박하다. 보안 체크리스트. React 규칙. 분석 대시보드. 마케팅 플레이북. 하나만 보면 그냥 체크리스트다.

놀라운 건 합쳤을 때다. Security reviewer가 verifier와 병렬로 돌고, 둘 다 프레임워크 skill 아래서 작동하고, 전부 라이브 시스템에 물려 있을 때. 결과는 부분의 합이 아니다. 내 컨디션과 상관없이 일관되는 품질이다.

투자는 실제했다. 29개 skill, 3개 부서의 7개 agent, 21개 plugin, 프로젝트마다 헌법. 몇 주 걸렸다. 근데 투자는 시스템에 한 거지, 특정 프로젝트에 한 게 아니다. 새 프로젝트가 자동으로 혜택을 받는다. 한계 비용이 거의 0이다.

혼자서 팀을 가질 수 있을까? 내가 가진 건 인간 팀은 아니다. 아직 이름도 없는 뭔가다. 3개 부서의 전문 관점들이 내가 만드는 모든 작업에 개입한다. 잘못된 문제를 풀고 있다고 말해주진 못한다. 인간 동료만이 가져올 수 있는 시각도 없다.

근데 한 사람이 전략적 사고를 하는 동안, 여러 차원의 품질을 예외 없이 유지하는 일? 이보다 나은 경험을 해본 적이 없다.

셋업이 투자다. 수익은 거기서 복리로 쌓인다.
