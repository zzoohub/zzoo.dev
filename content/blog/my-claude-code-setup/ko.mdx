혼자서 팀을 가질 수 있을까?

몇 달 동안 이 질문을 붙잡고 있었다. 답은 당연해 보였다. 팀은 사람들의 모임이다. 채용해야 한다. 각자 다른 전문성을 가져온다. 보안 담당자가 프로덕트 담당자의 맹점을 잡아내고, DBA가 프론트엔드 개발자는 생각도 안 하는 스키마 문제를 지적한다. 한 사람이 아무리 뛰어나도, 한 사람이 팀이 될 수는 없다.

그런데 지금은 생각이 다르다. "팀"이라는 단어가 우리가 생각한 것보다 더 구체적인 무언가를 가리키고 있었다. 팀에게 진짜 원하는 건 사람 수가 아니다. 서로 다른 관점이 일관되게 적용되는 것이다. 보안 엔지니어가 귀한 건 의자에 앉은 사람이 한 명 더 있어서가 아니다. 내가 가지지 않은 렌즈로 코드를 바라보고, 기분에 따라서가 아니라 매번 그렇게 하기 때문이다.

이 구분이 중요하다. 문이 하나 열린다. 여러 사람 없이도 여러 관점을 일관되게 적용할 수 있을까?

나는 솔로 개발자로 여러 프로덕트를 운영한다. 가장 힘든 건 컨텍스트 스위칭이었다. 보안, 데이터베이스, UX 사이만이 아니다. 프로덕트 방향, 분석, 그로스, 마케팅까지. 그 모든 걸 배울 수는 있다. 하지만 동시에 적용하는 건 불가능하다. 기능 구현에 몰입하고 있을 때 이탈률을 생각하는 사람은 없다. 코드를 쓰면서 런칭 전략을 세우는 사람도 없다. 그래서 회사에는 부서가 존재한다.

내 Claude Code 셋업이 이걸 바꿨다. 내가 이미 하는 일을 더 빨리 하게 만들어서가 아니다. 부서라고 부를 수밖에 없는 무언가를 만들어냈기 때문이다. 내가 요청하든 안 하든, 내 작업에 개입하는 지속적이고 전문화된 관점들. AI가 코딩을 도와준다는 건 놀랍지 않다. 놀라운 건 제대로 된 설정이 프로덕트 매니저, 보안 엔지니어, 마케터, 데이터 분석가를 동시에, 일관되게 만들어낸다는 점이다.

아키텍처를 설명하겠다. 개별 조각들은 단순하다. 조합이 예상 밖의 결과를 낳는다.

## 헌법

모든 프로젝트 루트에 `CLAUDE.md`라는 파일이 있다. 그 코드베이스에서 AI가 어떻게 행동해야 하는지를 규정하는 헌법이다. 기술 스택, 아키텍처 패턴, 제약사항을 정의하고, 가장 중요하게는 모든 변경에 어떤 skill과 agent를 사용해야 하는지를 명시한다. 내 프로젝트 전부에 들어가는 핵심 패턴은 이렇다:

```
1. 모든 구현은 반드시 지정된 skill을 사용
2. 구현 후, 두 agent를 병렬로 실행:
   - security-reviewer → 감사 → 수정
   - verifier → 테스트 실행, E2E, 브라우저 검증 → 보고
```

제안이 아니다. 프로젝트 DNA에 새겨진 하드 룰이다. Claude Code는 매 세션 시작 시 이 파일을 읽는다. 상기시킬 필요가 없다.

하지만 헌법이 하는 일은 품질 게이트를 강제하는 것만이 아니다. 작업 유형별 skill 파이프라인을 정의한다. API를 만든다면? 데이터베이스 설계부터, 그다음 API 스펙, 그다음 프레임워크별 쿼리 패턴으로 구현. 웹 기능을 만든다면? 디자인 시스템, 그다음 합성 패턴, 그다음 성능 최적화. 축적된 베스트 프랙티스를 올바른 순서로, 매번 강제하는 워크플로우다.

원래 나는 기분 나면 테스트 쓰고 기억나면 보안 리뷰 하는 유형이었다. 지금은 시스템이 그런 선택지를 주지 않는다. 일관성이 더 이상 자기 규율의 문제가 아니다. 구조의 문제다.

## 제약으로서의 Skill

29개 skill을 쓴다. 흥미로운 건, 거의 절반이 코드와 관련 없다는 점이다.

기술 skill은 당연히 있다. 아키텍처 설계. 데이터베이스 모델링. API 스펙. 프론트엔드 패턴. 모바일 개발. 국제화. 보안 체크리스트. 3D 웹. Vercel이 React best practices를 위한 공식 Claude Code skill을 제공한다. 8개 카테고리에 걸쳐 45개 규칙. 프레임워크를 만든 사람들이 "이렇게 써라"를 직접 인코딩한 거다. 읽을 수도 있는 가이드라인이 아니다. 무시할 수 없는 가드레일이다.

그런데 그 옆에 카피라이팅, 이메일 마케팅, 소셜 미디어 콘텐츠, 광고 크리에이티브, 가격 전략, 경쟁사 분석, 전환율 최적화, 이탈 방지, 검색 가시성, 프로덕트 브리프, PRD를 위한 skill들이 나란히 있다.

계획한 게 아니다. 코드 품질용 skill을 만들기 시작했는데 멈출 수가 없었다. SQL 인젝션을 방지하는 아키텍처가 나쁜 마케팅 카피도 똑같이 방지하더라. Skill은 문서가 아니다. 문서는 설명한다. Skill은 제약한다. Claude Code가 보안 체크리스트 skill을 로드하면, OWASP top 10을 아는 데 그치지 않는다. 매 항목을 확인하도록 강제된다. 카피라이팅 skill을 로드하면 설득 원칙과 브랜드 보이스 가이드라인을 따르도록 강제된다. 아는 것과 강제되는 것의 차이는 교과서와, 자기 기준을 충족하기 전까지 사인오프하지 않는 동료의 차이다.

## 가상 팀

여기서 "팀" 느낌이 진짜 나타난다. 7개 agent를 3개 부서로 운영한다.

**Dev**에 agent 2개. Security reviewer는 모든 구현 후 실행된다. 인젝션 취약점, 시크릿 노출, 레이스 컨디션, 접근 제어 결함을 스캔한다. 심각도별로 분류하고, 결제나 인증 관련은 나에게 직접 에스컬레이션한다. 자동화하면 안 되는 결정이 있기 때문이다. Verifier는 전체 테스트 스위트를 커버리지와 함께 실행하고, E2E 테스트를 돌리고, 실제 브라우저를 열어 UI가 정상 작동하는지 확인한다. 테스트를 작성하지 않는다. 있는 것을 검증하고, 뭐가 깨졌는지 정확히 보고한다. 이 둘은 모든 변경에 병렬로 실행된다.

**Plan**에 agent 2개. Product manager는 프로덕트 브리프와 PRD를 작성한다. 문제 정의, 성공 지표, 상세 요구사항. UX designer는 정보 구조, 사용자 흐름, 인터랙션 설계를 인지 원칙과 접근성 기준에 근거해서 수행한다. 새 기능을 시작할 때, 코드 한 줄 쓰기 전에 이 둘이 무엇을 만들지 정의한다.

**Biz**에 agent 3개. 이 부분은 예상 못 했다. Data analyst는 PostHog에 연결해서 주간 분석 리포트를 만들고, 리텐션 분석을 돌리고, 기능별 Kill/Keep/Scale 권고를 한다. Growth optimizer는 리퍼럴 프로그램을 설계하고, 전환 퍼널을 최적화하고, 이탈 방지를 담당한다. Marketer는 런칭 전략을 세우고, 콘텐츠 캘린더를 작성하고, 경쟁 포지셔닝을 관리한다.

이게 뭘 의미하는지 생각해 보자. 기능을 구현하면 security reviewer와 verifier가 코드를 검사한다. 하지만 그 전에 product manager와 UX designer가 무엇을 만들지 정의했다. 배포 후에는 data analyst가 영향을 측정하고, growth optimizer가 전환을 실험하고, marketer가 배포 전략을 짠다. 기획, 구현, 검증, 측정, 성장. 모든 단계에 전문화된 agent가 있다.

## 연결 조직

이 agent들이 스크립트가 아니라 동료처럼 느껴지는 건 실제 시스템에 연결되어 있기 때문이다. 21개 plugin이 인프라를 제공한다. 3개 LSP 서버가 TypeScript, Python, Rust에 타입 수준 인텔리전스를 준다. Expo, Pulumi, Vercel의 프레임워크 plugin이 각자의 컨벤션을 가져온다. 유틸리티 plugin이 다이어그램 생성, 코드 단순화, 새로운 skill 제작을 담당한다.

그 위에 MCP 서버가 agent를 살아 있는 시스템에 연결한다. Context7이 실시간 라이브러리 문서를 제공한다. Playwright가 테스트를 위해 브라우저를 구동한다. Sentry가 프로덕션 에러를 디버깅 세션으로 흘려보낸다. PostHog가 data analyst에게 라이브 분석 데이터를 먹인다. Neon이 마이그레이션과 쿼리 튜닝을 처리한다. GitHub이 PR과 이슈에 풀 액세스를 준다. Cloudflare가 Workers와 엣지 스토리지를 관리한다. Supabase, Slack, Gmail, Google Calendar이 나머지를 채운다.

각각은 단순한 연결이다. 하지만 합쳐지면 data analyst가 지표를 추측하는 게 아니라 실제 데이터를 쿼리한다는 뜻이다. Verifier가 UI가 동작하는지 추측하는 게 아니라 브라우저를 열어 직접 확인한다는 뜻이다.

## 복리 효과

개별 조각은 하나하나 소박하다. 보안 체크리스트. React 규칙 모음. 분석 대시보드. 마케팅 플레이북. 어느 하나만 보면 그냥 체크리스트같다.

복리 효과에서 놀라움이 온다. Security reviewer가 verifier와 병렬로 돌고, 둘 다 프레임워크별 skill 제약 아래에서 작동하고, product manager가 요구사항을 정의했고, data analyst가 결과를 측정할 것이고, marketer가 배포할 것이고, 그 모든 게 라이브 프로덕션 시스템에 연결되어 있을 때. 결과는 부분의 합이 아니다. 모든 변경, 모든 프로젝트에서, 내 에너지 상태와 무관하게 일관되는 운영이다.

투자는 실제했다. 29개 skill, 3개 부서에 걸친 7개 agent, 21개 plugin, 프로젝트마다 상세한 헌법. 몇 주가 걸렸다. 하지만 투자는 시스템에 한 것이지 특정 프로젝트에 한 게 아니다. 모든 새 프로젝트가 자동으로 혜택을 받는다. 이 운영 수준을 새 프로젝트에서 유지하는 한계 비용이 거의 0이다.

혼자서 팀을 가질 수 있을까? 내가 가진 건 인간적 의미의 팀이 아니다. 아직 좋은 이름이 없는 새로운 무언가다. 전문화된 관점들의 3개 부서가, 내가 만드는 모든 작업에 개입한다. 코드만이 아니라, 프로덕트 결정, 분석, 마케팅, 그로스 실험까지. 애초에 잘못된 문제를 풀고 있다고 말해줄 수는 없다. 인간 동료의 완전히 다른 삶의 경험에서 나오는 통찰을 줄 수도 없다.

하지만 한 사람이 창의적이고 전략적인 사고를 하는 동안, 프로덕트의 전체 생명주기에 걸쳐 품질과 운영 일관성을 유지하는 일에 대해서는? 이것보다 나은 경험을 해본 적이 없다.

셋업이 투자다. 수익은 거기서부터 복리로 쌓인다.
