혼자서 서비스를 만들 때 제일 부족한 건 뭘까?

능력은 아닌 것 같다. 배우면 된다. 보안도, DB도, 마케팅도. 진짜 부족한 건 동시성이다. 코드 짜고 있으면 코드만 생각하게 된다. 당연하다. 집중이니까. 근데 그 사이에 보안은 빠지고, 분석은 밀리고, 마케팅은 까먹는다. 하나에 몰두하면 나머지가 빠진다. 그래서 회사에 부서가 있는 것이기도 하다.

내가 하고 싶었던 건 모든 영역의 자동화다. 사람을 뽑는 게 아니라, 각 영역이 시스템으로 돌아가는 구조. 내가 코드에 몰두해도 보안 리뷰가 돌고, 분석이 쌓이고, 마케팅 초안이 나오는 것. Claude Code 셋업을 이 방향으로 만져왔다.

## 헌법

모든 프로젝트 루트에 `CLAUDE.md`를 둔다. AI가 이 코드에서 어떻게 작동할지를 정하는 규칙이다. 기술 스택, 아키텍처, 제약사항, 어떤 skill과 agent를 써야 하는지. 핵심 패턴은 이렇다:

```
1. 모든 구현은 지정된 skill을 사용
2. 구현 후, 두 agent를 병렬 실행:
   - security-reviewer → 감사 → 수정
   - verifier → 테스트, E2E, 브라우저 검증 → 보고
```

하드 룰로 넣어놨다. Claude Code가 세션 시작할 때 읽으니까, 상기시킬 필요 없이 돌아간다.

스캐폴딩도 표준화했다. 새 프로젝트마다 구조를 고민하는 대신, 템플릿을 복사한다. 앱 디렉토리, 문서(PRD, UX, 아키텍처), 비즈니스 운영(마케팅, 분석, 성장), 태스크 보드까지 들어있고, plan → build → ship 워크플로우도 잡혀 있다. 나는 원래 테스트를 기분 날 때 쓰고 보안 리뷰를 생각날 때 하는 타입이라, 이런 강제 구조가 나한테는 필요했다.

## 제약으로서의 Skill

34개 skill을 쓰고 있다. 거의 절반이 코드랑 상관없다.

기술 쪽은 당연히 있다. 아키텍처, DB 모델링, API 스펙, 프론트엔드 패턴, 모바일, 국제화, 보안 체크리스트, 3D 웹. 그 옆에 카피라이팅, 이메일 마케팅, 소셜 콘텐츠, 광고, 가격 전략, 경쟁사 분석, 전환율 최적화, 이탈 방지, 검색 노출, 프로덕트 브리프, PRD용 skill이 줄줄이 있다.

처음부터 계획한 건 아니었다. 코드 품질 skill을 만들기 시작했는데 계속 늘었다. 돌이켜보면, skill이 문서랑 다른 점이 하나 있는 것 같다. 문서는 설명한다. Skill은 강제한다. 보안 skill을 로드하면 OWASP 항목을 "아는" 데서 끝나지 않고, 하나하나 확인하게 된다. 카피라이팅 skill을 로드하면 설득 원칙을 따르게 된다. 아는 것과 매번 적용되는 것은 생각보다 차이가 컸다.

## 자동화 구조

14개 agent를 3개 영역으로 나눠서 쓰고 있다. 처음에는 7개였는데 필요할 때마다 하나씩 늘었다.

**Plan** — agent 4개. Product manager가 브리프와 PRD를 쓴다. Architect가 시스템 설계와 기술 결정을 잡는다. UX designer가 정보 구조와 사용자 흐름을 그린다. Task manager가 설계 문서에서 구현 태스크를 뽑아낸다. 새 기능을 시작할 때, 코드 한 줄 전에 이 넷이 먼저 뭘 만들지를 정하게 했다. 덕분에 "일단 짜고 보자"에서 벗어나는 데 도움이 됐다.

**Dev** — agent 6개. 처음에는 보안 리뷰어와 검증자 둘뿐이었다. 지금은 frontend, backend, mobile, desktop developer가 추가됐다. 각각 자기 영역의 skill을 들고 작업하고, 끝나면 reviewer가 보안 감사를, verifier가 테스트와 브라우저 검증을 병렬로 돌린다. 결제나 인증처럼 자동화하면 안 되는 결정은 나한테 올라온다. 이 선은 의도적으로 지키고 있다.

**Biz** — agent 4개. 아직 유저가 붙은 서비스가 없어서 실전 검증은 못 했다. Agent와 skill만 준비된 상태다. Data analyst가 PostHog에 붙어서 리텐션 분석을 돌리고, growth optimizer가 전환 퍼널과 이탈 방지를 맡고, marketer가 런칭 전략과 포지셔닝을 잡고, content marketer가 콘텐츠 캘린더를 챙기는 구조다. 서비스가 돌아가면 제일 먼저 검증해보고 싶은 영역이다.

## 연결 조직

agent가 쓸만해지는 건 실제 시스템에 물려 있을 때다. 19개 plugin이 인프라를 깔아준다. TypeScript, Python, Rust LSP가 타입 수준 분석을 하고, Expo, Vercel plugin이 프레임워크 컨벤션을 가져온다.

그 위에 MCP 서버가 agent를 살아 있는 데이터에 연결한다. Context7이 라이브러리 최신 문서를 주고, Playwright가 브라우저를 돌리고, Sentry가 프로덕션 에러를 보여주고, PostHog가 분석 데이터를 먹인다. Neon이 DB 마이그레이션을 처리하고, GitHub이 PR과 이슈를 열어주고, Slack과 Gmail이 커뮤니케이션을 맡는다.

하나하나는 단순한 연결이다. 합쳐졌을 때 달라지는 건, data analyst가 지표를 추측하는 게 아니라 실제로 쿼리하고, verifier가 UI를 짐작이 아니라 브라우저로 확인한다는 점이다.

## 자동화의 경계

잘 되는 영역이 있다. 보안 리뷰, 테스트, 코드 품질 검증 같은 반복적인 게이트. 컨디션이 안 좋은 날에도 빠지지 않는다. 여러 프로젝트를 넘나들 때도 품질이 어느 정도 유지된다.

안 되는 것도 분명하다. 잘못된 문제를 풀고 있다고 말해주지 못한다. 사용자랑 대화해서 진짜 니즈를 찾아오지도 못한다. 전략적 판단은 여전히 내 몫이다.

34개 skill, 14개 agent, 19개 plugin, 프로젝트마다 헌법과 스캐폴딩. 시스템에 투자한 거라 새 프로젝트에 적용하는 비용은 낮다.

솔로 개발자한테 부족한 건 능력이 아니라 동시성이었다. 한 가지에 몰두하면 나머지가 빠진다. 이 셋업은 그 문제를 자동화로 풀어본 거다. 완벽하진 않다. 근데 내가 코드에 빠져 있어도 나머지 영역이 멈추지 않는다는 건, 혼자 일하는 방식이 좀 달라지는 경험이었다.
