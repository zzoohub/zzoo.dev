혼자서 팀을 가질 수 있을까?

몇 달 동안 이 질문을 붙잡고 있었다. 답은 당연해 보였다. 팀은 사람들의 모임이다. 채용해야 한다. 각자 다른 전문성을 가져온다. 보안 담당자가 프로덕트 담당자의 맹점을 잡아내고, DBA가 프론트엔드 개발자는 생각도 안 하는 스키마 문제를 지적한다. 한 사람이 아무리 뛰어나도, 한 사람이 팀이 될 수는 없다.

그런데 지금은 생각이 다르다. 아니, 더 정확히 말하면, "팀"이라는 단어가 우리가 생각한 것보다 더 구체적인 무언가를 가리키고 있었다. 팀에게 진짜 원하는 건 사람 수가 아니다. 서로 다른 관점이 일관되게 적용되는 것이다. 보안 엔지니어가 귀한 건 의자에 앉은 사람이 한 명 더 있어서가 아니다. 내가 가지지 않은 렌즈로 코드를 바라보고, 기분에 따라서가 아니라 매번 그렇게 하기 때문이다.

이 구분이 중요하다. 문이 하나 열린다. 여러 사람 없이도 여러 관점을 일관되게 적용할 수 있을까?

나는 솔로 개발자로 여러 프로덕트를 운영한다. 반년 전이었다면 가장 힘든 건 보안 사고, 데이터베이스 사고, UX 사고, 그리고 실제 구현 사이의 컨텍스트 스위칭이라고 했을 것이다. 그 모든 걸 배울 수는 있다. 하지만 동시에 적용하는 건 불가능하다. 기능 구현에 몰입하고 있을 때 갑자기 멈추고 DBA처럼 생각하는 사람은 없다. 그래서 팀이 존재하는 거다.

내 Claude Code 셋업이 이걸 바꿨다. 내가 이미 하는 일을 더 빨리 하게 만들어서가 아니다. 진짜 특정 전문성을 가진 동료 같은 무언가를 만들어냈기 때문이다. AI가 코딩을 도와준다는 건 놀랍지 않다. 놀라운 건 제대로 된 설정이 "더 빠른 나"와는 질적으로 다른 무언가를 만들어낸다는 점이다. 내가 요청하든 안 하든, 내 작업에 개입하는 지속적이고 전문화된 관점들이 생긴다.

아키텍처를 설명하겠다. 개별 조각들은 단순하다. 조합이 예상 밖의 결과를 낳는다.

## 헌법

모든 프로젝트 루트에 `CLAUDE.md`라는 파일이 있다. 그 코드베이스에서 AI가 어떻게 행동해야 하는지를 규정하는 헌법이다. 기술 스택, 아키텍처 패턴, 제약사항을 정의하고, 가장 중요하게는 모든 변경에 어떤 skill과 agent를 사용해야 하는지를 명시한다. 내 프로젝트 전부에 들어가는 핵심 패턴은 이렇다:

```
1. 모든 구현은 반드시 지정된 skill을 사용
2. 구현 후, 두 agent를 병렬로 실행:
   - security-reviewer -> 감사 -> 수정
   - tester -> 변경된 코드 테스트 -> 수정
```

제안이 아니다. 프로젝트 DNA에 새겨진 하드 룰이다. Claude Code는 매 세션 시작 시 이 파일을 읽는다. 상기시킬 필요가 없다. 모든 기능, 모든 버그 수정, 모든 리팩토링이 같은 파이프라인을 거친다. 헌법이 강제한다.

프로젝트마다 다른 헌법을 쓴다. Next.js 사이트는 Vercel의 React best practices skill을 강제한다. Rust API는 `axum`과 `postgresql` skill을. React Native 앱은 모바일 전용 skill을. 하지만 보안-테스트 병렬 패스는 보편적이다. 예외 없고, "테스트는 나중에"도 없다.

효과는 미묘하지만 깊다. 나는 원래 기분 나면 테스트 쓰고 기억나면 보안 리뷰 하는 유형이었다. 지금은 시스템이 그런 선택지를 주지 않는다. 일관성이 더 이상 자기 규율의 문제가 아니다. 구조의 문제다.

## 제약으로서의 Skill

7개 도메인에 걸쳐 21개 skill을 쓴다. 아키텍처, 웹 프레임워크, 모바일, 백엔드, 데이터베이스, AI/ML, 품질. 직접 만든 것도 있고, 프레임워크 벤더가 제공하는 것도 있다. 여기서 얘기가 흥미로워진다.

Vercel이 React best practices를 위한 공식 Claude Code skill을 제공한다. 8개 카테고리에 걸쳐 45개 규칙. waterfall 방지, 번들 최적화, 서버사이드 패턴, 리렌더 방지. 프레임워크를 만든 사람들이 "이렇게 써라"를 직접 인코딩한 거다.

이게 뭘 의미하는지 생각해 보자. 수년간 best practice는 개발자가 읽을 수도 있고 안 읽을 수도 있는 문서에 살았다. 금방 낡는 블로그 글. 반쯤 기억하는 컨퍼런스 발표. 지금은 프레임워크 제작자가 AI가 반드시 따라야 하는 기계 판독 가능한 제약 조건을 배포하고 있다. 읽을 수도 있는 가이드라인이 아니다. 무시할 수 없는 가드레일이다.

이게 프레임워크 도입 방식에서 가장 중대한 변화 중 하나가 될 거라고 본다. 모든 주요 라이브러리가 skill을 제공하면, AI 기반 코드의 기본 품질이 전반적으로 올라간다. 내 커스텀 skill이 지금은 빈틈을 채운다. 공식 skill이 결국 대부분을 커버할 것이다.

Skill에 대한 핵심 통찰은 문서가 아니라는 점이다. 문서는 설명한다. Skill은 제약한다. Claude Code가 보안 체크리스트 skill을 로드하면, OWASP top 10을 아는 데 그치지 않는다. 매 항목을 확인하도록 강제된다. 데이터베이스 skill을 로드하면 N+1 쿼리를 아는 데 그치지 않는다. 찾아내도록 요구된다. 아는 것과 강제되는 것의 차이는 교과서와, 자기 기준을 충족하기 전까지 사인오프하지 않는 동료의 차이다.

## 가상 팀

여기서 "팀" 느낌이 진짜 나타난다. 7개 agent를 운영한다. 특정 작업을 자율적으로 처리하는 전문화된 서브 프로세스다.

Security reviewer는 모든 구현 후 실행된다. 인젝션 취약점, 시크릿 노출, 레이스 컨디션, IDOR을 스캔한다. 심각도별로 분류한다. 결제 로직이나 인증 관련은 나에게 직접 에스컬레이션한다. 자동화하면 안 되는 결정이 있기 때문이다. 이 agent는 같이 일해본 가장 편집증적인 보안 엔지니어의 성향을 갖고 있다. 다만 편집증에 지치는 법이 없다.

Tester는 라인, 브랜치, 함수, 구문 전부 100% 커버리지를 목표로 한다. 변경점을 분석하고, 테스트 전략을 세우고, 테스트를 작성하고, 실행하고, 커버리지가 찰 때까지 반복한다. 엄격한 규칙이 하나 있다. 애플리케이션 코드를 절대 수정하지 않는다. 테스트 파일만 건드린다. 이 제약이 중요한 이유가 있다. 구현이 뭘 내놓든 그걸로 작업해야 한다. "코드를 좀 고쳐서 테스트 가능하게 만들자"가 허용되지 않는다. 실제 QA 엔지니어처럼.

Database reviewer는 시니어 DBA 관점으로 접근한다. 핵심 질문은 하나다. "이 스키마가 현실에서 살아남을 수 있나?" 외래 키 제약 누락, 혼합된 ID 전략, 문서화 안 된 비정규화, 전부 nullable 안티패턴을 검사한다. 멀티 테이블 마이그레이션은 에스컬레이션. PK 전략 변경도 에스컬레이션. 내가 기능에 몰두하면 잊어버릴 질문들을 대신 한다.

거기에 UI engineer(비즈니스 로직 제로인 순수 컴포넌트), web vitals auditor(모든 변경에 자동 Lighthouse), UX designer(퍼스트 프린시플 인터랙션 설계), copywriter(A/B 변형 포함 마이크로카피)가 있다. UX designer와 copywriter는 파이프라인으로 동작한다. 플로우를 먼저 설계하고, 그다음 문구를 잡는다.

인상적인 건, 이 agent 하나하나가 내가 이론적으로는 갖고 있지만 동시에 유지할 수 없는 관점을 체화하고 있다는 점이다. 보안이 중요한 거 안다. 테스트 커버리지가 중요한 거 안다. 스키마 설계가 중요한 거 안다. 하지만 기능 구현에 빠져 있으면 기능만 생각한다. 이 agent들은 그 문제가 없다. 자기 한 가지만 생각하고, 매번 그렇게 한다.

## 연결 조직

Skill과 agent만으로도 쓸모 있다. 팀처럼 느껴지게 만드는 건 연결 조직이다. Claude Code를 내가 실제로 쓰는 시스템에 연결하는 MCP 서버들.

Context7이 모든 라이브러리의 최신 문서를 실시간으로 제공한다. 낡은 지식으로 작업하는 일이 없다. Playwright가 브라우저를 제어해서 테스트하고 검증한다. Sentry가 프로덕션 에러를 디버깅 세션으로 직접 흘려보낸다. 새벽 2시에 뭔가 터지면, 아침에 앉았을 때 컨텍스트가 이미 거기 있다. Neon이 마이그레이션 실행, 스키마 검사, 쿼리 튜닝을 터미널 사이에서 복붙 없이 가능하게 한다. GitHub이 PR, 이슈, 코드 검색에 풀 액세스를 준다. Cloudflare가 Workers, KV, D1을 관리하게 해준다.

각각은 단순한 연결이다. 하지만 합쳐지면 Claude Code가 실제 팀원이 가질 법한 컨텍스트로 작업한다는 뜻이다. 프로덕션 데이터베이스나 최신 에러율을 추측하며 고립된 상태로 작업하는 게 아니다. 살아 있는 시스템에 연결되어 있다.

## 복리 효과

예상 못 한 게 있다. 개별 조각은 하나하나 소박하다. 보안 체크리스트. React 규칙 모음. DBA 리뷰 프로세스. 에러 트래킹 연동. 어느 하나만 보면 이런 생각이 들 것이다. 괜찮네, 근데 그냥 체크리스트잖아.

복리 효과에서 놀라움이 온다. Security reviewer가 tester와 병렬로 돌고, 둘 다 database reviewer의 스키마 분석에 접근할 수 있고, 전부 프레임워크별 skill의 제약 아래에서 작동하고, 그 모든 게 MCP 서버를 통해 라이브 프로덕션 데이터에 연결되어 있을 때, 결과는 부분의 합이 아니다. 질적으로 다른 무언가다. 모든 프로젝트의 모든 변경에서, 내가 에너지가 넘치든 지쳐 있든, 집중하고 있든 산만하든, 일관되게 유지되는 품질 기준이다.

사실 이게 내가 경험한 대부분의 인간 팀보다 더 일관적이다. 인간 팀에는 컨디션이 안 좋은 날이 있다. 마감 압박에 대충 넘어가는 날이 있다. 평소엔 다 잡아내던 시니어 엔지니어가 이직 면접 때문에 딴생각을 한다. 품질 기준이 출렁인다. 내 agent 팀의 품질 기준은 출렁이지 않는다. 동기가 아니라 구조에 기반하기 때문이다.

투자는 실제했다. 21개 skill을 만들고, 7개 agent를 구성하고, 각 프로젝트에 상세한 헌법을 작성했다. 몇 주가 걸렸다. 하지만 투자는 시스템에 한 것이지, 특정 프로젝트에 한 게 아니다. 이제 모든 새 프로젝트, 모든 새 기능이 축적된 전문 지식의 혜택을 자동으로 받는다. 새 프로젝트에서 팀 수준의 품질을 유지하는 한계 비용이 거의 0이다.

처음 질문으로 돌아간다. 혼자서 팀을 가질 수 있을까? 솔직한 답은, 내가 가진 건 인간적 의미의 팀이 아니라는 것이다. 아직 좋은 이름이 없는 새로운 무언가다. 내가 만드는 모든 작업에 개입하는, 지속적이고 전문화되고 일관된 관점들의 집합이다. 어떤 특정한 면에서는 내가 속해본 팀들보다 신뢰할 수 있고, 다른 면에서는 당연히 부족하다. 인간 동료의 완전히 다른 삶의 경험에서 나오는 통찰을 줄 수 없다. 프로덕트 방향 자체에 반론을 제기할 수 없다. 애초에 잘못된 문제를 풀고 있다고 말해줄 수 없다.

하지만 한 사람이 창의적이고 전략적인 작업을 하는 동안, 모든 변경에서, 예외 없이, 여러 차원의 품질을 동시에 유지하는 그 특정한 일에 대해서는? 이것보다 나은 경험을 해본 적이 없다.

셋업이 투자다. 수익은 거기서부터 복리로 쌓인다.
