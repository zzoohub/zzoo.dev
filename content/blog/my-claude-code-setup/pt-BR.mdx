Quando usei Claude Code pela primeira vez, era só um autocomplete inteligente. Não é mais. Ele segue convenções do projeto, roda revisões de segurança, escreve testes com 100% de cobertura e revisa schemas de banco de dados como um DBA sênior. Todo projeto recebe a mesma barra de qualidade, garantida automaticamente.

Não é que a ferramenta seja excepcional. O setup é diferente. Aqui está minha configuração completa do Claude Code.

## O Sistema CLAUDE.md

Todo projeto tem um arquivo `CLAUDE.md` na raiz. Pense nele como uma constituição para o comportamento da IA naquela codebase. Ele define o tech stack, padrões de arquitetura, restrições chave e — mais importante — **quais skills e agentes devem ser usados**.

O padrão crítico em todos os meus projetos:

```
1. Toda implementação DEVE usar skills especificadas
2. Após implementação, rodar DOIS agentes EM PARALELO:
   - security-reviewer → auditoria → correção
   - tester → testar código alterado → correção
```

Isso não é uma sugestão. É uma regra fixa. Toda feature, todo bug fix, toda refatoração passa por esse pipeline. O arquivo `CLAUDE.md` torna isso automático — Claude Code lê no início de toda sessão e segue o fluxo sem ser pedido.

Para meu site Next.js, o CLAUDE.md exige as skills `vercel-react-best-practices` e `vercel-composition-patterns`. Para minha API Rust, é `axum` e `postgresql`. Para o app React Native, `react-native` e `vercel-react-native-skills`. A expertise certa é carregada antes da primeira linha de código ser escrita.

## 21 Skills: Expertise Codificada

Skills são arquivos de conhecimento que Claude Code carrega sob demanda. Alguns eu mesmo escrevi, outros vêm de fontes oficiais. Uso 21 em sete domínios:

**Arquitetura & Design** — `data-modeling`, `api-design`, `ux-design`, `design-system`. Esses codificam como penso sobre design de sistemas: extração de entidades, convenções REST, ergonomia cognitiva, design tokens.

**Frameworks Web** — `vercel-react-best-practices` (45 regras em 8 categorias), `vercel-composition-patterns`, `nextjs`. As skills com prefixo `vercel-*` são **fornecidas oficialmente pela Vercel** — não são algo que eu escrevi. Só a skill de melhores práticas de React cobre prevenção de waterfall, otimização de bundle, padrões server-side, prevenção de re-render e estratégias de renderização. Ter os criadores do framework publicando skills assim é um divisor de águas: as pessoas que construíram o framework codificam exatamente como querem que você use. `nextjs` é uma skill customizada que escrevi para preencher as lacunas que as oficiais não cobrem.

**Mobile** — `react-native`, `vercel-react-native-skills` (também oficial da Vercel). Performance de listas, animações, padrões de gestão de estado específicos para React Native.

**Backend** — `axum`, `axum-hexagonal`, `fastapi`, `fastapi-hexagonal`. Tanto stacks Rust quanto Python com padrões de arquitetura hexagonal embutidos.

**Banco de Dados** — `postgresql`, `sqlite`, `neon-postgres`. Padrões de resolução de problemas para paginação, full-text search, queries N+1, migrações zero-downtime, particionamento.

**IA/ML** — `langgraph`. Agentes ReAct, sistemas multi-agente, padrões human-in-the-loop.

**Qualidade** — `security-checklists`, `web-vitals-checklist`, `i18n-patterns`, `web-design-guidelines`.

O insight chave: skills não são documentação. São **restrições**. Quando Claude Code carrega `vercel-react-best-practices`, ele não apenas conhece padrões React — ele é obrigado a seguir 45 regras específicas. Prevenção de waterfall é marcada como prioridade CRITICAL. Otimização de tamanho de bundle é CRITICAL. Não são diretrizes; são guardrails.

Espero que isso se torne uma grande tendência. Conforme mais autores de frameworks e fornecedores de ferramentas publicam skills oficiais, a qualidade base de código gerado por IA vai subir em toda parte. Imagine toda grande biblioteca publicando uma skill que codifica suas melhores práticas — não como docs que você talvez leia, mas como restrições que a IA deve seguir. As skills customizadas que escrevi preenchem as lacunas hoje, mas as oficiais já são melhores para os domínios que cobrem.

## 7 Agentes: Um Time Virtual

Agentes são sub-processos especializados que lidam com tarefas específicas de forma autônoma. Tenho sete, divididos entre desenvolvimento e planejamento:

### Agentes de Desenvolvimento

**security-reviewer** — Roda uma auditoria de segurança após toda implementação. Escaneia OWASP top 10, verifica exposição de secrets, SQL injection, command injection, XSS, IDOR, race conditions. Classifica achados por severidade. Escala para mim em lógica de pagamento, mudanças de auth, criptografia ou código relacionado a compliance.

**tester** — Mira 100% de cobertura de testes em métricas de linha, branch, função e statement. Analisa código alterado, planeja estratégia de testes, escreve testes, roda, itera até a cobertura estar completa. Regra importante: ele nunca modifica código da aplicação, apenas arquivos de teste.

**database-reviewer** — Assume a perspectiva de um DBA sênior. Faz as perguntas que importam: "Esse schema vai sobreviver à realidade?" Verifica constraints de FK faltando, estratégias mistas de ID, desnormalização não documentada, anti-padrões de nullable-everything. Escala migrações multi-tabela e mudanças de estratégia de PK.

**ui-engineer** — Constrói componentes de UI puros sem lógica de negócio. Na web, usa shadcn/ui ou componentes customizados seguindo a skill de design system. No mobile, usa a skill `design-system` com padrões nativos da plataforma. Fronteira estrita: sem gestão de estado, sem fetch de dados, sem roteamento.

**web-vitals-auditor** — Auditoria de performance automatizada. Builda e serve o app, roda Lighthouse tanto em desktop quanto mobile, verifica contra os targets: LCP < 2.5s, INP < 200ms, CLS < 0.1, TTFB < 800ms, FCP < 1.8s. Produz um relatório com scores, análise de bundle e recomendações acionáveis.

### Agentes de Planejamento

**ux-designer** — Projeta a partir de primeiros princípios. Processo de cinco etapas: Definir o problema → Mapear jornadas do usuário → Projetar interações → Remover atrito → Validar acessibilidade. Produz fluxos de usuário, specs de tela com ações primárias e hierarquia de informação.

**copywriter** — Copy de produto e microcopy de UX com seis princípios de persuasão: uma única mensagem, certeza supera recompensa máxima, novidade é suficiente, especificar ação concreta, poder de agregação, escrever como uma mensagem. Entrega 2-3 variações com contagem de caracteres e sugestões de teste A/B.

O designer de UX e o copywriter trabalham como um pipeline: o designer cria o fluxo, depois passa para o copywriter para texto persuasivo, CTAs, copy de onboarding e mensagens de erro.

## Pipelines de Fluxo de Trabalho

O poder real não está nas ferramentas individuais — está em como elas se compõem em fluxos de trabalho repetíveis:

**Desenvolvimento Padrão:**
```
Implementação (com skills) → [security-reviewer + tester] (paralelo) → Correção → Feito
```

**Modelagem de Dados:**
```
skill data-modeling → agente database-reviewer → skill api-design → implementação
```

**Frontend:**
```
skill nextjs → implementação → vercel-react-best-practices (revisão)
```

**Design de UX:**
```
agente ux-designer → agente copywriter (apenas texto persuasivo)
```

Todo projeto segue o mesmo fluxo de trabalho de desenvolvimento através do CLAUDE.md. O pipeline de revisão de schema roda automaticamente quando mudanças de banco de dados estão envolvidas. O passe paralelo de segurança e testes acontece em toda mudança. Sem exceções.

## Servidores MCP: Integrações Externas

Claude Code se conecta a serviços externos através de servidores MCP (Model Context Protocol). Meu setup inclui:

- **Context7** — Lookup de documentação em tempo real para qualquer biblioteca
- **Playwright** — Automação de navegador para testes e screenshots
- **Sentry** — Integração de rastreamento de erros para debugging de problemas em produção
- **Neon** — Operações diretas de banco de dados e gestão de schema
- **Notion** — Integração de workspace para gestão de projetos
- **GitHub** — Fluxo completo do GitHub (PRs, issues, code search)
- **D2** — Geração de diagramas de arquitetura (compilados para SVGs temáticos)
- **Cloudflare** — Gestão de Workers, KV, R2, D1

Essas não são apenas conveniências. O Neon MCP significa que Claude Code pode rodar migrações, inspecionar schemas e otimizar queries sem eu trocar de contexto. O Sentry MCP significa que erros de produção alimentam diretamente as sessões de debugging. O D2 MCP gera diagramas de arquitetura que são publicados como parte da documentação do projeto.

## Como Isso Realmente Funciona

Uma sessão típica: eu descrevo uma feature em linguagem natural. Claude Code lê o CLAUDE.md, carrega as skills necessárias e começa a implementar. Quando termina, dois agentes iniciam em paralelo — security reviewer escaneando vulnerabilidades, tester escrevendo testes até 100% de cobertura. Se algum encontra problemas, são corrigidos. Então eu reviso e publico.

Não rodo linters manualmente. Não escrevo testes boilerplate. Não faço checklists de segurança de memória. O sistema cuida disso.

O investimento foi construir as 21 skills, configurar os 7 agentes e escrever arquivos CLAUDE.md detalhados para cada projeto. Isso levou tempo. Mas agora toda nova feature, em todo projeto, recebe o benefício daquele conhecimento acumulado — automaticamente.

## O Efeito Composto

Cada skill, agente e arquivo CLAUDE.md é individualmente simples. Um checklist de segurança. Um conjunto de regras React. Um processo de revisão de DBA. Mas compostos juntos, criam algo que parece ter um time: um arquiteto que conhece meus padrões, um engenheiro de segurança que nunca dorme, um líder de QA que insiste em 100% de cobertura e um DBA que pergunta "isso vai sobreviver à realidade?"

Para um desenvolvedor solo publicando múltiplos produtos, essa é a diferença entre burnout e escalabilidade.

As vibes são boas. Os agentes me protegem.
