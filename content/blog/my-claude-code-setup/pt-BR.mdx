Uma pessoa pode ter um time?

Passei meses nessa pergunta porque a resposta parecia óbvia. Um time é um grupo de pessoas. Você contrata. Cada um traz uma expertise diferente. A pessoa de segurança pega o que a de produto não vê. O DBA questiona schemas que o dev frontend nem considera. Uma pessoa sozinha não pode ser um time.

Só que a palavra "time" apontava pra algo mais específico do que a gente imaginava. O que você realmente quer de um time não são mais pessoas. São mais perspectivas aplicadas de forma consistente. A engenheira de segurança não é valiosa porque é mais um corpo na cadeira. É valiosa porque olha seu código por uma lente que você não tem, e faz isso toda vez — não só quando está com vontade.

Essa distinção abre uma porta. Dá pra conseguir múltiplas perspectivas consistentes sem múltiplas pessoas?

Toco vários produtos como developer solo. A parte mais difícil costumava ser a troca de contexto — não só entre pensamento de segurança, banco de dados e UX, mas indo além. Direção de produto. Analytics. Growth. Marketing. Dá pra aprender tudo isso. Não dá pra aplicar tudo ao mesmo tempo. Quando você está imerso implementando uma feature, não para pra pensar na taxa de churn. Quando está escrevendo código, não planeja espontaneamente uma estratégia de lançamento. É por isso que empresas têm departamentos.

Meu setup de Claude Code mudou isso. Não me deixando mais rápido no que já faço. Mas criando algo que genuinamente parece departamentos — perspectivas persistentes e especializadas que se envolvem com meu trabalho, eu pedindo ou não. A surpresa não é que IA ajuda a programar. A surpresa é que a configuração certa te dá um product manager, um engenheiro de segurança, um marketer e um analista de dados. Simultaneamente. Consistentemente.

Cada peça é simples. É a composição que faz algo inesperado.

## A constituição

Todo projeto começa com um arquivo chamado `CLAUDE.md` na raiz — uma constituição pra como a IA se comporta naquele codebase. Define o stack, padrões de arquitetura, restrições, e quais skills e agentes devem ser usados em cada mudança. O padrão central que aparece em todos os meus projetos:

```
1. Toda implementação DEVE usar os skills especificados
2. Depois de implementar, rodar dois agentes em paralelo:
   - security-reviewer → auditoria → correção
   - verifier → rodar testes, E2E, verificação no browser → relatório
```

Não é sugestão. É regra dura gravada no DNA do projeto. Claude Code lê esse arquivo no início de cada sessão. Não precisa ser lembrado.

Mas a constituição faz mais do que impor gates de qualidade. Define pipelines de skills — sequências forçadas específicas pra cada tipo de trabalho. Construindo uma API? Design de banco de dados primeiro, depois especificação da API, depois implementação com padrões de query do framework. Feature web? Design system, depois padrões de composição, depois otimização de performance. Cada workflow é uma marcha forçada pelas melhores práticas acumuladas, na ordem certa, toda vez.

Eu era o tipo de dev que escrevia testes quando dava vontade e fazia review de segurança quando lembrava. Agora o sistema não me dá essa opção. Consistência deixou de ser um problema de disciplina e virou estrutural.

## Skills como restrições

Tenho 29 skills, e aqui fica interessante: quase metade não tem nada a ver com código.

Os óbvios são técnicos. Design de arquitetura. Modelagem de banco de dados. Especificações de API. Padrões de frontend. Desenvolvimento mobile. Internacionalização. Checklists de segurança. Web 3D. A Vercel distribui skills oficiais de Claude Code pra melhores práticas de React — 45 regras em oito categorias. Os autores do framework codificaram como querem que você use as ferramentas deles. Não são guias que você talvez leia. São barreiras que você não consegue ignorar.

Mas junto a eles existem skills de copywriting, email marketing, conteúdo pra redes sociais, criativos de anúncio, estratégia de preço, análise de concorrentes, otimização de conversão, prevenção de churn, visibilidade em buscadores, product briefs e PRDs.

Não planejei isso. Comecei fazendo skills pra qualidade de código e não parei. A mesma arquitetura que previne SQL injection acabou prevenindo copy de marketing ruim igualmente bem. Um skill não é documentação — documentação descreve, skill restringe. Quando Claude Code carrega o checklist de segurança, é obrigado a verificar cada item OWASP. Quando carrega o skill de copywriting, é obrigado a seguir princípios de persuasão e diretrizes de voz de marca. A diferença entre saber e ser obrigado é a diferença entre um livro-texto e um colega que não aprova até o trabalho atingir o padrão dele.

## O time virtual

Aqui é onde a sensação de "time" realmente aparece. Rodo sete agentes organizados em três departamentos.

**Dev** tem dois agentes. O security reviewer roda depois de cada implementação, escaneando injeções, exposição de secrets, race conditions e falhas de controle de acesso. Classifica por severidade e escala tudo que toca pagamento ou autenticação — certas decisões não devem ser automatizadas. O verifier roda a suite completa de testes com cobertura, executa testes E2E, e abre um browser real pra verificar que a UI funciona corretamente. Não escreve testes. Valida o que existe e reporta exatamente o que quebrou. Esses dois rodam em paralelo em toda mudança.

**Plan** tem dois agentes. O product manager escreve product briefs e PRDs — definição de problema, métricas de sucesso, requisitos detalhados. A UX designer cuida de arquitetura de informação, fluxos de usuário e design de interação, baseada em princípios cognitivos e padrões de acessibilidade. Quando começo uma feature nova, esses dois definem o que vai ser construído antes de uma linha de código ser escrita.

**Biz** tem três agentes. Essa é a parte que eu não esperava. O data analyst conecta no PostHog e produz relatórios analíticos semanais, roda análise de retenção e faz recomendações Kill/Keep/Scale sobre features. O growth optimizer projeta programas de referral, otimiza funis de conversão e cuida de prevenção de churn. O marketer monta estratégias de lançamento, escreve calendários de conteúdo e gerencia posicionamento competitivo.

Pensa no que isso significa. Quando construo uma feature, o security reviewer e o verifier checam o código. Mas antes disso, o product manager e a UX designer definiram o que construir. E depois do deploy, o data analyst mede o impacto, o growth optimizer experimenta com conversão, e o marketer resolve distribuição. Planejar, construir, verificar, medir, crescer. Agentes especializados em cada etapa.

## O tecido conectivo

O que faz esses agentes parecerem colegas em vez de scripts é a conexão com sistemas reais. Vinte e um plugins fornecem a infraestrutura — três servidores LSP dão inteligência em nível de tipo pra TypeScript, Python e Rust. Plugins de frameworks do Expo, Pulumi e Vercel trazem suas próprias convenções. Plugins utilitários cuidam de geração de diagramas, simplificação de código e criação de novos skills.

Por cima disso, servidores MCP conectam os agentes aos sistemas vivos que precisam. Context7 fornece documentação de bibliotecas em tempo real. Playwright controla um browser pra testes. Sentry canaliza erros de produção pras sessões de debug. PostHog alimenta o data analyst com analytics ao vivo. Neon cuida de migrações de banco de dados. GitHub dá acesso completo a PRs e issues. Cloudflare gerencia Workers e armazenamento edge. Supabase, Slack, Gmail e Google Calendar cobrem o resto.

Cada conexão é simples. Juntas significam que o data analyst não adivinha métricas — consulta dados reais. O verifier não adivinha se a UI funciona — abre um browser e verifica.

## O efeito composto

Cada peça isoladamente é modesta. Um checklist de segurança. Regras de React. Um dashboard analítico. Um playbook de marketing. Qualquer uma delas parece só um checklist.

A surpresa é o que acontece quando se compõem. Quando o security reviewer roda em paralelo com o verifier, e ambos operam sob restrições de skills específicos do framework, e o product manager definiu os requisitos, e o data analyst vai medir os resultados, e o marketer vai distribuir, e tudo isso está conectado a sistemas de produção ao vivo — o resultado não é a soma das partes. É uma operação que se mantém consistente em cada mudança, cada projeto, esteja eu cheio de energia ou exausto.

O investimento foi real. Vinte e nove skills, sete agentes em três departamentos, vinte e um plugins, constituições detalhadas pra cada projeto. Semanas de setup. Mas o investimento foi no sistema, não em nenhum projeto individual. Cada projeto novo se beneficia automaticamente. O custo marginal de manter essa operação num projeto novo é perto de zero.

Uma pessoa pode ter um time? O que tenho não é um time no sentido humano. É algo que ainda não tem um bom nome — três departamentos de perspectivas especializadas que se envolvem com cada peça de trabalho que produzo. Não só o código, mas as decisões de produto, o analytics, o marketing, os experimentos de growth. Não consegue questionar se estou resolvendo o problema certo. Não consegue trazer a perspectiva de uma experiência de vida completamente diferente.

Mas pra manter qualidade e consistência operacional ao longo de todo o ciclo de vida de um produto, enquanto uma pessoa faz o pensamento criativo e estratégico? Nada que experimentei chega perto.

O setup é o investimento. Os retornos se compõem a partir daí.
