A coisa mais difícil de ser um desenvolvedor solo não é o que a maioria das pessoas imagina. Não é a quantidade de trabalho. Não é a solidão. É que você só consegue pensar de um jeito por vez.

Quando você está imerso numa feature, pensando em estados de componente e fluxos de usuário, você não para espontaneamente para pensar como um DBA. "Essa tabela tem as foreign keys certas? Esse schema vai aguentar cem mil registros?" Você sabe que essas perguntas importam. Você até sabe as respostas, na maioria dos casos. Mas quando está construindo, está construindo. Seu cérebro não faz auditoria de segurança por conta própria. O de ninguém faz. É por isso que times existem.

Ou pelo menos, era isso que eu pensava. Que times existiam para ter mais pessoas. Mas acho que a palavra "time" aponta para algo mais preciso do que percebíamos. O que você realmente quer de um time não são mais corpos em cadeiras. São mais perspectivas aplicadas de forma consistente. A engenheira de segurança é valiosa não porque é uma segunda pessoa na sala. É valiosa porque olha cada mudança no código por uma lente que você não tem, e faz isso sempre, não só quando está com vontade.

Essa distinção importa porque abre uma pergunta real: se o que você precisa são perspectivas consistentes, você precisa mesmo de pessoas para isso?

Eu toco vários produtos como desenvolvedor solo. Minha configuração de Claude Code mudou a forma como trabalho de um jeito que não esperava. Não me deixou mais rápido escrevendo código. Criou algo que genuinamente parece ter colegas com expertise específico. A surpresa não é que IA pode ajudar a programar. A surpresa é que a configuração certa produz algo qualitativamente diferente de "um você mais rápido." Cria perspectivas persistentes e especializadas que se engajam com seu trabalho quer você lembre de pedir ou não.

Cada peça é simples. É a composição que faz algo inesperado.

## A constituição

Todo projeto começa com um arquivo chamado `CLAUDE.md` na raiz. Eu penso nele como uma constituição: as regras fundamentais de como a IA se comporta naquele código. Define o stack, padrões de arquitetura, restrições, e o mais importante, quais skills e agentes devem ser usados em cada mudança.

O padrão central que aparece em todos os meus projetos é:

```
1. Toda implementação DEVE usar os skills especificados
2. Depois de implementar, rodar DOIS agentes EM PARALELO:
   - security-reviewer -> auditoria -> correção
   - tester -> testar código alterado -> correção
```

Isso não é sugestão. É regra dura. Claude Code lê esse arquivo no início de cada sessão. Não precisa ser lembrado. Cada feature, cada fix, cada refactor passa pelo mesmo pipeline. A constituição garante.

Projetos diferentes recebem constituições diferentes. Meu site Next.js exige o skill oficial de React best practices da Vercel. Minha API Rust exige skills de `axum` e `postgresql`. Meu app React Native exige skills específicos de mobile. Mas o passe paralelo de segurança e testes é universal. Sem exceção. Sem "coloco testes depois."

O efeito é sutil mas profundo. Eu era o tipo de desenvolvedor que escrevia testes quando tinha vontade e fazia revisão de segurança quando lembrava. Agora o sistema não me dá essa opção. Consistência não é mais questão de disciplina. É estrutura.

## Skills como restrições

Tenho 21 skills em sete domínios: arquitetura, frameworks web, mobile, backend, bancos de dados, IA/ML e qualidade. Alguns escrevi eu mesmo. Outros vêm dos criadores dos frameworks.

A Vercel distribui skills oficiais de Claude Code para melhores práticas de React. Quarenta e cinco regras em oito categorias. Prevenção de cascata, otimização de bundles, padrões server-side, prevenção de re-renders. As pessoas que construíram o framework codificaram exatamente como querem que você o use.

Pensa no que isso significa. Durante anos, melhores práticas moravam em documentação que desenvolvedores podiam ou não ler. Posts de blog que ficavam desatualizados. Palestras de conferência meio lembradas. Agora os autores de frameworks estão distribuindo restrições legíveis por máquina que a IA é obrigada a seguir. Não são guias que você talvez leia. São barreiras que você não consegue ignorar.

Eu espero que isso se torne uma das mudanças mais significativas na forma como frameworks são adotados. Quando toda biblioteca importante distribuir um skill, a qualidade base de código assistido por IA sobe no geral. Meus skills customizados preenchem as lacunas hoje. Os oficiais vão cobrir a maior parte eventualmente.

O ponto chave sobre skills é que eles não são documentação. Documentação descreve. Skills restringem. Quando Claude Code carrega um skill de checklist de segurança, ele não apenas sabe sobre OWASP top 10. Ele é compelido a verificar cada item. Quando carrega o skill de banco de dados, não apenas sabe sobre queries N+1. É obrigado a procurá-las. A diferença entre saber e ser obrigado é a diferença entre um livro-texto e um colega que não assina até o trabalho atender ao padrão dele.

## O time virtual

É aqui que a sensação de "time" realmente aparece. Rodo sete agentes: sub-processos especializados que lidam com tarefas específicas de forma autônoma.

O revisor de segurança roda depois de cada implementação. Escaneia injeções, exposição de segredos, race conditions, IDOR. Classifica achados por severidade. Escala pra mim qualquer coisa que toque lógica de pagamento ou autenticação, porque certas decisões não devem ser automatizadas. Esse agente tem a disposição do engenheiro de segurança mais paranoico com quem você já trabalhou, só que nunca se cansa de ser paranoico.

O tester mira cobertura total: linhas, branches, funções, statements. Analisa o que mudou, planeja uma estratégia de teste, escreve testes, roda e itera até completar a cobertura. Uma regra estrita: nunca modifica código de aplicação. Só arquivos de teste. Essa restrição importa porque significa que o tester tem que trabalhar com o que a implementação dá, igual um QA real que não pode "dar um jeito no código pra ficar testável."

O revisor de banco de dados assume a perspectiva de um DBA senior. Pergunta central: "Esse schema vai sobreviver a realidade?" Foreign keys faltando, estratégias mistas de ID, desnormalização sem documentar, o anti-padrão de tudo-nullable. Migrações multi-tabela são escaladas. Mudanças de estratégia de primary key são escaladas. Faz as perguntas que eu esqueceria quando estou focado em features.

Cada agente encarna uma perspectiva que eu teoricamente tenho mas não consigo manter simultaneamente. Sei que segurança importa. Sei que cobertura de teste importa. Sei que design de schema importa. Mas quando estou imerso construindo uma feature, estou pensando na feature. Esses agentes não têm esse problema. Só pensam na coisa deles, e pensam toda vez.

## O tecido conectivo

Skills e agentes seriam úteis sozinhos. O que os faz parecer um time é o tecido conectivo: MCP servers que plugam Claude Code nos sistemas reais que uso.

Context7 dá documentação atualizada de qualquer biblioteca, então nunca trabalha com conhecimento obsoleto. Playwright permite controlar um navegador pra testes e verificação. Sentry canaliza erros de produção diretamente pras sessões de debug, então quando algo quebra de madrugada, o contexto já está lá quando sento pra trabalhar de manhã. Neon permite rodar migrações, inspecionar schemas e tunar queries sem eu copiar e colar entre terminais. GitHub dá acesso completo a PRs, issues e busca de código. Cloudflare permite gerenciar Workers, KV stores e bancos D1.

Cada conexão é individualmente simples. Juntas significam que Claude Code opera com o mesmo contexto que um membro humano do time teria. Não trabalha isolado, adivinhando sobre o banco de produção ou as taxas de erro mais recentes. Está conectado ao sistema vivo.

## O efeito composto

Cada peça, individualmente, é modesta. Uma checklist de segurança. Um conjunto de regras de React. Um processo de revisão de DBA. Uma integração de error tracking. Olhando qualquer uma delas isoladamente, você pensaria: legal, mas é só uma checklist.

O efeito composto é o que surpreende. Quando o revisor de segurança roda em paralelo com o tester, e ambos têm acesso à análise de schema do revisor de banco de dados, e todos operam sob as restrições de skills específicos do framework, e tudo isso está conectado a dados de produção em tempo real via MCP servers, o resultado não é a soma das partes. É algo qualitativamente diferente. É um padrão de qualidade que se mantém consistente em cada mudança, em cada projeto, esteja eu energizado ou exausto, focado ou disperso.

Isso é mais consistente que a maioria dos times humanos em que trabalhei. Times humanos têm dias ruins. Pessoas cortam caminhos sob pressão de prazo. O engenheiro senior que normalmente pega tudo está distraído porque está entrevistando pra outra empresa. O padrão de qualidade flutua. O do meu time de agentes não flutua porque é estrutural, não motivacional.

O investimento foi real. Construir 21 skills, configurar 7 agentes, escrever constituições detalhadas pra cada projeto. Levou semanas. Mas o investimento foi no sistema, não em nenhum projeto individual. Agora cada projeto novo, cada feature nova, se beneficia automaticamente do expertise acumulado. O custo marginal de manter qualidade de time num projeto novo é perto de zero.

Volto à pergunta original. Um time de um é possível? O que tenho não é um time no sentido humano. É algo novo que ainda não tem um bom nome. É um conjunto de perspectivas persistentes, especializadas e consistentes que se engajam com cada pedaço de trabalho que produzo. Não pode ter a intuição que vem da experiência de vida completamente diferente de um colega humano. Não pode questionar a direção do produto. Não pode me dizer que estou resolvendo o problema errado.

Mas para o trabalho específico de manter qualidade em múltiplas dimensões simultaneamente, em cada mudança, sem exceção, enquanto uma pessoa faz o trabalho criativo e estratégico, nada do que experimentei chega perto.

A configuração é o investimento. Os retornos se compõem a partir daí.
