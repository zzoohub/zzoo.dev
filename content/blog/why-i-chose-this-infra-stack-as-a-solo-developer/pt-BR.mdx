Semana passada eu fui olhar minha conta da GCP e encontrei um banco de dados rodando para um projeto que eu abandonei em setembro. Cinco meses. Cinco meses de compute ativo para um projeto que eu sequer lembro por que comecei. O valor era pequeno, uns oito dólares. Mas quando fui investigar, encontrei mais dois. E um bucket no S3 com fotos de teste. E registros DNS apontando para lugar nenhum.

Eu tinha um cemitério inteiro de infraestrutura e não sabia.

Todo mundo conhece o conceito de dívida técnica. Código que funciona hoje mas vai te custar caro amanhã. Mas existe uma versão desse problema que ninguém discute: dívida de infraestrutura. E ela é pior que a técnica por um motivo simples. A dívida técnica você vê toda vez que abre o código. Ela incomoda. Ela grita. A dívida de infraestrutura é silenciosa. Ela mora em dashboards que você não visita, em contas que debitam automaticamente, em serviços que rodam em segundo plano sem que ninguém perceba.

Desenvolvedores solo matam mais projetos do que entregam. Isso não é fracasso. É como exploração funciona. Você testa uma ideia, constrói um protótipo, descobre que o mercado não liga ou que a abordagem técnica não se sustenta, e segue em frente. O problema é que cada projeto abandonado deixa para trás um rastro de recursos na nuvem. E esses rastros se acumulam.

Quando percebi isso, mudou completamente como eu avalio ferramentas. A pergunta deixou de ser "como isso escala?" e virou "como isso morre?" E a resposta me levou a uma conclusão estranha: o comando mais importante de toda a minha infraestrutura não é `pulumi up`. É `pulumi destroy`.

Parece derrotista. Na verdade, é libertador.

Para compute, eu uso GCP Cloud Run. O motivo principal: escala a zero. Não "quase zero." Zero de verdade. Quando ninguém está usando o serviço, eu não pago nada. Eu mantenho cinco ou seis projetos ao mesmo tempo. Em qualquer dia, no máximo dois têm tráfego real. Um VPS de cinco dólares por mês parece barato. Mas multiplica por seis projetos que ninguém usa e são trinta dólares por mês pelo privilégio de manter fantasmas vivos.

Cloud Run também usa containers. Isso significa que eu não estou preso a nenhum runtime. Um projeto é Rust, outro é Node.js. Se amanhã eu quiser reescrever algo em Go, a infraestrutura não muda. Só muda o que vai dentro do container. Essa portabilidade tem uma consequência que vai além da conveniência: eu consigo trocar a aplicação sem tocar na infraestrutura ao redor. É outra forma de teardown fácil.

Para banco de dados, eu uso Neon. PostgreSQL serverless. E o importante aqui é que é PostgreSQL de verdade. Não "compatível com PostgreSQL." Não PostgreSQL com camadas extras. PostgreSQL. Quando eu faço verificação de queries em tempo de compilação com SQLx, eu quero falar com o motor real.

Mas o que realmente me vendeu no Neon foi o branching. Eu posso criar uma branch do banco de produção, testar uma migração nela, e jogar fora. Times grandes resolvem isso com ambientes de staging dedicados. Desenvolvedor solo não tem esse luxo. O branching do Neon me dá a rede de segurança sem o custo operacional.

E como tudo nesse stack, Neon escala a zero. Branches de desenvolvimento não custam nada quando estão paradas. Quando eu abandono um projeto, não tem compute rodando em segundo plano cobrando por existir.

O padrão deveria estar ficando claro: tudo nessa infraestrutura hiberna quando você para de usar.

Para frontend e armazenamento de objetos, eu uso Cloudflare. Workers para servir aplicações, R2 para storage. A comparação óbvia é Vercel. E sim, a experiência de desenvolvimento do Vercel com Next.js é mais polida. Mas o modelo de precificação do Cloudflare joga em outra liga. R2 cobra zero de egress. Zero. Se você já serviu imagens pelo S3 e viu a linha de egress subir quando alguma coisa viralizou, você entende por que isso importa. Um momento viral transforma uma conta de dois dólares em duzentos. Com R2, fica em dois.

Workers roda em mais de trezentas localidades no mundo. Tudo é rápido em todo lugar sem eu precisar pensar em regiões. Para um desenvolvedor solo, cada decisão que eu não preciso tomar é energia que eu posso gastar em algo que importa.

Agora a peça que conecta tudo: Pulumi.

Pulumi é infraestrutura como código, como Terraform. A diferença é que eu escrevo em TypeScript. O mesmo TypeScript que eu uso todo dia. Sem HCL. Sem YAML. Sem sintaxe nova para aprender. Types e autocomplete no IDE. A curva de aprendizado foi basicamente zero.

Mas o motivo pelo qual Pulumi é a peça central desse stack não é a linguagem. É o comando `destroy`. Quando eu rodo `pulumi destroy`, tudo desaparece. O serviço do Cloud Run, o banco do Neon, o bucket do R2, os bindings de IAM, os secrets. Tudo. Na ordem correta de dependências. Limpo. Completo. Sem deixar rastro.

Isso é o que viabiliza ser um desenvolvedor solo com múltiplos projetos. Eu consigo levantar um projeto novo numa tarde. Copiar o template do Pulumi, ajustar os nomes, rodar `pulumi up`. E consigo derrubar tudo com a mesma velocidade quando decido que não vale a pena continuar. A fricção de criar é baixa. Mas o que realmente importa é que a fricção de destruir é zero.

Tem um detalhe que faz dessa destruição algo que eu posso fazer sem medo: a flag `protect`. Recursos que realmente importam, como o banco por trás do meu projeto que gera receita, eu marco como protegidos. Recursos protegidos sobrevivem ao `pulumi destroy`.

```typescript
const db = new neon.Project("production-db", { ... }, { protect: true });
```

Essa linha é a diferença entre "melhor ter cuidado" e "deixa eu limpar isso." Eu posso ser agressivo na limpeza sem risco de apagar algo que importa. A flag `protect` transforma `pulumi destroy` de um comando perigoso numa rotina.

Olhando o stack como um todo, cada escolha otimiza para a mesma coisa: custo baixo em repouso e teardown fácil. Cloud Run escala a zero. Neon escala a zero. Cloudflare não cobra egress. Pulumi destrói tudo limpo.

A maioria dos conselhos de infraestrutura ensina a escalar para cima. Assume que seu projeto vai dar certo e crescer. Mas se você é um desenvolvedor solo, o resultado esmagadoramente provável para qualquer projeto é que você vai abandoná-lo. Otimizar para esse caso não é pessimismo. É realismo. E é o que te dá liberdade para começar o próximo projeto sem arrastar o cadáver do anterior.

Infraestrutura que não morre limpa é infraestrutura que vai te assombrar. `pulumi destroy` não é o fim triste de um projeto. É a página em branco que torna o próximo possível.
