先月、クラウドの請求書を見て気づいたことがある。9月に放棄したプロジェクトのデータベースがまだ動いていた。5ヶ月間、誰もアクセスしていないデータベースに金を払い続けていたわけだ。金額は大したことない。8ドル程度。だが調べてみると、似たようなものがさらに2つ見つかった。テスト画像が入ったままのストレージバケット。どこにも繋がっていないDNSレコード。気づかないうちに、デジタルの墓地を築いていた。

ソロ開発者は、出荷するプロジェクトよりも殺すプロジェクトの方がはるかに多い。これは失敗ではない。探索とはそういうものだ。アイデアを試し、プロトタイプを作り、市場が反応しないか技術的に筋が悪いと分かったら、次に進む。問題は、放棄されたプロジェクトが必ずクラウドリソースの残骸を残すことだ。そしてその残骸は、静かに蓄積する。

技術的負債という概念は誰もが知っている。今日は動くが明日問題になるコード。だがもう一つ、あまり語られない負債がある。インフラ負債だ。技術的負債より厄介な理由は単純である。技術的負債はコードを開くたびに目に入る。不快だし、気になる。インフラ負債は沈黙している。普段見ないダッシュボードの中に住み、自動引き落としの請求書の中に潜み、バックグラウンドで誰にも気づかれずに動き続ける。

金銭的コストだけではない。もっと深刻なのは認知的コストだ。放置したリソースの一つ一つが、頭の片隅を占有する。「AWSのあのデータベース、まだ動いてるんだっけ?」こういう小さな不安が十数プロジェクト分積み重なると、集中力を蝕むバックグラウンドノイズになる。

この問題に気づいてから、ツールの評価基準が根本的に変わった。「どうスケールするか」ではなく「どう死ぬか」を聞くようになった。プロジェクトの綺麗な死が、最も重要な要件になった。そして一つの奇妙な結論に辿り着いた。自分のインフラで最も重要なコマンドは `pulumi up` ではない。`pulumi destroy` だ。

ニヒリスティックに聞こえるかもしれない。だが実態は逆である。綺麗に壊せることが、新しく始める自由を与えてくれる。

コンピュートにはGCP Cloud Runを使っている。最大の理由はscale to zeroだ。「ほぼゼロ」ではない。ゼロだ。誰もサービスにアクセスしていなければ、課金は発生しない。自分は常時5つか6つのプロジェクトを並行して走らせている。ある日に実際のトラフィックがあるのは、せいぜい1つか2つだ。月5ドルのVPSは安く見える。だが誰も使っていない6プロジェクト分で月30ドル。ゴーストを維持するための出費だ。

Cloud Runはコンテナベースでもある。あるプロジェクトはRust、別のプロジェクトはNode.js。来年Goで書き直したくなっても、インフラは何も変わらない。変わるのはコンテナの中身だけだ。これは単なる便利さではない。アプリケーション層をインフラ層に触れずに入れ替えられるということだ。もう一つのeasy teardownの形である。

データベースにはNeonを使っている。サーバーレスPostgreSQLだ。重要なのは、これが本物のPostgreSQLであること。「PostgreSQL互換」ではない。抽象化レイヤーが被さったものでもない。生のPostgreSQLだ。SQLxでcompile-time query verificationをするとき、話している相手は本物のエンジンでなければならない。

Neonを選んだ決め手はbranchingだ。本番データベースをforkしてブランチを作り、そこでmigrationをテストし、終わったらブランチを捨てる。大きなチームはこれを専用のstaging環境で解決する。ソロ開発者にはそんな余裕はない。Neonのbranchingは、運用コストなしでセーフティネットを提供してくれる。

そしてこのスタックの他のすべてと同様、Neonもscale to zeroだ。開発用ブランチはアイドル時にコストがかからない。プロジェクトを放棄しても、裏でcomputeが回り続けて課金されることはない。

パターンはもう明らかだろう。このスタックのすべてが、使われなくなった瞬間に休眠する。

フロントエンドとオブジェクトストレージにはCloudflareを使っている。Workersでアプリケーションを配信し、R2でファイルを保存する。明らかな比較対象はVercelだ。確かに、Next.jsに対するVercelの開発体験の方が洗練されている。だがCloudflareの料金モデルは次元が違う。R2のegressは無料だ。ゼロ。S3で画像を配信してバズった経験があれば、これがなぜ重要か分かるはずだ。バズった瞬間、2ドルの請求が200ドルになる。R2なら2ドルのままだ。

Cloudflare Workersは世界300箇所以上のロケーションで動く。リージョンを考える必要なく、どこからでも高速だ。ソロ開発者にとって、自分が取らなくていい判断の一つ一つが、本当に重要なことに使えるエネルギーになる。

そして、すべてを繋ぐ部品がPulumiだ。

PulumiはInfrastructure as Codeツールで、Terraformと同じカテゴリに属する。違いは、インフラをTypeScriptで書けること。毎日使っている言語だ。HCLもYAMLも新しい構文も不要。型安全性とIDEの補完がそのまま使える。学習コストは実質ゼロだった。

だがPulumiがこのスタックの中核である理由は、言語ではない。`destroy` コマンドだ。`pulumi destroy` を実行すると、すべてが消える。Cloud Runのサービス、Neonのデータベース、R2のバケット、IAMバインディング、シークレット。すべてが依存関係の正しい順序で、綺麗に、完全に消える。何も残らない。

これがソロ開発者として複数プロジェクトを回すことを可能にしている。新しいプロジェクトを午後一つで立ち上げられる。Pulumiのテンプレートをコピーし、名前を変え、`pulumi up` を実行するだけだ。そして追求する価値がないと判断したら、同じ速さで壊せる。作る摩擦は低い。だが本当に重要なのは、壊す摩擦がゼロであることだ。

恐れなく壊せるようにしている仕組みが一つある。`protect` フラグだ。本当に大事なリソース、たとえば収益を生んでいるプロジェクトのデータベースには、protectedマークを付ける。protectedなリソースは `pulumi destroy` を生き残る。

```typescript
const db = new neon.Project("production-db", { ... }, { protect: true });
```

この一行が、「慎重にやらないと」と「さっさと片付けよう」の違いを生む。重要なものを消すリスクなしに、積極的にクリーンアップできる。`protect` フラグは `pulumi destroy` を危険なコマンドからルーティンに変える。

スタック全体を俯瞰すると、すべての選択が同じものに最適化されている。休止時の低コストと、容易な破棄だ。Cloud Runはscale to zero。Neonもscale to zero。Cloudflareはegressを課金しない。Pulumiはすべてを綺麗に破壊する。

世のインフラ指南の大半は、スケールアップの方法を教える。プロジェクトが成功して成長することを前提にしている。だがソロ開発者にとって、あらゆるプロジェクトの圧倒的に高い確率の結末は、放棄だ。そのケースに最適化することは悲観主義ではない。現実主義だ。そしてそれこそが、前のプロジェクトの亡骸を引きずらずに次を始める自由を与えてくれるものである。

綺麗に死ねないインフラは、いずれ後悔する。`pulumi destroy` はプロジェクトの悲しい結末ではない。次のプロジェクトを可能にする白紙だ。
