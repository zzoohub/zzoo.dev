ソロ開発者として、複数のプロジェクトを同時に運用している。それぞれ運用コストが安く、デプロイが簡単で — そして誰も話題にしない部分 — 撤退するときに**簡単に壊せる**必要がある。

いくつかの組み合わせを試行錯誤した結果、[Mealio](https://github.com/zzoohub/mealio)（写真ファーストの食事記録アプリ）のようなプロジェクトで横断的に使うスタックに落ち着いた。各ピースがなぜ採用されたのかを説明する。

## GCP Cloud Run — アイドル時にコストゼロのサーバー

Cloud Runはスケールトゥゼロのフルマネージドコンテナプラットフォームだ。この「スケールトゥゼロ」がソロ開発者にとって決定的に重要。

僕のプロジェクトの大半はトラフィックが不規則だ。数百リクエストの日もあれば、ゼロの日もある。Cloud Runなら、アイドル中は一切料金がかからない。リクエストが来た瞬間にコンテナが数秒で立ち上がる。

MealioのRust/Axum APIでは、Cloud Runの設定はこんな感じだ：

- **0-3インスタンス**のオートスケーリング（スケールトゥゼロ）
- インスタンスあたり**1 CPU、512Miメモリ**
- インスタンスあたり**最大80同時リクエスト**
- スタートアップとライブネスのヘルスチェック

稼働時間の90%がアイドルの$5/月のVPSと比べて、Cloud Runなら本格的なオートスケーリング、TLS、ヘルスチェック付きのプロダクションサービスをデプロイでき — 実際の使用量に対してのみ支払う。トラフィックの少ないサイドプロジェクトなら、月額1桁ドルで済むことも多い。

コンテナベースのモデルは、特定のランタイムにロックインされないことも意味する。今日はRustバイナリをデプロイしているが、明日GoやNode.jsに切り替えたくなっても、デプロイの話は同じ：Dockerイメージをビルドしてプッシュすれば完了。

## Neon — オーバーヘッドなしのマネージドPostgreSQL

データベースレイヤーには[Neon](https://neon.tech)を選んだ。サーバーレスPostgreSQL — 使用量に基づいてデータベースがスケールし、ブランチングが無料で使える。

### なぜSupabaseじゃないのか？

Supabaseは素晴らしいプロダクトで、ユーザーがアジアに集中しているなら真剣に検討すべきだ。**Neonはまだソウルリージョンをサポートしていない**ので、韓国のユーザー向けに作るなら重要だ。Supabaseはサポートしているし、認証、ストレージ、リアルタイム機能をバンドルしていて、別々のサービスが不要になる。

でも僕のユースケースでは、いくつかの理由でNeonが勝つ：

1. **ピュアPostgreSQL** — 余計な抽象化レイヤーなし。RustでSQLxのコンパイル時チェック付きクエリを使っていて、データベースは単なるPostgreSQLであってほしい。
2. **ブランチング** — 本番に適用する前にマイグレーションをテストするためのデータベースブランチを作成できる。専用のステージング環境がない場合、これは非常に便利。
3. **スケールトゥゼロのコンピュート** — Cloud Runと同様に、Neonはアイドルなコンピュートをサスペンドする。開発ブランチのコストはほぼゼロ。
4. **充実した無料枠** — サイドプロジェクトやアーリーステージのプロダクトには十分すぎる。

Pulumiのセットアップでは、Neonプロジェクトの立ち上げは数行のコードだ：

```typescript
const neonProject = new neon.Project("mealio", {
  name: "mealio",
  regionId: "aws-us-east-1",
  pgVersion: 18,
  historyRetentionSeconds: 21600,
});
```

## Cloudflare — より安いエッジネットワーク

Cloudflareを多用している：フロントエンドホスティングにWorkers、オブジェクトストレージにR2、グローバル配信にエッジネットワーク。

### なぜVercelじゃないのか？

VercelはNext.jsデプロイのデファクトで、洗練されたプロダクトだ。でもソロ開発者として複数プロジェクトを運用すると、コストがすぐに嵩む。

Cloudflareの料金設定は劇的に寛大だ：

- **Workers**：無料枠で1日100,000リクエスト。Vercelの無料枠は月100GB帯域幅でファンクションの制限がより厳しい。
- **R2**：エグレス料金ゼロ。画像やファイルを配信するアプリにとって、これは非常に大きい。S3のエグレスコストはスケール時に驚きの請求になりかねない。
- **グローバル配信**：Cloudflareは世界中300以上のデータセンターで稼働。すべてのデプロイが即座にどこでもアクセス可能で、少数のリージョンだけじゃない。

Mealioでは、ユーザーがアップロードした食事写真の保存にR2を使っている。エグレス料金がゼロなので、バズった瞬間が驚きの請求に変わる心配がない。

トレードオフは？CloudflareのNext.js向けの開発体験はVercelほど洗練されていない。`@cloudflare/next-on-pages`が必要で、互換性の端に当たることもある。でもコスト削減とグローバルパフォーマンスは、自腹で払っている人間にとって十分に価値がある。

## Pulumi — 壊すことはセットアップと同じくらい重要

このスタックの中で僕が最も強い意見を持っている部分で、ほとんどのソロ開発者が見落としているもの。

**ソロ開発者として、リリースするより多くのプロジェクトを終了させることになる。** これは正常で健全なことだ。実験し、アイデアを検証し、先に進む。しかし終了したすべてのプロジェクトは、インフラの痕跡を残す：データベース、クラウドストレージバケット、IAMロール、コンテナレジストリ、DNSレコード、シークレット。

これらを手動で削除するのは面倒で、ミスしやすく、気が滅入る。必ず何かを忘れて、誰も使っていないデータベースに課金され続ける。

Pulumiはこれを1コマンドで解決する：

```bash
pulumi destroy
```

すべてが消える。Cloud Runサービス、Neonデータベース、R2バケット、IAMバインディング、シークレット — すべてが正しい依存関係の順序でクリーンに削除される。

### なぜTerraformじゃなくてPulumiなのか？

どちらも優れたIaCツールだが、PulumiはTypeScriptでインフラを書ける — 僕が日常的に使っている同じ言語で。HCLを学ぶ必要なし、コンテキストスイッチングなし。型安全性、IDEの自動補完、ループ、条件分岐、抽象化を自然に使える。

Mealioのインフラエントリーポイントの簡略版はこんな感じだ：

```typescript
import { cloudRunService } from "./src/gcp-cloudrun";
import { neonProject } from "./src/neon";
import { r2Bucket } from "./src/cloudflare";
import { secrets } from "./src/gcp-secrets";
import { registry } from "./src/gcp-registry";
import { wif } from "./src/gcp-iam";
```

各モジュールは集中したTypeScriptファイルだ。新しいプロジェクトを始めるときは、構造をコピーし、名前と設定を調整して`pulumi up`。プロジェクトが終わったら`pulumi destroy`。残留リソースなし、忘れた請求なし。

### protectフラグ

誤ったdestroyを生き延びるべき本番リソースには、Pulumiの`protect`オプションがある：

```typescript
const db = new neon.Project("my-db", { ... }, { protect: true });
```

これにより、重要なリソースが誤って削除されないと確信しながら`pulumi destroy`を実行でき、それ以外はすべてクリーンアップされる。

## 全体像

典型的なプロジェクトでこれらのピースがどう組み合わさるか：

| レイヤー | サービス | 理由 |
|-------|---------|-----|
| **サーバー** | GCP Cloud Run | スケールトゥゼロ、コンテナベース、従量課金 |
| **データベース** | Neon | サーバーレスPostgreSQL、ブランチング、スケールトゥゼロ |
| **フロントエンド / CDN** | Cloudflare Workers | グローバルエッジ、エグレスゼロ、充実した無料枠 |
| **ストレージ** | Cloudflare R2 | S3互換、エグレス料金ゼロ |
| **シークレット** | GCP Secret Manager | Cloud Runとのネイティブ統合 |
| **CI/CD** | GitHub Actions | 公開リポジトリは無料、GCPとのWIF認証 |
| **IaC** | Pulumi | TypeScript、簡単な解体、リソース保護 |

## このスタックが最適化するもの

これは10人のエンジニアとベンチャー資金を持つスタートアップ向けのスタックではない。以下のようなソロ開発者のためのスタックだ：

- **複数プロジェクトを運用**し、それぞれ休止時に安い必要がある
- KubernetesやVMの管理なしに**速くリリース**したい
- 優雅なシャットダウンを含む**フルライフサイクルを考える**
- **自腹で払い**、予測可能な従量課金を望む
- 一つのベンダーにすべてのマネージドサービスを委ねるより**シンプルさを重視**する

これらすべての選択に共通するのは、**スケールトゥゼロの経済性と運用のシンプルさ**だ。プロジェクトがアクティブなときはインフラがスケールアップして対応する。そうでないときは、ほぼゼロのコスト。そしてお別れの時が来たら、1コマンドですべてクリーンアップ。

それがソロ開発者に本当に必要なインフラの自由だ。
