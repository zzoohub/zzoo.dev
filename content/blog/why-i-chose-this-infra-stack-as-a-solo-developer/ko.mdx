내 인프라에서 가장 중요한 명령어는 `deploy`가 아니다. `pulumi destroy`다.

이상하게 들릴 수 있다. 인프라라고 하면 보통 만드는 걸 떠올린다. 서버 띄우고, 데이터베이스 붙이고, 스토리지 설정하고. 업계 전체가 "어떻게 올릴 것인가"에 맞춰져 있다. 그런데 솔로 개발자가 인프라로 가장 자주 하는 일은 세팅이 아니다. 철거다.

솔로 개발자로 여러 프로젝트를 운영하면서 깨달은 게 있다. 출시하는 프로젝트보다 죽이는 프로젝트가 훨씬 많다는 것. 이건 실패가 아니다. 탐색이란 원래 그렇다. 아이디어를 시도하고, 프로토타입을 만들고, 시장이 반응하지 않거나 기술적으로 안 되면 다음으로 넘어간다. 문제는 버려진 프로젝트마다 클라우드 리소스의 잔해가 남는다는 거다. 여기 데이터베이스 하나, 저기 IAM role 하나, 테스트 이미지 200MB가 든 스토리지 버킷, 아무 데도 안 가는 DNS 레코드. 하나하나는 별거 아니다. 그런데 프로젝트 열두 개가 쌓이면 얘기가 달라진다. 돈보다 더 나쁜 건 "저기 뭐가 아직 살아있지?" 하는 인지적 부담이다.

나는 이걸 인프라 부채라고 부른다. 솔로 개발자판 기술 부채다. 다만 더 교활하다. 소리 없이 쌓이니까. 클라우드 청구서를 확인하고서야 삭제 안 한 데이터베이스 세 개를 발견하게 된다.

이 깨달음이 현재 스택의 모든 선택을 이끌었다. 철거가 세팅보다 중요하다는 것.

컴퓨팅에는 GCP Cloud Run을 쓴다. 이유는 단순하다. scale-to-zero가 된다. 아무도 API를 호출하지 않으면 비용이 0이다. "조금 나가는" 게 아니라 진짜 0이다. 프로젝트 대여섯 개를 돌리는데 특정 날에 트래픽이 있는 건 한두 개뿐이라면, 이게 결정적이다. 월 $5 VPS가 싸 보이지만 몇 달째 안 건드린 프로젝트가 여덟 개면 월 $40이다.

Cloud Run은 컨테이너 기반이라 런타임에 종속되지 않는다는 장점도 있다. 한 프로젝트에선 Rust 바이너리를, 다른 프로젝트에선 Node.js 앱을 배포한다. 방법은 동일하다. 컨테이너 빌드하고, 푸시하고, 끝. 내년에 Go로 다시 쓰고 싶으면 인프라는 건드릴 필요가 없다. 이 이식성도 일종의 손쉬운 철거다. 인프라 계층을 건드리지 않고 애플리케이션 계층만 갈아낄 수 있다.

데이터베이스로는 Neon을 쓴다. 서버리스 PostgreSQL인데, 핵심은 PostgreSQL이라는 점이다. 뭔가 덧붙인 PostgreSQL이 아니라 그냥 PostgreSQL. Rust에서 SQLx로 컴파일 타임에 쿼리를 검증하는데, "대체로 호환되는" 추상화가 아니라 진짜 PostgreSQL이어야 한다.

그런데 실제로 Neon을 선택하게 만든 건 branching이다. 프로덕션 데이터베이스를 fork해서 migration을 테스트한 다음, branch를 버리면 된다. 큰 팀은 이 문제를 전용 staging 환경으로 해결한다. 솔로 개발자에겐 그런 여유가 없다. Database branching이 운영 부담 없이 안전망을 제공한다.

Cloud Run과 마찬가지로 Neon도 scale-to-zero다. 개발용 branch는 비용이 거의 안 든다. 프로젝트를 포기해도 뒤에서 컴퓨트가 돌아가지 않는다. 패턴이 보일 것이다. 이 스택의 모든 것은 사용하지 않으면 잠든다.

프론트엔드 호스팅과 스토리지는 Cloudflare를 쓴다. Workers로 앱을 서빙하고, R2로 파일을 저장한다. Vercel과 비교하면, 맞다, Vercel이 Next.js에 대한 개발 경험은 더 좋다. 하지만 Cloudflare의 가격은 차원이 다르다. R2의 egress 비용은 0이다. 진짜 0. 사진이 많은 앱을 S3에서 돌려보고 egress 비용이 올라가는 걸 지켜본 적이 있다면 이게 왜 중요한지 안다. 바이럴 한 번이면 스토리지 비용이 $2에서 $200이 된다. R2에서는 $2 그대로다.

Cloudflare Workers는 전 세계 300개 넘는 edge에서 돌아간다. 리전 고민 없이 어디서든 빠르다. 솔로 개발자에게 생각할 거리가 적다는 건 큰 장점이다. 도메인 연결하고 잊어버리면 된다.

여기서 모든 게 하나로 모인다. Pulumi.

Pulumi는 Terraform 같은 Infrastructure as Code 도구다. 클라우드 리소스를 코드로 정의하면, 도구가 생성, 수정, 삭제를 알아서 처리한다. 차이점은 Pulumi가 TypeScript로 작성할 수 있게 해준다는 것이다. 매일 쓰는 언어 그대로. HCL도, YAML도, 새로 배울 문법도 없다. 타입 안전성에 IDE 자동완성까지 된다.

그런데 Pulumi가 이 스택에서 가장 중요한 이유는 언어가 아니다. `destroy` 명령어 때문이다. `pulumi destroy`를 실행하면 모든 게 사라진다. Cloud Run 서비스, Neon 데이터베이스, R2 버킷, IAM 바인딩, secret까지. 의존성 순서에 맞춰 깔끔하게. 완전하게. 남는 건 없다.

이것이 이 스택을 솔로 개발자에게 작동하게 만드는 핵심이다. 오후 한나절이면 새 프로젝트를 띄울 수 있다. Pulumi 템플릿 복사하고, 이름 바꾸고, `pulumi up`. 그리고 안 되겠다 싶으면 똑같이 빠르게 없앨 수 있다. 인프라를 만드는 마찰이 낮다. 하지만 더 중요한 건, 파괴하는 마찰이 0이라는 거다.

이 두려움 없는 파괴를 가능하게 하는 기능이 있다. `protect` 플래그다. 실제로 중요한 프로덕션 리소스에는 보호를 건다. 보호된 리소스는 `pulumi destroy`를 실행해도 살아남는다. 그래서 나머지를 공격적으로 정리하면서도 중요한 것이 날아갈 걱정은 안 해도 된다. protect 플래그가 `pulumi destroy`를 위험한 명령어에서 일상적인 명령어로 바꿔준다.

```typescript
const db = new neon.Project("production-db", { ... }, { protect: true });
```

이 한 줄이 "조심해야 하는데"와 "그냥 정리하자" 사이의 차이다.

한 발 물러서서 보면, 이 스택의 모든 선택에는 같은 맥락이 흐른다. 유휴 비용은 낮게, 철거는 쉽게. Cloud Run은 scale-to-zero. Neon도 scale-to-zero. Cloudflare는 egress 무료. Pulumi는 깔끔한 파괴. 안 쓸 때 싸고, 끝나면 사라진다.

대부분의 인프라 조언은 반대 방향을 가리킨다. scale up하는 법을 알려준다. 프로젝트가 성공하고 성장할 거라고 가정한다. 그런데 솔로 개발자라면, 특정 프로젝트의 압도적으로 높은 확률은 포기다. 그 경우에 최적화하는 것은 비관이 아니다. 현실이다. 그리고 이전 프로젝트를 질질 끌지 않고 다음 프로젝트를 시작할 자유를 주는 것이기도 하다.

쉽게 부술 수 없는 인프라는 후회할 인프라다. `pulumi destroy`는 프로젝트의 슬픈 결말이 아니다. 다음 프로젝트를 가능하게 하는 백지다.
