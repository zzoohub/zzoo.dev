[이전 글](/ko/blog/my-claude-code-setup)에서 21개 스킬, 7개 에이전트, 8개 MCP 서버로 구성한 Claude Code 세팅을 공유했다. 그 글을 쓰면서도 느꼈다 — 저건 빌드 파이프라인의 이야기일 뿐이라고. 프로덕트는 빌드로 끝나지 않는다.

지금은 아이디어 검증부터 런칭, 측정, 성장까지 전체 생명주기를 커버한다. 스킬 30개+, 에이전트 10개+, MCP 서버 15개+, 플러그인 20개+. 이 전체 시스템이 스캐폴드 프로젝트에 템플릿화되어 있어서 새 프로덕트를 시작할 때 복사하면 된다.

## 전체 생명주기

```
0. Validate  — 가설 하나, 최소 테스트, 유기적 풀이 보일 때까지
      ↓
1. Product → Design → UI
      ↓
2. Architecture → Build  (+ 트래킹 플랜 병렬)
      ↓
3. Deploy   — CI/CD, 인프라, 첫 배포
      ↓
4. Launch   — 마케팅, 프라이싱, 법무
      ↓
5. Measure  — Kill / Keep / Scale
      ↓
6. Grow     — Scale 판정 후에만
```

코드를 한 줄도 안 쓰는 단계가 절반 이상이다. 그리고 이전 글에서 빠져 있던 핵심 — **빌드 이후**가 추가됐다.

## Phase 0. Validate

만들기 전에 죽여라.

가설 하나를 쓴다. 최소한의 테스트를 몇 시간 안에 만들어서 실제 사용자에게 보낸다. 유기적 풀이 보이면 진행, 2-3번 피벗해도 시그널이 없으면 Kill.

이전 글에서는 Product Brief부터 시작했다. 지금은 아니다. Brief를 쓰기 전에 "이 문제가 진짜인가?"를 먼저 검증한다.

## Phase 1. Product → Design → UI

검증된 아이디어를 문서로 만들고, 설계하고, UI로 만든다:

| 스텝 | 스킬/도구 | 산출물 |
|------|-----------|--------|
| Product Brief | `z-product-brief` | 문제 정의, 가설, 성공 기준 |
| PRD | `z-prd-craft` | 상세 요구사항, P0/P1/P2 |
| UX Design | `z-ux-designer` 에이전트 | 사용자 플로우, 화면 스펙 |
| UI Mockup | Stitch MCP (병렬) | 시각적 목업 |
| UI 구현 | `frontend-design` 스킬 | 프로덕션급 컴포넌트 |

각 스킬의 상세 동작은 [이전 글](/ko/blog/my-claude-code-setup)에서 다뤘으니 여기선 생략한다. 달라진 건 Stitch MCP가 추가되어 UX 설계와 UI 프로토타이핑을 병렬로 진행할 수 있게 된 것이다.

## Phase 2. Architecture → Build

핵심 변화: **트래킹 플랜이 아키텍처와 병렬로 설계된다.**

```
z-software-architect    → 설계 문서
z-database-design       → DB 스키마
z-rest-api-design       → OpenAPI 스펙

z-data-analyst (병렬)   → 트래킹 플랜, 퍼널, 대시보드, Kill 기준
```

트래킹을 나중에 붙이는 게 아니라 처음부터 bake-in한다. "이 이벤트를 왜 추적하는가?"가 아키텍처 결정 시점에 답해져야 한다.

구현 스킬은 프로젝트 유형에 따라 CLAUDE.md가 강제한다:

- **Web**: `vercel-composition-patterns` → `vercel-react-best-practices`
- **API (Rust)**: `z-axum-hexagonal` + `z-postgresql`
- **API (Python)**: `z-fastapi-hexagonal` + `z-postgresql`
- **Mobile**: `expo-app-design:building-native-ui` → `vercel-react-native-skills`

구현 후 `z-security-reviewer` + `z-verifier`가 병렬로 품질 게이트를 수행한다. 이전과 동일하되, 문서나 카피만 바꾼 경우에는 건너뛴다 — CLAUDE.md에 명시되어 있다.

## Phase 3. Deploy

Pulumi로 인프라를 코드로 관리한다. 이전 글과 동일하지만, 배포 체크리스트가 추가됐다:

- [ ] Staging → smoke test → Production
- [ ] PostHog 이벤트 firing 확인
- [ ] Sentry 연결 (전체 서비스)
- [ ] Aha Moment 이벤트 E2E 작동 확인

배포는 "코드가 서버에 올라가는 것"이 아니라 **"측정 인프라가 작동하는 것"**까지 포함한다.

## Phase 4. Launch

이전 글에 없었던 것. `z-marketer` 에이전트가 런칭 전체를 조율한다:

```
z-marketer (허브)
  → 포지셔닝, 오디언스, 채널 전략
  → 프라이싱 → z-pricing 스킬
  → 이메일 시퀀스 → z-email-marketing 스킬
  → SEO → z-search-visibility 스킬
  → 런치 자료 (Product Hunt, HN, Reddit)
  → 법무 (ToS, 개인정보처리방침)
```

`z-marketer`가 허브 역할을 하고, 프라이싱은 `z-pricing`, 이메일은 `z-email-marketing`, SEO는 `z-search-visibility` 같은 전문 스킬에 위임한다. 런치 당일: 체크리스트대로 포스팅하고, 모든 코멘트에 2시간 내 응답하고, 피드백을 로깅한다.

## Phase 5. Measure → Decide

이것도 이전 글에 없었다. `z-data-analyst` 에이전트가 매주 리포트를 만들고, Kill / Keep / Scale을 추천한다.

**Week 1**: 관찰만. 피드백에 응답. 최적화하지 않는다.
**Week 2**: Kill 기준에 따라 첫 평가.
**이후**: 주간 재평가. 판정이 명확할 때까지.

수정 우선순위: **Retention → Activation → Acquisition.** 새 사용자를 데려오기 전에 기존 사용자가 남는지 먼저 확인한다.

- **Kill** → 포스트모템, 아카이브, 같은 날 다음으로.
- **Keep** → Phase 1이나 2로 돌아가서 가장 큰 문제를 고친다.
- **Scale** → Phase 6으로.

## Phase 6. Grow

새는 제품은 키우지 않는다. Scale 판정 후에만 진입한다.

`z-growth-optimizer` 에이전트:
- CRO 실험 (ICE 스코어로 우선순위)
- 리퍼럴/바이럴 루프
- 이탈 방지 (`z-churn-prevention`)
- 결제 실패 복구
- 콘텐츠 SEO

이 단계에서야 E2E 테스트를 추가한다. 프로덕트가 살아남았으니까 핵심 플로우를 보호할 가치가 생긴 것이다.

## 스캐폴드

이 전체 시스템이 하나의 프로젝트 템플릿에 들어 있다:

```
_scaffold/
├── CLAUDE.md        # 프로젝트 공통 룰 + 스킬 매핑
├── workflow.md      # 위 6단계 워크플로우
├── docs/            # 기획 문서 (brief, PRD, 설계 문서)
├── biz/             # 비즈니스 (마케팅, 분석, 성장, 법무, 운영)
│   ├── analytics/   # 트래킹 플랜, 퍼널, Kill 기준
│   ├── marketing/   # 전략, 런치, 콘텐츠, SEO
│   ├── growth/      # CRO, 리퍼럴, 이탈 방지
│   ├── legal/       # ToS, Privacy Policy
│   └── ops/         # FAQ, 런북, 피드백 로그
├── services/        # API, Worker
├── clients/         # Web, Mobile
├── db/              # 마이그레이션, 시드
├── infra/           # Pulumi IaC
├── openapi/         # API 계약서
└── scripts/
```

새 프로덕트를 시작하면 이 폴더를 복사한다. CLAUDE.md와 workflow.md가 세팅되어 있으니, Claude Code는 첫 세션부터 전체 파이프라인을 안다.

`docs/`에 기획 문서가 채워지고, `biz/`에 비즈니스 문서가 채워지면서 프로덕트가 자란다. 문서와 코드가 같은 레포에 있으니 AI가 항상 최신 맥락을 읽는다.

## 달라진 것

| 항목 | 이전 | 지금 |
|------|------|------|
| 범위 | 빌드 파이프라인 (9단계) | 전체 생명주기 (7단계) |
| 스킬 | 21개 | 30개+ |
| 에이전트 | 7개 | 10개+ |
| MCP 서버 | 8개 | 15개+ |
| 플러그인 | — | 20개+ |
| 마케팅 | 없음 | `z-marketer` + 전문 스킬 5개 |

| 성장 | 없음 | `z-growth-optimizer` + `z-churn-prevention` |
| 템플릿 | 없음 | `_scaffold/` 프로젝트 |

## 바이브 코딩 v2

이전 글에서 "타이피스트가 아니라 디렉터"라고 했다. 여전히 맞다. 하지만 이전에는 프로덕트를 **만드는** 디렉터였고, 지금은 프로덕트를 만들고, 런칭하고, 측정하고, 성장시키는 디렉터다.

한 사람이 전체 프로덕트 생명주기를 운영한다. 각 단계에서 전문가 수준의 스킬이 로드되고, 에이전트가 병렬로 실행되고, MCP 서버가 외부 세계와 연결해준다. 내가 하는 건 방향을 잡고 판단을 내리는 것이다.

바이브가 좋다. 파이프라인이 뒤를 봐준다.
