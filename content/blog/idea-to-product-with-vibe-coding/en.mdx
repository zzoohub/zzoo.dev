In a [previous post](/en/blog/my-claude-code-setup), I shared my Claude Code setup ‚Äî 21 skills, 7 agents, 8 MCP servers. Even while writing it, I knew that was only half the story. A product doesn't end at build.

Now the pipeline covers the full lifecycle: idea validation, launch, measurement, and growth. 30+ skills, 10+ agents, 15+ MCP servers, 20+ plugins. The entire system is templatized in a scaffold project ‚Äî copy it and start building.

## The Full Lifecycle

```
0. Validate  ‚Äî One hypothesis, minimum test, until organic pull
      ‚Üì
1. Product ‚Üí Design ‚Üí UI
      ‚Üì
2. Architecture ‚Üí Build  (+ tracking plan in parallel)
      ‚Üì
3. Deploy   ‚Äî CI/CD, infrastructure, first deploy
      ‚Üì
4. Launch   ‚Äî Marketing, pricing, legal
      ‚Üì
5. Measure  ‚Äî Kill / Keep / Scale
      ‚Üì
6. Grow     ‚Äî Only after Scale
```

More than half these steps produce zero lines of code. And the key addition since the last post: **everything after build**.

## Phase 0. Validate

Kill before you build.

Write one hypothesis. Build the minimum test in hours. Ship to real users. If you see organic pull, proceed. No signal after 2-3 pivots? Kill.

The previous post started with Product Brief. Not anymore. Before writing a Brief, validate whether the problem is real.

## Phase 1. Product ‚Üí Design ‚Üí UI

Validated ideas become documents, then designs, then UI:

| Step | Skill/Tool | Output |
|------|-----------|--------|
| Product Brief | `z-product-brief` | Problem definition, hypotheses, success criteria |
| PRD | `z-prd-craft` | Detailed requirements, P0/P1/P2 |
| UX Design | `z-ux-designer` agent | User flows, screen specs |
| UI Mockup | Stitch MCP (parallel) | Visual mockups |
| UI Implementation | `frontend-design` skill | Production-grade components |

Each skill's detailed behavior was covered in the [previous post](/en/blog/my-claude-code-setup). The key change: Stitch MCP now enables parallel UX design and UI prototyping.

## Phase 2. Architecture ‚Üí Build

Key change: **the tracking plan is designed in parallel with architecture.**

```
z-software-architect    ‚Üí design doc
z-database-design       ‚Üí DB schema
z-rest-api-design       ‚Üí OpenAPI spec

z-data-analyst (parallel) ‚Üí tracking plan, funnels, dashboards, kill criteria
```

Tracking isn't bolted on later ‚Äî it's baked in from the start. "Why are we tracking this event?" gets answered at architecture time.

Implementation skills are enforced per project type via CLAUDE.md:

- **Web**: `vercel-composition-patterns` ‚Üí `vercel-react-best-practices`
- **API (Rust)**: `z-axum-hexagonal` + `z-postgresql`
- **API (Python)**: `z-fastapi-hexagonal` + `z-postgresql`
- **Mobile**: `expo-app-design:building-native-ui` ‚Üí `vercel-react-native-skills`

After implementation, `z-security-reviewer` + `z-verifier` run the quality gate in parallel. Same as before, except docs/copy-only changes skip it ‚Äî codified in CLAUDE.md.

## Phase 3. Deploy

Pulumi manages infrastructure as code. Same as before, but with a deploy checklist:

- [ ] Staging ‚Üí smoke test ‚Üí Production
- [ ] PostHog events firing
- [ ] Sentry connected (all services)
- [ ] Aha Moment event working end-to-end

Deployment isn't "code is on the server." It's **"measurement infrastructure is working."**

## Phase 4. Launch

This wasn't in the previous post. The `z-marketer` agent orchestrates the entire launch:

```
z-marketer (hub)
  ‚Üí Positioning, audience, channel strategy
  ‚Üí Pricing ‚Üí z-pricing skill
  ‚Üí Email sequences ‚Üí z-email-marketing skill
  ‚Üí SEO ‚Üí z-search-visibility skill
  ‚Üí Launch materials (Product Hunt, HN, Reddit)
  ‚Üí Legal (ToS, Privacy Policy)
```

`z-marketer` acts as the hub, delegating to specialized skills: `z-pricing` for pricing, `z-email-marketing` for email, `z-search-visibility` for SEO. Launch day: post per checklist, respond to all comments within 2 hours, log feedback.

## Phase 5. Measure ‚Üí Decide

Also new. The `z-data-analyst` agent produces weekly reports and recommends Kill / Keep / Scale.

**Week 1**: Observe only. Respond to feedback. Don't optimize.
**Week 2**: First evaluation against kill criteria.
**After**: Weekly re-evaluation until the decision is clear.

Fix priority: **Retention ‚Üí Activation ‚Üí Acquisition.** Confirm existing users stay before bringing new ones.

- **Kill** ‚Üí Post-mortem, archive, move on same day.
- **Keep** ‚Üí Back to Phase 1 or 2. Fix the biggest problem.
- **Scale** ‚Üí Proceed to Phase 6.

## Phase 6. Grow

Don't grow a leaky product. Enter only after Scale.

`z-growth-optimizer` agent:
- CRO experiments (ICE-scored priority)
- Referral/viral loops
- Churn prevention (`z-churn-prevention`)
- Payment failure recovery
- Content SEO

This is when E2E tests get added. The product survived ‚Äî now core flows are worth protecting.

In the previous post I said "not a typist ‚Äî a director." Still true. But before, I was directing the **build**. Now I'm directing the build, launch, measurement, and growth.
