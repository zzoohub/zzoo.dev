Cuando la gente escucha "vibe coding," la mayoría imagina decirle a la IA "hazme una página de login." Eso está mal. Al menos, eso no es lo que yo hago.

Mi vibe coding comienza con una sola idea y recorre product briefs, PRDs, diseño de arquitectura, esquemas de base de datos, flujos de UX, implementación, testing, auditorías de seguridad y deploy. **Escribir código es solo un paso en ese pipeline.**

En un post anterior dije "vive como constructor" y compartí mi setup de Claude Code. Este post muestra cómo esas herramientas realmente se combinan para convertir una idea en producto — paso a paso.

## El Pipeline Completo

```
Idea
  ↓
Paso 1. Product Brief — Definición del problema, hipótesis, criterios de éxito
  ↓
Paso 2. PRD — Requisitos detallados, prioridades P0/P1/P2
  ↓
Paso 3. Arquitectura de Software — Diseño de sistema, ADRs, trade-offs
  ↓
Paso 4. Diseño de Base de Datos — Schema, índices, migraciones
  ↓
Paso 5. Diseño de UX — Flujos de usuario, specs de pantalla, principios cognitivos
  ↓
Paso 6. Prototipado de UI — Wireframes, scaffolds visuales
  ↓
Paso 7. Implementación — Desarrollo guiado por skills
  ↓
Paso 8. Quality Gate — [Auditoría de seguridad + Testing] en paralelo
  ↓
Paso 9. Deploy y Operaciones
```

Más de la mitad de estos pasos producen cero líneas de código. Ese es el verdadero poder del vibe coding. Vamos paso por paso.

## Paso 1. Product Brief — "¿Por Qué Estamos Construyendo Esto?"

Cuando tengo una idea, lo primero que hago no es abrir un editor de código. Ejecuto `/product-brief`.

Este skill opera sobre el **Framework de Cuatro Riesgos** de Marty Cagan (SVPG). Cuando digo "quiero construir una app de seguimiento de comidas," el skill pregunta sobre cuatro riesgos:

| Riesgo | Pregunta | Validación |
|--------|----------|------------|
| **Valor** | ¿Los usuarios realmente usarán esto? | Entrevistas, análisis competitivo |
| **Usabilidad** | ¿Los usuarios podrán entenderlo? | Testing de prototipos |
| **Factibilidad** | ¿Podemos construirlo? | Spike técnico |
| **Viabilidad** | ¿Funciona como negocio? | Modelo de ingresos, revisión legal |

El principio central es **empezar por el problema, no la solución**. No "construyamos una app de seguimiento de comidas" sino "la gente no registra sus comidas porque el input es muy tedioso." El skill fuerza esta dirección.

El output es un documento de 1-2 páginas:

- **Problema** — Quién sufre, cuál es el dolor, cuál es la evidencia
- **Hipótesis y Riesgos** — Supuestos listados explícitamente con planes de validación
- **Dirección Propuesta** — La dirección (no listas de features, sino experiencias)
- **No-Objetivos** — Lo que esto NO es (previene el scope creep)
- **Criterios de Éxito** — 2-4 resultados medibles
- **Preguntas Abiertas** — Cosas que aún no sabemos

Esa última sección es la más importante. Escribir lo que no sabes. La mayoría de los proyectos fallan no por respuestas incorrectas sino por preguntas no formuladas. El mayor valor del product brief es sacar la incertidumbre a la luz temprano.

En este punto hay cero código. Pero qué construir, por qué, para quién y dónde parar está claro. Sin esto, programar con IA es solo autocompletado.

## Paso 2. PRD — "¿Qué Exactamente Estamos Construyendo?"

Una vez que el Brief establece la dirección, el skill `prd-craft` crea un documento de requisitos detallado. Si el Brief es el "por qué," el PRD es el "qué."

El skill comienza con una **Entrevista de Descubrimiento**. Esto no se puede saltar. Basándose en el Brief, pregunta lo que falta:

- ¿Cuáles son los personas de usuario específicos y contextos de uso?
- ¿Cómo resuelven los usuarios este problema actualmente? ¿Qué es doloroso?
- ¿Cómo mediremos el éxito? ¿Números específicos?
- ¿Qué está explícitamente fuera de alcance?

Luego escribe el PRD:

**Problema / Oportunidad** — La sección más crítica. "Necesitamos un sistema de notificaciones" no es un problema. "El 23% de los usuarios se pierden actualizaciones urgentes porque solo revisan la app una vez al día" sí lo es. El skill fuerza esta distinción — hay un checklist: "no plantees una solución como el problema."

**Requisitos Funcionales** — Features clasificadas como P0 (imprescindible para el lanzamiento), P1 (debería tener), P2 (sería bueno tener). Cada requisito se mapea a un **user journey**, no a un componente técnico. "El usuario puede resetear su contraseña por email" — funcional. "Crear índice B-tree en PostgreSQL" — detalle de implementación que no va aquí.

**Métricas de Éxito** — Cada objetivo tiene exactamente una métrica primaria con números específicos y plazos. Nada de lenguaje de "mejorar." "Tasa de completado de onboarding del 60% al 80%, dentro de 3 meses post-lanzamiento."

**User Journeys Críticos** — Seis journeys revisados sistemáticamente: primera experiencia (onboarding), loop de uso core, experiencia de usuario recurrente, gestión/mantenimiento, casos edge y estados de error, off-boarding/exportación de datos. Los últimos dos se omiten frecuentemente — el skill los fuerza.

En este punto existen dos documentos de planificación. Product Brief (dirección estratégica) y PRD (requisitos detallados). 10-15 páginas combinadas. Todavía cero líneas de código.

## Paso 3. Arquitectura de Software — "¿Cuál Es la Estructura?"

El PRD va al skill `software-architect`. Lee el PRD y produce un **Documento de Diseño de Arquitectura de Software (SADD)**.

Primero hace preguntas que el PRD no puede responder:

- ¿Usuarios esperados, requests/seg, volumen de datos?
- ¿Cuáles rutas son críticas en latencia?
- ¿Dónde se requiere consistencia fuerte? ¿Dónde está bien la consistencia eventual?
- ¿Se necesitan features en tiempo real? (WebSocket, SSE, polling)
- ¿Sensibilidad al costo? (solopreneur vs enterprise)

Luego escribe el documento de diseño. La idea clave es que las decisiones de arquitectura se dividen en **dos ejes independientes**:

**Eje 1: Arquitectura del Sistema** — Cómo se comunican los servicios. Request-response, event-driven, CQRS, event sourcing, o monolito modular.

**Eje 2: Estructura del Código** — Cómo se organiza internamente cada servicio. Hexagonal (puertos y adaptadores), clean, o por capas.

Estos se combinan libremente. La mayoría de mis proyectos solo caen en "monolito modular + hexagonal." No puedo costear la complejidad operacional de microservicios solo, y la arquitectura hexagonal facilita el testing y el intercambio de adaptadores.

Cada elección de tecnología tiene un **ADR (Architecture Decision Record)**. No solo "usamos Axum" sino:

> **ADR-1: Lenguaje y Framework del Backend**
> - Contexto: Desarrollador solo, servicio B2C, sensible al costo
> - Decisión: Rust + Axum
> - Alternativas: Python/FastAPI (rechazado — overhead de runtime, mayor memoria) / Go/Gin (rechazado — sistema de tipos más débil)
> - Consecuencias: (+) Respuesta sub-ms, 10-30MB de memoria, verificación en compile-time / (-) Tiempos de compilación, ecosistema más pequeño

El skill tiene referencias de preferencias de infraestructura integradas para el contexto de solopreneur:
- Base de datos: Neon (serverless, scale-to-zero, branching)
- Compute backend: GCP Cloud Run (basado en contenedores, auto-scaling, free tier generoso)
- Frontend: Cloudflare Workers + Next.js (edge SSR, precios predecibles)
- Móvil: Expo (React Native)
- Object storage: Cloudflare R2 (cero cargos por egress)
- IaC: Pulumi (nativo en TypeScript)
- Error tracking: Sentry / Analytics: PostHog

Estos defaults existen para que no delibere desde cero cada vez. Cuando los requisitos del proyecto difieren de los defaults, solo entonces registro la desviación y su justificación como ADR.

Para mi proyecto Mealio: Expo (móvil) + Axum (API) + Neon (DB) + GCP Cloud Run (compute) + Cloudflare R2 (almacenamiento de fotos) + Pulumi (infraestructura). Esta combinación se decidió en el SADD, con justificación documentada para cada elección.

## Paso 4. Diseño de Base de Datos — "¿Cómo Almacenamos los Datos?"

Una vez establecida la arquitectura, el skill `database-design` diseña el schema. Este skill tiene siete archivos de referencia:

| Referencia | Propósito |
|-----------|---------|
| `acid-transactions.md` | Niveles de aislamiento de transacciones, estrategias de bloqueo, prevención de deadlocks |
| `data-types-guide.md` | Guía de selección de tipos de PostgreSQL |
| `design-patterns.md` | Herencia de tablas, soft delete, audit trails, datos temporales |
| `indexing-strategy.md` | Tipos de índices, cálculo de selectividad, índices parciales |
| `migration-patterns.md` | Migraciones sin downtime, scripts de rollback |
| `normalization-guide.md` | Criterios de decisión de normalización/desnormalización |
| `performance-patterns.md` | Optimización de patrones de consulta, orden de columnas, particionamiento |

El flujo de trabajo es sistemático:

**Paso 1 — Recopilar requisitos.** Si existe un SADD, extraer entidades de dominio, flujos de datos, estrategia de almacenamiento y modelo de consistencia. Seguir las decisiones a nivel de sistema del SADD (plataforma de DB, patrón de arquitectura), pero evaluar independientemente las decisiones de dominio de DB (nivel de normalización, estrategia de particionamiento, diseño de índices, niveles de aislamiento de transacciones).

**Paso 2 — Diseño de schema.** Identificar entidades core, normalizar al menos a 3NF, evaluar necesidades de desnormalización, elegir tipos de datos apropiados, definir constraints (PK, FK, UNIQUE, CHECK, NOT NULL). Ordenar columnas de mayor a menor para minimizar el desperdicio por padding de alineación.

**Paso 3 — Diseño de transacciones y concurrencia.** Identificar operaciones que necesitan diseño explícito de transacciones: pagos, inventario, flujos multi-paso, recursos de alta contención. Para cada uno, decidir nivel de aislamiento, estrategia de bloqueo, lógica de reintento.

**Paso 4 — Escribir DDL.** PostgreSQL CREATE TABLE, INDEX, COMMENT.

**Paso 5 — Estrategia de índices.** PostgreSQL NO indexa automáticamente las columnas FK — un descuido común que el skill detecta automáticamente. Identificar columnas en WHERE, JOIN, ORDER BY frecuentes. Calcular selectividad. Elegir tipos de índice apropiados: Partial, Expression, Covering (INCLUDE), GIN/GiST/BRIN.

**Paso 6 — Revisión de performance.** Verificar planes de ejecución con EXPLAIN ANALYZE. Evaluar necesidades de connection pooling y particionamiento.

**Paso 7 — Plan de migración.** Scripts de migración versionados con scripts de rollback obligatorios. Estrategias de zero-downtime.

El MCP de Neon brilla aquí. Diseñar el schema, correr migraciones y verificar resultados en la misma sesión. Cero cambio de contexto.

Reglas duras: nunca usar FLOAT para valores monetarios (usar NUMERIC), siempre usar TIMESTAMPTZ, incluir created_at/updated_at en cada tabla, siempre crear índices en columnas FK. El skill las aplica para que yo no tenga que recordarlas.

## Paso 5. Diseño de UX — "¿Cómo Experimentan Esto los Usuarios?"

El agente `ux-designer` maneja el diseño de experiencia de usuario. A diferencia de otros agentes, este se ubica en la categoría de **planificación** y usa el modelo Opus. El diseño requiere razonamiento profundo.

La primera acción del agente es cargar tres archivos de referencia:

**`cognitive-principles.md`** — Ley de Hick (más opciones = decisiones más lentas), Ley de Fitts (objetivos más pequeños/lejanos = interacción más difícil), teoría de Carga Cognitiva, Gradiente de Meta (la motivación aumenta cerca de completar), Regla del Pico y Final (la memoria de una experiencia está dominada por el momento pico y el final). Cada decisión de diseño se etiqueta con estos principios.

**`design-process.md`** — Proceso de 5 pasos: Definir → Mapear → Diseñar → Eliminar → Validar. Los pasos no se pueden saltar ni reordenar.

**`ergonomics.md`** — Touch targets de 44px+, contraste de color 4.5:1 (texto), 3:1 (UI), outline de foco 2px+, ubicación en zona del pulgar para móvil. Umbrales numéricos específicos.

Cada tarea comienza con el **Checklist de Primeros Principios**:

1. ¿Cuál es la **UNA meta** del usuario? (No features, no métricas de negocio — su intención real)
2. ¿Cuál es el **mínimo necesario** para lograrlo? (Información, acciones, pantallas)
3. ¿Qué **se puede eliminar**? (Si eliminarlo no bloquea la meta, elimínalo)

El tercer punto es clave. Agregar features es fácil. Eliminarlas es difícil. El agente aplica "¿esto ayuda a la meta del usuario?" a cada elemento. ¿No pasa la prueba? Eliminado.

Outputs:

- **Flujo de Usuario** — Punto de entrada hasta completar la meta, puntos de decisión con defaults recomendados, manejo de errores y casos edge
- **Especificaciones de Pantalla** — Una acción principal por pantalla (visualmente dominante), solo la información necesaria para la decisión actual, acciones secundarias (visualmente discretas), mecanismos de feedback
- **Notas de Accesibilidad** — Contraste, estados de foco, navegación por teclado, lector de pantalla
- **Justificación del Diseño** — Qué principio cognitivo justifica cada decisión

Lista de anti-patrones: copy de marketing, hero sections con propuestas de valor vagas, secciones decorativas, onboarding innecesario, diálogos de confirmación para acciones no destructivas, elementos que existen "porque otras apps los tienen." Todos prohibidos.

## Paso 6. Prototipado de UI — "¿Cómo Se Ve?"

Con las specs de UX listas, el agente `ui-engineer` crea scaffolds visuales. La regla central de este agente: **cero lógica de negocio**.

Para web, usa shadcn/ui por defecto. Los componentes que no puede cubrir se construyen custom siguiendo el skill `design-system`. Para móvil, patrones nativos de la plataforma.

Los outputs son componentes de UI puros:
- Sin `useState`, `useEffect`
- Sin data fetching
- Sin lógica de negocio en event handlers
- Props solo para control visual

```tsx
// Lo que produce el ui-engineer: scaffold de UI puro
export function LoginPage() {
  return (
    <Card>
      <Card.Header>
        <Card.Title>Login</Card.Title>
      </Card.Header>
      <Card.Content className="flex flex-col gap-4">
        <FormField label="Email">
          <Input type="email" placeholder="email@example.com" />
        </FormField>
        <FormField label="Password">
          <Input type="password" />
        </FormField>
      </Card.Content>
      <Card.Footer>
        <Button variant="primary" fullWidth>Sign In</Button>
      </Card.Footer>
    </Card>
  );
}

// Agregado después por mí (u otro agente):
// - const [email, setEmail] = useState('')
// - const { mutate: login } = useLogin()
// - onSubmit handler
// - error states
// - redirect after success
```

¿Por qué esta separación? Cuando los límites de UI y lógica están claros, los cambios de diseño no tocan la lógica y los cambios de lógica no tocan la UI. En la práctica, esta separación hace una diferencia enorme en el mantenimiento.

El agente ejecuta un auto-review después de completar cargando el skill `frontend-design`. Verifica valores hardcodeados, todos los estados visuales definidos, cumplimiento de accesibilidad.

## Paso 7. Implementación — "Los Skills Imponen Convenciones"

Finalmente, se escribe código. Pero no libremente.

**CLAUDE.md fuerza los skills correctos para cada proyecto.** Cada raíz de proyecto tiene un archivo CLAUDE.md con "toda implementación debe usar los skills designados" como regla dura.

- Proyecto Next.js → skill `nextjs` + `vercel-react-best-practices` + `vercel-composition-patterns`
- API en Rust → skill `axum` + skill `postgresql`
- App React Native → skill `react-native` + `vercel-react-native-skills`
- Arquitectura hexagonal → skill `axum-hexagonal` o `fastapi-hexagonal`

Lo que cada skill hace concretamente:

**Skill `axum`** — Carga patrones de producción para Axum 0.8+. Composición de router, firmas de handlers, manejo de errores, stacks de middleware, uso de SQLx. El archivo de referencia solo tiene más de 14,000 caracteres de ejemplos de código.

**Skill `axum-hexagonal`** — Guía los límites de arquitectura hexagonal (dominio, puertos, adaptadores) con código Rust concreto. Tres archivos de referencia: ejemplos de dominio, ejemplos de adaptadores, ejemplos de bootstrap. Más de 18,000 caracteres combinados.

**`vercel-react-best-practices`** — Un skill oficial de Vercel. 45 reglas en 8 categorías. Prevención de waterfall, optimización de bundle, patrones server-side, prevención de re-renders — todo con prioridad CRITICAL. La gente que construyó el framework codificó "úsalo así."

**Los servidores MCP aceleran la implementación:**
- **Context7** — Obtiene documentación actualizada de librerías en tiempo real. Previene el uso de APIs obsoletas
- **Neon MCP** — Correr migraciones, inspeccionar schemas, optimizar queries en la misma sesión
- **D2 MCP** — Escribir diagramas de arquitectura en lenguaje D2, compilar a SVGs con tema claro/oscuro
- **Playwright MCP** — Automatización de navegador para verificación de renderizado, capturas de pantalla
- **Sentry MCP** — Los errores de producción fluyen directamente a las sesiones de debugging
- **PostHog MCP** — Consultar datos de analytics dentro del contexto del código

**Plugins LSP** — TypeScript LSP, Pyright LSP, Rust Analyzer LSP. Claude Code obtiene verificación de tipos y autocompletado en tiempo real mientras escribe. Rust Analyzer detecta errores de compilación a nivel de editor antes de que lleguen a la terminal.

Cero cambio de contexto. Una sesión de Claude Code maneja código, migraciones de DB, consulta de documentación y generación de diagramas.

## Paso 8. Quality Gate — "Sin Compromisos"

Después de la implementación, **dos agentes corren en paralelo.** Esto está hardcodeado en CLAUDE.md. Cada implementación, cada cambio, sin excepciones.

### Agente Tester

**Meta: 100% de cobertura.** Líneas, branches, funciones, statements — todos.

Proceso:
1. **Analizar** — Mapear cada archivo, función y branch en el código modificado
2. **Correr tests existentes** — Encontrar gaps de cobertura
3. **Planificar** — Listar cada línea, branch y función sin cobertura
4. **Escribir** — Crear tests para cada gap (prioridad: rutas críticas → lógica de negocio → puntos de integración → CRUD → UI → utilidades → rutas de error)
5. **Correr** — Con reporte de cobertura
6. **Iterar** — Hasta 100%. 99% no es 100%

Checklist de cobertura de branches: if/else ambos lados, switch/match cada caso + default, try/catch éxito y fallo, ternarios ambos resultados, short-circuits lógicos evaluados y saltados, null/optional presente y ausente, early returns activados y no activados, loops cero/una/muchas iteraciones.

**Regla central: este agente nunca modifica código de aplicación.** Solo escribe archivos de test. Si encuentra un problema en el código, me lo reporta.

### Agente Security Reviewer

**Mentalidad de asumir la brecha** — cada input es malicioso, cada dependencia está comprometida.

Proceso:
1. **Escaneo Rápido** — Secrets hardcodeados, exposición de .env, modo debug, patrones peligrosos
2. **Análisis de Dominio** — Basado en los cuatro checklists del skill `security-checklists`: Auth, API, Lógica de Negocio, Supply Chain
3. **Clasificación de Severidad** — Critical / High / Medium

| Patrón de Auto-Fallo | Riesgo |
|--------------------|------|
| Secrets en código o logs | Exposición de credenciales |
| Input de usuario en concatenación de query string | SQL Injection |
| Input de usuario en llamadas shell/system | Command Injection |
| Input de usuario en eval/exec | Remote Code Execution |
| Contenido crudo de usuario en output HTML | XSS |
| Request body completo pasado a model update | Mass Assignment |

**Reglas de escalación** — Lógica de pagos/financiera, cambios en sistema de autenticación, implementaciones criptográficas, integraciones sensibles con terceros, cumplimiento (GDPR, PCI-DSS) — todos escalados a mí incondicionalmente. La IA no debería tomar decisiones sola en estas áreas.

Los dos agentes corren en paralelo porque son tareas independientes. Correrlos secuencialmente desperdicia tiempo. Uno escribe tests mientras el otro escanea vulnerabilidades. ¿Ambos encuentran problemas? Se corrigen y se vuelve a correr. Hasta que todo pase.

## Paso 9. Deploy y Operaciones — "A Producción"

El código pasa el quality gate, hora de deployar. La infraestructura ya se decidió en el Paso 3.

**Pulumi** gestiona la infraestructura como código. TypeScript define GCP Cloud Run, Secret Manager, Artifact Registry, Cloudflare R2 y Neon. No hay HCL que aprender. El plugin `pulumi-authoring` guía la autoría de ComponentResource, manejo de Output y patrones de gestión de secrets.

Post-deploy:
- **Sentry MCP** — Errores de producción inspeccionados y debuggeados directamente desde sesiones de Claude Code
- **PostHog MCP** — Datos de comportamiento de usuarios consultados en contexto de código. Análisis de funnels, event tracking, A/B tests
- **Cloudflare MCP** — Gestionar Workers, KV, R2, D1

Los issues descubiertos en producción vuelven al pipeline. Incluso los bug fixes cargan skills, y tester + security-reviewer corren en paralelo después de la implementación. Sin excepciones.

## Los Documentos Se Mantienen Vivos

En mi flujo de trabajo, la documentación no es overhead — es **parte del desarrollo**. Dos reglas:

**1. Cuando los requisitos cambian, todos los documentos relacionados se actualizan.** Si un feature se elimina del PRD, el SADD, diseño de DB y specs de UX se actualizan. En el momento en que los documentos se desfasan de la realidad, la IA confía en documentos incorrectos y escribe código incorrecto.

**2. Después de unidades de desarrollo importantes, CLAUDE.md se actualiza.** Cuando aparece un nuevo módulo, la estructura de carpetas, convenciones y relaciones de dependencia se reflejan en CLAUDE.md. En la próxima sesión, Claude Code lee este archivo y arranca con comprensión completa del estado actual del proyecto.

Mira el CLAUDE.md de Mealio para ver esto en acción. La estructura Feature-Sliced Design del móvil, arquitectura dual-mode (auth/guest), comportamiento de cola de uploads, patrón de trigger-and-poll de análisis con IA, gestión de keys de TanStack Query, gotchas de Jest — todo documentado. Un documento vivo de 189 líneas.

El plugin `claude-md-management` ayuda. `/revise-claude-md` refleja los aprendizajes de la sesión actual en CLAUDE.md.

## Una Persona, Muchos Roles

Con este pipeline, una persona se mueve entre múltiples roles:

| Paso | Rol | Skill/Agente |
|------|------|-------------|
| 1. Product Brief | PM | skill `product-brief` |
| 2. PRD | Product Manager | skill `prd-craft` |
| 3. Arquitectura | Software Architect | skill `software-architect` |
| 4. Base de Datos | DBA | skills `database-design` + `postgresql` |
| 5. Diseño UX | UX Designer | agente `ux-designer` |
| 6. Prototipo UI | UI Engineer | agente `ui-engineer` |
| 7. Implementación | Developer | `axum`, `nextjs`, `react-native`, etc. |
| 8. Quality Gate | QA + Security Engineer | agentes `tester` + `security-reviewer` |
| 9. Deploy | DevOps | Pulumi + Cloudflare + GCP |

Antes, cada uno de estos roles era una persona diferente. Ahora, el skill correcto se carga para cada rol, aplicando los frameworks y constraints de ese rol. No necesito ser experto en cada dominio — el pensamiento del experto está codificado en el skill.

## Limitaciones

Seré honesto. Este sistema tiene límites.

**El gusto de dominio no se puede reemplazar.** Los skills guían "define el problema con evidencia" pero cuál problema realmente duele, dónde está el dinero, qué quieren realmente los usuarios — ese es mi juicio. La IA escribe el Product Brief, pero los insights que lo alimentan vienen de observar usuarios.

**El gusto de UX es igual.** El diseño basado en principios cognitivos previene malas decisiones, pero crear experiencias genuinamente naturales y placenteras al final viene de usar el producto tú mismo y sentirlo.

**La calidad de los skills es todo.** Skills mal escritos propagan malos patrones en cada proyecto. Los skills necesitan revisión y actualizaciones regulares.

Pero estos límites no son bugs del sistema — son el diseño. La IA provee frameworks, yo proveo los ingredientes. División del trabajo.

## El Efecto Compuesto

El valor real de este sistema aparece con el tiempo.

Los skills y agentes creados para el primer proyecto se llevan directamente al segundo. Los patrones de CLAUDE.md mejorados en el segundo proyecto se aplican al tercero. Nuevos patrones descubiertos durante las revisiones de seguridad se agregan a los skills y se verifican automáticamente en cada proyecto futuro.

## Lo Que Realmente Significa el Vibe Coding

El vibe coding no es extraer código de la IA. Es **dirigir a la IA a través de todo el proceso de construir un producto.**

Partiendo de una idea, definir el problema (Product Brief), especificar requisitos (PRD), diseñar la estructura (SADD), modelar datos (DB Design), mapear la experiencia de usuario (UX), crear scaffolds visuales (UI), implementar código (Skills) y verificar calidad (Tester + Security). En cada paso, se cargan frameworks de nivel experto, y yo establezco la dirección y tomo las decisiones.

No un mecanógrafo. Un director.

En un post anterior dije "vive como constructor." Este pipeline es la infraestructura del constructor. Solo necesitas la idea — el sistema construye el resto.

Las vibes son buenas. El pipeline te cubre las espaldas.
